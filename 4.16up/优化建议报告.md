# 智能体舱（Agent-01）优化建议报告

**审查日期**：2026-04-16  
**审查范围**：性能、用户体验、API 设计、架构质量  

---

## 一、性能优化（PERFORMANCE）

### P1：首页组件过大，应拆分
| 项目 | 内容 |
|------|------|
| **文件** | `app/page.tsx`（1166 行） |
| **问题** | 单个文件包含 HomePage、AgentCard、WorkflowStepButton、UserAgentCard 等多个组件，解析慢、维护难、无法 tree-shake。 |
| **建议** | 将 AgentCard、WorkflowStepButton、UserAgentCard 抽离为独立文件。 |

---

### P2：首页多个 localStorage 读取未合并
| 项目 | 内容 |
|------|------|
| **文件** | `app/page.tsx` 约第 133-143 行 |
| **问题** | 三个独立的 `useEffect` 分别读 `wf_collapsed`、`agent_collapsed`、`my_agents_collapsed`，触发三次 re-render。 |
| **建议** | 合并为一个 `useEffect`，一次性读取所有 localStorage 状态。 |

---

### P3：分类切换无缓存，重复请求 API
| 项目 | 内容 |
|------|------|
| **文件** | `app/page.tsx` 约第 329-346 行 |
| **问题** | 每次切换分类都发起 API 请求，切回同一分类会重复请求。 |
| **建议** | 用 Map 缓存已请求过的分类结果，避免重复调用。 |

---

### P4：分类筛选使用 O(n*m) 遍历
| 项目 | 内容 |
|------|------|
| **文件** | `app/page.tsx` 约第 486 行 |
| **问题** | `categories.filter(c => group.categoryIds.includes(c.id))` 是 O(n*m)。 |
| **建议** | 构建一次 `Map<id, category>` 查找表，将复杂度降为 O(n)。 |

---

### P5：语音识别轮询过于频繁
| 项目 | 内容 |
|------|------|
| **文件** | `app/agents/[id]/page.tsx` 约第 323-336 行 |
| **问题** | 语音识别结果以 500ms 间隔轮询，最多 300 次（5 分钟），最差情况发 300+ 次请求。 |
| **建议** | 改用指数退避策略（500ms → 1s → 2s → 4s，上限 30s），或改为 WebSocket/SSE 推送。 |

---

### P6：聊天消息列表无虚拟化，大量消息时卡顿
| 项目 | 内容 |
|------|------|
| **文件** | `app/agents/[id]/page.tsx` 约第 454-470 行 |
| **问题** | 每次新消息块到达时整个消息列表 re-render，消息多时 UI 明显卡顿。 |
| **建议** | 用 `React.memo` 包裹单条消息组件，或引入 `react-window` 做列表虚拟化。 |

---

### P7：Agent 分类展示逻辑涉及 5 次数据库查询
| 项目 | 内容 |
|------|------|
| **文件** | `app/api/agents/route.ts` 约第 113-208 行 |
| **问题** | 单次分类展示需查 workflow_categories、workflows、workflow_steps、category_agent_display、agent_categories 共 5 张表。 |
| **建议** | 将展示逻辑封装为数据库视图（View）或存储过程，减少网络往返。 |

---

### P8：管理后台用户列表 5000 行内存排序
| 项目 | 内容 |
|------|------|
| **文件** | `app/api/admin/users/route.ts` 约第 57-80 行 |
| **问题** | 查询最多 5000 行用户数据到 Node.js 内存中做排序，然后再分页返回。 |
| **建议** | 将排序逻辑下推到数据库层（`.order()`），让 DB 做排序和分页。 |

---

### P9：用户分组成员数 N+1 查询
| 项目 | 内容 |
|------|------|
| **文件** | `app/api/admin/user-groups/route.ts` 约第 17-21 行 |
| **问题** | 对每个分组单独查一次成员数量，100 个分组就是 100 次查询。 |
| **建议** | 改为单次聚合查询，按 `group_id` 分组统计。 |

---

### P10：管理后台代理权限弹窗每次都重新请求数据
| 项目 | 内容 |
|------|------|
| **文件** | `app/admin/agents/page.tsx` 约第 119-129 行 |
| **问题** | 每次打开权限弹窗都重新请求部门、团队、分组数据，关闭再打开又请求一次。 |
| **建议** | 页面加载时缓存一次部门/团队/分组数据，弹窗直接使用缓存。 |

---

### P11：管理后台 Agent 列表缺少 useMemo
| 项目 | 内容 |
|------|------|
| **文件** | `app/admin/agents/page.tsx` 约第 287-293 行 |
| **问题** | `filteredAgents` 在每次 render 时重新计算，即使依赖数据未变化。 |
| **建议** | 用 `useMemo` 包裹过滤逻辑。 |

---

### P12：批量分类展示配置逐条发请求
| 项目 | 内容 |
|------|------|
| **文件** | `app/admin/agents/page.tsx` 约第 550-553 行 |
| **问题** | "一键全选" 功能对每个分类单独发 PATCH 请求，50 个分类就是 50 次请求。 |
| **建议** | 新增批量更新接口，一次请求处理多条记录。 |

---

### P13：分析统计接口每次请求都做重量级聚合
| 项目 | 内容 |
|------|------|
| **文件** | `app/api/admin/analytics/route.ts` 约第 33-107 行 |
| **问题** | 每次请求都扫全表 logs 做内存聚合，数据量大时非常慢。 |
| **建议** | 创建物化视图（Materialized View）预聚合统计数据，定时刷新。 |

## 三、API 设计与后端架构优化

### A1：列表接口响应格式不统一
| 项目 | 内容 |
|------|------|
| **文件** | 多个 admin 路由 |
| **问题** | `users` 路由返回 `{ users: [], total, page, pageSize }`，`agents`/`workflows`/`logs` 等返回裸数组。 |
| **建议** | 统一为 `{ data: T[], pagination: { page, pageSize, total } }` 信封格式。 |

---

### A2：多数列表接口缺少分页
| 项目 | 内容 |
|------|------|
| **文件** | `agents`、`workflows`、`categories`、`departments`、`teams`、`notices`、`resource-permissions`、`user-groups` 等路由 |
| **问题** | 硬编码 limit 或无 limit，无法获取超出限制的数据且无告知。 |
| **建议** | 所有列表接口统一支持 `page` + `pageSize` 参数。 |

---

### A3：错误响应格式不统一
| 项目 | 内容 |
|------|------|
| **问题** | 有的返回 `{ error: "msg" }`，有的返回空数组 `[]`，有的暴露数据库原始错误信息。缺少错误码字段。 |
| **建议** | 统一为 `{ error: { message, code, status } }` 格式，添加错误码（如 `VALIDATION_ERROR`、`PERMISSION_DENIED`）供前端程序化处理。 |

---

### A4：缺少请求日志和链路追踪
| 项目 | 内容 |
|------|------|
| **文件** | 所有 API 路由 |
| **问题** | 只有零散的 `console.error`，无请求 ID、无耗时记录、无结构化日志。 |
| **建议** | 创建 `lib/request-logger.ts`，每个请求生成 `requestId`，记录 method、path、userId、duration。 |

---

### A5：20+ 个路由重复鉴权代码
| 项目 | 内容 |
|------|------|
| **文件** | 所有 admin 路由 |
| **问题** | 每个路由都有 `const admin = await getActiveAdmin(); if (!admin) return ...` 重复代码。 |
| **建议** | 抽取 `requireAdmin()` / `requireUser()` 辅助函数，内部抛出统一错误，减少重复。 |

---

### A6：缺少健康检查接口
| 项目 | 内容 |
|------|------|
| **问题** | 无法在不鉴权的情况下检测 API 和数据库是否正常。 |
| **建议** | 新建 `app/api/health/route.ts`，返回 `{ status, timestamp, checks: { database } }`。 |

---

### A7：硬编码配置项散落各处
| 项目 | 内容 |
|------|------|
| **文件** | `lib/rate-limit.ts`、`lib/auth.ts`、多个路由 |
| **问题** | Token TTL（30 天）、登录失败上限（5 次）、锁定时间（15 分钟）、分页上限（100）等全部硬编码，修改需重新部署。 |
| **建议** | 统一到 `lib/config.ts`，从环境变量读取，提供合理默认值。 |

---

### A8：缺少数据库索引
| 项目 | 内容 |
|------|------|
| **文件** | `supabase/schema.sql` |
| **问题** | 以下高频查询字段缺少索引：`users(tenant_code)`、`users(role)`、`agents(enabled)`、`conversations(user_id)`、`agent_categories(agent_id)`、`workflow_categories(workflow_id)`、`resource_permissions(resource_type, resource_id)`。 |
| **建议** | 在 schema.sql 和新 migration 文件中补充索引。 |

---

### A9：请求体缺少大小限制和 Schema 校验
| 项目 | 内容 |
|------|------|
| **文件** | 所有 POST/PATCH 路由 |
| **问题** | 直接 `await req.json()` 无大小限制，无字段格式校验（如 `model_params` 可以传任意深度嵌套对象）。 |
| **建议** | 引入 `zod` 做 Schema 校验，必要时在 `next.config.ts` 中配置请求体大小限制。 |

---

### A10：适配器无重试机制
| 项目 | 内容 |
|------|------|
| **文件** | `lib/adapters/index.ts` 各适配器错误处理 |
| **问题** | API 请求失败立即 throw，不做任何重试。临时网络抖动直接失败。 |
| **建议** | 对非流式请求（如 Qingyan 认证）增加指数退避重试（1s → 2s → 4s，最多 3 次）。 |
