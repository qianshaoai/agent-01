# 体验版模块极简开发方案

## 1. 目标

为了节前先快速做出一版体验版，本阶段不做完整的体验用户后台、智能体后台配置和复杂会话管理，只实现一个最小可用版本。

本期目标：

```text
1. 做一个体验版界面，展示代码中配置的智能体，并支持站内聊天
2. 只开放一个预置体验账号进入体验版
3. 智能体配置由开发人员写在代码配置中，敏感值放到环境变量
4. 体验账号不能访问或调用正式平台功能
```

预置体验账号：

```text
账号：18700000008
密码：12345678
```

注意：该账号只是本期预置账号，权限判断不建议硬编码手机号，统一使用 `userType = trial` 判断。

## 2. 本期不做内容

为了保证开发速度，本期明确不做以下内容：

- 不改注册页面。
- 不做体验用户注册。
- 不新增体验版用户管理后台。
- 不新增体验版智能体管理后台。
- 不做后台动态配置智能体。
- 不做智能体上下架管理。
- 不调整企业用户、个人用户体系。
- 不让前端保存 API Token 和 Bot ID。
- 不跳转第三方智能体网页。

## 3. 动工前确认点

### 3.1 使用 userType 判断体验账号

不要用手机号判断体验权限。

建议：

```text
1. 在数据库中将 18700000008 账号的 user_type 设置为 trial
2. JWT 中继续携带 userType
3. 后端接口统一通过 payload.userType === "trial" 判断体验账号
```

原因：

- 手机号可能变更。
- `user_type` 是稳定权限标识。
- 后续增加更多体验账号时不用改代码。

动工前确认：

```text
1. 检查 users.user_type 字段的取值约束（CHECK 约束或 enum 类型）：
   - 如果约束已包含 trial：直接 INSERT 体验账号即可
   - 如果约束未包含 trial：必须先做一次 migration 扩展约束，否则插入会失败

2. 同步扩展 lib/auth.ts 中 UserPayload.userType 的 TypeScript 类型：
   现状：userType: "personal" | "organization"
   改为：userType: "personal" | "organization" | "trial"
   否则后端 payload.userType === "trial" 的判断会被 TS 报"无重叠类型"。
```

### 3.2 体验账号 tenant_code

体验账号建议设置：

```text
tenant_code = PERSONAL
```

原因：

- 现有登录逻辑中，非 PERSONAL 用户会校验租户是否启用、是否到期。
- 体验账号不属于正式企业租户，设置为 PERSONAL 最省改动。

### 3.3 API Token 和 Bot ID 使用环境变量

不要把 API Token 和 Bot ID 明文提交到源码。

建议放到 `.env.local`：

```text
TRIAL_AGENT_001_BOT_ID=xxx
TRIAL_AGENT_001_API_TOKEN=xxx
```

源码中只写环境变量读取：

```ts
const trialAgents = [
  {
    id: "agent_001",
    name: "体验版智能体",
    description: "用于体验智能体问答能力",
    botId: process.env.TRIAL_AGENT_001_BOT_ID,
    apiToken: process.env.TRIAL_AGENT_001_API_TOKEN,
  },
]
```

### 3.4 聊天回复使用 SSE 流式返回

建议体验版聊天直接复用现有 `streamChat` 的 SSE 流式能力。

原因：

- 前端体验更好。
- 不需要后端把流聚合成一次性结果。
- 和现有聊天能力保持一致，开发量基本持平。

接口返回不建议做成：

```json
{
  "reply": "一次性回复内容"
}
```

建议采用：

```text
请求：
  POST /api/trial/chat
  Content-Type: application/json
  Body: { "agent_id": "agent_001", "message": "你好" }

响应：
  Content-Type: text/event-stream
  SSE 持续推送回复 token
```

前端用 `fetch` POST JSON，读取 `response.body` 的 ReadableStream 持续渲染。

### 3.5 正式接口必须做后端守卫

体验账号不能访问正式功能，不能只靠前端跳转。

必须同时做：

```text
1. 前端路由拦截
2. 后端接口拦截
```

建议抽一个 helper：

```ts
function requireFullUser(payload) {
  if (payload.userType === "trial") {
    return apiError("体验账号不可调用此接口", "FORBIDDEN", 403)
  }
}
```

在正式业务接口入口调用，例如：

```text
/api/agents/*
/api/conversations/*
/api/files/*
其他会写入或读取正式业务数据的接口
```

### 3.6 多轮上下文边界

本期建议不做多轮会话管理。

最简方案：

```text
每次发送消息都新建会话，不传 conversation_id
```

影响：

- 实现简单。
- 不依赖数据库和历史会话存储。
- 模型不会记住上一轮上下文。

如果业务方明确需要短期记忆，再做增强：

```text
前端用 sessionStorage 维护 conversation_id
conversation_id 可按 trial_${userId}_${botId} 生成
```

### 3.7 体验版聊天不跳转外部网页

体验用户始终在本平台 `/trial` 页面内完成聊天。

调用链路：

```text
前端 /trial 页面
→ 调用本平台后端 /api/trial/chat
→ 后端读取环境变量中的 Bot ID 和 API Token
→ 后端代理调用智能体 API
→ 前端在当前页面展示回复
```

## 4. 登录逻辑

登录流程：

```text
1. 用户在原登录页输入账号和密码
2. 后端登录接口校验账号密码
3. 登录成功后 JWT 中携带 userType，并在响应体中返回 userType
4. 前端读取响应体的 userType
5. 如果 userType = trial，前端跳转 /trial
6. 如果 userType != trial，保持现有正式平台跳转逻辑
```

需要修改的现有代码：

```text
app/api/auth/login/route.ts
  在登录成功的响应体中新增 userType 字段：
  { ok, firstLogin, tenantName, userType }
  否则前端拿不到 userType，无法判跳。

app/login/page.tsx
  登录成功后读取 data.userType；
  若 data.userType === "trial"，router.push("/trial")；
  否则保留现有跳转逻辑。
```

体验账号与"首次登录改密"的优先级（必处理）：

```text
现状（app/login/page.tsx:52）：
  登录成功后先判 data.firstLogin → 弹首次改密弹窗
  否则才走 router.push("/")

体验账号必须绕开首次改密弹窗，不然进不了 /trial。两选一：

方案 A（推荐，DB 侧消除歧义）：
  体验账号在 DB 中预置 first_login = false
  既不弹改密弹窗，也不影响别的逻辑

方案 B（前端判断侧消除歧义）：
  在 app/login/page.tsx 中把 trial 分支放在 firstLogin 判断之前：
    if (data.userType === "trial") {
      router.push("/trial");
      router.refresh();
      return;
    }
    if (data.firstLogin) { setShowFirstLogin(true); return; }
    ...
```

二选一即可，建议方案 A——一处 SQL 解决，前端不用改优先级。

前端跳转建议：

```text
体验账号登录成功：
跳转 /trial

其他账号登录成功：
保持现有跳转逻辑
```

说明：

- 不改注册流程。
- 不影响现有正式用户登录。
- 不影响企业用户、个人用户原有逻辑。
- `18700000008 / 12345678` 需要提前在数据库中创建，并设置 `user_type = trial`、`tenant_code = PERSONAL`。

## 5. 体验版页面

体验版页面以前端现有页面风格为模板开发，不重新设计复杂界面。

页面路径建议：

```text
/trial
```

页面内容：

```text
顶部：体验版标题
中间：智能体展示区域
底部：选中智能体后的聊天输入框和发送按钮
```

智能体展示内容包括：

```text
智能体名称
智能体简介
智能体图标或默认头像
```

展示范围来自后端代码中写好的 `trialAgents` 配置。

如果代码中配置 1 个智能体，则页面展示 1 个智能体。

如果代码中配置多个智能体，则页面展示多个智能体，用户可以选择其中一个进入聊天。

## 6. 智能体配置方式

本期智能体列表写在后端代码中，敏感值放环境变量。

由业务方提供：

```text
API Token
Bot ID
智能体名称
智能体简介
```

后端代码中维护智能体配置列表：

```ts
const trialAgents = [
  {
    id: "agent_001",
    platform: "coze",        // 必填，决定走哪个适配器
    name: "体验版智能体",
    description: "用于体验智能体问答能力",
    avatar: "",
    apiEndpoint: "https://api.coze.cn/v3/chat",
    botId: process.env.TRIAL_AGENT_001_BOT_ID,        // 平台侧的 bot/assistant 标识
    apiToken: process.env.TRIAL_AGENT_001_API_TOKEN,  // 平台侧的鉴权 token
  },
]
```

`platform` 取值（必须与 `lib/adapters/index.ts` 现有适配器对齐）：

```text
coze     → cozeStream
dify     → difyStream
yuanqi   → yuanqiStream
qingyan  → qingyanStream
其他任意值 → openaiCompatibleStream（默认分支）
```

不要写 `zhipu`、`glm` 等不存在的取值，会被默认分支当 OpenAI 兼容协议处理。

`trialAgents` 字段到 `AdapterConfig` 的映射（调 `streamChat` 时构造）：

```ts
const cfg: AdapterConfig = {
  platform: trialAgent.platform,
  apiEndpoint: trialAgent.apiEndpoint,
  apiKey: trialAgent.apiToken,             // apiToken → apiKey
  agentCode: trialAgent.botId ?? "",       // botId 兜底放 agentCode
  modelParams: {
    bot_id: trialAgent.botId,              // Coze 用 modelParams.bot_id
    // 不同平台还可能用 assistant_id / app_id 等键，按平台补
  },
};
```

注意：

- API Token 不能写在前端。
- Bot ID 不能写在前端。
- API Token 和 Bot ID 不要明文提交到 Git。
- 前端只展示 `id`、`name`、`description`、`avatar`。
- 真实调用由后端完成。

环境变量缺失的失败处理（必须明确）：

```text
约束：禁止带着空 token / 空 botId 去调第三方平台
原因：
  - 第三方平台对空鉴权的响应不一致：可能返回 401，也可能返回看似成功但是无效的回包
  - 排查成本高、日志噪音大、还可能误扣额度

实现：
  GET /api/trial/agents
    在序列化前过滤掉 botId 或 apiToken 为空的条目；
    若全部条目都缺失，返回 500 + error: "trial agents not configured"

  POST /api/trial/chat
    定位到 trialAgent 后立刻校验 botId、apiToken 是否非空；
    任一缺失：直接返回 500 + error: "trial agent <id> missing botId/apiToken"
    不向第三方发起请求

可选增强：
  在应用启动时（或 lib/trial-agents.ts 模块加载时）打印一条 warn 日志，
  列出哪些 trial agent 因为环境变量缺失被禁用，便于运维快速发现。
```

## 7. 后端接口

本期只需要两个体验版接口。

### 7.1 获取体验版智能体列表

```text
GET /api/trial/agents
```

返回示例：

```json
[
  {
    "id": "agent_001",
    "name": "体验版智能体",
    "description": "用于体验智能体问答能力",
    "avatar": ""
  }
]
```

说明：

- 只允许 `userType = trial` 的用户访问。
- 不返回 API Token。
- 不返回 Bot ID。
- 只返回前端展示需要的信息。

### 7.2 发送聊天消息

```text
POST /api/trial/chat
```

请求参数：

```json
{
  "agent_id": "agent_001",
  "message": "你好"
}
```

处理流程：

```text
1. 校验当前登录用户 payload.userType === "trial"
2. 根据 agent_id 查找后端代码中的智能体配置
3. 读取该智能体对应的 Bot ID 和 API Token
4. 后端代理调用智能体接口
5. 通过 SSE 流式返回智能体回复
```

响应方式：

```text
text/event-stream
```

说明：

- 不聚合成一次性 `reply` 返回。
- 直接复用现有流式聊天体验。
- 体验版聊天在当前页面完成，不跳转外部网页。

## 8. 权限隔离

虽然本期是极简版，但仍然需要保证体验账号不能影响正式功能。

### 8.1 路由限制（必须 server-side 拦截）

体验账号登录后，只允许访问：

```text
/trial
```

实现方式（修改 `middleware.ts`）：

```text
1. 把 /trial 加入 USER_PROTECTED 列表
   现状（middleware.ts:5）：
     USER_PROTECTED = ["/", "/agents", "/settings", "/user-agents"]
   改为：
     USER_PROTECTED = ["/", "/agents", "/settings", "/user-agents", "/trial"]
   作用：未登录访问 /trial 也会被中间件跳转到 /login，与正式页面保持一致。

2. 在 USER_PROTECTED 通过登录校验后追加 trial 分流：
   if (payload.userType === "trial" && !pathname.startsWith("/trial")) {
     return NextResponse.redirect(new URL("/trial", req.url));
   }
```

注意：

- 不要依赖前端 useEffect 重定向。前端跳转在 React 水合后才执行，会出现"正式页面闪一下再跳走"的体验问题，并可能在重定向前就发出正式接口请求触发后端 403 噪音。
- middleware 在请求最早期介入，零闪烁、零无效请求。
- /trial 必须先要求 user JWT，否则未登录也能直接访问体验页面。

### 8.2 后端接口限制

后端正式业务接口需要拦截：

```text
如果 payload.userType === "trial"：
    禁止访问正式业务接口
```

建议优先保护这些接口（持 user JWT 就能调的正式业务接口）：

```text
/api/agents/*
/api/conversations/*
/api/files/*
/api/users/*
/api/workflows/*
其他持 user JWT 即可访问的正式业务读写接口
```

不需要在此清单内的：

```text
/api/admin/*
原因：admin 接口校验 ai_portal_admin_token（admin JWT），
trial 用户持有的是 ai_portal_token（user JWT），
天然通不过 admin 校验，不需要在 trial 守卫里再列一遍。
```

推荐做法：

```text
在 lib/auth.ts 中新增两个 helper：

  requireFullUser(payload)
    用于正式业务接口入口
    payload.userType === "trial" 时返回 403 FORBIDDEN

  requireTrialUser(payload)
    用于体验版接口入口（/api/trial/*）
    payload.userType !== "trial" 时返回 403 FORBIDDEN
```

调用规则（避免互相误拦）：

```text
/api/trial/*  → 调用 requireTrialUser，禁止调用 requireFullUser
其他正式接口 → 调用 requireFullUser，禁止调用 requireTrialUser
```

这样即使体验用户手动请求正式接口，也不会影响正式业务数据；正式用户也不会误进体验接口产生混淆日志。

## 9. 数据存储

本期可以先不新增体验版智能体配置表。

建议只做最小记录，全部复用现有 `logs` 表，不新建表、不改表结构。

现有 `logs` 表字段（`supabase/schema.sql`）：

```text
user_phone    TEXT
tenant_code   TEXT
agent_code    TEXT
agent_name    TEXT
action        TEXT NOT NULL    -- 现有取值: login | chat | upload | speech
status        TEXT NOT NULL CHECK (status IN ('success', 'error'))
duration_ms   INTEGER
error_msg     TEXT
created_at    TIMESTAMPTZ
```

体验版写入约定：

```text
体验账号登录：
  action      = "trial_login"
  user_phone  = 体验账号手机号
  tenant_code = "PERSONAL"
  status      = "success" / "error"

体验版聊天调用：
  action      = "trial_chat"
  user_phone  = 体验账号手机号
  tenant_code = "PERSONAL"
  agent_code  = trialAgent.id（如 "agent_001"）
  agent_name  = trialAgent.name
  duration_ms = 后端耗时
  status      = "success" / "error"
  error_msg   = 失败原因（仅 error 时）
```

不写入 message 内容、不写入 user_id（logs 表没有 user_id 字段，定位用 user_phone）。

如果时间紧，聊天记录可以先只写日志，不做页面历史会话。

本期不做：

- 不做聊天历史列表。
- 不做智能体配置表。
- 不做智能体管理后台。
- 默认不做多轮会话管理。

多轮上下文策略：

```text
默认每次请求新会话，不传 conversation_id。
如果业务方要求短期记忆，再用 sessionStorage 维护 conversation_id。
```

## 10. 开发步骤

建议按以下顺序开发：

```text
 1. 确认 users.user_type 字段约束是否包含 trial；不含则先做 migration
 2. 扩展 lib/auth.ts 中 UserPayload.userType 类型
    "personal" | "organization" → "personal" | "organization" | "trial"
 3. 创建或确认体验账号 18700000008 / 12345678
 4. 设置该账号 user_type = trial、tenant_code = PERSONAL
 5. 修改 app/api/auth/login/route.ts 响应体，新增返回 userType
 6. 修改 app/login/page.tsx，登录成功后 data.userType === "trial" → /trial
    并按 §4 处理 firstLogin 优先级（推荐 DB 侧把体验账号 first_login 设为 false，
    或在前端把 trial 跳转分支放在 firstLogin 判断之前）
 7. 修改 middleware.ts：
      - USER_PROTECTED 增加 "/trial"
      - 在 USER_PROTECTED 通过登录校验后增加 trial 分流（非 /trial 路径 → /trial）
 8. 在 lib/auth.ts 中新增 requireFullUser 和 requireTrialUser 两个 helper
 9. 在持 user JWT 的正式业务接口入口统一调用 requireFullUser
10. 配置 TRIAL_AGENT_001_BOT_ID 和 TRIAL_AGENT_001_API_TOKEN 到 .env.local
11. 新增 lib/trial-agents.ts，包含 trialAgents 配置（含 platform、apiEndpoint）
12. 新增 GET /api/trial/agents 接口（入口调用 requireTrialUser）
13. 新增 POST /api/trial/chat SSE 接口（入口调用 requireTrialUser，
      复用 lib/adapters/index.ts 的 streamChat，注意字段映射）
14. 新增 app/trial/page.tsx，包含智能体展示区和流式聊天区
15. 在 logs 表中按约定字段记录 trial_login 和 trial_chat
16. 联调登录、展示、SSE 聊天、权限隔离
```

## 11. 验收标准

本期完成后需要满足：

- 使用 `18700000008 / 12345678` 可以登录。
- 该账号在数据库中为 `user_type = trial`。
- 该账号建议设置 `tenant_code = PERSONAL`。
- 该账号登录后自动进入体验版页面。
- 体验版页面展示代码中配置的智能体。
- 体验版页面可以在站内发送消息并流式收到智能体回复。
- 聊天过程不跳转第三方网页。
- 前端看不到 API Token 和 Bot ID。
- API Token 和 Bot ID 从环境变量读取，不明文提交到 Git。
- 体验账号不能进入正式平台功能。
- 体验账号不能调用正式业务接口。
- 其他正式账号登录和使用不受影响。

- 支持短期记忆或多轮会话管理。

## 13. 总结

本期采用极简但不埋安全坑的方案：预置一个体验账号，通过 `userType = trial` 识别权限；智能体列表写在代码配置中，Bot ID 和 API Token 放环境变量；体验版聊天在本平台内通过后端代理和 SSE 流式完成。

这样可以最大限度减少对现有平台的改动，同时保证节前交付一个可登录、可展示、可聊天、不会影响正式功能的体验版。
