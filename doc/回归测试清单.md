### 模块 2 · 多租户 / 配额

📂 *改了这些才需要测：* `app/api/agents/[id]/chat`、`supabase/rpc.sql`、tenants 相关接口

- [ ] 组织用户聊天 → 扣该组织配额（`quota_used` 增加）
- [ ] 个人用户聊天 → **不扣**配额
- [ ] 配额耗尽 → 拦截、提示余额不足

## ⑤ 高风险点 · 重点中的重点

历史上「改一处坏一片」或隐蔽误判的链路，**每次回归都扫一遍**：

1. **chat 路由配置解析链** —— `provider_id` 优先级、provider 异常时阻断不回退旧 key。改 chat / adapters / model-providers 必复测。
2. **配额扣费链** —— 加权扣费、aborted 不扣、个人用户不扣、余额边界不超扣 / 不漏扣。
3. **「fetch failed 被当成 NOT_FOUND」** —— 项目里多处 `.single() + !data → 报"不存在"`；跨境 Supabase 抖动时会把网络错误误判成"数据不存在"。改这类查询时，确认有 `error` 检查。
4. **加密 key** —— rekey / trial 脚本 / fallback 解密，任一处不对都会让「配了 key 却聊不通」。
5. **React 严格模式 useEffect 双调** —— dev 期列表请求会双发，靠 retry 兜底；dev 偶发红条属已知、prod 不受影响，但别把 retry 逻辑改没了。
6. **智能体发布默认 `enabled=false`** —— 发布链路改动后确认这条还在，避免「发布即对外」。
7. **分页上限 `MAX_PAGE_SIZE=100`** —— 列表页在前端分组 / 排序时，超过 100 条会「只看到当前页」。
8. **多分类 `categoryIds` vs 遗留 `category_id`** —— 智能体 / 工作流按分类筛选时用对字段，否则多分类对象会被筛漏。
