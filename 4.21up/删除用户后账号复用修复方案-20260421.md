# 4.21 修复方案：删除用户后允许原用户名与手机号重新注册

**日期**：2026-04-21  
**优先级**：P0  
**适用范围**：`agent-01` 当前代码库  
**目标执行者**：Claude / Codex / 任何接手实现的工程代理

---

## 1. 问题定义

当前系统中，管理员在后台执行“删除用户”后，前台列表里看不到该用户，但注册同一个用户名或同一个手机号时，系统仍会提示“已存在”。

这不是前端缓存问题，也不是列表刷新问题，而是后端语义不一致：

1. 管理员删除用户时，并没有物理删除数据库记录，只是把 `users.status` 更新为 `deleted`
2. 用户管理列表默认过滤 `status = 'deleted'` 的记录，所以看起来像“已经删掉了”
3. 注册接口对“用户名是否已占用”的校验，仍然把这些 `deleted` 记录算进去
4. 数据库层仍保留唯一约束，因此即使业务代码放行，若旧值未释放，插入也会冲突

---

## 2. 本次目标

本次按“方案 A”实施，目标只有一个：

`管理员删除用户后，原用户名和原手机号都允许被重新注册。`

同时必须满足：

1. 继续保留软删除能力，不改成物理删除
2. 历史数据仍然保留，便于审计和追踪
3. 删除后的记录不再占用原 `username`
4. 删除后的记录不再占用原 `phone`
5. 新注册用户可以复用被删除用户原本的用户名和手机号

---

## 3. 总体策略

不要只改注册校验。  
正确做法是：

1. 删除用户时，除了把状态设为 `deleted`，还要把 `username` 和 `phone` 改写成“墓碑值”
2. 注册时，只把“有效用户”视为占用者
3. 补一份历史数据修复 SQL，把数据库里已经软删除但仍占用原账号信息的旧数据清理掉

一句话概括：

`删除时释放账号字段，注册时忽略已删除用户，数据库层保持唯一性可成立。`

---

## 4. 明确边界

本次只允许处理“删除用户后账号复用”这一件事。

可以做的事：

1. 修改管理员删除用户接口
2. 修改注册接口的用户名/手机号判重逻辑
3. 新增一份 SQL 修复脚本，用于处理历史 `deleted` 数据
4. 补充必要的注释，解释“为什么要改写为墓碑值”

不能做的事：

1. 不能把软删除改成物理删除
2. 不能顺手重构用户系统
3. 不能顺手调整登录、权限、组织、日志、会话等无关模块
4. 不能顺手修乱码、样式、lint 历史问题
5. 不能自行扩大需求到“注销账号”“组织删除”“数据归档”

如果实现过程中发现其他问题，也只允许记录为备注，不允许顺手改。

## 5. 强约束

下面这些约束是硬性要求，不允许自由发挥。

### 5.1 删除语义约束

管理员删除用户时，仍然只能是软删除：

1. 必须保留 `status = 'deleted'`
2. 不允许执行 `delete from users`
3. 不允许清空整条用户记录

### 5.2 字段释放约束

删除时必须同步释放以下字段：

1. `username`
2. `phone`

推荐墓碑值规则如下：

1. `username = deleted_<userId>`
2. `phone = del_<userId>`

要求：

1. 墓碑值必须稳定且唯一
2. 墓碑值必须不再等于用户原来的用户名和手机号
3. 不依赖当前时间戳生成，避免不可预测性

### 5.3 nickname 处理约束

允许一并改写 `nickname`，但不是必须。

如果改写，统一写成：

`已删除用户`

不要保留原用户名或原手机号到 `nickname` 里，避免后台详情页继续暴露原账号信息。

### 5.4 注册判重约束

注册接口中：

1. 手机号判重只能检查有效用户：`active`、`disabled`
2. 用户名判重也只能检查有效用户：`active`、`disabled`
3. `deleted` 用户不能继续占用用户名或手机号
4. `cancelled` 用户本次不要改语义，保持现有行为

这里最重要的一条是：

`不要把 deleted 和 cancelled 混为一谈。`

管理员删除 = `deleted`  
用户主动注销 = `cancelled`

本次只处理 `deleted`。

### 5.5 数据库约束兼容性

不允许通过“删除唯一索引”来规避问题。  
当前唯一性约束应继续保留，方案必须与现有唯一约束兼容。

也就是说：

1. 不能删除 `username` 的唯一约束
2. 不能删除 `phone + tenant_code` 的唯一约束
3. 必须通过“删除时改写字段值”来释放占用

## 7. 具体实施要求

### 7.1 管理员删除接口

目标文件：

`app/api/admin/users/[id]/route.ts`

当前问题：

删除时仅执行：

```ts
update({ status: "deleted" })
```

必须改成一次更新完成以下内容：

1. `status: "deleted"`
2. `username: "deleted_<userId>"`
3. `phone: "del_<userId>"`
4. 可选：`nickname: "已删除用户"`

要求：

1. 一次 `update` 完成，不要拆成多次更新
2. 不要新增异步串行步骤
3. 不要把原手机号或原用户名拼进墓碑值里
4. 不要把 `tenant_code` 改掉

原因：

1. 一次更新能减少部分成功、部分失败的风险
2. 不保留原手机号可降低泄露风险
3. 保留 `tenant_code` 可减少对其他关联逻辑的影响

### 7.2 注册接口

目标文件：

`app/api/auth/register/route.ts`

必须调整两处判重逻辑：

1. 手机号判重逻辑保持只查有效状态
2. 用户名判重逻辑补上只查有效状态

即用户名判重不能再是：

```ts
.eq("username", normalizedUsername)
```

而要与手机号判重语义一致，只统计有效用户。

要求：

1. 只补充状态过滤，不重写整段注册逻辑
2. 不要顺手改注册成功后的登录逻辑
3. 不要顺手改字段命名、错误消息风格、返回结构

### 7.3 历史数据修复脚本

必须新增一个 SQL 脚本，处理历史上已经被软删除、但仍占用原账号信息的用户。

建议文件名：

`supabase/migration_v21.sql`

脚本目标：

1. 找出 `status = 'deleted'` 的用户
2. 把其 `username` 改成 `deleted_<id>`
3. 把其 `phone` 改成 `del_<id>`
4. 如要处理 `nickname`，统一设为 `已删除用户`

要求：

1. SQL 必须幂等，可重复执行
2. 只处理 `status = 'deleted'`
3. 不处理 `cancelled`
4. 不动非删除用户

1. 禁止把“删除用户”改成真删除
2. 禁止删除数据库唯一索引

4. 禁止顺手改登录逻辑
5. 禁止顺手改列表页展示逻辑
6. 禁止把手机号墓碑值设计成带原手机号明文

8. 禁止为了“代码更优雅”而扩大重构范围

10. 禁止额外改动数据库 schema 结构，除非只是新增本次修复 SQL 文件

## 9. 验收标准

实现完成后，必须满足以下用例。

### 用例 1：删除后可复用同用户名同手机号

步骤：

1. 创建一个用户，用户名为 `18700000002`，手机号为 `18700000002`
2. 管理员在后台删除该用户
3. 再次使用相同用户名与相同手机号注册

期望：

1. 注册成功
2. 不再提示“该用户名已被使用”
3. 不再提示“该手机号已注册”

### 用例 2：已删除用户从后台默认列表中消失

步骤：

1. 删除一个用户
2. 打开用户管理列表默认页

期望：

1. 默认列表仍看不到该用户
2. 本次修复不破坏现有列表过滤行为

### 用例 3：非删除用户仍然保持唯一性

步骤：

1. 保留一个正常用户 A
2. 尝试注册与 A 相同的用户名
3. 尝试注册与 A 相同的手机号

期望：

1. 仍然提示冲突
2. 正常用户的唯一性保护不被破坏

### 用例 4：取消账号语义不变

步骤：

1. 准备一个 `status = 'cancelled'` 的账号
2. 用其原用户名或原手机号尝试重新注册

期望：

1. 本次不强行改变其现有行为
2. `cancelled` 是否可复用，不在本次实现范围
