## P3 环境与流程

### E1：本地 dev 启停规范化

#### 问题背景
运行检查报告中指出：
- `npm run dev` 报 3000 端口被占用，尝试切到 3001
- 再次启动出现 `.next/dev/lock` 锁文件冲突

这不是代码 bug，而是**运维流程没规范**。

#### 修改方案

**第 1 步：在 `package.json` 中新增辅助脚本**

```json
{
  "scripts": {
    "dev": "next dev --turbopack",
    "dev:stop": "npx kill-port 3000 || true",
    "dev:clean": "rimraf .next/dev/lock && rimraf .next/cache",
    "dev:fresh": "npm run dev:stop && npm run dev:clean && npm run dev"
  }
}
```

需要安装：
```bash
npm install -D kill-port rimraf
```

**第 2 步：写一份 `DEV_WORKFLOW.md` 文档**

放在项目根目录或 `doc/` 下：

```markdown
# 开发流程规范

## 启动
npm run dev

## 启动遇到端口占用
npm run dev:stop   # 杀掉 3000 端口进程
npm run dev        # 重新启动

## 完全重置（清缓存 + 锁文件）
npm run dev:fresh

## 停止
Ctrl+C 即可，无需手动清锁文件
```

**第 3 步（可选）：加 pre-dev hook**

在 `package.json` 中：

```json
"predev": "npm run dev:clean"
```

每次 `npm run dev` 前自动清锁文件。但会加长启动时间，权衡使用。

#### 风险
- `kill-port` 会无差别杀掉占用 3000 的进程，包括误杀其他应用
- `rimraf .next/cache` 会清掉构建缓存，下次启动变慢

#### 验证
- 故意占着 3000 端口，执行 `npm run dev:fresh`，应能自动清理并启动成功

### E2：`npm run build` 末尾 `spawn EPERM` 复验

#### 问题背景
上次 build 通过编译阶段但最后报 `spawn EPERM`。判断是 Windows 沙箱环境的子进程权限问题，不是业务代码问题。

#### 修改方案

**第 1 步：在干净环境复验**

在 3 种环境各跑一次 `npm run build`，看是否稳定：

1. **干净的本地终端**：关闭 VSCode 所有集成终端，用系统 PowerShell / cmd 直接跑
2. **GitHub Actions CI**：提交一个只跑 build 的 workflow 文件
3. **Docker 容器**（Linux 环境）：在纯 Linux 中验证

```yaml
# .github/workflows/build-check.yml
name: Build Check
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npm run build
```

**第 2 步：分析结果**

| 场景 | 现象 | 结论 |
|------|------|------|
| 三种环境都过 | EPERM 是本地特定环境问题 | 可关闭此项，在文档中记录即可 |
| Linux 过但 Windows 不过 | Windows 特有的子进程或 antivirus 干扰 | 需细查 Windows Defender 或杀毒软件例外 |
| CI 也不过 | 真实构建问题 | 需要打开 stack trace，按具体错误定位 |

**第 3 步：如果是 Windows Defender 导致**

把项目目录加入 Defender 排除列表：
- Windows 安全中心 → 病毒和威胁防护 → 管理设置 → 添加排除项 → 文件夹 → 选择项目根目录

#### 风险
- 排除 Defender 仅本地生效，不能解决 CI 问题
- 如果是 Node 版本问题，需要统一到 LTS

#### 验证
- 在 GitHub Actions 上连续触发 3 次 build，全部通过
- 本地干净环境跑一次，观察是否稳定

### E3：CI/CD 质量门禁补齐

#### 问题背景
项目目前没有 CI 配置（只有 Jenkinsfile，未确认是否启用）。质量门禁全靠人工跑 lint/build，容易漏掉。

#### 修改方案

**第 1 步：建立 GitHub Actions workflow**

新建 `.github/workflows/ci.yml`：

```yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      
      - name: Lint
        run: npm run lint
      
      - name: Type check
        run: npx tsc --noEmit
      
      - name: Build
        run: npm run build
        env:
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY }}
          SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}
          JWT_SECRET: ${{ secrets.JWT_SECRET }}
```

**第 2 步：在 GitHub 仓库中配置 Secrets**

Settings → Secrets and variables → Actions → New repository secret，添加上述环境变量。

**第 3 步：强制 PR 通过 CI 才能合并**

Settings → Branches → Branch protection rules → main 分支勾选 "Require status checks to pass before merging"。

**第 4 步（可选）：加 Jenkinsfile 或保留现状**

如果团队仍用 Jenkins，补上同等的质量门禁：

```groovy
pipeline {
  stages {
    stage('Quality') {
      steps {
        sh 'npm ci'
        sh 'npm run lint'
        sh 'npx tsc --noEmit'
        sh 'npm run build'
      }
    }
  }
}
```

#### 风险
- 首次跑 CI 需要配置 secrets，可能环境变量缺失导致 build 失败
- 如果团队没用过 GitHub Actions，需要培训成本

#### 验证
- 提交一个 PR，CI 三个检查（lint/type/build）都跑过
- 故意引入一个 lint error，确认 CI 会阻止合并

## R 延期项重新评估

### R1：P13 分析统计物化视图（用 RPC 替代）

#### 原延期理由
Supabase JS Client 不支持原生 `GROUP BY`。

#### 新方案：用 Postgres 函数替代

既然客户端不支持 GROUP BY，可以在数据库层面写一个 RPC 函数，把整个聚合逻辑封装进去，然后前端调用。

**新建 `supabase/migration_v21.sql`**：

```sql
-- 分析统计：top agents（按调用次数）
CREATE OR REPLACE FUNCTION get_top_agents(
  p_tenant_code TEXT DEFAULT NULL,
  p_days INT DEFAULT 30,
  p_limit INT DEFAULT 10
)
RETURNS TABLE (
  agent_code TEXT,
  agent_name TEXT,
  call_count BIGINT
) AS $$
  SELECT 
    l.agent_code,
    l.agent_name,
    COUNT(*)::BIGINT AS call_count
  FROM logs l
  WHERE l.action = 'chat'
    AND l.created_at > NOW() - (p_days || ' days')::INTERVAL
    AND (p_tenant_code IS NULL OR l.tenant_code = p_tenant_code)
  GROUP BY l.agent_code, l.agent_name
  ORDER BY call_count DESC
  LIMIT p_limit;
$$ LANGUAGE sql STABLE;

-- 用户维度统计
CREATE OR REPLACE FUNCTION get_user_usage(
  p_tenant_code TEXT DEFAULT NULL,
  p_dept_id UUID DEFAULT NULL,
  p_team_id UUID DEFAULT NULL,
  p_days INT DEFAULT 30,
  p_search TEXT DEFAULT NULL
)
RETURNS TABLE (
  user_phone TEXT,
  call_count BIGINT,
  last_used TIMESTAMPTZ
) AS $$
  SELECT 
    l.user_phone,
    COUNT(*)::BIGINT AS call_count,
    MAX(l.created_at) AS last_used
  FROM logs l
  LEFT JOIN users u ON u.phone = l.user_phone AND u.tenant_code = l.tenant_code
  WHERE l.action = 'chat'
    AND l.created_at > NOW() - (p_days || ' days')::INTERVAL
    AND (p_tenant_code IS NULL OR l.tenant_code = p_tenant_code)
    AND (p_dept_id IS NULL OR u.dept_id = p_dept_id)
    AND (p_team_id IS NULL OR u.team_id = p_team_id)
    AND (p_search IS NULL OR l.user_phone ILIKE '%' || p_search || '%')
  GROUP BY l.user_phone
  ORDER BY call_count DESC
  LIMIT 100;
$$ LANGUAGE sql STABLE;
```

**改造 `app/api/admin/analytics/route.ts`**：

```ts
// 旧：拉全量 logs 到内存聚合
const { data: allLogs } = await db.from("logs").select("*")...

// 新：调用 RPC
const { data: topAgents } = await db.rpc("get_top_agents", {
  p_tenant_code: tenantCode || null,
  p_days: days,
  p_limit: 10,
});

const { data: userUsage } = await db.rpc("get_user_usage", {
  p_tenant_code: tenantCode || null,
  p_dept_id: deptId || null,
  p_team_id: teamId || null,
  p_days: days,
  p_search: userSearch || null,
});
```

#### 收益
- 数据量大时性能从 O(n) 降到 O(log n + limit)
- 内存占用从全表加载降到返回集大小
- 真正解决了之前"暂不改"的根源问题

#### 风险
- RPC 函数需要在 Supabase SQL Editor 执行
- 需要对账聚合结果是否与原应用层实现一致

### R3：Bug 23 分类排序并发

#### 原延期理由
`sort_order` 无 UNIQUE 约束，重复不报错，最差结果是两个分类排序值相同。

#### 一次性修复方案

其实只需改一行 SQL，把"先查 max、再 +1"合并为一条 INSERT：

**改造 `app/api/admin/categories/route.ts` POST 部分**：

```ts
// 旧
const { data: maxRow } = await db
  .from("categories")
  .select("sort_order")
  .order("sort_order", { ascending: false })
  .limit(1)
  .single();
const nextSortOrder = (maxRow?.sort_order ?? 0) + 1;

const { data, error } = await db.from("categories").insert({
  name,
  sort_order: nextSortOrder,
}).select().single();

// 新：用子查询原子化
const { data, error } = await db.rpc("insert_category_with_next_order", { p_name: name });
```

**新建 `supabase/rpc_category.sql`**：

```sql
CREATE OR REPLACE FUNCTION insert_category_with_next_order(p_name TEXT)
RETURNS categories AS $$
  INSERT INTO categories (name, sort_order)
  VALUES (p_name, COALESCE((SELECT MAX(sort_order) FROM categories), 0) + 1)
  RETURNING *;
$$ LANGUAGE sql;
```

#### 收益
- 从"TOCTOU 竞态"变为原子操作
- 修复成本极低，但彻底解决

## L 长期建设

### L1：自动化测试基础设施

#### 问题背景
CLAUDE.md 明确写了 **"There are no automated tests in this project"**。目前所有质量保证靠人工跑 build 和肉眼测试。

随着 Bug 修复 + 优化积累到 58 项（24 Bug + 34 优化），**回归风险越来越大**。

#### 修改方案

**第 1 步：选型**

| 测试类型 | 工具 | 优先级 |
|---------|------|--------|
| 单元测试 | Vitest | 高 |
| API 集成测试 | Vitest + msw | 高 |
| 端到端测试 | Playwright | 中（可后置） |
| 组件测试 | Vitest + @testing-library/react | 中 |

**第 2 步：搭建 Vitest 基础**

```bash
npm install -D vitest @vitest/ui @testing-library/react @testing-library/jest-dom jsdom
```

`vitest.config.ts`：

```ts
import { defineConfig } from 'vitest/config';
import path from 'path';

export default defineConfig({
  test: {
    environment: 'jsdom',
    setupFiles: ['./test/setup.ts'],
  },
  resolve: {
    alias: { '@': path.resolve(__dirname, '.') },
  },
});
```

`test/setup.ts`：

```ts
import '@testing-library/jest-dom';
```

`package.json` 加 script：

```json
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run",
    "test:ui": "vitest --ui"
  }
}
```

**第 3 步：先覆盖高风险模块的单元测试**

优先写测试的模块（按价值排序）：

1. **`lib/auth.ts`** — JWT 签发/校验，安全关键
2. **`lib/crypto.ts`** — 加密/解密，数据安全
3. **`lib/permissions.ts`** — 资源权限判断，权限关键
4. **`lib/rate-limit.ts`** — 限流，安全关键
5. **`lib/adapters/index.ts`** — AI 适配器，业务关键
6. **`lib/validate.ts`** — Schema 校验辅助

每个模块至少覆盖：
- 正常路径
- 边界条件（空值、超长、异常类型）
- 错误路径

示例 `lib/crypto.test.ts`：

```ts
import { describe, it, expect } from 'vitest';
import { encrypt, decrypt } from '@/lib/crypto';

describe('crypto', () => {
  it('encrypt → decrypt round trip', () => {
    const plain = 'sk-test-1234567890';
    const cipher = encrypt(plain);
    expect(cipher).not.toBe(plain);
    expect(cipher.split(':').length).toBe(3);
    expect(decrypt(cipher)).toBe(plain);
  });

  it('encrypt empty string returns empty', () => {
    expect(encrypt('')).toBe('');
  });

  it('decrypt invalid format throws', () => {
    expect(() => decrypt('invalid')).toThrow();
  });
});
```

**第 4 步：覆盖率目标**

- 第一阶段：核心 lib 层 70%+ 覆盖率
- 第二阶段：关键 API 路由（auth、chat、quota）集成测试
- 第三阶段：E2E 关键流程（登录 → 聊天 → 登出）

**第 5 步：CI 集成**

E3 的 workflow 中加一步：

```yaml
- name: Test
  run: npm run test:run
```

#### 收益
- 回归风险大幅降低，后续修改有安全网
- 重构时更敢动手
- 新人上手时测试用例即文档

#### 风险
- 搭建成本不低（8-16 小时）
- 业务已存在的代码要补测试需要理解原有逻辑
- 测试本身也需要维护

### L2：监控告警接入

#### 问题背景
F8 接入了请求日志（仅控制台），但：
- 线上日志要聚合，控制台看不见

#### 修改方案

**第 1 步：接入错误上报（Sentry）**

```bash
npm install @sentry/nextjs
npx @sentry/wizard@latest -i nextjs
```

自动生成 `sentry.client.config.ts`、`sentry.server.config.ts`、`sentry.edge.config.ts`。

在 `withRequestLog` 中手动上报错误：

```ts
import * as Sentry from '@sentry/nextjs';

export function withRequestLog(handler) {
  return async (req, ...args) => {
    try {
      return await handler(req, ...args);
    } catch (err) {
      Sentry.captureException(err, {
        tags: { path: req.url },
        extra: { requestId },
      });
      throw err;
    }
  };
}
```

**第 2 步：接入指标采集（可选）**

- 前端性能：`@vercel/analytics`
- 后端延迟：Prometheus + Grafana（如果自托管）

**第 3 步：告警规则**

- 错误率 > 1% 触发告警
- p95 响应时间 > 2s 触发告警

#### 风险
- Sentry 免费额度有限，超出要付费
- 过度监控会产生告警疲劳

## 需要提醒的
