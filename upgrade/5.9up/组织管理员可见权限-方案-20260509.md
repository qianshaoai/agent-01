# 组织管理员可见权限可选 — 方案

**版本**：5.9up
**日期**：2026-05-09
**分支**：`devB`
**状态**：设计稿，待评审

---

## 一、问题

当前 `org_admin`（组织管理员）创建 / 编辑工作流时，可见权限被**强制锁死**为「全组织可见」：

- 后端 `POST /api/admin/workflows` line 100-105 强制 `visibleTo='org_only'` + `permissions=[{scope_type:'org', scope_id: tenantCode}]`
- 后端 `PATCH /api/admin/workflows/[id]` line 64 / 100 阻止 org_admin 修改 `visibleTo` 和 `permissions`
- 前端 `app/admin/workflows/page.tsx` line 1019-1022 把"可见权限"那块替换成蓝色提示"本工作流将自动对您所在组织的所有成员可见（不可修改）"

**真实需求**：org_admin 应该能在**本组织范围内**自由选三种粒度：
1. 我所在组织全员可见（当前默认）
2. 我所在组织的某个 / 某些**部门**可见
3. 我所在组织的某个 / 某些**小组**可见

跨组织（指定别的 tenant、别的 dept/team）一律拒。

---

## 二、数据库

**不需要 migration**。`resource_permissions` 表早就支持三种 `scope_type`（org / dept / team），super_admin 也是这套模型。本期只是开放给 org_admin 用。

## 三、后端改动

### 3.1 工具函数：校验 org_admin 提交的 permissions 是否都在本组织

新增 `lib/admin-scope.ts`（或就放在 workflows route 文件里）：

```ts
async function validateOrgAdminPermissions(
  tenantCode: string,
  permissions: Array<{ scope_type: string; scope_id: string | null }>
): Promise<{ ok: true } | { ok: false; reason: string }> {
  if (permissions.length === 0) {
    return { ok: false, reason: "至少选择一个可见范围" };
  }

  // 拉本组织下所有 dept/team 的 id 集合（用于校验）
  const [{ data: depts }, { data: teams }] = await Promise.all([
    db.from("departments").select("id").eq("tenant_code", tenantCode),
    db.from("teams").select("id").eq("tenant_code", tenantCode),
  ]);
  const deptIds = new Set((depts ?? []).map((d) => d.id));
  const teamIds = new Set((teams ?? []).map((t) => t.id));

  for (const p of permissions) {
    if (p.scope_type === "org") {
      // 必须等于本组织 code
      if (p.scope_id !== tenantCode) return { ok: false, reason: "只能选本组织" };
    } else if (p.scope_type === "dept") {
      if (!p.scope_id || !deptIds.has(p.scope_id)) return { ok: false, reason: "部门不属于本组织" };
    } else if (p.scope_type === "team") {
      if (!p.scope_id || !teamIds.has(p.scope_id)) return { ok: false, reason: "小组不属于本组织" };
    } else {
      return { ok: false, reason: `不支持的 scope_type=${p.scope_type}` };
    }
  }
  return { ok: true };
}
```

### 3.2 `POST /api/admin/workflows` 改造

把 line 100-105 的强制锁死改成校验：

```ts
if (admin.role === "org_admin") {
  if (!admin.tenantCode) return apiError("组织管理员未绑定组织", "FORBIDDEN");

  // 5.9up：org_admin 现在可以自选 permissions，但只能在本组织范围内
  // 默认（前端没传 permissions）→ 全组织可见，行为同 5.7up
  if (!Array.isArray(permissions) || permissions.length === 0) {
    visibleTo = "org_only";
    permissions = [{ scope_type: "org", scope_id: admin.tenantCode }];
  } else {
    // 校验 permissions 都在本组织
    const result = await validateOrgAdminPermissions(admin.tenantCode, permissions);
    if (!result.ok) return apiError(result.reason, "FORBIDDEN");
    // 不论选了 org/dept/team，对外都是 visible_to='custom'
    visibleTo = "custom";
  }
}
```

下面写 `permsToInsert` 那段同步调整：

```ts
const permsToInsert: Array<{ scope_type: string; scope_id: string | null }> = [];
if (admin.role === "org_admin") {
  // 上面已经把 permissions 标准化了，直接用
  permsToInsert.push(...permissions);
} else if (visibleTo === "custom" && Array.isArray(permissions)) {
  permsToInsert.push(...permissions);
}
```

### 3.3 `PATCH /api/admin/workflows/[id]` 改造

放开两处 org_admin 写权限：

**line 64-65**：
```ts
// 之前：仅 super_admin/system_admin 可改 visibleTo
// 改为：org_admin 也可改，但只能在 org_only / custom 之间切（不能放出本组织外）
if (body.visibleTo !== undefined) {
  if (admin.role === "org_admin") {
    if (body.visibleTo !== "org_only" && body.visibleTo !== "custom") {
      return apiError("组织管理员只能在本组织范围内设置可见权限", "FORBIDDEN");
    }
  }
  updates.visible_to = body.visibleTo;
}
```

**line 100-110**（permissions 写入）：
```ts
if (body.permissions !== undefined) {
  const perms = Array.isArray(body.permissions) ? body.permissions : [];

  // 5.9up：org_admin 也能写，但要校验都在本组织
  if (admin.role === "org_admin" && admin.tenantCode) {
    const result = await validateOrgAdminPermissions(admin.tenantCode, perms);
    if (!result.ok) return apiError(result.reason, "FORBIDDEN");
  }

  await db.from("resource_permissions")
    .delete()
    .eq("resource_type", "workflow")
    .eq("resource_id", id);

  if (perms.length > 0) {
    await db.from("resource_permissions").insert(
      perms.map((p) => ({
        resource_type: "workflow",
        resource_id: id,
        scope_type: p.scope_type,
        scope_id: p.scope_id,
      }))
    );
  }
}
```

> ⚠️ 注意：`ensureOrgAdminCanTouch` 这个守卫已经先校验过工作流当前归属本组织。改后要确保 PATCH 完成后**新的 permissions 仍然让本组织能看到**——否则 org_admin 改完自己看不见、也不能再改了。这个由前端表单约束（至少留一条本组织内 permission），后端再加一道兜底：保存后如果 permissions 全部清空且角色是 org_admin，自动补一条 scope=org 的；或者直接拒绝空提交。

### 3.4 风险点

- **org_admin 能否选「我所在组织全员可见」+ 同时勾几个部门**：建议**不让混选**。要么"全员"，要么"指定部门"，要么"指定小组"，三者互斥。前端 UI 已经是 select 单选 scope，沿用即可。
- **edge case**：org_admin 把工作流改成只对一个不包含自己的小组可见 → 自己就看不到了。两种处理：
  - 选项 A：保存后若 admin 不在所选范围内，前端弹提示"保存后您将看不到此工作流，确认？"
  - 选项 B：自动追加一条本组织 admin 的可见性（粗暴但安全）
  - 推荐 A，让用户主动确认。

## 四、前端改动

文件：`app/admin/workflows/page.tsx`

### 4.1 把蓝色提示框换成下拉

把 line 1018-1023 的:

```tsx
{isOrgAdmin ? (
  <div className="px-4 py-2.5 rounded-[12px] bg-blue-50 border border-blue-100 text-sm text-blue-700">
    本工作流将自动对您所在组织的所有成员可见（不可修改）
  </div>

  <select>...super_admin 的下拉</select>
)}
```

改为：org_admin 也用 select，但只显示三个选项，且 dept/team picker 自动过滤到本组织。

```tsx
<select
  ...
  onChange={...}
>
  {isOrgAdmin ? (
    <>
      <option value="org_only">我所在组织全员可见</option>
      <option value="custom:dept">指定本组织部门可见</option>
      <option value="custom:team">指定本组织小组可见</option>

  ) : (
    <>
      <option value="all">全部用户可见</option>
      <option value="org_only">仅组织用户可见</option>
      <option value="personal_only">仅个人用户可见</option>
      <option value="custom:org">指定组织可见</option>
      <option value="custom:dept">指定部门可见</option>
      <option value="custom:team">指定小组可见</option>
    </>
  )}

```

### 4.2 dept / team picker 数据过滤

org_admin 模式下：

- 部门 picker：`allDepts.filter(d => d.tenant_code === currentAdminTenantCode)`
- 小组 picker：先按本组织 dept 过滤，再按 dept 取 teams

省去外层「按组织分组」那层（因为只有一个组织，没必要分组），直接平铺该组织的部门 / 小组列表更清爽。

### 4.3 提交时构造 permissions

`handleSave`（line 274 附近）当前已经能处理 `custom` + permIds，无需大改，只要把上面 select 的 `org_only`/`custom:dept`/`custom:team` 几个值正确映射成 permissions 就行：

```ts
let permissions;
if (wfForm.visibleTo === "org_only" && isOrgAdmin) {
  // org_admin 选"全员"→ 写一条 scope=org 的
  permissions = [{ scope_type: "org", scope_id: adminTenantCode }];
} else if (wfForm.visibleTo === "custom") {
  permissions = wfForm.permIds.map(id => ({
    scope_type: wfForm.permScope,
    scope_id: id,
  }));
} else {
  permissions = undefined;
}
```

### 4.4 组织管理员的 admin 信息从哪里拿

`/api/admin/me` 已经返回 `tenantCode`。前端 `useState` 一份 `adminTenantCode` 跟 `adminRole` 同时拉即可（line 87-88 旁边）。

## 六、关键决策点（请你拍板）

1. **三种粒度互斥还是可混选？**（推荐互斥，与现有 super_admin UI 保持一致）
2. **org_admin 把范围改成不包含自己时**：弹确认（推荐）/ 自动兜底 / 直接拒绝
3. **不传 permissions 提交**（即直接点创建未做选择）：自动按"全组织"走（5.7up 行为，向后兼容）vs 报错要求选择。推荐自动兜底「全组织」
4. **撤销路径 B 的"边界外不可改"语义**——之前 5.7up 是 org_admin 完全不能改可见性，本期开放了。是否需要做老数据的 migration？答：不需要，老数据 visible_to 字段保持不变，只是后续创建 / 编辑能选了
