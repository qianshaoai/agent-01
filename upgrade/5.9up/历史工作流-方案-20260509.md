# 历史工作流功能方案

**版本**：5.9up
**日期**：2026-05-09
**状态**：设计稿，待评审

---

## 一、目标

让用户能：
1. 查看自己**之前用过**的工作流（已完成 / 已放弃 / 已结束）
2. 进入历史会话，**只读**回看完整对话记录（每一步智能体说了什么）
3. 不影响"我的进行中工作流"的现有体验

---

## 二、术语与边界

| 术语 | 含义 |
|---|---|
| **进行中会话** | `workflow_sessions.status = 'in_progress'` |
| **已完成会话** | `status = 'completed'`（最后一步走完自动转） |
| **已放弃会话** | `status = 'abandoned'`（用户主动停止 / 软删除） |
| **历史会话** | 已完成 + 已放弃 的合集 |

**不在范围内**：
- 老路径（`session_id IS NULL`）的对话不计入"历史工作流"，仍按"按智能体看历史对话"逻辑
- 删除整个工作流模板时联动行为按现有 `ON DELETE CASCADE`

---

## 三、数据库变更

**结论：不需要 migration**。

| 现有支持 | 说明 |
|---|---|
| `workflow_sessions.status` 已支持三态 | 不用加列 |
| `idx_wf_sessions_status (user_id, status)` 已建索引 | 列表查询走索引 |
| `conversations.session_id` 已建立 | 对话已能按 session 隔离查 |

可选（暂不实施）：加一列 `ended_at timestamptz` 显式记录结束时间，比 `updated_at` 更准。本期复用 `updated_at`。

---

## 四、状态流转重设计

### 4.1 当前问题

- `DELETE /api/workflow-sessions/[id]` 现在是**物理删除**，整个 session + 关联 conversations 一并消失
- 用户一旦点 ✕，对话历史也没了，无法事后回看
- "已放弃"状态从来没有路径写入

### 4.2 改动方案（推荐 A）

| 操作位置 | 当前行为 | 新行为 |
|---|---|---|
| 进行中会话卡片 ✕ 按钮 | 物理 DELETE | 软删除：`PATCH status='abandoned'`，从"进行中"消失，进"历史" |
| 历史列表行 ✕ 按钮 | （不存在） | 物理 DELETE，弹确认"永久删除该会话及全部对话，不可恢复" |
| 自然完成 | `status='completed'`（已实现） | 不变 |

**好处**：用户的"删除"语义保留（卡片消失），同时数据可在历史里找回；只有显式去历史里点永久删除才真删。

### 4.3 备选方案 B（不推荐）

进行中会话 ✕ 弹两个选项："移到历史 / 永久删除"。多一步交互，对常用操作不友好。

## 五、API 设计

### 5.1 修改

| 路由 | 改动 |
|---|---|
| `GET /api/workflow-sessions` | 现已支持 `?status=` 单值过滤；扩展为支持 `?status=history` 表示 `IN (completed, abandoned)` |
| `DELETE /api/workflow-sessions/[id]` | 默认行为改为软删除（`UPDATE status='abandoned'`）。新增 `?hard=1` 时执行物理删除，且仅允许 status 已是 `completed`/`abandoned`，避免误删进行中会话 |

### 5.2 新增（可选，本期不做）

| 路由 | 用途 |
|---|---|
| `POST /api/workflow-sessions/[id]/restore` | 把已放弃的会话恢复成进行中（撤销软删） |
| `DELETE /api/workflow-sessions?clear=history` | 一键清空所有历史 |

> 本期不实现，等用户实际用了再看是否需要。

### 5.3 对话查询复用

- 列表 `GET /api/conversations?sessionId=xxx` —— 已有
- 单条消息 `GET /api/conversations/[id]/messages` —— 已有
- 写接口（POST 消息、编辑、重发、PATCH abort）—— 在 readonly 模式下前端**不发起**即可，后端无需新增校验

## 六、前端页面结构

### 6.1 首页入口

`app/page.tsx` 的"我的进行中工作流"区块标题栏右侧（折叠按钮和左右箭头之间）加一个文字按钮：

```
📁 历史
```

点击 → `router.push('/workflows/history')`。即使没有进行中会话也显示该入口（条件外提一层）。

### 6.2 历史列表页 `/workflows/history`

新建 `app/workflows/history/page.tsx`。

**布局**：

```
┌ 顶部 ─────────────────────────────────────────────┐
│ ← 返回    历史工作流                              │
│ [搜索框: 按会话名 / 工作流名]   [筛选: 全部 ▾]   │
└────────────────────────────────────────────────────┘
┌ 列表 ────────────────────────────────────────────┐
│ 项目A资料整理 [已完成]                            │
│ 高净值客户面谈前准备工作流 · 完整 6 步 · 5/8 18:32│
│                              [查看] [永久删除]    │
├────────────────────────────────────────────────────┤
│ 测试工作流2 [已放弃]                              │
│ 高净值客户面谈前准备工作流 · 1/6 步 · 5/8 14:15  │
│                              [查看] [永久删除]    │
└────────────────────────────────────────────────────┘
[ 上一页 ] 1/2 [ 下一页 ]
```

**筛选下拉**：全部 / 已完成 / 已放弃。

**搜索**：前端模糊匹配 `name` 或 `workflow.name`，不开后端 LIKE。

**每页 50 条**，按 `updated_at DESC` 排序。

### 6.3 历史会话详情 `/workflows/history/[sessionId]`

新建 `app/workflows/history/[id]/page.tsx`。

**布局**：

```
┌─ 顶部 ───────────────────────────────────────────┐
│ ← 返回    项目A资料整理 [已完成]                 │
│ 工作流: 高净值客户面谈前准备工作流                │
│ 时间: 2026-05-08 18:32 完成                       │
└────────────────────────────────────────────────────┘
┌─ 步骤切换 (横向) ────────────────────────────────┐
│ [1.明确需求✓] [2.萃取素材✓] [3.搭架构✓] ...    │
└────────────────────────────────────────────────────┘
┌─ 当前步骤 ───────────────────────────────────────┐
│ 步骤标题 / 类型徽章 / 智能体名                   │
│ 描述：xxx                                         │
└────────────────────────────────────────────────────┘
┌─ 对话回看（只读）────────────────────────────────┐
│ 用户：xxx                                         │
│ 助手：xxx                                         │
│ ...                                              │
│                                                   │
│ ✕ 输入框（禁用 / 不渲染）                         │
└────────────────────────────────────────────────────┘
```

**步骤切换**：每个步骤是一个 chip，点击切换显示该步骤对应 conversation 的消息流。

**人工步骤**（exec_type ≠ agent）：显示"该步骤为人工执行 / 审核 / 外部工具，无对话记录"。

**只读保证**：
- 所有写入按钮（发送 / 编辑 / 重发 / 删除消息）不渲染
- 输入框不渲染（最稳）

### 6.4 实现选型

**选项 A（推荐）**：复用 `app/agents/[id]/page.tsx`，加 `?readonly=1` 参数

- 同一份组件，加一个 `readonly` 状态
- 输入框区域整块根据 `readonly` 隐藏
- 进度条点击行为改：在 readonly 下点其他步骤跳 `/workflows/history/[sessionId]?step=N`，而不是 `/agents/[code]`
- URL 形态：`/agents/[code]?session=xxx&step=N&readonly=1`

**选项 B**：单独写 readonly 视图组件 `<ReadonlyChatView>`

- 纯净，但要复刻 markdown 渲染、代码块复制、附件展示等
- 工作量更大

**推荐 A**：6 处需要在 readonly 下隐藏 / 改行为，加几个 `if (readonly)` 即可，工作量可控。详见 §七。

## 七、智能体页 readonly 改造点

`app/agents/[id]/page.tsx` 需要在 `readonly` 为 true 时：

| 位置 | 改动 |
|---|---|
| 顶部 banner | 加一行黄色提示"只读模式 · 历史回看" |
| 输入框 / 发送按钮 / 麦克风 / 附件按钮 | 整块不渲染 |
| 消息上的"重新生成 / 编辑 / 复制" | "复制"保留，"重新生成 / 编辑"不渲染 |
| 进度条点击 | 跳 `/workflows/history/[sessionId]?step=N` 而不是 `/agents/[code]?step=N` |
| 「下一步」按钮 | 不渲染（已完成的不需要推进） |
| 「停止」按钮 | 不渲染（不会有流式中的状态） |
| 中断徽章 | 仍然渲染（让用户看到当时是停止的） |

## 十、风险与注意

1. **DELETE 改软删，要观察前端调用方**：当前只有首页 `deleteSession` 调用，影响面可控
2. **会话内对话有 abort 状态的消息**：readonly 模式要照常显示"已停止"徽章
3. **没有对话的步骤**（人工步骤 / 智能体页从未进过的步骤）：详情页要给空态，不报错
4. **隐私**：只读路由仍然要做 owner 校验，已有 `user_id` 过滤，复核一遍即可
