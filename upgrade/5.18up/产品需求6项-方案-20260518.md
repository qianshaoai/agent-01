## 四、R3 · 已放弃的工作流可恢复

### 现状

- `workflow_sessions` 表（migration_v29）：`status` = `in_progress | completed | abandoned`。
- 用户在「进行中」放弃工作流 → `DELETE /api/workflow-sessions/[id]`（软删）→ `status='abandoned'`。
- 历史页 `app/workflows/history/page.tsx`：展示 `completed + abandoned`，每行有「查看」+「永久删除」。
- `PATCH /api/workflow-sessions/[id]` **已支持 `status` 字段**，且 `current_step_idx` 一直保留。

→ 恢复在技术上 = 把 `abandoned` 改回 `in_progress`。后端能力基本现成，主要是补前端入口 + 一道守卫。

### 要做什么

1. **前端**：历史页 `abandoned` 状态的行加「恢复」按钮（`completed` 行不显示）。
   点击 → `PATCH status='in_progress'` → 该会话从历史页消失、回到「进行中」列表，
   `current_step_idx` 不变，员工可继续做。
2. **后端守卫**：当前 `PATCH /api/workflow-sessions/[id]` 是**复用**接口，同时承担
   改名、更新进度（`currentStepIdx`）、标记完成 —— **不能**简单收成"只允许
   `abandoned→in_progress`"，否则会把改名 / 进度 / 完成一起堵死。
   正确做法是按"目标状态"列**状态流转白名单**：
   - 请求把 `status` 改为 `in_progress` → 校验当前必须是 `abandoned`（这就是"恢复"）；
   - 请求把 `status` 改为 `completed` → 校验当前必须是 `in_progress`；
   - 请求把 `status` 改为 `abandoned` → 校验当前必须是 `in_progress`；
   - 不带 `status` 的请求（纯改名 / 改进度）→ 不受状态校验影响，照常放行。
   - 非白名单内的流转一律 422 拒绝。

### 涉及文件

```
修改：app/workflows/history/page.tsx
修改：app/api/workflow-sessions/[id]/route.ts   （状态流转守卫）
```

### 决策点

| 编号 | 问题 | 推荐 |
|---|---|---|
| D3-1 | 恢复后 `current_step_idx` 是否保留 | 保留 —— 从放弃时的进度继续 |
| D3-2 | 后端如何限制状态流转 | 按"目标状态"列状态流转白名单（见上）；不带 `status` 的改名 / 改进度不受影响 |
| D3-3 | 底层 workflow 已被管理员**停用**时，是否还允许恢复 | 允许恢复（数据仍在），继续页对已停用工作流给提示，不在恢复处硬拦 |
| D3-4 | 底层 workflow 已被**删除**（FK `ON DELETE CASCADE`，会话已随之物理删除）| 不存在该场景，无需处理 |

## 五、R4 · 智能体排序方式优化

### 现状

- 智能体多分类：`agent_categories` M2M（`categoryIds` / `categoriesAll`），另有遗留单列 `category_id`。
- `categories` 表有 `sort_order`。
- 后台 `app/admin/agents/page.tsx`：智能体是一张**扁平表格**，无分组、不按分类排序，
  顺序取决于 `/api/admin/agents` 返回顺序；新增弹窗分类为多选，且"不选则不出现在任何分类下"。

### 要做什么（方向，细节待决策）

让后台智能体列表「按分类整齐排列」。核心矛盾：**一个智能体可属于多个分类**，
分组归属不唯一，必须先拍板归组规则。

- 展示：后台列表从扁平表格改为**按分类分组的区块**（每个分类一个分区，分区按
  `categories.sort_order` 排，分区内智能体按名称 / 编号排）。
- 排序「自动」：分类增删改后列表自动按上述规则重排，管理员无需手动拖。

### ⚠️ 两个必须先解决的现状问题（小B 评审补充）

1. **分页限制 —— "前端客户端排序、0 接口改动"是错的**：`/api/admin/agents`
   是分页接口，默认 `pageSize=50`，且按 `created_at desc` 返回。只在前端对
   "当前页拿到的 50 条"做分组，会变成**"当前页内分组"**，某个分类的智能体可能
   分散在多页、当前页里看不全 —— 不是真正的"按分类完整展示"。
   两条出路（D4-3 抉择）：
   - 后端给**全量 / 分组接口**，或前端**分页循环拉完所有页**再分组，才能真正完整。
     ⚠️ `parsePagination` 有 `MAX_PAGE_SIZE=100` 上限 —— 单次带大 `pageSize` 最多 100 条，
     智能体一旦超过 100 个就又不全，`pageSize=100` 只能算临时方案，不是正解；
   - 或维持前端分组，但**文案 / UI 必须写清"仅对当前页分组"**，不夸大成完整展示。
2. **分类筛选用错字段**：当前 `filteredAgents` 的分类筛选判断的是遗留单列
   `a.category_id`（`page.tsx` 第 445 行），而展示用的是多分类 `categoriesAll`。
   多分类智能体只要"主类"对不上就会被筛漏。
   → **R4 至少要把筛选逻辑改成 `a.categoryIds.includes(filter)`**，这是本项的下限。

### 决策点（**必须先拍板才能开工**）

| 编号 | 问题 | 选项 | 推荐 |
|---|---|---|---|
| D4-1 | 多分类智能体归到哪个分区 | 甲：引入"主分类"单选字段（需改表单 + 可能改 DB）<br>乙：在每个所属分类分区下都出现一次<br>丙：只挂在 `sort_order` 最小的分类下 | **乙**（最贴"在多个分类下显示"的现有语义，0 DB 改动） |
| D4-2 | "创建时有明确的分类"：分类是否改为必选 | 保持可选 + 设"未分类"分区 / 改必选 | 保持可选 + "未分类"分区兜底 |
| D4-3 | 完整分组 vs 当前页分组 | 甲：后端给全量/分组接口，或前端分页循环拉完所有页，做到真完整展示<br>乙：前端分组，但文案写清"仅当前页" | 看产品对"完整"的刚性；**只要"完整展示"就必须走甲，并重估工时** |
| D4-4 | 分类筛选改 `categoryIds.includes`（修漏筛 bug） | —— | **必做**，与归组方案无关，是本项下限 |

> ⚠️ P2、估时 1h —— 只够「乙 + 前端当前页分组 + 修筛选 bug」这种 0-DB / 0-接口方案。
> 若 D4-1 选「甲（主分类）」或 D4-3 选「甲（全量/分组接口）」，需加迁移 / 改接口，工时翻倍，需重估。

### 涉及文件（按推荐的 0-DB 方案）

```
修改：app/admin/agents/page.tsx        （列表改分类分组渲染 + 筛选改 categoryIds.includes）
（若 D4-3 走甲）修改：app/api/admin/agents/route.ts   （全量 / 分组返回）
```

| 编号 | 问题 | 选项 | 推荐 |

### ⚠️ 批量重排接口的原子性要求（小B 评审补充）

新增 `PUT` 方向对，但"批量"不等于"原子"。若后端实现成**循环逐行 `update`
`workflow_steps.step_order`**，中途失败一半，照样会留下串号 / 重号的脏顺序 ——
和现在 `moveStep()` 多次 PATCH 的问题本质一样，只是挪了个位置。

落库实现要求（按优先级）：

- **首选**：DB 端事务 / RPC 一次性重排（参考 `supabase/rpc.sql` 里 `increment_quota_*`
  的 RPC 写法），整批要么全成、要么全不动。
- **次选**（暂不做 RPC 时）：必须做到 ——
  - 接口先**整批校验**（id 集合与该工作流当前步骤集合完全一致）再逐行写；
  - 任一行写失败 → 接口返回明确错误，前端**强制重新拉取真实顺序**回显，
    并提示"排序未完全保存，请重试"；
  - 文档**明示**此实现仍存在"部分失败"残余风险，不能宣称已消除竞态。

## 七、R6 · 工作流排序优化

### 现状

- 工作流分类：`workflow_categories` M2M（`categoryIds`）+ `wf_categories` 表（有 `sort_order`）。
- 工作流自身有全局 `sort_order`（新增/编辑弹窗里是手填数字）。
- 后台 `app/admin/workflows/page.tsx`：工作流是一串扁平卡片，顶部有分类**筛选**下拉，
  但**不按分类分组展示**，排序是全局一套 `sort_order`。

### 要做什么（方向，细节待决策）

需求原文：「明确工作流分类及不同标签下工作流的显示；分类和排序各个标签隔离；
后台显示干净利索」。即后台按分类（标签）分区展示，且每个标签内的排序互不干扰。

两个核心矛盾：

1. **多分类 + 全局排序**：一个工作流可属多个分类，而 `workflows.sort_order` 是
   **全局单值**，天然无法表达"每个分类内独立排序"。
2. **分页限制**（与 R4 同源）：`/api/admin/workflows` 也是分页接口、按 `sort_order`
   全局返回。只在前端分区，同样只能做到"当前页内分区"，分类不完整。

### ⚠️ 结论先行（小B 评审强化）

**如果产品原话坚持"分类和排序各个标签隔离"，方案乙不算满足需求**：

- 方案乙不改 DB，只能做到"看起来按分类分区"，**排序仍是全局 `sort_order` 一套**，
  不存在"各标签独立排序"，且受分页限制只能分当前页。
- 要做到需求字面的"真隔离"，**必须走方案甲** —— 在 `workflow_categories` 关联表
  加 `sort_order` 列（迁移 v38），每个工作流在每个所属分类下各有一个排序值；
  同时分页 / 列表接口要能按分类取全量，否则分区仍不完整。

所以 D6-1 不是"推荐乙"，而是**先让产品把话说死**："真隔离"→甲；"看着整齐就行"→乙。

### 决策点（**必须先拍板**）
