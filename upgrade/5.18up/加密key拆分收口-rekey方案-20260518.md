## 一、背景

Fix 3（拆分 AES 加密 key 与 JWT_SECRET）的**代码已在 5.14up 完成** ——
[lib/crypto.ts](../../lib/crypto.ts) 实现「路径 A · Fallback Chain」：

- `encrypt()` 永远用新 key `ENCRYPTION_KEY`（base64 的 32 字节，不再 sha256）
- `decrypt()` 先试新 key → 失败回退旧 key `sha256(JWT_SECRET)` → 都失败抛错

> **⚠ 小B 评审修正**：crypto.ts **没有启动期校验**。`decrypt()` 在缺新 key 时会
> **静默回退旧 key** —— `getNewKey()` 抛的「ENCRYPTION_KEY required」错被 `decrypt()`
> 的 `catch` 吞掉、走到旧 key 分支；只有 `encrypt()` 路径会在被调用时因缺 key 抛错。
> 所以**不能依赖「进程启动失败」当兜底**。真正的部署门禁是 **Jenkins 凭据** ——
> `Jenkinsfile` 用 `credentials('agent01_ENCRYPTION_KEY')` 注入，凭据不存在则**构建
> 直接失败**，prod 不会带着空 key 上线。

**路径 A 只「向前修复」**：上线后新加密的数据用新 key 保护，但**上线前已入库的
存量密文仍是 `sha256(JWT_SECRET)` 加密的** —— JWT_SECRET 泄漏，存量密钥照样可解。
用户决策（5/18）：**补一次性 rekey，彻底关闭。**

本方案 = 路径 A 之上的存量 rekey + 把会反向污染的部署脚本一并收口。

---

## 二、目标

把所有加密列的存量密文从「旧 key 加密」整体重写为「新 key 加密」，并保证**后续每次
部署不会再写回旧 key 密文**。完成后 JWT_SECRET 泄漏不再暴露任何加密字段。

### 3.2 ⚠ 阻断点（小B 首轮）：`migrate-trial-agents.mjs` 必须一并收口

[scripts/migrate-trial-agents.mjs](../../scripts/migrate-trial-agents.mjs) 自带内联

**后果**：rekey 做完后，下一次部署的这个 stage 会把这 3 行又写回旧 key 密文，收口被
反向打开。

**修复（已实现）**：`getKey()` 改为读 `ENCRYPTION_KEY`、base64 解码 32 字节（与
crypto.ts `getNewKey()` 一致，不 sha256）；`REQUIRED_ENVS` 用 `ENCRYPTION_KEY` 替换
`JWT_SECRET`（该脚本里旧 key 只用于加密，改完即不再需要 JWT_SECRET）；移除多余的
`createHash` import。Jenkins pipeline 的 `environment` 块已注入 `ENCRYPTION_KEY`，
stage 内 `node` 进程直接读得到，stage 本身无需改。

### 3.3 收口点（小B 复评）：`migrate-agent-keys-to-providers.mjs` 补 fallback decrypt

[scripts/migrate-agent-keys-to-providers.mjs](../../scripts/migrate-agent-keys-to-providers.mjs)
不是部署脚本（一次性、已跑过），**不构成反向污染**。但它原 `decrypt()` 只用
`sha256(JWT_SECRET)` 旧 key —— rekey 之后若有人误跑，解不开新密文。

**修复（已实现）**：`decrypt()` 改为「新 key 优先、回退旧 key」，与 crypto.ts 路径 A
一致；`ENCRYPTION_KEY` 未配时退化为「仅旧 key」，兼容 rekey 之前的历史跑法。脚本头部
加 5.16up 废弃注记（一次性脚本，rekey 后不建议再跑）。

> **其它脚本已核对**：`setup-db.mjs` 只建表不加密。**每次部署都会重新加密的脚本只有
> `migrate-trial-agents.mjs` 这一个。**

### 3.4 陈旧注释订正（小B 复评小建议）

- [lib/crypto.ts](../../lib/crypto.ts)：原注释「production 不允许 fallback，进程启动
  期直接抛错」与实际不符（见 §一 修正）。已改为准确表述：无启动期校验，`encrypt()`

- [Jenkinsfile](../../Jenkinsfile)：trial stage 注释原写「用线上 JWT_SECRET 重新
  加密」，已同步改为 `ENCRYPTION_KEY`。

## 四、脚本设计 · `scripts/rekey-encryption.mjs`

### 4.1 核心安全约束

脚本**自带 encrypt/decrypt 实现，绝不 import `lib/crypto.ts`**。

原因：crypto.ts 的 `encrypt()` 在未配 `ENCRYPTION_KEY` 时会 **dev fallback 到旧 key**。
脚本若误用它、又在没有真 key 的环境跑 → **用旧 key 重写、静默废掉收口**。
所以脚本独立实现，强制显式指定新 key 来源，**缺失即 abort、绝不 fallback**。

**启动期两道强校验（小B 复评）**：

1. `ENCRYPTION_KEY` base64 解码必须正好 **32 字节**，否则 abort。
2. **新旧 key 不得相等** —— 若有人误把 `ENCRYPTION_KEY` 配成 `base64(sha256(JWT_SECRET))`，
   旧密文会被「新 key」直接解开、统计全落 `new_hit`，但安全上等于没拆分（伪拆分）。
   脚本启动即 `if (NEW_KEY.equals(OLD_KEY)) abort`。

### 4.2 入参（key 不进 argv）

```
node scripts/rekey-encryption.mjs                     # dry-run（默认，不写库）
node scripts/rekey-encryption.mjs --apply             # 实写
node scripts/rekey-encryption.mjs --new-key-env NAME  # 改新 key 来源变量名
```

- **新 key 来源：环境变量**，默认变量名 `ENCRYPTION_KEY`，可用 `--new-key-env` 改名。
  **绝不接受 `--new-key <值>` 把明文塞进 argv** —— argv 会进 shell history、能被 `ps`
  看到。脚本读 `process.env[VAR]`。
- 旧 key 来源：环境变量 `JWT_SECRET`。
- 脚本自带 `.env.local` 解析（与其它脚本一致；`node --env-file` 也兼容）。
- 两个 key 都不打印、不写日志。
- `--apply` 缺省为 dry-run。

### 4.3 加解密口径（与 crypto.ts 路径 A 严格一致）

- 新 key = `Buffer.from(env[VAR], "base64")`（32 字节，**不 sha256**）
- 旧 key = `sha256(JWT_SECRET)`
- decrypt：先新 key、失败回退旧 key
- encrypt：永远新 key，AES-256-GCM，输出 `iv:authTag:cipher`（hex）

### 4.4 逐行分类 + 计数

对范围内每张表每列，分页拉全（每页 1000，防 Supabase 默认上限漏行；按 `id` 升序
稳定排序，防分页过程中漏扫 / 重复扫），逐行归类：

| 分类 | 判定 | dry-run | `--apply` |
| --- | --- | --- | --- |
| `empty` | 空字符串 | 跳过 | 跳过 |
| `plaintext_shape` | 非三段 `iv:tag:cipher` 格式 | **记为问题** | **记为问题，不重写** |
| `new_hit` | 新 key 直接解开 | 计数 | 重写（幂等） |
| `old_hit` | 新 key 失败、旧 key 解开 | 计数 | 重写为新 key 密文 |
| `failed` | 两个 key 都解不开 | **记为问题** | **记为问题，不重写** |

- 写库前**回环自校验**：新密文必须能被新 key 解回原文（`decrypt(新密文)===原文`），
  否则计入 `verify_failed`、不 `update`。dry-run 也跑这步自校验。
- **CAS 写库**：`update` 带 `.eq(col, 读取时的密文)` 条件 —— 仅当该行密文仍是读取
  时的值才写。若 rekey 读取后管理员并发改了 key，CAS 不命中、**不覆盖**，计入
  `concurrent_changed`，提示重跑脚本确认（重跑时该行会落 `new_hit`）。
- `update` 报错计入 `update_failed`。
- `plaintext_shape`：对密钥列，非三段格式极可能是**历史明文 API key**（比密文泄漏
  更严重），不当「正常跳过」—— 列出 id、计入问题。
- **退出码**：`plaintext_shape / failed / verify_failed / update_failed /
  concurrent_changed` 任一 > 0 → **退出码非 0**，不显示「成功」。前四类需人工处理，
  `concurrent_changed` 重跑脚本即可确认。

### 4.5 输出与幂等

- 每表输出 `empty / new_hit / old_hit / plaintext_shape / failed`（`--apply` 再加
  `rewritten / concurrent_changed`），问题行附 id 清单；不打印明文。
- 幂等：已是新 key 的行再跑 `decrypt(新 key 命中)→encrypt(新 key)`，结果等价、无害。

## 五、执行时序（**严格按序**）

```
[1] 生成新 key + 配 Jenkins
    - node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
    - Jenkins → 凭据 → Secret text，ID = agent01_ENCRYPTION_KEY
    - key 自己在密码管理器留存一份（丢 = 用它加密的数据永久解不开）

[2] 备份生产 DB
    - Supabase 控制台 backup，或至少导出三张表的 *_enc 列；留存 ≥ 7 天

[3] 合并 pr-d → master + 部署
    - ⚠ 本次合并必须包含 migrate-trial-agents.mjs 的修复（§3.2），否则部署的
      trial 迁移 stage 会用旧 key 污染 3 行
    - prod 拿到 ENCRYPTION_KEY；新数据 + 3 个 trial agent 起用新 key
    - 存量老数据靠 crypto.ts fallback 链仍可解，业务不中断

[4] 部署后尽快跑 rekey —— 通过 Jenkinsfile.rekey 一次性 job
    ⚠ 运行位置：部署目录只有 standalone 产物、不含 scripts/，不能在部署目录跑。
      用仓库内 Jenkinsfile.rekey：Jenkins 新建 Pipeline job（Script from SCM、
      Script Path = Jenkinsfile.rekey），它 checkout 仓库 + npm ci + 凭据注入
      ENCRYPTION_KEY/JWT_SECRET/SUPABASE_*，跑 scripts/rekey-encryption.mjs。
    - 先「Build with Parameters」不勾 APPLY → dry-run：确认 0 plaintext_shape /
      failed / verify_failed
    - 再勾 APPLY → 实写

[5] 验证
    - 再跑一次 dry-run，要求：old_hit = 0、failed = 0、plaintext_shape = 0，
      所有行都落 new_hit —— old_hit > 0 即说明还有旧密文残留
    - 抽查几个 agent 对话、几个命名 API 调用正常
```

**为什么 rekey 必须在 [3] 之后**：部署前 / dev 环境没有真 `ENCRYPTION_KEY`，脚本会因
§4.1 强校验直接 abort —— 故意如此，防用错 key 白跑。

**rekey 不挂部署流水线（小B 复评决定）**：rekey 是一次性动作，做成 Jenkins deploy
stage 会每次部署空跑、且耦合主流程。按步骤 [4] 使用 `Jenkinsfile.rekey` 建一个**手动触发**
的一次性 Pipeline job；它不挂主部署 pipeline，但保留 Jenkins 构建日志可追溯。

## 六、风险与兜底

| 风险 | 兜底 |
| --- | --- |
| 部署反向污染（trial 迁移 stage 用旧 key） | §3.2 已修 `migrate-trial-agents.mjs` |
| 误把 ENCRYPTION_KEY 配成 sha256(JWT_SECRET)（伪拆分） | §4.1 启动校验新旧 key 不相等，相等即 abort |
| 新 key 进 shell history / 进程列表 | §4.2 key 只从 env 读，不进 argv |
| 看不出旧密文残留 | §4.4 新旧 key 分类计数，§5 验证以 `old_hit=0` 为准 |
| 历史明文密钥被当正常跳过 | §4.4 `plaintext_shape` 记为问题、退出码非 0 |
| 脚本写坏密文 | §2 跑前备份；§4.4 写前回环自校验 |
| 在错误环境用旧 key 重写 | §4.1 强制显式 key 来源 + 长度校验，缺失即 abort |
| rekey 读取后管理员并发改 key | §4.4 CAS 写库：密文已变即不覆盖、计 concurrent_changed，重跑确认 |
| rekey 后误跑 migrate-agent-keys 解不开新密文 | §3.3 已补 fallback decrypt + 废弃注记 |

`JWT_SECRET` rekey 后**不轮换**（项目无 staging 环境，永久保留 fallback 链是合理
选择），退化为仅 JWT 签名用。
