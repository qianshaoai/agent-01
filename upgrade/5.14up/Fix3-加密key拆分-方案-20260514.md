## 一、背景

5.13up 安全审查发现：[lib/crypto.ts:6](../../lib/crypto.ts#L6) 直接用 `JWT_SECRET` 派生 AES-256-GCM 加密 key。这意味着：

1. **职责重叠**：同一个 secret 承担"签 JWT"和"加密 API key"两个职责
2. **轮换不可能**：一旦轮换 JWT_SECRET，所有历史 `_enc` 密文全部失效，对应智能体直接不可用
3. **泄露面更大**：JWT_SECRET 在更多代码路径里被引用，泄露概率高于专用加密 key

Fix 3 的目标：把加密 key 单独抽到 `ENCRYPTION_KEY` 环境变量，与 JWT_SECRET 解耦。

## 二、当前现状（已验证代码）

### 2.1 `lib/crypto.ts` 的三个问题

```ts
// L6  getKey() 用 JWT_SECRET 派生 → 职责重叠
function getKey(): Buffer {
  const secret = process.env.JWT_SECRET;
  if (!secret) throw new Error("JWT_SECRET is not set");
  return createHash("sha256").update(secret).digest();
}

// L25 decrypt() 格式不对 → 原样返回密文当明文穿透 ⚠️
if (parts.length !== 3) return ciphertext;

// L35 decrypt() 解密失败 → 也原样返回密文当明文穿透 ⚠️
} catch {
  return ciphertext;
}
```

**风险**：当上游业务把 `decrypt()` 返回值直接当 `apiKey` 用，且密文损坏 / key 变化时，**密文字符串会被直接当成 API key 发到外部 LLM 服务商**。等于把密文当明文外泄了一遍。

5.13up 小B v2 review 已明确："decrypt() catch must throw not return ciphertext"——本方案承接落地。

### 2.2 加密字段清单（DB 层）

|---|---|---|---|
| `agents` | `api_key_enc` | schema.sql:59 | 平台级智能体的 API key（最多） |
| `tenants` | `openai_key_enc` | migration_v25 | 组织级 OpenAI key（5.7up 引入，目前实际使用少） |
| `user_agents` | `api_key_enc` | migration_v4 | 用户私有智能体的 API key |

## 三、两条实施路径对比

### 路径 A · Fallback Chain，无 DB Migration（推荐）

**思路**：新增 `ENCRYPTION_KEY` 环境变量。`encrypt()` 用新 key；`decrypt()` 先用新 key，失败 fallback 到 JWT_SECRET 派生的旧 key；都失败 **抛错**（不再原样返回密文）。

| 项 | 内容 |
|---|---|
| DB migration | ❌ 不需要 |
| 历史密文处理 | 保留原样，运行时 fallback 解密 |
| 新写入密文 | 用 ENCRYPTION_KEY 加密 |
| 上线方式 | 部署即生效，零停机 |
| 风险点 | ENCRYPTION_KEY 一旦上线必须**永久保留**（删掉会丢新写入的密文） |
| JWT_SECRET 是否可轮换 | ❌ 仍然不可以（轮换会破坏老密文）|
| 跟 5.14up 关系 | 不占 migration 编号，5.14up 走路径 A：v34/v35/v36 |
| 代码改动 | `lib/crypto.ts` + 3 个 decrypt 调用点兜底（greeting / chat / user-agents/chat） + `.env.local.example` + crypto 单元测试 |

**优点**：实施快、风险低、零数据迁移、不阻塞 5.14up
**缺点**：fallback chain 代码常驻；JWT_SECRET 仍然不能轮换（这本来也不是当前痛点）

### 路径 B · 重加密所有历史密文（彻底但风险高）

**思路**：新增 `ENCRYPTION_KEY`。写一次性脚本把三张表里所有 `_enc` 字段用 JWT_SECRET 解密、用 ENCRYPTION_KEY 加密、写回 DB。完成后 `decrypt()` 只用 ENCRYPTION_KEY。

| 项 | 内容 |
|---|---|
| DB migration | ✅ 占 v35（5.14up 后移到 v36/v37）|
| 历史密文处理 | 全部重加密 |
| 新写入密文 | 用 ENCRYPTION_KEY |
| 上线方式 | 需 DB 备份 + dry-run + 维护窗口 |
| 风险点 | 任一行解密失败 / 写入中断 → 该 agent 不可用 |
| JWT_SECRET 是否可轮换 | ✅ 可以（密文跟 JWT_SECRET 解耦） |
| 跟 5.14up 关系 | 占 v35，5.14up 编号顺延：v34/v36/v37 |
| 代码改动 | `lib/crypto.ts` + migration_v35 + 一次性迁移脚本 |

**优点**：彻底、JWT_SECRET 解耦完成
**缺点**：迁移风险大；当前没有 staging 环境跑 dry-run；JWT_SECRET 轮换在当前阶段并非业务需要

### 推荐：**路径 A**

理由：

1. **5.13up 收口方案已经表态**："没有 staging 环境，JWT_SECRET 轮换是优先级很低的事，fallback chain 永久存在可接受"
2. **JWT_SECRET 当前不需要轮换**：不是因为泄露事件触发的修复，而是预防性架构整改
3. **路径 B 的迁移风险显著高于 fallback chain 的代码成本**：3 张表数据量虽然小（百级别），但密文损坏或脚本中断会导致 agent 直接不可用，需要从备份恢复
4. **不阻塞 5.14up 节奏**：5.14up 迁移编号确定走 v34/v35/v36，不再有路径 B 时的"+1"摇摆

## 四、路径 A 实施步骤

### Step 1 · 生成 ENCRYPTION_KEY

```bash
# 32 字节随机值，base64 输出
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

- dev：写入本地 `.env.local`（已 .gitignore 忽略，不入仓）
- prod：写入 Jenkins 凭据中心（**ID = `agent01_ENCRYPTION_KEY`**，Kind = Secret text，Scope = Global），再由 `Jenkinsfile.environment` 段 `credentials('agent01_ENCRYPTION_KEY')` 注入到构建环境，Deploy 阶段通过 `writeFile` 写到目标主机的 `.env.local`
- 加到 `.env.local.example`（占位值 + 注释）

### Step 2 · 重构 `lib/crypto.ts`

完整改写后的结构（不写出实际代码，只说接口）：

```ts
function getNewKey(): Buffer {
  // 优先读 ENCRYPTION_KEY；
  // 解析规则见下方 "ENCRYPTION_KEY 格式与校验"，不合法的处理同样见下方。
}

function getOldKey(): Buffer {
  // 永远用 JWT_SECRET 派生，专门用于解旧密文
}

export function encrypt(plaintext: string): string {
  // 用 getNewKey() 加密，输出格式不变（"iv:authTag:ciphertext"）
}

export function decrypt(ciphertext: string): string {
  if (!ciphertext) return "";

  // 兼容历史明文（无 ":" 三段）
  const parts = ciphertext.split(":");
  if (parts.length !== 3) {
    if (process.env.NODE_ENV !== "production") {
      console.warn("[crypto] decrypt got plaintext-shaped value, length =", ciphertext.length);
    }
    return ciphertext; // 短期内保留兼容
  }

  // 1) 先尝试新 key
  try { return decryptWith(getNewKey(), parts); } catch {}

  // 2) 失败 fallback 到旧 key
  try { return decryptWith(getOldKey(), parts); } catch {}

  // 3) 都失败：抛错（不再原样返回密文）
  throw new Error("decrypt failed: ciphertext could not be decoded with either key");
}
```

**ENCRYPTION_KEY 格式与校验规则**（写死，避免实现时漂移）：

| 维度 | 规则 |
|---|---|
| 格式 | **Base64 编码字符串** |
| 长度 | **base64 decode 后必须正好 32 bytes**（256-bit）|
| 不做 sha256 派生 | base64 decode 后直接当 AES key（**禁止** `sha256(ENCRYPTION_KEY)`，否则跟 ENCRYPTION_KEY 的"高熵随机"语义冲突）|
| Prod 校验失败 | `getNewKey()` 直接 `throw new Error("ENCRYPTION_KEY missing or invalid (must be base64 of 32 bytes)")`，整个进程启动期失败 |
| Dev 校验失败 | `console.warn` + fallback 到 `getOldKey()`（JWT_SECRET 派生）确保本地开发顺畅，但日志明确提示当前不是真正的 Fix 3 状态 |
| Test 环境 | 走 dev 同样规则（fallback + warn）|

判断 prod 的标准：`process.env.NODE_ENV === "production"`。

**关键决策内嵌**：
- `decrypt()` 失败 throw（落地小B 在 5.13up 已提的建议）
- 所有 catch 必须 throw 或返回明确错误标识，**禁止返回 ciphertext 当明文**
- 历史明文兼容路径加 dev warning，便于将来逐步收紧
- ENCRYPTION_KEY 校验规则写死在 `getNewKey()` 里，单元测试覆盖 4 个分支：合法 / 长度不对 / 非 base64 / 未设置

### Step 3 · 调用点防御（**必须同步改，不只是改 lib/crypto.ts**）

`decrypt()` 改成失败 throw 之后，**现有 8 个调用点不能保持原样**。已扫描代码确认 3 处存在硬伤：

| 调用点 | 现状 | 改 throw 后的问题 | 必须改成 |
|---|---|---|---|
| `app/api/agents/[id]/greeting/route.ts:49` | `decrypt(agent.api_key_enc)` 在 try 外 | 直接 **500 Internal Server Error** 给前端 | 用 try-catch 包裹，失败时返回 `{ prologue: null }`（跟原"无开场白"路径一致），同时 server log warn |
| `app/api/agents/[id]/chat/route.ts:57/65` | 在外层 try-catch 内 | 走"服务器错误"路径，**非友好 API key 错误**，且原始错误信息可能漏到前端 | catch 内识别 decrypt 错误，**返回明确业务错误**："该智能体 API key 不可用，请联系管理员" |
| `app/api/user-agents/[id]/chat/route.ts:55` | 在 `streamChat()` 参数构造里 | 错误冒到 **SSE 流**，前端直接看到 "decrypt failed: ciphertext could not be decoded with either key" 这种内部错误文案 | 在调 streamChat 前先单独 `try { apiKey = decrypt(...) } catch { return SSE error chunk }`，错误文案脱敏 |

剩余 5 个调用点（admin/agents 增删改 / admin/tenants 设 openai-key / user-agents 增删改）**主要是 encrypt 而非 decrypt**，需要确认：
- encrypt 抛错时（理论上不会，加密本身不依赖密文格式）路径有兜底
- 写入数据库前的 try-catch 包住 encrypt + insert/update 整段

| C1 `lib/crypto.ts` 重构 | getNewKey 校验 + decrypt 失败 throw + 历史明文兼容 |
| C2 调用点兜底 | 3 个 decrypt 调用点改防御逻辑（greeting / chat / user-agents/chat）|
| C3 单元测试 | crypto.ts 4 个分支用例（合法/长度不对/非 base64/未设置）|

### Step 4 · 自检

- `npm run ci:check` 全过
- 本地 dev：用一个测试 agent 走完整对话流程（验证 fallback 解密路径）
- 本地 dev：故意删 ENCRYPTION_KEY 重启，验证 fallback 到 JWT_SECRET 仍能工作

### Step 5 · 部署（**env 必须先行**）

部署顺序硬性要求：**先配 env，再部署代码**。原因：新代码在 prod 启动时会调 `getNewKey()` 做 ENCRYPTION_KEY 校验，没配会直接 throw 进程不起来；如果代码先部署、env 后配，会出现一段"启动失败"的窗口期。

正确顺序（Jenkins + PM2 部署链路）：

| 步骤 | 操作 | 验证 |
|---|---|---|
| 1 | **本机** 生成 ENCRYPTION_KEY（32 bytes base64） | `node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"` 输出 44 字符 |
| 2 | **Jenkins → 凭据管理 → 添加凭据**：Kind=Secret text，Scope=Global，ID=`agent01_ENCRYPTION_KEY`，Secret 粘贴步骤 1 的 base64 | 凭据列表能看到 `agent01_ENCRYPTION_KEY`（System 全局作用域）|
| 3 | 合并 Fix 3 PR（含 Jenkinsfile 改两行：environment 段加 `ENCRYPTION_KEY = credentials('agent01_ENCRYPTION_KEY')`，Deploy 阶段 writeFile 加 `ENCRYPTION_KEY=${env.ENCRYPTION_KEY}`）| Jenkins 自动触发 build |
| 4 | Build 通过后 Deploy 阶段会把 `ENCRYPTION_KEY=<value>` 写入 `/www/wwwroot/o.qianshao.ai/.env.local` | SSH 到目标主机 `cat .env.local | grep ENCRYPTION_KEY` 能看到该行 |
| 5 | PM2 reload `o-app`（Jenkinsfile 已自动做 `pm2 reload ecosystem.config.js --update-env`） | `pm2 status` 显示 online + uptime 重置 |
| 6 | Health Check 通过（Jenkinsfile 自动 `curl localhost:3000`）+ 手测 chat / greeting 流程 | 全部 200，无 decrypt failed 日志 |
| 7 | 24h 监控（见下方）| 见下方监控项 |

**禁止操作**：
- ❌ Jenkins 凭据没加就合并 Fix 3 PR（Pipeline 在 environment 段会因 `credentials('agent01_ENCRYPTION_KEY')` 找不到直接挂）
- ❌ 部署后再删/改 Jenkins 凭据中心里的 `agent01_ENCRYPTION_KEY`（见 §6.1 路径 A 回滚约束）
- ❌ 直接 SSH 到目标主机改 `.env.local` 里的 ENCRYPTION_KEY（下次 Jenkins Deploy 会被 writeFile 覆盖，且会与 Jenkins 凭据不一致）

**24h 监控项**：
- 服务器 PM2 日志（`pm2 logs o-app`）里 `decrypt failed` 错误数量（应接近 0；非 0 时定位是哪个 agent 的历史脏数据）
- chat / greeting / user-agents/chat 三个接口的错误率（不应高于上线前基线，可通过 audit_logs / Supabase 监控）
- 日志里 `getNewKey` 是否走过 dev fallback warn（prod 走过即异常，说明 Jenkins 注入失败）

### Step 6 · 长期治理

- `lib/crypto.ts` 加 dev mode 监控：统计新 key 命中 vs 旧 key fallback 命中比例
- 每写入一条新密文都用新 key → 时间推移后 fallback 命中率会下降（每个 agent 编辑一次就迁移一条）

## 五、备份与 Dry-run（路径 B 备选时才用）

**如果小B 拍板走路径 B**，必须先做以下准备：

### 5.1 DB 备份

- Supabase Dashboard → Database → Backups → 触发即时备份
- 验证可恢复（在备份列表里能看到带时间戳的 snapshot）

### 5.2 Dry-run 脚本

新建独立脚本 `scripts/dry-run-rekey.ts`：

```text
读取三张表所有 _enc 非空行
  -> 用 JWT_SECRET 派生 key 解密
  -> 用 ENCRYPTION_KEY 派生 key 加密
  -> 验证：解密结果可以用 new key 解回原文
  -> 仅打印统计（成功 N / 失败 M / 历史明文 K / 空 P），不写入 DB
  -> 把失败行的 (table, row_id) 输出到 JSON 报告
```

### 5.3 失败行处理

- 解密失败 → 该行密文损坏或格式错误，**手动逐条核查**
- 历史明文 → 视为兼容数据，迁移时跳过 _enc 字段（用 new key 重加密）
- 全部成功才允许执行真实迁移

### 5.4 真实迁移脚本

`supabase/migration_v35_rekey_encryption.sql` + 对应 Node 脚本（SQL 只做表层标记，实际加解密在 Node 完成因为 PostgreSQL 没有 AES-GCM 内置函数）。

### 5.5 路径 B 回滚

- 必须从 DB 备份恢复（_enc 字段已被改写，没法靠 revert 代码恢复）
- 此期间业务停摆约 5-15 分钟
- 严禁在没有完成 dry-run + 没有备份的情况下执行

## 六、回滚预案

### 6.1 路径 A 回滚

| 情况 | 处理 |
|---|---|
| 上线后发现 decrypt throw 率异常高 | 立即 revert 代码 → fallback 走 try/return ciphertext 旧行为（短期权宜）|
| ENCRYPTION_KEY 误删 | **不能 revert ENCRYPTION_KEY**——已用它加密的新密文无法解；必须把删掉的 key 找回来恢复环境变量 |
| 想退回到只用 JWT_SECRET | 部署后写入的新密文需先重加密回 JWT_SECRET，否则丢失。等同于做一遍路径 B |

**结论**：路径 A 上线后 ENCRYPTION_KEY **永久不可删**。env 文档要明确标注。

### 6.2 路径 B 回滚

| 情况 | 处理 |
|---|---|
| 迁移中断（部分行 rekey 完，部分未完）| 从备份恢复整张表，重跑 dry-run |
| 迁移完成但代码出 bug | 从备份恢复 + revert 代码 |
| 维护窗口超时 | 视恢复时间，决定继续迁移 or 回滚 |
