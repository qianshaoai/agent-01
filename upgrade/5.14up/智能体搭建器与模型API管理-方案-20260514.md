## 二、前置依赖（重要）

### 2.1 Fix 3 加密 key 拆分必须先做（阻塞项）

工作流 §3.1 `model_providers.api_key_enc` 用现有 `lib/crypto.ts` 加密。当前 `lib/crypto.ts` 仍是 5.13up 安全方案里标识的"用 JWT_SECRET 派生 AES key"的状态——Fix 3 已 memory 记录"上线前再做"。

**如果先做本方案再做 Fix 3**：所有 `model_providers.api_key_enc` 的密文要重新加密一遍，迁移脚本风险翻倍（既要兼容现有 `agents.api_key_enc`，又要兼容刚做完的 `model_providers.api_key_enc`）。

**结论**：Fix 3 必须在 PR-A 开干前先做掉。提到 5.14up 实施周期的第 0 天。

如果业务侧坚持先做 5.14up，必须在工作流 §11 的风险表里独立加一行"Fix 3 延期导致的二次密钥迁移成本"，并写明谁负责后续重做。

### 2.2 logs.status 扩展为 ('success','error','aborted')（migration_v34）

**当前实际情况**（已验证代码）：
- [supabase/schema.sql:110](../../supabase/schema.sql#L110) `CHECK (status IN ('success', 'error'))`
- [app/api/agents/[id]/chat/route.ts:406](../../app/api/agents/%5Bid%5D/chat/route.ts#L406) 已写 `status: wasAborted ? "aborted" : "error"`

**这是一个已存在的 silent bug**：chat route 写入 `'aborted'` 时违反 CHECK，insert 失败但被 `try/finally` 吞掉，导致用户中断的对话**根本没有日志**。本方案的测试聊天也会复用同一套 logging，必须先修。

**结论**：新建 `supabase/migration_v34_logs_status_aborted.sql`，扩展 CHECK 为 `('success', 'error', 'aborted')`，在 PR-A 之前合入。短工作量（~10 行 SQL）。

## 四、对工作流 §13 决策点的小A 推荐

| ID | 问题 | 工作流推荐 | 小A 推荐 | 理由 |
|---|---|---|---|---|
| D-1 | API Key 全平台共用 vs 组织独立 | 第一阶段全平台 | **同意**（全平台 provider） | 组织级 key 涉及配额隔离 / 计费分摊，复杂度跳两个量级。先全平台跑通 |
| D-2 | org_admin 是否能搭建智能体 | 第一阶段不开放 | **同意不开放** | 权限和费用风险均不可控（org_admin 能创建任意 system prompt） |
| D-3 | 草稿发布默认可见范围 | 默认本组织 | **改为：默认仅本人可见**，发布后单独点"扩大可见范围" | 防止误发布 → 全平台用户立刻看到测试期智能体。本组织可见仍属于"广播"风险 |
| D-4 | API Key 存 provider 还是 copy 到 agents | 存 provider | **同意 provider** | DRY、密钥轮换不需要逐个改 agent |
| D-5 | 测试聊天是否扣额度 | 不扣用户额度 | **同意不扣，但加管理员级测试配额**（每 provider 每天 100 次测试上限） | 工作流 §11 提到的"测试聊天消耗费用"风险，单靠 rate limit 不够，要有明确硬上限 |
| D-6 | 是否做知识库 | 本次不做 | **同意延后** | 知识库 = 独立的 4-6 人天工作量，跟本方案合做必然延期 |

**新增小A 拍板项 D-7（工作流没列）**：

| ID | 问题 | 小A 推荐 |
|---|---|---|
| D-7 | 草稿改完再次发布，agent 已被工作流绑定，要不要做"发布前 diff 提示" | **第一阶段不做** UI，但**发布日志要含 diff 字段**（before/after JSON），方便审计追溯。UI 在 P4 路线图里做。 |

## 五、工作流没覆盖的边界（小A 补）

### 5.1 草稿测试聊天的 SSE 超时定义

工作流 §4.3 写"单次测试超时 60 秒"。但 SSE 流是分 chunk 的——是 **首字节超时** 60 秒，还是 **全流超时** 60 秒？

**推荐**：首字节 30 秒超时（拉不到第一个 chunk 即报错），全流不设硬超时（让 GPT-4o 长输出能完整跑完），但 UI 加"取消"按钮，用户主动 abort。

### 5.2 published_agent_id 反查的一致性

工作流 §3.2 `published_agent_id` 是 FK to `agents`。但 `agents` 可能被超级管理员**直接在原后台删除**（绕开搭建器），此时草稿的 `published_agent_id` 会 SET NULL。

**推荐**：
- 草稿列表 UI 显示"已发布的 agent 已被删除"红色提示，允许用户点击"重新发布"
- agents 表的删除接口加一个软警告（"该智能体由草稿 X 发布而来，确认删除？"）

### 5.3 现有 agents.api_key_enc 何时下线

工作流 §6.1 配置合并优先级保留了 `agents.api_key_enc` 作为兜底。但长期来看这是反 DRY 的。

**推荐**：本方案 **不下线** agents.api_key_enc（向后兼容）。但在 5.14up 变更记录里记一条"长期 TODO：当所有现有智能体都迁移到 provider_id 后，agents.api_key_enc 改为 NULL，下个版本删除该列"。

### 5.4 provider 删除时的引用扫描

工作流 §11 提到"被引用时禁止硬删除"，但没明确扫描范围。

**推荐**：删除前 SQL 扫这三处引用：
```sql
SELECT COUNT(*) FROM agents WHERE provider_id = $1;
SELECT COUNT(*) FROM agent_drafts WHERE provider_id = $1;
SELECT COUNT(*) FROM logs WHERE detail->>'provider_id' = $1 LIMIT 1;
```
前两个有引用 → 拒绝删除（建议改禁用）；第三个仅警告（历史日志，不阻止）。

### 5.5 测试聊天的 system_admin 边界

工作流 §7.1 说 system_admin 可"只读 + 测试" provider，但**没说 system_admin 能不能用自己搭的草稿做测试聊天**——他能选 provider，但 provider 的 key 他看不到，调用是后端代理。

**推荐**：system_admin 可以测试聊天，**调用通过后端代理**（前端发 draft_id + message，后端读 provider key，前端永远拿不到 key）。这点工作流隐含但没明说。

### 5.6 灰度策略

工作流没说上线 strategy。**推荐**：
- 后台新入口"模型接入" / "智能体搭建"用环境变量 `NEXT_PUBLIC_FEATURE_AGENT_BUILDER=true` 控制显隐
- 开发期默认 true，生产首次部署设 false，小范围放给特定 super_admin 账号验证 1 周后再设 true
- DB 迁移不走 feature flag（迁移是不可逆的，必须先部署）

## 六、风险加固（在工作流 §11 基础上补）

| 风险 | 工作流提及 | 小A 补充缓解 |
|---|---|---|
| API Key 泄露 | ✅ | 加：前端不要把 provider 列表请求结果存 localStorage / SWR 持久缓存，每次拉新 |

| provider 频繁切换导致 chat 异常 | ❌ 未提 | 加：chat route 增加 5 秒 provider 配置缓存（避免每次 chat 都查 model_providers）|

## 七、上线与回滚

### 7.1 上线步骤

```
1. Day 0 部署：Fix 3 加密 key 拆分 + migration_v34_logs_status_aborted，验证现有功能不退化（顺手修了 chat aborted 静默丢日志的 bug）
2. Day 2 部署：PR-A（migration_v35_model_providers + 模型接入 UI 隐藏）
3. Day 4 部署：PR-B（migration_v36_agent_drafts + 搭建器 UI 隐藏）
4. Day 6 部署：PR-C（测试聊天 + 发布，**发布出的 agent 默认 enabled=false**）
5. Day 7 部署：PR-D（chat 链路兼容），回归通过后批量把 PR-C 时发布的 agent 改成 enabled=true
6. Day 8 部署：PR-E + 翻开 feature flag，灰度给指定 super_admin
7. Day 14 验证无回归 → 全员翻开 flag
```

### 7.2 回滚预案

| 部署阶段 | 出问题怎么办 |
|---|---|
| Day 2-4 出问题 | 直接 revert PR，DB 迁移保留（迁移是兼容的，不删表只关接口） |
| Day 5-6 出问题 | revert PR-C；已写入 `agents` 的草稿发布产物默认 `enabled=false`，不对用户可见，DB 数据保留，待修复后继续处理 |
| Day 7 出问题（最危险）| revert PR-D 立刻回退 chat 链路；同时**立即批量将 `published_from_draft_id IS NOT NULL` 的新 agent 改回 `enabled=false`**（或关闭 feature flag），避免新 agent 已可见却聊不通的体验断层；旧 agent 不受影响 |
| Day 8 出问题 | feature flag 关掉，所有 builder 入口立即隐藏；DB 数据保留，等修复后重开 |
