# 体验版聊天体验优化方案

**文档日期**：2026-04-30
**适用范围**：`/trial` 体验版聊天模块（trial agents、trial_messages、trial_conversations）
**文档目的**：统一本轮 A+B 优化的开发边界，避免在改善体验的过程中越位重写正式版、引入大改、或借机扩展无关功能

---

## 一、本轮目标

本轮不是新功能迭代，是一次**"补齐聊天软件标准动作"**的小步收口。

目标只有 3 个：

1. 把昨天调试 PDF/xlsx 时暴露的体感问题（"不知道附件解析有没有成功""不知道 bot 在不在思考"）补齐反馈链路
2. 让体验版聊天具备一线产品该有的标准动作：重命名、删除、重发、编辑、拖拽、搜索
3. 不影响正式版任何接口、任何 UI、任何数据库表

一句话概括：

**只在 `/trial` 这条线里加体感反馈和聊天管理动作，不重写聊天主链路、不动正式版、不动数据库结构。**

## 三、本轮 5 个批次

每批做完独立 commit + 本地自测，确认无回归才进下一批。

### 批次 1 · 反馈链路
- A1 附件解析进度：发送时如果有需要后端提取的文件，气泡里显示"附件解析中…"占位
- A2 解析结果标签：每个附件 chip 加 ✓/⚠ 状态角标
- A6 附件大小前置校验：前端选择文件时立即校验，超限不上传
- A5 退出登录二次确认（原生 confirm 即可）
- B12 流式 typing 光标：流式中 assistant 气泡末尾显示闪烁 ▎

**约束**：
- A1+A2 的"状态"不入库，仅本次会话内存里维护
- 只在体验版聊天页加，不动 `app/agents/**` 的正式版

**SSE 帧时机（强约束）**：
- 后端 `/api/trial/chat` 当前已发：`meta.chat_id`（最早一帧）→ delta 流 → `done`
- 本轮新增一帧：`data: {meta: {attachment_status: [{file_name, ok, reason?}]}}`
- 该帧**必须**在 `extractTextFromUrl` 全部完成之后、`for await (const chunk of streamChat...)` 进入第一次 `controller.enqueue(delta)` **之前**发出
- 没有 file 类附件需要后端提取时（图片 / native 平台 / 无附件），**不发**该帧
- 前端收到该帧后：把 pending 占位 chip 的状态翻成 ✓ 或 ⚠；A1 的"解析中…"占位在收到此帧后或第一段 delta 后清除（哪个先到）

### 批次 2 · 聊天管理（重命名 / 删除）
- A4 hover 聊天行显示「⋯」菜单：重命名（inline 编辑）/ 删除（confirm）
- 后端：**新增** `PATCH /api/trial/conversations/[id]`（改 title）；**保留**已有的 `DELETE /api/trial/conversations/[id]`（trial_messages 走表的 ON DELETE CASCADE 自动级联）

**约束**：
- 删除时**保留**原 `trial_messages` 的 ON DELETE CASCADE 行为，不写额外 SQL
- 重命名只允许改 `title` 字段，不允许扩展为改 agent_id 等其他列
- 删除后如果是 activeChatId，前端回退到"无活跃聊天"（即 activeChatId = null）。**禁止**沿用现 `page.tsx` deleteChat 里 `remaining[0].id` 的自动 fallback 行为——本批次必须把那段改掉

### 批次 3 · 重发 + 编辑
- A3 hover assistant 气泡显示"重新生成"：复用上一条 user message 重发
- B10 hover user 气泡显示"编辑"：编辑后从该位置之后清空、重新生成

**前置依赖（必做）**：
- `GET /api/trial/conversations/[id]/messages` 在 **本地分支**（line 62-77）的 map 里追加 `id` 字段（取 `trial_messages.id`）。Coze 回退分支无 id（line 178-183），保持 undefined
- 前端消息类型 `Msg` 增加可选 `id?: string` 字段；只有带 id 的消息才显示"编辑"按钮

**新增接口（必做）**：
- `DELETE /api/trial/messages/[id]?from=true` — 删除该消息及之后所有消息
  - 鉴权：先查 `trial_messages → chat_id → trial_conversations.user_id`，必须等于当前用户
  - 实现：用消息的 `created_at` 作为基准，删 `chat_id = X AND created_at >= ?` 的所有行
  - **不带 `?from=true` 直接返回 400**（本轮只支持"该条及之后"模式，不预留单条删除）

**约束**：
- 不引入"消息分支/树形会话"概念。编辑/重新生成 = 调上述 DELETE → 然后调现有 `POST /api/trial/chat`（带相同 chat_id）重发
- 重新生成 = 找到最后一条 assistant 前面的 user 消息 → 调 `DELETE /api/trial/messages/[userId]?from=true` **连同该 user + 之后所有一起删** → 用该 user 的 content + attachments 重新 POST chat。**严禁只删 assistant 不删 user**——`POST /api/trial/chat` 自带 user 入库，会造成 DB 里出现 `user → user(dup) → assistant` 历史污染
- 编辑 = 删除该 user 消息及之后所有 → 用新内容 POST chat
- "重新生成"按钮只对**最后一条** assistant 消息显示，避免越改越乱
- 前端调用 DELETE 失败时不允许继续 POST chat（防止历史错乱）

**消息 id 获取链路（必做）**：
- 现有 `POST /api/trial/chat` 是 SSE，user/assistant 消息在 `finally` 里入库，**不**回传 message id 给前端。前端乐观插入的气泡本地无 id
- 不改动 chat 路由的流时序（避免引入并发/时序复杂度）。改用：**流结束后前端立即调用 `GET /api/trial/conversations/[chatId]/messages` 刷新本地 body**，拿到带 id 的消息列表覆盖乐观气泡
- "流结束"覆盖三种路径：
  1. **正常完成**：收到 `done` 帧后立即 GET 刷新
  2. **后端报错**：收到 `error` 帧后立即 GET 刷新（assistant partial 也已入库）
  3. **用户 abort（前端 `AbortController.abort()`）**：客户端拿不到 `done` 帧，需要在 `catch (AbortError)` 或 finally 里**延迟 500ms 后 GET 刷新一次**（给后端 finally 里的 insert 留入库时间）
- 三种路径共用同一个刷新函数；如果刷新返回的 messages 长度未覆盖到末尾两条乐观气泡，再延迟 500ms 重试一次，最多重试 2 次后放弃
- "编辑 / 重新生成"按钮 enable 条件：
  - **流式中**：末尾两条气泡无 id，按钮一律不显示
  - **重新生成**：仅当最后一条 assistant 和其前一条 user **都**有 id 时启用，否则按钮置灰并 tooltip 提示"消息保存中，稍候"
  - **编辑**：仅当该 user 消息有 id 时启用
- 历史消息附件（`B13`）也走这条刷新链路，保证 chat_id 切换 + 流式结束两种入口都拿到 id

### 批次 4 · 浏览/输入提速
- B7 拖拽上传：拖文件到聊天区任意位置 → 复用现有上传逻辑
- B8 聊天搜索：左侧聊天历史顶部加搜索框，**纯前端**对 title 做模糊匹配
- B9 回到最新按钮：滚动离底超过 N 像素时浮动"↓"按钮

**约束**：
- B7 实现要点：从现有 `onFilePicked(e: ChangeEvent)` 抽出 `handleFiles(files: File[])` 共用函数。`onFilePicked` 和 `onDrop` 都调用它。**禁止**让 `onDrop` 伪造 `ChangeEvent` 去喂 `onFilePicked`
- B7 拖拽时图片粘贴 / 现有点击上传链路不动
- B8 不查后端、不查消息内容，**只**对已加载的 chats 数组的 title 做 includes 过滤
- B9 不实现"未读消息数"，仅滚动锚点

### 批次 5 · 附件细节
- B11 图片点击 lightbox 放大；文件 chip 点击下载（直接打开 `att.url`）
- B13 历史消息附件回显：进入老聊天后，user 气泡里附件 chip 显示正确并可点击

**约束**：
- B11 不引入新依赖：图片 lightbox 自己写一个固定遮罩层（fixed inset-0 + 点击关闭），不用 react-image-lightbox 等库
- B13 验证 `trial_messages.attachments` 反序列化是否正确，必要时只在前端兜底渲染，**不**改后端入库格式
- B13 **可点击范围限定**：只保证 4.30 之后入库的本地 `trial_messages` 附件可点击（带 url）。`messages route.ts` 的 Coze 回退分支返回的旧附件只有 `file_id`，没有公开 URL，前端**仅显示 chip + 文件名**，不可点击；这是已知限制不算 bug

## 四、全局约束（贯穿 5 批次）

### 1. 不动正式版

`/agents/[id]/chat`、`/api/agents/[id]/chat` 这条主链路 0 改动。任何"顺手优化"正式版的想法本轮一律拒绝。

### 2. 不动数据库

不允许：
- 新建表
- 新建索引
- 新增 / 删除列
- 修改约束
- 新增触发器或 RPC

trial_messages、trial_conversations 现有结构足够支撑本轮所有功能。

### 3. 不引入新依赖

本轮不允许 `npm install` 任何 runtime 依赖（dev 依赖也尽量不加）。需要的：
- 拖拽：原生 HTML5 drag-and-drop API
- lightbox：自写 div fixed 遮罩
- 搜索：原生 `Array.filter + String.includes`

### 4. 不改全局样式

- 不动 `tailwind.config`、`globals.css`
- 不动现有色板（深蓝 #002FA7 系）
- 新增 UI 优先复用现有组件结构（hover 菜单、modal、chip 等）

### 5. 不做 C 档功能

明确**不在**本轮：暗色主题、跨智能体草稿、导出 Markdown、语音输入、tab 标题动态。这些等老板确认。

### 6. 错误处理保守优先

新接口 / 新动作出错时：
- 后端：返回明确 code（400/404/500）+ msg，不静默吞错
- 前端：toast 或 inline 提示，**不阻塞**用户继续聊天

### 7. 不引入新的状态管理

继续用现有 `agentStates` Map + `useState`，不要为搜索/拖拽/编辑新建 zustand / context。

### 8. 每批一个 commit + 本地自测

每批做完：
1. 本地用桌面 ceshi 文件夹的 4 份测试文件复跑一遍
2. 看 console 无 error
3. 提交单批 commit，commit message 标明批次（如 `feat(trial): batch1 反馈链路`）
