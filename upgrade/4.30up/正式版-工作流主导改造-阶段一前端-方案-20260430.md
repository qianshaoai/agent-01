# 正式版 · 工作流主导改造 · 阶段一前端方案

**文档日期**：2026-04-30
**适用范围**：正式版用户主页 `app/page.tsx`，不含 trial / admin / login / 后端接口
**文档目的**：把主页从“分类/智能体入口”调整为“工作流入口”，阶段一**只改前端界面，不动后端、不动 DB**

---

## 一、需求原文对齐

阶段一只完成用户明确提出的 3 件事：

1. 取消原有“智能体菜单/分类菜单”，将左侧内容改为“我的工作流”；“全部”栏展示当前账号可用工作流，下面展示工作流名称；点击工作流名称后进入当前工作流详情。
2. “智能体展示栏”只展示工作流绑定智能体；用户点击对应工作流后，只展示该工作流中的智能体。
3. 删除“我的智能体”栏。

一句话概括：

**主页主入口从“找智能体”改成“选工作流”；智能体只作为所选工作流的一部分出现。**

## 三、Mock 登录态处理

`app/page.tsx` 当前在 `/api/me` 返回未登录时，会进入 mock 兜底分支。阶段一删除 `agents`、`displayAgents`、`categories` 等状态后，该分支必须同步收口。

本阶段采用 **方案 A：保留 mock 分支，但改成工作流主页 mock**。

**原因**：
- 不改变现有“未登录也能看到 demo UI”的开发/演示行为。
- 避免把“前端界面改造”扩大成登录跳转逻辑调整。
- `lib/mock-data.ts` 已在白名单内，必要时可以补充 mock workflow。

**实现要求**：
- mock 分支只允许设置：
  - `setUser(...)`
  - `setNotices(...)`
  - `setWorkflows(mockWorkflows 或 [])`
  - `setHasAnyWorkflow(...)`（如该 state 保留）
  - `setActiveWorkflowId(null)`
- mock 分支必须删除：
  - `setAgents(...)`
  - `setDisplayAgents(...)`
  - `setCategories(...)`
  - `/api/agents` 相关兜底数据
- 如果 `lib/mock-data.ts` 暂不补 mock workflow，则 mock 未登录状态下右侧“全部”显示空工作流态；这可接受。
- 不采用方案 B（未登录直接跳 `/login`）。如果后续要调整登录策略，单独立项。

## 四、页面结构改造

### 1. 左侧侧栏改为“我的工作流”

**数据源**：复用 `GET /api/workflows`，不传 `categoryId`，获得当前账号可见的全部工作流。后端 0 改动。

**侧栏内容**：
- 标题：`我的工作流`
- 第一项：`全部`
- 下方列表：当前账号可见的所有工作流名称

**删除 / 不再使用**：
- 原分类菜单
- 分类分组逻辑
- `catGroups` 相关读取、写入、编辑弹窗
- `activeCategory` / `switchCategory` / 按分类拉取工作流和智能体的逻辑

**点击行为**：
- 点击 `全部`：右侧显示当前账号全部可见工作流卡片网格。
- 点击某个工作流名称：设置 `activeWorkflowId = wf.id`，右侧进入该工作流详情。

**状态约定**：
- `activeWorkflowId = null` 表示“全部”视图。
- 页面初始加载完成后默认保持 `activeWorkflowId = null`，即默认选中“全部”。
- 不新增工作流独立路由，本阶段仍在 `app/page.tsx` 内部切换视图。

**排序约定**：
- 前端不新增排序规则，信任 `/api/workflows` 当前返回顺序。
- 现有接口已按 `sort_order ASC` 返回；因为阶段一不改 `lib/types.ts`，前端不要直接依赖 `wf.sort_order` 字段。

### 2. 右侧“全部”视图

点击侧栏 `全部` 时，右侧展示当前账号全部可见工作流卡片网格。

**卡片字段**：
- 工作流名称
- 工作流描述
- 工作流分类文本（若接口返回 `category`）

**交互**：
- 点击工作流卡片 = 点击左侧同名工作流，进入该工作流详情。

**布局**：
- 响应式网格：`sm:1 / md:2 / lg:3` 列。
- 沿用当前页面的白底卡片、蓝色主题、8-16px 圆角风格，不做视觉重构。

**不做**：
- 搜索 / 筛选 / 排序切换
- 封面图
- 进度条
- 最近使用
- 推荐 / 热门 / 收藏 / 置顶

### 3. 点击工作流后的详情视图

点击某条工作流后，右侧进入该工作流详情。

**详情内容**：
- 工作流名称 / 描述
- 工作流步骤列表
- 每个步骤里的智能体操作入口

**重要口径**：
- 现有代码里的工作流详情是“步骤时间轴 + `WorkflowStepButton`”，不是完整 `AgentCard` 网格。
- 阶段一不重写详情视图，只保证其中出现的智能体入口来自 `wf.workflow_steps[].agents`。
- 若后续老板要求在详情里展示完整智能体卡片，那属于阶段二 UI 改造。

**智能体显示规则**：
- 只显示当前工作流步骤中绑定的智能体。
- 同一智能体如果出现在多条工作流里，在不同工作流详情中各自出现，互不去重。
- 未绑定智能体、智能体已删除、外链未配置等状态沿用现有 `WorkflowStepButton` 行为。

### 4. 删除独立智能体展示栏

删除主页中独立的“智能体展示”模块。

**删除范围**：
- `AgentCard` 网格
- `displayAgents`
- `/api/agents?categoryId=...` 分类过滤调用
- `categoryCache`
- `filtered`
- `agentCollapsed`
- `agent_collapsed` localStorage

**保留范围**：
- `/agents/[id]` 详情页
- `/api/agents/[id]/chat` 聊天接口
- 工作流步骤按钮跳转 `/agents/<agent_code>` 的能力

### 5. 删除“我的智能体”栏

根据最新需求，用户自建智能体功能从主页剔除。

**删除范围**：
- “我的智能体”整块 UI
- `UserAgentCard` 列表
- 添加 / 编辑 / 删除用户自建智能体弹窗
- `userAgents`
- `showMyAgentsSettings`
- `editingUA`
- `uaForm`
- `uaSaving`
- `uaError`
- `/api/user-agents` fetch / POST / PATCH / DELETE 调用
- `my_agents_collapsed` localStorage

**保留范围**：
- 后端 `app/api/user-agents/**` 不动。
- 数据库不动。
- 已有用户自建智能体数据本阶段不删除，只是不在主页展示和管理。

## 十一、审核结论与修改意见

对照最新需求，原方案需要收口：

1. 删除“工作流分组”设计。需求只要求 `全部 + 工作流名称列表`，不需要 `+ 新建分组`，也不需要 `workflowGroups`。
2. 明确删除用户自建“我的智能体”模块。原方案只说删除“我的智能体栏”，但没有把 `/api/user-agents` 调用、弹窗、状态一起列入删除范围，现在已补齐。
3. 不再保留 `/api/agents` 调用。主页不再有独立智能体展示，工作流详情已通过 `/api/workflows` 内嵌的 `workflow_steps[].agents` 获得智能体入口。
4. 默认选中“全部”，因此 `activeWorkflowId = null` 必须代表全部视图，而不是加载后自动选第一条工作流。
5. “智能体卡”口径改成“智能体入口 / WorkflowStepButton”。阶段一不重写工作流详情 UI，避免超出前端小改范围。

本方案现在可以作为阶段一前端改造执行依据。
