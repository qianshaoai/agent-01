# 工作流流程图可视化 · 阶段三行动方案

**日期**：2026-04-27
**依据**：`upgrade/4.27up/工作流流程图可视化优化方案-20260427.md` 第 10 章 阶段三
**前置**：阶段一、阶段二均已稳定上线，且**收集到管理员明确的"步骤排序不便"反馈**
**目标**：降低多步骤工作流的调序成本
**预计工作量**：上移/下移按钮 0.5 天

---

## 1. 本阶段目标

**第一轮只做**：上移 / 下移按钮（零依赖、乐观更新、失败时强制 `load()` 重拉 + 补偿尝试）

**第一轮不做**：
- 拖拽排序（**默认不进入第一轮实施**）
- 拖拽必须等"按钮排序上线后用户仍明确反馈不够用"再独立立项评估，且评估通过后才动

阶段三**不**重构阶段一、阶段二的产出。

### 2.2 拖拽排序（**第一轮不做**）

拖拽排序**不属于**阶段三第一轮实施范围。理由：

- 上移 / 下移已经能解决"调序不便"的核心痛点
- 拖拽涉及批量 `PATCH` 局部成功 / 失败，回滚和数据一致性比按钮排序复杂得多
- 引入 `@dnd-kit` 等依赖偏离总优化规划

**触发条件**：等按钮排序在线上稳定使用一段时间，用户明确反馈"按钮排序不够用"，再独立立项评估，本方案不在此处展开实现细节。

## 5. 实现要点（不构成代码强约束）

### 5.1 上移 / 下移逻辑
- 上移：当前步骤与上一步交换 `step_order`，发 2 次 PATCH
- 下移：当前步骤与下一步交换 `step_order`，发 2 次 PATCH
- 操作期间所有上移 / 下移 / 删除 / 插入按钮禁用，按钮上展示 loading icon

### 5.2 失败处理（必须严格执行）

两次 PATCH 交换 `step_order` 存在"第一次成功、第二次失败"的局部成功风险。**前端 state 回滚 ≠ 数据库回滚**，必须按以下流程处理：

1. **第一次 PATCH 成功** → 进入第二次 PATCH
2. **第二次 PATCH 失败** → 启动失败处理流程：
   - **(a) 尝试补偿**：发一次"反向 PATCH"把第一次的步骤 `step_order` 写回原值
   - **(b) 无论补偿成功还是失败，强制 `await load()`** 重新从后端拉取最新 `workflows` 数据，覆盖前端 state
   - **(c) 提示文案**：
     - 补偿成功 → toast：`"排序未生效，已恢复原顺序"`
     - 补偿失败 → toast：`"排序可能未完全保存，请刷新确认"`
3. **第一次 PATCH 就失败** → 直接 toast `"排序失败"`，无须补偿（数据库未变更）

**约束**：
- 失败处理流程**禁止**简化为"只回滚前端 state"
- `load()` 是真实的源真相获取动作，必须执行
- 补偿动作只尝试一次，不重试，避免引入循环风险

### 5.3 乐观更新
- 前端先按目标顺序重排数组渲染
- 两次 PATCH 都成功保留新顺序
- 任意失败按 §5.2 流程处理

### 5.3 拖拽
拖拽不在本轮范围，禁止实现，需独立立项。

## 6. 验证方式

### 6.1 功能验证
1. 准备一个 5+ 步骤的工作流
2. 逐步上移 / 下移，每次刷新验证顺序持久化
3. 第一步上移按钮 / 最后一步下移按钮置灰
4. 触发**第一次 PATCH** 失败 → toast `"排序失败"`，顺序不变
5. 触发**第二次 PATCH** 失败（局部成功场景）→ 补偿成功时 toast `"排序未生效，已恢复原顺序"`；补偿失败时 toast `"排序可能未完全保存，请刷新确认"`；两种情况都必须看到 `load()` 重拉触发
6. 阶段一、阶段二功能不受影响

## 7. 风险点

| 风险 | 缓解 |
|------|------|
| 上移 / 下移并发触发顺序错乱 | 操作期间所有相邻按钮禁用 + loading |
| 两次 PATCH 一成功一失败导致脏数据 | §5.2 补偿写回 + 强制 `load()` 重拉，**禁止**只回滚前端 state |
| 补偿写回也失败 | 用 `"排序可能未完全保存，请刷新确认"` 显式提示用户介入 |
| 修改幅度失控 | 仅在阶段一节点渲染内插入按钮和状态，不动节点结构 |
| 拖拽诱惑顺手做了 | 文档强制：拖拽**不在**第一轮范围；任何拖拽相关依赖、组件、UI 暗示均禁止引入 |
