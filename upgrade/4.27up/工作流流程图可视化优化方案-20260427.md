# 4.27 优化方案：工作流流程图可视化与步骤编排体验升级

**日期**：2026-04-27  
**优先级**：P1  
**适用范围**：`agent-01` 后台工作流管理模块  
**目标页面**：`app/admin/workflows/page.tsx`  
**目标用户**：平台管理员、系统管理员、组织管理员  

---

## 1. 背景与问题

当前后台“工作流管理”已经具备创建工作流、添加步骤、绑定智能体、插入步骤、启停步骤等能力，但整体呈现仍偏列表式。

在实际使用中，管理员需要反复阅读每一条步骤内容，才能判断：

1. 工作流一共有几步。
2. 每一步之间的先后关系是什么。
3. 哪些步骤绑定了智能体。
4. 哪些步骤尚未绑定智能体。
5. 插入新步骤应该放在哪两个步骤之间。
6. 某个工作流是否存在断点、空配置或禁用步骤。

这导致工作流配置不够直观，尤其是在步骤数量较多、步骤说明较长、需要频繁插入智能体时，管理员操作成本偏高。

本方案目标是为工作流管理增加一套“流程图式”的可视化编排视图，让管理员不用细读列表也能快速理解整个工作流结构。

---

## 2. 建设目标

本次优化目标不是重做工作流引擎，也不是引入复杂流程编排系统，而是在现有数据结构和接口基础上增强后台可视化体验。

核心目标：

1. 让工作流步骤关系一眼可见。
2. 让“插入步骤 / 插入智能体”的位置更直观。
3. 让未绑定智能体、禁用步骤等异常状态明显可见。
4. 保留现有列表操作能力，避免影响已习惯旧页面的管理员。
5. 不改变数据库结构，不改变前台用户使用流程。
6. 优先做低风险、可快速落地的前端增强。

---

## 3. 总体方案

在后台工作流管理页中，对每个展开的工作流增加“流程图视图”。

建议采用双视图模式：

1. **流程图视图**
   用于快速理解步骤顺序、节点类型、绑定智能体状态和插入位置。

2. **列表视图**
   保留当前已有列表式步骤管理方式，用于兼容旧习惯和查看较长说明。

初期可以默认展示流程图视图，也可以在页面顶部提供“流程图 / 列表”切换，管理员可按需选择。

---

## 4. 页面结构设计

### 4.1 工作流列表层

工作流卡片头部保持现有结构：

1. 工作流名称
2. 所属分类
3. 可见范围
4. 启停状态
5. 步骤数量
6. 复制、编辑、删除等操作

点击展开后，进入工作流详情区。

### 4.2 工作流详情区

展开区建议拆成三部分：

1. **视图切换栏**
   - 流程图
   - 列表

2. **流程图画布**
   - 横向展示步骤节点
   - 节点之间用箭头连接
   - 箭头中间提供插入按钮

3. **底部操作区**
   - 添加步骤
   - 可选：展开全部说明

---

## 5. 流程图节点设计

每一个工作流步骤展示为一个节点卡片。

节点建议包含：

1. 步骤序号
2. 步骤标题
3. 执行类型
4. 绑定智能体名称
5. 启用 / 停用状态
6. 编辑按钮
7. 删除按钮

执行类型用颜色和图标区分：

| 执行类型 | 显示样式 | 说明 |
|---|---|---|
| `agent` | 蓝色，Bot 图标 | 智能体执行 |
| `manual` | 橙色，User 图标 | 人工执行 |
| `review` | 紫色，Eye 图标 | 人工审核 |
| `external` | 灰色或绿色，Wrench / ExternalLink 图标 | 外部工具 |

### 5.1 智能体绑定状态

对于 `exec_type = agent` 的步骤，需要突出展示绑定状态：

1. 已绑定智能体：显示智能体名称。
2. 未绑定智能体：显示醒目提示“未绑定智能体”。
3. 绑定智能体已不存在：显示错误提示“智能体已删除或不可用”。
4. 绑定外链型智能体：显示外链标识。

### 5.2 禁用步骤状态

禁用步骤不应从流程图中消失，否则管理员无法理解配置。

建议表现：

1. 节点降低透明度。
2. 标记“已停用”。
3. 连接线仍保留，但颜色变浅。

---

## 6. 连接线与插入交互

流程图中步骤之间使用箭头连接。

每两个节点之间放置一个小型插入按钮：

1. 第一步之前：允许插入到最前面。
2. 任意两步之间：允许插入到指定位置。
3. 最后一步之后：允许追加步骤。

插入按钮点击后复用现有步骤弹窗逻辑：

1. 自动设置 `insertAfterOrder`。
2. 新步骤保存后调用现有重排逻辑。
3. 保持当前工作流展开状态。

这样能直接降低“我应该把智能体插到哪一步”的判断成本。

---

## 7. 交互细节

### 7.1 节点点击

建议：

1. 点击节点主体：打开编辑步骤弹窗。
2. 点击编辑图标：打开编辑步骤弹窗。
3. 点击启停图标：切换步骤启用状态。
4. 点击删除图标：弹出确认框后删除。

### 7.2 智能体快捷绑定

第一阶段建议先复用现有编辑弹窗，不新增复杂的节点内下拉绑定。

第二阶段可以增强：

1. 未绑定节点上直接显示“绑定智能体”按钮。
2. 点击后弹出智能体选择浮层。
3. 选择后直接保存当前步骤。

### 7.3 横向滚动

当步骤较多时，流程图采用横向滚动。

要求：

1. 节点宽度固定，避免内容撑乱。
2. 节点标题超长时截断显示，悬浮显示完整标题。
3. 说明文字不在节点中大段展示，只显示摘要。
4. 移动端保持横向滚动，不强行压缩成难读的小卡片。

---

## 8. 数据与接口影响

本方案第一阶段不改数据库。

继续使用现有字段：

1. `workflows`
2. `workflow_steps`
3. `agents`
4. `workflow_categories`
5. `resource_permissions`

继续使用现有接口：

1. `GET /api/admin/workflows`
2. `POST /api/admin/workflows/:id/steps`
3. `PATCH /api/admin/workflow-steps/:id`
4. `DELETE /api/admin/workflow-steps/:id`

不新增接口，不改变已有接口返回结构。

---

## 9. 实施边界

### 9.1 本次建议做

1. 后台工作流展开区新增流程图视图。
2. 节点展示步骤类型、绑定智能体、启停状态。
3. 节点之间增加插入按钮。
4. 保留原列表视图。
5. 支持步骤较多时横向滚动。
6. 对未绑定智能体、已停用步骤做视觉提示。

### 9.2 本次暂不做

1. 拖拽排序。
2. 条件分支。
3. 并行节点。
4. 节点连线自定义。
5. 流程运行状态追踪。
6. 工作流版本管理。
7. 引入 React Flow 等大型流程图库。

原因：

1. 当前工作流数据模型是线性步骤，不支持分支和并行。
2. 直接引入复杂流程图库会扩大改造范围。
3. 用户当前痛点主要是“看不清顺序”和“插入智能体麻烦”，轻量流程图足够解决第一阶段问题。

---

## 10. 推荐实施步骤

### 阶段一：轻量流程图视图

目标：快速解决可读性问题。

实施内容：

1. 在 `app/admin/workflows/page.tsx` 中增加流程图渲染区域。
2. 提取步骤节点渲染函数。
3. 增加执行类型视觉映射。
4. 将现有 `openInsertStep` 接入节点间插入按钮。
5. 保留原步骤列表作为列表视图。

验收结果：

1. 管理员展开工作流后可以看到横向流程图。
2. 每一步的类型、绑定智能体、启停状态清晰可见。
3. 可以在任意两步之间插入步骤。

### 阶段二：智能体绑定体验增强

目标：减少编辑弹窗往返。

实施内容：

1. 未绑定智能体节点增加“绑定智能体”快捷入口。
2. 节点内显示绑定智能体的类型和名称。
3. 绑定外链型智能体时显示外链标识。
4. 智能体失效时明确提示。

验收结果：

1. 未绑定步骤能一眼识别。
2. 管理员能更快完成智能体绑定。

### 阶段三：排序与批量维护增强

目标：提升复杂工作流维护效率。

实施内容：

1. 支持上移 / 下移按钮。
2. 可选：拖拽排序。
3. 排序后统一调用现有 `PATCH /api/admin/workflow-steps/:id` 更新 `step_order`。
4. 增加保存中状态，避免重复操作。

验收结果：

1. 管理员可以快速调整步骤顺序。
2. 重排后刷新页面顺序保持一致。

---

## 11. UI 建议

整体视觉应保持后台管理系统风格，避免做成过重的营销式视觉。

建议：

1. 画布使用浅灰背景。
2. 节点使用白色卡片，边框保持克制。
3. 不使用大面积渐变。
4. 不使用复杂装饰背景。
5. 节点圆角不超过当前系统风格。
6. 图标使用现有 `lucide-react`。
7. 连接线使用细线和箭头。
8. 插入按钮常态弱化，悬浮时突出。

节点宽度建议：

1. 桌面端：220px 到 260px。
2. 移动端：200px 左右，横向滚动。
3. 节点高度保持稳定，避免因为文案长短导致流程图上下跳动。

---

## 12. 风险与注意事项

### 12.1 不要破坏现有列表操作

现有列表视图已经承载完整管理能力。流程图第一阶段应该是增强，而不是替换。

### 12.2 不要提前设计复杂分支

当前数据模型是线性工作流，流程图也应表达线性结构。不要在 UI 上暗示支持条件分支或并行节点。

### 12.3 不要引入重型依赖

第一阶段用普通 React + Tailwind + lucide 即可完成。除非后续明确要做拖拽、缩放、分支编排，否则不建议引入 React Flow。

### 12.4 注意编码问题

当前项目中部分中文文件在终端输出中存在乱码现象。实施时应避免用会重写整文件编码的方式处理已有文件，优先使用编辑器或精确补丁修改。

---

## 13. 验收标准

第一阶段完成后，应满足以下标准：

1. 后台工作流展开后存在流程图视图。
2. 流程图按 `step_order` 正确展示步骤顺序。
3. 节点能显示步骤标题、执行类型、绑定智能体状态。
4. 禁用步骤有明显但不过度干扰的状态提示。
5. 未绑定智能体的步骤有醒目提示。
6. 任意步骤之间可以点击插入按钮新增步骤。
7. 新增步骤后顺序正确，刷新页面后顺序不乱。
8. 原有列表管理能力仍可使用。
9. 不新增数据库字段。
10. 不影响前台用户查看和使用工作流。
