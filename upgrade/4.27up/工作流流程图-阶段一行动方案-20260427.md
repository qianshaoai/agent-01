# 工作流流程图可视化 · 阶段一行动方案

**日期**：2026-04-27
**依据**：`upgrade/4.27up/工作流流程图可视化优化方案-20260427.md` 第 10 章 阶段一
**目标**：管理员展开工作流后，不读列表也能一眼看出步骤数 / 顺序 / 类型 / 绑定状态 / 启停 / 异常
**前置**：无
**预计工作量**：0.5 - 1 天

---

## 1. 本阶段目标

完成"流程图视图"作为工作流展开区的**新增视图**（与列表视图并存）：

1. 展开工作流即默认看到流程图
2. 节点按 `step_order` 横向排列，箭头连接
3. 节点显示：序号、标题、执行类型徽章、绑定智能体、启停状态、编辑/删除按钮
4. 节点之间提供"+ 插入步骤"按钮（复用现有 `openInsertStep` + `insertAfterOrder`）
5. 异常状态可视化：未绑定智能体 / 智能体已删除 / 步骤已停用
6. 顶部"流程图 / 列表"切换 Tab，列表视图保留全部原有能力

## 5. 实现要点（不构成代码强约束，仅指导）

### 5.1 节点结构（建议）
```
┌─────────────────────────┐
│ ① [类型徽章] 标题        │
│ ─────────────────────── │
│ 🤖 案例撰写专家          │
│ 已启用 · [✎] [⊘] [🗑]   │
└─────────────────────────┘
       ↓
       ⊕ ← 插入按钮（hover 突出）
       ↓
┌─────────────────────────┐
│ ② ...                    │
└─────────────────────────┘
```

### 5.2 连接线
- 用 SVG 三角箭头或 CSS `::after` 三角；不引入第三方
- 禁用步骤前后的连接线 `opacity` 降为 30-50%

### 5.3 横向滚动容器
- `overflow-x-auto`，节点 `flex-shrink-0`
- 移动端不强行压缩

### 5.4 视图切换状态
- `viewMode: "flow" | "list"`，按 workflow.id 维度记忆当前选择
- 初始默认 `"flow"`

## 6. 验证方式

### 6.1 功能验证
1. 进入 `/admin/workflows`
2. 展开任意工作流（建议挑一个有 5+ 步骤、有 1 个未绑定智能体、有 1 个已停用的工作流）
3. 对照"§3 验收清单"逐项打勾

### 6.2 回归验证
1. 切到列表视图，对所有按钮（编辑步骤、删除步骤、启停切换、插入步骤）做一次完整操作
2. 各操作行为必须与改造前完全一致
3. 切回流程图，状态同步刷新

### 6.3 质量门禁
```
npm run ci:check
```
零 error。

## 7. 风险点

| 风险 | 缓解 |
|------|------|
| 节点高度不稳定导致流程图上下抖 | §5.3 节点高度固定，长标题 / 长说明截断 |
| 横向滚动条与外层 Card 滚动冲突 | 流程图容器自带 `overflow-x-auto`，不做嵌套竖向滚动 |
| 视图切换后状态不同步 | `viewMode` 状态在父组件，切换不卸载子树 |
| 中文文件被整文件重写引入乱码 | 全程用精确补丁式编辑，不 Write 整文件覆盖 |
| 列表视图与流程图操作 race 不一致 | 两视图共用同一份 `workflows` state，单源数据 |
