# 开场白展示（Greeting / Prologue）方案

**文档日期**：2026-05-06
**适用范围**：正式版聊天页 `app/agents/[id]/page.tsx` + 新接口 `app/api/agents/[id]/greeting/route.ts`
**文档目的**：进入新对话空白态时，自动拉取 bot 在原平台后台配置的"开场白"（Coze 的 onboarding prologue），渲染为第一条 assistant 消息

> **状态：已上线**。本文档为事后归档，方便后续审查 / 回滚 / 扩展到其它平台。

---

## 一、需求

聊天页进入空白态时（新对话、还没发过消息），按现状是空的——但平台后台 bot 通常配了开场白（"您好，我是 xxx，可以帮你 xxx"）。希望把这个开场白同步过来主动展示，对齐其它 AI 产品（如 ChatGPT / Claude / 体验版）的体感。

---

## 二、范围

### 已实现（Coze 平台）

- **新增** `app/api/agents/[id]/greeting/route.ts`：调 Coze `GET /v1/bot/get_online_info`，提取 `onboarding_info.prologue` 返回前端
- **改动** `app/agents/[id]/page.tsx`：
  - 新对话空白态触发 `useEffect` 拉 prologue
  - 渲染为第一条 assistant 消息，id 形如 `greeting-${agentCode}`
  - sessionStorage 缓存 30 分钟，避免来回切聊天频繁打 Coze
  - 用户发送第一条真消息后，乐观插入接在 greeting 后；DB 拉历史时不含 greeting，bot 看到的是干净上下文
  - 导出 Markdown 时跳过 greeting 虚拟消息

### 未做（其它平台）

- 元器（yuanqi）：理论上有 assistant detail 接口，待补
- Dify / Qingyan：各自有开场白机制，待补
- 后端接口对非 Coze 平台**静默返回 `{ prologue: null }`**——前端拿到就当没开场白，不报错

---

## 三、关键设计

### 1. 不入库 / 不扣配额

- greeting 仅前端渲染，**不写 DB**（`messages` 表 不出现 greeting- 行）
- 每次进新对话都重拉一次（命中缓存的话直接读）
- 不调 chat 路径，不消耗 quota

### 2. 不污染 bot 上下文

- greeting 只活在前端 messages 数组里
- 用户发消息时 chat 路由从 DB 拉历史 → 看不到 greeting
- bot 永远收到干净的对话历史，不会被"自己之前说过开场白"这件事影响

### 3. 容错策略

后端任何错误（agent 不存在 / 平台非 Coze / token 解密失败 / Coze 4xx/5xx / 超时 / JSON 不对）一律静默返回 `{ prologue: null }`。前端拿到 null 就当没开场白处理，不弹错。

### 4. 缓存

- key：`agent_greeting:${agentCode}`
- 范围：sessionStorage（关闭浏览器 tab 即清）
- TTL：30 分钟
- 失效后重新打 Coze

---

## 四、依赖

后端解密链路：`agents.api_key_enc` 用 `lib/crypto.ts` 的 `JWT_SECRET` 解密。**dev 本地的 `JWT_SECRET` 必须跟线上 Jenkins `agent01_JWT_SECRET` 一致**，否则解密失败 → token 错误 → Coze 返 4101 → prologue 取不到。
