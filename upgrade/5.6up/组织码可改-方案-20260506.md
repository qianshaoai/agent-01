# 组织码可改方案

**文档日期**：2026-05-06
**状态**：**草案**，未开工
**适用范围**：admin 后台 `/admin/tenants` 编辑表单 + 后端 PATCH 接口 + DB 事务级联更新

---

## 一、需求

当前 `tenants.code` 创建后不能改（admin 后台编辑组织时这个字段是只读的）。希望允许管理员修改组织码，改完后所有该组织下的用户、智能体可见性、配额等关联都跟着同步过去。

---

## 二、关键事实

### tenants.code 是软主键，被多张表引用 — 实施前必须做"全库引用扫描"

> ⚠ 下表是**当前能确认**的引用列；实际项目中 `tenant_code` / 组织码 这种列还可能散落在更多业务表里。
> **实施前置工作（必做）**：在 supabase 跑全库扫描确认完整列表：
>
> ```sql
> -- 找所有名字像 tenant_code / org_code / org 字符串字段
> SELECT table_name, column_name, data_type
> FROM information_schema.columns
> WHERE table_schema = 'public'
>   AND (column_name ~* '(tenant|org).*code' OR column_name = 'tenant_code');
> ```
>
> 同时 grep 代码 `tenant_code` 的所有读写位置确保没漏。

**当前已识别的引用**：

| 表 | 列 | 引用方式 | 备注 |
| --- | --- | --- | --- |
| `users` | `tenant_code` | **疑似软引用**（实施前需查 `information_schema.table_constraints` 确认是否真有 FK） | schema.sql 里没看到 `REFERENCES tenants(code)` |
| `tenant_agents` | `tenant_code` | 外键 → `tenants.code`，当前 `ON DELETE CASCADE` | 改 FK 时**保留 `ON DELETE CASCADE`**，仅追加 `ON UPDATE CASCADE` |
| `resource_permissions` | `scope_id`（当 `scope_type='org'`）| 软引用，无 FK | rpc 里手工 UPDATE |
| `logs` | `tenant_code` | 软引用 | **是否要改见 §三-3 历史归属决策** |
| `departments` | `tenant_code` | 软引用 | rpc 里手工 UPDATE |
| `teams` | `tenant_code` | 软引用 | rpc 里手工 UPDATE |
| `notices` | `tenant_code` | 软引用 | rpc 里手工 UPDATE |
| `tenant_categories` | `tenant_code` | 软引用（如该表存在）| 实施前 grep 确认 |
| `user_groups` | `tenant_code` | 软引用 | rpc 里手工 UPDATE |
| `admins` | 关联组织码（org_admin 角色）| 字段名待确认 | grep `tenant_code` / `org_code` 确认 |
| `workflows.visible_to` | 旧版逗号分隔组织码兼容数据 | 文本搜索匹配 | 已 migration 到 resource_permissions，但**遗留数据**可能仍存逗号格式，需 grep 验证 |

> 改 code 不是单表 UPDATE，是**跨 N 张表的级联同步**，N 必须先扫清楚。

### JWT 里存了 tenantCode（**普通用户 + 组织管理员都要管**）

`lib/auth.ts` 签的 token payload 里 `tenantCode = 'CESHI'`。改了之后老 cookie 仍然带 `'CESHI'`，但 DB 里已经是 `'NEWCODE'`。后续请求拿 user.tenantCode 去查 tenants 会查不到。

**两种 token 都要清**：
- 普通用户 token（`type: "user"`）
- 组织管理员 token（`type: "admin"`，可能含 tenantCode 字段或通过 admin 表关联组织）

**`middleware.ts`** 中间件目前同时验普通用户 cookie 和 admin cookie，方案的"force_relogin_at"机制必须**两个登录态都生效**——避免改完组织码后 org_admin 还能用旧 admin token 操作。

### 企业用户首次登录用组织码当密码

`app/api/auth/login/route.ts` 有逻辑：企业用户首登用 `tenant_code` 作为初始密码。改了 code 后**老用户不影响**（已经用过初始密码登录过），但首登行为会受影响（极小概率场景）。

### 外部系统可能硬编码

如果有其它系统（前哨内部别的服务、文档、Excel 名单）写死了 tenant_code，改了之后那些地方失效。**这是业务风险，不在代码层处理。**

## 三、实现方案

### 1. 后端：新增/扩展 PATCH `/api/admin/tenants/[id]`

加 `code` 字段允许更新。处理流程：

1. **校验**：
   - 新 code 格式 — **沿用现有创建组织时的规则**（4-8 位英文字母）；如果改这次需要放宽到 16 位 / 含数字，**必须同步放宽创建规则**，避免"创建不能用、修改能用"的产品割裂
   - 新 code 不能与其它已存在 tenants.code 重复
   - 旧 code 必须存在，且不是 `PERSONAL` 保留字
2. **事务（必须用 PostgreSQL RPC 函数；禁止手动 begin/commit）**：

   写一个 `rename_tenant_code(old TEXT, new TEXT)` PostgreSQL 函数（语言 plpgsql），函数体内是隐式单事务，supabase-js 通过 `db.rpc("rename_tenant_code", { old, new })` 调用。

   函数体里依次 UPDATE：
   ```sql
   UPDATE tenants              SET code = new WHERE code = old;
   UPDATE users                SET tenant_code = new WHERE tenant_code = old;
   UPDATE tenant_agents        SET tenant_code = new WHERE tenant_code = old;
   UPDATE departments          SET tenant_code = new WHERE tenant_code = old;
   UPDATE teams                SET tenant_code = new WHERE tenant_code = old;
   UPDATE resource_permissions SET scope_id    = new WHERE scope_type = 'org' AND scope_id = old;
   UPDATE notices              SET tenant_code = new WHERE tenant_code = old;
   UPDATE user_groups          SET tenant_code = new WHERE tenant_code = old;
   -- ... 全库扫描确认的所有引用表都加进来
   -- logs 是否更新见 §三-3
   ```

   **不允许**用 supabase-js 在 client 侧串多次 UPDATE 假装事务（任何一步失败前面已落库）。
3. **返回**：成功后返回新 tenant 行

### 2. 后端：FK 处理

**实施前置**：先用以下 SQL 查清楚 `users.tenant_code` 和 `tenant_agents.tenant_code` 当前**真实**的 FK 状态（schema.sql 与线上 DB 可能因为历次 migration 已不一致）：

```sql
SELECT
  tc.table_name,
  kcu.column_name,
  rc.update_rule,
  rc.delete_rule,
  ccu.table_name AS ref_table,
  ccu.column_name AS ref_column
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.referential_constraints rc
  ON rc.constraint_name = tc.constraint_name
JOIN information_schema.constraint_column_usage ccu
  ON rc.unique_constraint_name = ccu.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND ccu.table_name = 'tenants'
  AND ccu.column_name = 'code';
```

**根据真实状态决定 migration**：

- 如果 `users.tenant_code` 实际上**没有 FK**（schema.sql 表象） → **不需要在 users 上加 FK**，rpc 里手工 UPDATE 即可
- 如果 `tenant_agents.tenant_code` 已是 `ON DELETE CASCADE`（schema.sql 表象） → migration 写：
  ```sql
  ALTER TABLE tenant_agents
    DROP CONSTRAINT tenant_agents_tenant_code_fkey,
    ADD CONSTRAINT tenant_agents_tenant_code_fkey
      FOREIGN KEY (tenant_code) REFERENCES tenants(code)
      ON UPDATE CASCADE ON DELETE CASCADE;  -- ⚠ 删除语义保留 CASCADE，仅追加 ON UPDATE
  ```
  **不要把 `ON DELETE CASCADE` 改成 `ON DELETE RESTRICT`**——会改变现有"删除组织时一并清空 tenant_agents"的行为。
- 其它没 FK 的表（resource_permissions / logs / departments / teams / notices / user_groups / ...）继续在 rpc 里手工 UPDATE

### 3. 后端：JWT 失效策略（普通用户 + 组织管理员都覆盖）

改了 code 后，该组织所有用户的 cookie 里还是旧 tenantCode。**必须强制踢出**（方案 a），方案 b（容忍）会让所有依赖 tenantCode 的查询持续报错。

**实施**：
- `users` 表 + `admins` 表（如果 admin 表也存了组织关联）各加一个 `force_relogin_at` timestamp 列；改 code 后**把该 tenant 下所有 user 和 admin** 的 `force_relogin_at = NOW()`
- `lib/auth.ts` 验证 token 时多查一次目标账号的 `force_relogin_at`；如果 token 的 `iat < force_relogin_at` → 视为失效
- `middleware.ts` 中间件统一处理两种 token：
  - 普通用户 cookie：失效 → 清 cookie + 重定向 `/login`
  - admin cookie：失效 → 清 cookie + 重定向 `/admin`（登录页）
- 用户/管理员第二次访问时被踢出，提示"组织信息已变更，请重新登录"

### 3.5 logs 表更新策略（决策点）

`logs.tenant_code` 是否随之改：

| 选项 | 优点 | 缺点 |
| --- | --- | --- |
| **方案 i · 改**（rpc 一并 UPDATE）| 按"当前组织"过滤 logs 时不会遗漏旧 code 的历史日志 | 历史日志失真——一年前 `CESHI` 做的操作日志变成 `NEWCODE` 做的，审计角度不严谨 |
| **方案 ii · 不改** | 历史日志保留"事实归属"，符合审计语义 | 后台按组织过滤日志要 OR 旧 code 的支持；改了一次 code 就出现这种麻烦 |
| **方案 iii · 改 + 留痕** | 改 logs.tenant_code，同时在 logs 里**追加一条**"组织码变更"事件（旧 code → 新 code，操作人，时间）做审计 | 折中方案，工作量稍大 |

**默认推荐 iii**——改完 + 加一条 audit 事件，既保查询体验又留审计痕迹。**实施前请用户确认 i / ii / iii。**

### 4. 前端：admin 后台编辑表单

`app/admin/tenants/page.tsx` 编辑组织 modal 里：
- 把 `code` 字段从只读 → 可编辑
- 字段下方加红字警告：
  - "修改组织码会影响该组织下所有用户登录、配额、权限关联。改完后所有该组织用户必须重新登录。"
  - "如果有外部系统（Excel 名单、其它服务）记录了旧组织码，请同步修改。"
- 保存按钮加二次确认 modal："确认要把 `CESHI` 改为 `XXXX` 吗？"

### 5. 登录页提示

如果用户用老 code 登录失败（不存在），错误提示加一行："如果你的组织最近修改过组织码，请联系管理员获取最新组织码。"

## 六、已知风险与不做

### 风险

1. **外部系统硬编码 tenant_code**：本方案不处理（业务方人肉同步）
2. **登录被踢的体验**：所有该组织用户立刻退出，可能正在工作的人骂街——admin 端必须在改之前发通知。前端可以加一行警告"建议在低峰期操作"
3. **企业初始密码逻辑**：老用户已用过 → 不影响；从未登录过的新用户首次登录密码就是新 code，admin 要分发新 code 给他们

### 不做

- 不做"码改了之后旧 code 自动重定向"——加这层逻辑收益小，复杂度高
- 不**新增独立的**审计表 / 审计专属 schema —— 但**会在现有 `logs` 表里追加一条"组织码变更"事件**（属于 §三-3.5 推荐方案 iii 的一部分），用于事后追查。
- 不做"批量导入旧 code → 新 code 映射"——用不到

## 七、风险与回退

### 改动落地后能否回滚？

可以，但要逆向跑一次 rpc：`rename_tenant_code('NEWCODE', 'OLDCODE')`，再回退代码。`migration_v23.sql` 加的 ON UPDATE CASCADE 不影响业务，可以保留不回退。

### 灾难场景

如果 rpc 函数有 bug 把数据搞乱（比如某张表 UPDATE 漏了一行）：
- supabase 自带 PITR (Point in Time Recovery)，可恢复到操作前几分钟的状态
- 强烈建议**改之前手动备份 6 张表的关键数据**：
  ```sql
  CREATE TABLE _backup_users_YYYYMMDD       AS SELECT * FROM users;
  CREATE TABLE _backup_tenant_agents_YYYYMMDD AS SELECT * FROM tenant_agents;
  -- ... 6 张表
  ```
