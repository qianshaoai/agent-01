## 二、关键决策点（先定再开干）

### Q1：docx 在前端生成还是后端生成？

| 选项 | 优点 | 缺点 |
| --- | --- | --- |
| **前端**（用 `docx` npm 包）| 不调接口，不扣配额；与现有 .md 导出实现风格一致；离线可用 | bundle size +200KB（gzip 后约 60-80KB）；用户首次打开聊天页要多下点 JS |
| **后端**（新接口 `/api/conversations/[id]/export?format=docx`）| 不增前端体积；后续加 PDF 也走同一接口好维护 | 多一个接口，要鉴权 / 校验 / 防滥用；用户必须在线；图片要服务端 fetch 处理 |

**推荐前端方案**：
- 现有 .md 导出已经是纯前端，**保持一致性**
- 避免引入新接口的鉴权 / 限流复杂度
- 200KB 体积可接受（项目已经引了 react-markdown / rehype-highlight 等更大的包）
- 用 dynamic import + chunk 分割：`docx` 库只在用户**点击导出**时才加载，不影响首屏

### Q2：用什么库

调研结果：

| 库 | star | 体积 (gzip) | 中文支持 | 图片支持 |
| --- | --- | --- | --- | --- |
| **`docx`** | 4k+ | ~60KB | ✓（系统字体） | ✓（嵌 ArrayBuffer） |
| `mammoth` | 5k+ | - | 仅"读"docx，不"写"，**不适用** | - |
| `office-clone-template` | - | - | 不维护 | - |

**选 `docx`**。它有完整 Builder API，支持段落、标题、表格、图片、列表、超链接，文档完整。

### Q3：图片附件如何处理

聊天里图片附件的 Markdown 用 `![filename](supabase_public_url)` 嵌入。docx 需要：
- `fetch(supabase_url)` 拿 ArrayBuffer
- 用 `docx` 的 `ImageRun` 嵌入

**选项**：
- **A · 真嵌入图片**：每张图 fetch + 等 → 导出时间增加（图多时几秒），但文档完整
- **B · 只放 URL 链接**：跟 .md 类似，文档里写"📎 图片：[name](url)"
- **C · 选项 A 但加进度提示 + 失败降级**：导出按钮变 loading，下方显示"正在打包 3/5 张图..."；**单张图片失败（CORS / 过期 / 超大）不阻断整体导出**，在 docx 里降级为"图片：filename — 链接 url"，并在导出完成时 UI 提示"3 张图嵌入成功，2 张失败已降级为链接"

**推荐 C**：用户体验好，且失败兜底策略清晰。

**失败降级实现**：
- 单张图 fetch 用 try/catch + 5s 超时
- 失败的图记下 filename 和原因（CORS / timeout / >10MB / 非图）
- docx 段落降级为：`📷 图片附件: filename（嵌入失败：原因，URL: url）`
- export 函数 return 时附带 `partialFailures: [{ filename, reason }]` 数组，UI 据此 toast

### Q4：UI 怎么呈现"选格式"

| 选项 | 描述 | 工作量 |
| --- | --- | --- |
| **i · 按钮 hover dropdown** | 鼠标悬停才出现 / 点击展开。桌面 OK，移动端不友好 | 2h |
| **ii · 点击弹出 popover 菜单** | 点按钮 → 弹小菜单 → 选格式 → 下载 | 3h（写 popover 组件 + click-outside 关闭） |
| **iii · 点击进入弹窗模态** | 大模态框列出格式 + 每种说明 | 4h（过度设计） |

**推荐 ii**：popover 是这种"二级选择"的标准模式，移动端也工作良好（点击展开）。

## 五、实施细节

### 1. 引入 `docx` 库（dynamic import）

```ts
async function generateDocxBlob(messages: Msg[], agentName: string): Promise<Blob> {
  const docx = await import("docx"); // 动态加载，仅在用户点 "Word" 才下
  // ... 用 docx.Document / docx.Paragraph / docx.TextRun / docx.ImageRun 构造
  return docx.Packer.toBlob(doc);
}
```

`webpack` / Turbopack 会自动把 `docx` 打成独立 chunk，首屏不加载。

### 2. 抽公共 export service

新建 `lib/export-conversation.ts`：

```ts
// ⚠ CLIENT-ONLY: 该模块用 Blob / URL.createObjectURL / 动态 import docx / fetch 图片，
// 严禁从 server route 或 Server Component 导入。
"use client";

export type ExportFormat = "markdown" | "docx";

export type ExportProgress = (stage: string) => void; // 例 "正在打包 3/5 张图..."

export type ExportResult = {
  blob: Blob;
  filename: string;
  partialFailures: Array<{ filename: string; reason: string }>; // 图片嵌入失败列表
};

export async function exportConversation(
  format: ExportFormat,
  ctx: {
    agentName: string;
    agentCode: string;
    conversationTitle: string;
    messages: Msg[];
  },
  onProgress?: ExportProgress
): Promise<ExportResult> { ... }
```

**约束**：
- 文件首行写 `"use client"`，并加注释明确 client-only
- 模块顶层**禁止** `import { Document } from "docx"`——docx 必须在 generateDocx 函数内部用 `await import("docx")` 动态加载，让 webpack/Turbopack 把它打成独立 chunk
- markdown 路径继续走纯字符串拼接，不走 docx 库

聊天页只调这一个函数，把"按格式分发 + 文件名构造 + 下载触发 + 进度上报 + 失败列表"封装好。

### 3. UI · popover

- 导出按钮 onClick 改成 toggle popover 显隐
- popover 里两个选项：
  - 📝 Markdown (.md)
  - 📄 Word (.docx)
- 点击 → 关闭 popover → 调 exportConversation（传 onProgress 回调）→ 下载
- 导出过程中按钮显示 loading spinner + 文字（来自 onProgress 回调，例如 "正在打包 3/5 张图..."）
- 完成后如果 partialFailures.length > 0 → toast 提示"X 张图嵌入失败（CORS / 超时），已降级为链接"

**popover 定位规则**（移动端必须明确）：
- 桌面（≥sm）：右对齐导出按钮的右边缘，宽度 auto
- 移动端（<sm）：右对齐 viewport 右边 16px（防溢出），最大宽度 240px
- 点遮罩 / 点 popover 外 / Esc 关闭
- popover 内点 option 关闭并触发导出
- z-index 高于 header 的 z-30，建议 z-50

| 段落 | 内容 |

| Paragraph | "导出时间：YYYY-MM-DD HH:mm" |

| 每条消息 |  |
| - Heading 2 | "👤 用户  ·  时间" / "🤖 助手  ·  时间" |

| - Paragraph | "📎 附件：filename"（文件附件，如有） |
| - Paragraph | 正文（多段保留换行） |
| - Paragraph (italic gray) | "⚠ 已停止生成"（如有） |
| (分隔符) |  |
| Footer Paragraph | "由「人机协同工作舱」生成 · {agent_code}" |

#### Markdown → docx 保真等级（明确声明）

聊天里 assistant 经常返回 Markdown（表格、有序/无序列表、代码块、加粗、斜体、链接、引用块）。docx 转换的保真等级**本期目标**：

| 元素 | 处理 |
| --- | --- |
| 段落换行（`\n\n`）| **保留** — 转成新 Paragraph |
| 行内换行（`\n`）| **保留** — Paragraph 内 break |
| 加粗 `**x**` | **不解析** — 文档里仍是 `**x**` 字面量 |
| 斜体 `*x*` / `_x_` | **不解析** |
| 链接 `[text](url)` | **不解析** |
| 列表 `- x` / `1. x` | **不解析** — 字面量保留，至少不丢内容 |
| 代码块 ` ```` ` | **不解析** — 字面量保留 |
| 表格（GFM `\| a \| b \|`）| **不解析** — 字面量保留（用户在 Word 里看到的就是文本表格，可以阅读但不是真正的 docx 表格） |
| 标题 `#`、`##` | **不解析** — 字面量保留 |

**理由**：完整解析 Markdown → docx 节点要引第二个库（如 `markdown-it` + 自写 visitor），工作量从 1 天涨到 3 天。本期先做"纯文本换行级保真"，足够用户阅读。

**未来扩展（不在本次）**：要做完整保真，可以引 `markdown-it` 解析为 AST，再 visitor 模式映射到 docx 节点；或者用 `pandoc` 服务端转换。本期只把这条**写进 §不做**。

### 5. 文件名

跟 markdown 同口径：`{agentName}-{conversationTitle}-{date}.docx`，特殊字符过滤。

## 八、风险

1. **docx 库 bundle**：开工前置必做——`npm install docx` 后跑 `npm run build`，检查 `.next/static/chunks/` 目录里是否多了 docx 相关文件，且 main / app 入口 chunk 未变胖；浏览器 Network 面板验证：进入聊天页时**不**加载 docx，点击导出 Word 才**按需加载**。如需更精细分析再临时配 analyzer。如果发现意外进了首屏，立刻排查 `lib/export-conversation.ts` 是否有顶层 `import { Document } from "docx"`。
2. **图片 fetch CORS**：Supabase 公开 URL 默认带 CORS 头，浏览器 fetch 应该 OK；但实际可能存在缓存 / CDN 中间层差异。**靠 §三-Q3-C 失败降级保护**——单图失败不阻塞整体。
3. **大对话内存**：上百条消息 + 多张图，可能瞬时占内存几十 MB，但浏览器扛得住，不用流式生成
4. **docx 库与 React 19 / Next 16 兼容性**：纯 JS 库，无 React 依赖，应该没问题；**落地前 npm install 看 peer warning + tsc/lint 全过**
