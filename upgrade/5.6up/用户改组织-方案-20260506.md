**文档日期**：2026-05-06

**适用范围**：`/admin/users` 编辑用户时允许修改 `tenant_code`，含个人↔组织、组织↔组织双向转移

## 二、关键事实

### users.tenant_code 现状

```
schema.sql:27  tenant_code TEXT NOT NULL DEFAULT 'PERSONAL'
```

无 FK（schema 里没看到 REFERENCES）。值含义：
- `'PERSONAL'` = 个人用户
- 其它字符串 = 组织码（应为 `tenants.code` 中存在的某条）

### 受影响的列（决策 ii 下）

> 注：`logs` 表当前**无 user_id 字段**，按 `user_phone` 关联（schema.sql / migrations 已确认）。下表追溯条件为 `user_phone + 旧 tenant_code`。

`users` 表内必须同步更新的字段（单 user 维度）：

| 列 | 处理 | 原因 |
| --- | --- | --- |
| `tenant_code` | 改值（核心）| 转组织 |
| `user_type` | `'PERSONAL' → 'personal'`，否则 `→ 'organization'` | 权限计算读这个，不同步会错 |
| `dept_id` | **置 NULL** | 旧组织的部门 id 在新组织无意义；新组织部门由管理员后续再分配 |
| `team_id` | **置 NULL** | 同上 |
| `role` | org_admin → 自动降级为 `'user'`（见 §四-6 角色规则） | 跨组织保留管理员权限会错乱 |
| `force_relogin_at` | `NOW()` | 强制重登 |

跨表（按 user 维度）：

| 表 | 处理 | 条件 |
| --- | --- | --- |
| `logs.tenant_code` | **追溯改**（决策 ii） | `user_phone = X AND tenant_code = old` |
| `user_group_members` | **删除**该用户在所有"非新组织且非全平台"分组里的成员关系 | `user_id = X AND group_id IN (SELECT id FROM user_groups WHERE tenant_code IS NOT NULL AND tenant_code <> new)`<br>保留：全平台分组（`tenant_code IS NULL`）+ 新组织分组（理论上一般没有但留兜底） |

**不动**的表：

- `conversations` / `messages` / `files` — user 维度，无 tenant_code 字段
- `tenant_agents` / `departments` / `teams` / `resource_permissions` — 组织维度，与单 user 转组织无关
- `notices` — tenant 维度公告，无关

### 配额流转

- 改前用户消耗"个人配额"（取决于业务规则——若个人无独立 quota 表则不消耗）
- 改后用户的 chat 调用走 `tenants.quota / quota_used` 共享池（chat route 现有逻辑）
- **不补/不退个人剩余配额**——直接切换归属

### Cookie 失效

用户表加 `force_relogin_at`，**`middleware.ts` 在 verify token 之后多查一次 DB**：若 `token.iat < user.force_relogin_at` 视为失效，清 cookie + redirect 登录。

> 校正之前文档表述：当前 `middleware.ts` 用的是 `verifyToken()` / `getPayloadFromRequest()`，**不是** `getCurrentUser()`。本方案需要在 `lib/auth.ts` 新增一个 helper，例如 `validateTokenFreshness(payload)`，由 middleware 调用，发现失效就清 cookie 重定向。
> `lib/auth.ts` 的 `getCurrentUser()` 也加同样的检查（用户端业务接口走这条）。

差别是这次只对**单个用户**触发，不触及 admin token / 同组织其它人。

### 唯一约束（必须前置校验）

`schema.sql` 有 `UNIQUE(phone, tenant_code)`：同一组织里不能有两个相同手机号的账号。

转入 PERSONAL → 某组织时，如果该组织里**已经有同手机号的人**（比如另一个真人也是这个手机号注册），rpc 会因 UNIQUE 冲突失败。

**rpc 必须在改之前先校验**：
```sql
IF EXISTS (
  SELECT 1 FROM users
  WHERE phone = v_user_phone
    AND tenant_code = p_new_tenant_code
    AND id != p_user_id
) THEN
  RAISE EXCEPTION 'tenant "%" already has a user with phone %', p_new_tenant_code, v_user_phone;
END IF;
```
前端拿到错误显示："目标组织 X 已存在手机号为 138**** 的用户，无法转入"。

## 四、实施细节

### 1. RPC 函数（同步更新所有相关字段）

```sql
CREATE OR REPLACE FUNCTION change_user_tenant(
  p_user_id UUID,
  p_new_tenant_code TEXT
) RETURNS VOID
LANGUAGE plpgsql
AS $$
DECLARE
  v_old_tenant_code TEXT;
  v_user_phone TEXT;
  v_old_role TEXT;
  v_new_user_type TEXT;
  v_new_role TEXT;
BEGIN
  -- 校验目标组织存在或为 PERSONAL
  IF p_new_tenant_code <> 'PERSONAL'
     AND NOT EXISTS (SELECT 1 FROM tenants WHERE code = p_new_tenant_code) THEN
    RAISE EXCEPTION 'tenant_code "%" not found', p_new_tenant_code;
  END IF;

  -- 取旧值
  SELECT tenant_code, phone, role
  INTO v_old_tenant_code, v_user_phone, v_old_role
  FROM users WHERE id = p_user_id;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'user % not found', p_user_id;
  END IF;
  IF v_old_tenant_code = p_new_tenant_code THEN
    RAISE EXCEPTION 'user % already in tenant %', p_user_id, p_new_tenant_code;
  END IF;

  -- 唯一约束前置校验：UNIQUE(phone, tenant_code)
  IF EXISTS (
    SELECT 1 FROM users
    WHERE phone = v_user_phone
      AND tenant_code = p_new_tenant_code
      AND id <> p_user_id
  ) THEN
    RAISE EXCEPTION 'tenant "%" already has a user with phone %', p_new_tenant_code, v_user_phone;
  END IF;

  -- 派生新值
  v_new_user_type := CASE WHEN p_new_tenant_code = 'PERSONAL' THEN 'personal' ELSE 'organization' END;
  -- 角色规则：org_admin 跨组织一律降为 user；其它角色保持
  v_new_role := CASE WHEN v_old_role = 'org_admin' THEN 'user' ELSE v_old_role END;

  -- 1. 改 users（同步 tenant_code / user_type / role / 清 dept_id+team_id / 强制重登）
  UPDATE users
     SET tenant_code      = p_new_tenant_code,
         user_type        = v_new_user_type,
         role             = v_new_role,
         dept_id          = NULL,
         team_id          = NULL,
         force_relogin_at = NOW()
   WHERE id = p_user_id;

  -- 2. 清理跨组织的分组成员关系（更稳的写法）
  --    保留：tenant_code IS NULL 的全平台分组成员关系
  --    保留：tenant_code = p_new_tenant_code 的新组织分组成员关系（一般不会有，但理论可能）
  --    删除：其它任何"非新组织 + 非全平台"的分组成员关系
  --   这样即使历史上误加过别的组织的分组（不只是旧组织那条），也不会残留跨组织权限
  DELETE FROM user_group_members
   WHERE user_id = p_user_id
     AND group_id IN (
       SELECT id FROM user_groups
       WHERE tenant_code IS NOT NULL
         AND tenant_code <> p_new_tenant_code
     );

  -- 3. 追溯改 logs（决策 ii）— 按 user_phone + 旧 tenant_code
  UPDATE logs
     SET tenant_code = p_new_tenant_code
   WHERE user_phone = v_user_phone
     AND tenant_code = v_old_tenant_code;

  -- 4. 审计：写一条变更事件
  INSERT INTO logs (user_phone, tenant_code, action, status, error_msg)
  VALUES (
    v_user_phone,
    p_new_tenant_code,
    'admin_change_user_tenant',
    'success',
    'from ' || v_old_tenant_code || ' to ' || p_new_tenant_code
      || CASE WHEN v_old_role = 'org_admin' THEN ' (role demoted: org_admin -> user)' ELSE '' END
  );
END;
$$;
```

### 2. PATCH 接口

```ts
// app/api/admin/users/[id]/route.ts
export async function PATCH(...) {
  const admin = await requireAdmin();
  if (admin instanceof Response) return admin;
  const { tenantCode, ...other } = await req.json();

  if (typeof tenantCode === "string") {
    // 校验：format / 不能为空 / 长度 / 与现有组织/PERSONAL 一致
    const { error } = await db.rpc("change_user_tenant", {
      p_user_id: userId,
      p_new_tenant_code: tenantCode,
    });
    if (error) return NextResponse.json({ error: error.message }, { status: 400 });
  }

  // 其它字段照常更新（nickname / role 等）
  ...
}
```

### 3. UI · 编辑表单

`/admin/users` 编辑用户 modal 里加一个"所属组织"区域：
- 下拉选项：
  - "个人空间（PERSONAL）"
  - 该 admin 可见的所有组织（按 `tenants` 列表 + 当前 admin 角色权限过滤）
- 当前值高亮选中
- 改了选项后保存 → 弹二次确认：
  - 文案："确认把用户 `{nickname / phone}` 从 `{old}` 转入 `{new}`？此操作会：1) 切换该用户的所属组织和配额来源；2) 历史日志的归属一并迁移到新组织；3) 该用户当前登录态会失效，必须重新登录。"
  - 按钮："取消" / "确认转移"
- 保存调 PATCH，成功后刷新列表 + toast"用户已转入 {new}，将在该用户下次访问时被强制重登"

### 4. 失效检查

- `users` 加 `force_relogin_at TIMESTAMPTZ`（如未存在）
- `lib/auth.ts` 新增导出 helper `validateTokenFreshness(payload)`：
  - 入参：JWT payload（含 iat 和 userId / phone+tenantCode）
  - **实施提醒**：当前 `UserPayload` 类型可能没显式声明 `iat?: number`，开发时如有 TS 报错，先在类型定义里补上 `iat?: number`（jose 解出来的 payload 实际带这个字段，只是类型未显式声明）
  - 行为：查对应 user 的 `force_relogin_at`，与 `payload.iat` 比较
  ```ts
  if (user.force_relogin_at && new Date(payload.iat * 1000) < new Date(user.force_relogin_at)) {
    return false; // token 已被强制失效
  }
  return true;
  ```
- `lib/auth.ts` 内部的 `getCurrentUser()` 调用同一 helper，校验失败返回 null（业务接口走这条）
- `middleware.ts`：当前用 `verifyToken()` / `getPayloadFromRequest()` 解 token；在 verify 通过后**追加调用 `validateTokenFreshness(payload)`**；失败 → 清 cookie + redirect 登录页（不沿用 getCurrentUser，因为 middleware 不在 server-side 层调它）

### 5. 权限隔离

按现有角色严格层级（super_admin > system_admin > org_admin > user）收紧：

| 操作者角色 | 可操作目标 | 限制 |
| --- | --- | --- |
| `super_admin` | 任意用户 | 无限制 |
| `system_admin` | **严格低于自己**的用户（org_admin / user）| **不可操作 super_admin**（哪怕 PERSONAL 下的 super_admin）；不可操作其他 system_admin |
| `org_admin` | 不允许此操作 | 接口侧直接 403 |
| `user` | 不允许 | 接口侧直接 403 |

实施：

- 接口侧（`PATCH /api/admin/users/[id]`）：先查目标用户的 role，对比当前 admin 角色，不满足层级则 403 + msg "权限不足，无法修改该用户所属组织"
- UI 侧：org_admin / user 不显示"所属组织"下拉；system_admin 看到下拉但点了 super_admin 用户时下拉禁用并 tooltip"不可修改更高权限管理员"

### 6. 被转用户角色规则

| 旧角色 | 转入 PERSONAL | 转入其它组织 |
| --- | --- | --- |
| `super_admin` | 保留（系统级，与组织无关） | 保留 |
| `system_admin` | 保留 | 保留 |
| `org_admin` | **降级为 `user`** | **降级为 `user`** |
| `user` | 保持 `user` | 保持 `user` |

org_admin 的"管理范围"是它原所属组织。一旦它被转出原组织，旧的管理边界失效；如果保留 org_admin 角色到新组织，会出现"凭老身份操作新组织"的语义错位。

**rpc 自动降级**：`v_new_role := CASE WHEN v_old_role = 'org_admin' THEN 'user' ELSE v_old_role END`。

audit 事件里追加注释：`(role demoted: org_admin -> user)`，便于事后追查。

如果之后要把这个用户重新提为新组织的 org_admin，由超级管理员在用户管理页另外操作（已有的 role 编辑入口）。

## 六、风险与回退

### 风险

1. **强制重登的体验**：被转用户正在工作时被踢——需要让 admin 操作前**自行通知**该用户。前端二次确认文案明确写"会强制重登"。
2. **logs 追溯改是不可逆的**：如果改错了用户，再改回去 logs 也会跟着回去，但中间已经写过的"audit 事件"会留两条记录形成历史。可接受。
3. **org_admin 误操作**：本方案排除 org_admin 改 tenant_code 的能力，从源头规避。
4. **PERSONAL 用户配额**：转入组织后该用户开始消耗组织配额。如果业务上有"个人用户独立计费"的需求，本方案不处理（个人用户当前没独立配额表）。

### 回退

- 操作错误：再调一次 rpc 把 tenant_code 改回去（rpc 可重入）
- 灾难：rpc 函数有 bug → supabase PITR 恢复
- 代码层：本方案改动集中在 5 个文件，删除新增的 PATCH 分支 / UI 区块即可还原；rpc 函数可保留不删

## 七、不做

- 不做"批量转移用户"
- 不做"用户多组织成员身份"（一个 user 只对应一个 tenant_code）
- 不做"配额迁移 / 退款 / 补差"
- 不做 personal 独立配额体系
- 不做被转用户的邮件/短信通知（admin 操作前人肉通知）
- 不做组织码可改方案里的全表级联（本方案不动 tenant_agents / departments / teams / resource_permissions / 等等，因为那些是"组织"维度的关联，跟单个用户转组织无关）
