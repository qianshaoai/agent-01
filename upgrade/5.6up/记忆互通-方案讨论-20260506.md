# 同工作流多 agent 记忆互通 · 方案讨论

**文档日期**：2026-05-06
**状态**：**讨论中，暂缓实施**
**目的**：留底"记忆互通"3 种实现方案的取舍，等需求收紧后再选其一

---

## 当前架构现状

- `conversations` 表是 **user × agent** 一对一（一条会话只属于一个智能体）
- `messages` 表只有 `role: user/assistant`，**没有 `agent_id` 字段**
- 进 agent A 跟 agent B 是两条独立 conversation，互不可见
- 工作流（`workflows` + `workflow_steps`）目前只是 UI 上的"步骤展示"，**没有执行引擎**

---

## 方案 A · 共享 conversation（"真"互通）

### 改动

- `conversations` 改成 user × **workflow** 一对一，所有 step 里的 agent 共用一条
- `messages` 加 `agent_id` 字段区分谁回的
- chat route 拉 history 时带上"其他 agent 的发言"，给 prompt 加来源前缀（例 `[来自智能体B] xxx`）

### 优点

- 体验贴近"团队协作"——agent2 真能看到 agent1 跟你说过什么
- 切 agent 时无需任何"接力"操作

### 代价

- DB schema 改动：conversations 加 workflow_id、messages 加 agent_id、迁移历史数据
- chat route 几乎重写
- **所有 adapter 拼 history 的逻辑都要重写**：Coze 自维护 `conversation_id`，得改成 `auto_save_history: false` + 前端送整段 history 模式，与元器 / Dify 对齐
- bot 看到"另一个 assistant 说过的话"会困惑，需要明确角色标记

### 工作量

3-5 天（核心逻辑 + 测试 + 数据迁移脚本）

---

## 方案 B · 切换时塞摘要（轻量）

### 改动

- step 1 → step 2 时，先调一次 agent1 让它输出"截至目前的对话摘要"
- 摘要作为 agent2 系统 prompt 或 user 第一条消息的"前情提要"
- 各 agent 仍各自维护自己的 conversation

### 优点

- **不动 DB**，纯 chat route + 前端切换流程加一步
- 每个 agent 自己的会话不被污染

### 代价

- 摘要质量依赖 agent1 的能力
- 每次切换多调一次 LLM（多扣 1 次配额）
- agent2 拿到"截图版"上下文，遇到细节追问会失忆

### 工作量

1-2 天

---

## 方案 C · 工作流变量传参（真正的工作流引擎）

### 改动

- 工作流定义"上下文变量"（如 `客户姓名` / `理财偏好`）
- 每个 step 声明 input / output schema
- 平台引擎执行时把变量在 agent 间传递
- Coze / Dify 自家工作流就是这种

### 优点

- 业务规则清晰，可控
- 长期方向最合理

### 代价

- 你目前的"工作流"只是 UI 步骤展示组件，**没有任何执行引擎**
- 要做这条得写整套 DSL + 调度器 + UI 编辑器 + 变量绑定器

### 工作量

2-4 周

---

## 决策点（先收集需求再选）

要回答的问题：

1. **使用场景到底是哪种？**
   - "同事之间互相知道你说过啥" → A
   - "接力做事时把上下文递交" → B
   - "参数化工作流，按变量驱动" → C

2. **是否能容忍"摘要丢细节"**？
   - 能 → B 性价比最高
   - 不能 → 必须 A 或 C

3. **是否有正经的工作流编排需求**（不只是聊天接力）？
   - 有 → 长期 C
   - 没有 → A / B 二选一

4. **预算 / 时间窗**？
   - 1 周内 → 只能 B
   - 2 周 → A
   - 1 月+ → C

---

## 暂缓原因

- 需求还不明确（用户还没明确说想要哪种感觉）
- A、C 工作量大，要先有明确业务收益再投
- 现阶段先把"对话导出"这种简单需求做完，再回头判断
