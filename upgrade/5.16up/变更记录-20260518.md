### W2 · 按模型加权扣额度

`app/api/agents/[id]/chat/route.ts`：

- **解析顺序重排（小B 复评）**：把"provider/key 解析 + 算模型 + 算 weight"
  整体从原步骤 6 提到**步骤 2**，在配额检查、建会话、写用户消息**之前**完成。
- provider 查询 `select` 补 `default_model`（小B finding 2）。
- 当前模型 `modelUsed`：`agent.model_params.model || provider.default_model ||
  （仅 openai）"gpt-4o-mini"`，与适配器内部兜底对齐；非空时回填
  `resolvedModelParams.model`，保证 日志/扣费/真实调用 三者一致。
- 算 `weight`：查 `model_quota_weights`，命中且 `enabled` → `weight_per_call`；
  查不到 / 未 enabled → `weight = 1`（决策 3/4，模型驱动、软放过）。
- **带权重配额预检查（小B finding 1，方案甲）**：
  `quota_used + weight > quota` 即拦，与 `increment_quota_used_weighted` 内部
  守卫口径一致 —— 不漏扣、不超扣。
- 扣费：`weight > 1` 走 `increment_quota_used_weighted`，否则原
  `increment_quota_used`；**移出 `Promise.all`**，检查 `{ data, error }`，
  `error` 或加权 RPC 返回 `false`（余额不足/并发）→ `console.error` 告警（小B 复评）。

## PR-2 补丁 · 剩余次数计数器识别 weight

### 问题

W2 已完成后端加权扣费与后台日志展示，但用户对话页顶部的"剩余次数"计数器仍固定
每次 `left - 1`。加权模型（如 `gpt-4o` 实扣 5）发完后，页面只少 1 次，刷新后才恢复
真实余额，容易让员工误以为额度还很多。

### 修复

- `app/api/agents/[id]/chat/route.ts`：SSE `done` 事件带上本次 `weight`：
  `{ done: true, conversationId, weight }`。
- `app/agents/[id]/page.tsx`：收到 `done` 后按 `obj.weight ?? 1` 乐观扣减剩余次数。
  服务端未带 `weight` 时仍按 1 扣，兼容旧行为。

### 不涉及

- `user-agents` / `trial` 路由没有加权扣费，保持原行为。
- 扣费 RPC 失败导致的乐观扣减漂移仍属"先调模型、后扣费"架构残余风险，刷新即可恢复；
  彻底解决需后续改成调用前预占 / 或 route 直接返回真实剩余额度。

## Fix 3 收口 · 加密 key 拆分（存量 rekey + 部署脚本收口）

方案见 [加密key拆分收口-rekey方案-20260518.md](./加密key拆分收口-rekey方案-20260518.md)（v3）。

### 背景

5.14up 已实现「路径 A · Fallback Chain」（`lib/crypto.ts`），但只「向前修复」：上线前
已入库的存量密文仍用 `sha256(JWT_SECRET)` 旧 key 加密。用户决策（5/18）：补一次性
rekey，把存量密文整体重写为新 key `ENCRYPTION_KEY`，彻底关闭「JWT_SECRET 泄漏 = 全
平台密钥沦陷」。

### 改动

- **新建 `scripts/rekey-encryption.mjs`**：把 `agents` / `model_providers` /
  `user_agents` 的 `api_key_enc` + `tenants` 的 `openai_key_enc`（4 表，主键 `id`）
  逐行从旧 key 重写为新 key。自带加解密不 import crypto.ts；新 key 只从 env 读不进
  argv；启动校验 base64 32 字节 + 新旧 key 不相等（防伪拆分）；逐行五分类
  `empty/new_hit/old_hit/plaintext_shape/failed`；写前回环自校验；问题行退出码非 0；
  dry-run 默认、`--apply` 实写。
- **`scripts/migrate-trial-agents.mjs`**（阻断点收口）：该脚本挂在 Jenkins 部署
  流水线、每次部署重写 3 个 trial agent 的 `api_key_enc`。原内联加密用旧 key —— rekey
  后会被它反向污染。已改为用 `ENCRYPTION_KEY` 新 key（`getKey()` base64 解码 32 字节、
  不 sha256）；`REQUIRED_ENVS` 用 `ENCRYPTION_KEY` 替换 `JWT_SECRET`；移除多余的
  `createHash` import。
- **`scripts/migrate-agent-keys-to-providers.mjs`**：一次性脚本、不反向污染，但
  `decrypt()` 原只认旧 key。已补「新 key 优先、回退旧 key」fallback；头部加废弃注记。
- **`lib/crypto.ts`**：订正陈旧注释（原写「production 不允许 fallback，进程启动期
  直接抛错」，实际无启动校验、`decrypt()` 静默回退、`encrypt()` 才抛）。**无逻辑改动。**
- **`Jenkinsfile`**：trial stage 注释由「用线上 JWT_SECRET」同步改为 `ENCRYPTION_KEY`。
  **无逻辑改动。**

### 小B 两轮评审
