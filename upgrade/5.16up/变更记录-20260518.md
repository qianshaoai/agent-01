## PR-2 · W1 token 统计 + W2 加权扣额度

### W1 · Token 用量统计

  - `AdapterConfig` 加 `onUsage?` 回调 + 导出 `TokenUsage` 类型。
  - openai 兼容适配器：**仅 `platform === "openai"`** 时请求带

    不认会 400）；SSE 末 chunk 带 `usage` 时回调 `onUsage`。
- `app/api/agents/[id]/chat/route.ts`：`streamChat` 传 `onUsage` 捕获
  `capturedUsage`；流结束写 `logs` 带 `model_used`（模型已知就记）+
  `prompt_tokens / completion_tokens`（有 usage 才记）。
- `app/admin/logs/page.tsx`：日志表加「模型」「Token(in/out)」两列展示
  （小B finding 5 用户选乙）。

### W2 · 按模型加权扣额度

`app/api/agents/[id]/chat/route.ts`：

- **解析顺序重排（小B 复评）**：把"provider/key 解析 + 算模型 + 算 weight"
  整体从原步骤 6 提到**步骤 2**，在配额检查、建会话、写用户消息**之前**完成。
- provider 查询 `select` 补 `default_model`（小B finding 2）。
- 当前模型 `modelUsed`：`agent.model_params.model || provider.default_model ||
  （仅 openai）"gpt-4o-mini"`，与适配器内部兜底对齐；非空时回填
  `resolvedModelParams.model`，保证 日志/扣费/真实调用 三者一致。
- 算 `weight`：查 `model_quota_weights`，命中且 `enabled` → `weight_per_call`；
  查不到 / 未 enabled → `weight = 1`（决策 3/4，模型驱动、软放过）。
- **带权重配额预检查（小B finding 1，方案甲）**：
  `quota_used + weight > quota` 即拦，与 `increment_quota_used_weighted` 内部
  守卫口径一致 —— 不漏扣、不超扣。
- 扣费：`weight > 1` 走 `increment_quota_used_weighted`，否则原
  `increment_quota_used`；**移出 `Promise.all`**，检查 `{ data, error }`，
  `error` 或加权 RPC 返回 `false`（余额不足/并发）→ `console.error` 告警（小B 复评）。

### 小B PR-2 评审修复

小B 评审 PR-2 提了 2 点，均已补（`app/api/agents/[id]/chat/route.ts`）：

- **P2 · weight=1 扣费失败漏检**：原来 `ded.data === false` 只在 `weight > 1`
  时查。`increment_quota_used` 也返回 boolean，并发下 weight=1 的请求也可能
  返回 false（没扣成）。改为**两种 RPC 都查 `data === false`**，没扣成即告警。
- **P3 · 权重表查询 error 被当 weight=1**：原来 `model_quota_weights` 查询只看
  `data` 不看 `error`，迁移漏跑 / 表异常 / DB 故障时贵模型会静默按 1 扣。
  改为查询出错即 `console.error` + **返回 503 阻断**（与"配额检查 DB 故障即 503"
  同口径，不静默少扣）。
