## 四、实现约束

1. 不读、不改 `.env.local`。涉及环境变量只改 `.env.local.example` 或文档，除非用户明确要求。
2. 不做无关重构，不顺手改样式体系、组件库或接口返回结构。
3. 前端改动必须保留现有权限、筛选、跳转、高亮和空状态逻辑。
4. 后端改动必须保留现有鉴权与租户 / 组织隔离逻辑。
5. R3 的 `PATCH /api/workflow-sessions/[id]` 是复用接口，状态白名单只约束带 `status` 的请求；纯改名 / 改进度不受影响。
6. R4 若做“完整展示”，不能只依赖 `pageSize=100`，必须后端全量 / 分组接口，或前端分页循环拉完所有页。
7. R5 批量重排必须优先 DB 事务 / RPC；若暂用逐行写，必须整批校验、失败后强制刷新真实顺序，并在文档注明残余风险。
8. R6 方案乙不能宣传为“标签内独立排序”；只有 `workflow_categories.sort_order` 方案甲才是真隔离。

## 五、验证约束

每个 PR 至少完成对应模块的手工回归，并在 PR 描述或变更记录里写清：

- 改了哪些文件。
- 选了哪个方案 / 决策点。
- 跑了哪些检查，没跑的要说明原因。
- 手工验收了哪些路径。

建议检查：

```text
npm run typecheck
npm run lint
npm run diff-check
```

如本地构建仍遇到 Next worker OOM，记录为环境限制，不把它当作业务改动失败。

1. 方案 A 不碰 `app/admin/workflows/page.tsx`。
