## 一、为什么"干净重做"而不是直接 merge devA

5/18 试过 `git merge devA` → 已 abort。原因：

- devA 落后 master 52 个提交，它的 GPT 接入是 5.7up 时代的代码。
- 那时 pr-d 的架构（API 管理模块、provider 解析、空响应保护…）**都还不存在**。
- `git merge` 把 devA 的 chat route 和 pr-d 演进后的 chat route auto-merge 成了
  "缝合怪"，语义不可靠、排查成本比重做还高。

**结论**：不 raw merge。改为 —— 把 devA GPT 接入里 **pr-d 还缺的部分**，对着
pr-d 当前架构**重新实现一遍**。

## 四、要做的三块

### W1 · Token 用量统计

- `lib/adapters/index.ts`：`AdapterConfig` 加 `onUsage?` 回调。以 pr-d 现有 openai
  适配器为基线**增量改**，只加 `onUsage`，不整段替换。
- ⚠️ **必须修正点（小B finding 3）**：`stream_options: { include_usage: true }`
  **只对 `platform === "openai"` 开**。默认分支还接着 zhipu 等 OpenAI 兼容平台，
  有些兼容接口不认 `stream_options` 会直接 400 —— 不能一刀切给所有 openai-compatible。
  zhipu 暂不统计 token，但不会被打挂。
- `app/api/agents/[id]/chat/route.ts`：传 `onUsage` 捕获 `capturedUsage`；流结束
  写 `logs` 时带 `prompt_tokens / completion_tokens / model_used`。
- **日志页展示（小B finding 5，用户选乙）**：`app/admin/logs/page.tsx` 加
  `prompt_tokens / completion_tokens / model_used` 三列展示。
  `/api/admin/logs` 已 `select("*")`，后端不用改，只补前端列。
- 只有 openai 平台才有 token 数；Coze / 清言 / zhipu 等不返回 → 字段留空，属正常。

### W2 · 按模型加权扣额度

- **当前模型取值（三级 fallback，须与 adapter 对齐 —— 小B 复评补充）**：
  `agent.model_params.model || provider.default_model || "gpt-4o-mini"`。
  这条链跟 openai 适配器内部兜底一致，保证**日志模型 / 扣费模型 / 真实调用模型
  完全对齐**。
  - ⚠️ **必须修正点（小B finding 2）**：当前 chat route 解析 provider 时
    `select` 没取 `default_model`。PR-2 必须在 provider 查询里**加上 `default_model`**，
    并把上面这条 fallback 回填进 `resolvedModelParams.model`。
- ⚠️ **PR-2 实现顺序（小B 复评补充）**：现 chat route 的配额检查在 provider
  解析**之前**。W2 要把顺序改成 —— **① 解析 provider / default_model → ② 算
  weight → ③ 带权重配额检查**；且这一串必须**在创建会话、写用户消息之前**完成，
  余额不足要在任何 DB 写入前拦下。
- **算 weight**：解析完模型查 `model_quota_weights` —— 命中且 `enabled` →
  `weight = weight_per_call`；查不到 / 未 enabled → `weight = 1`（决策 3、4：模型
  驱动、软放过，不依赖是否拿到 usage）。同一个 `weight` 给"预检查"和"扣费"复用。
- ⚠️ **必须修正点（小B finding 1，账务口径，方案甲）—— 不超扣**：
  - 配额预检查**带权重**：`tenant.quota_used + weight > tenant.quota` 即拦截，
    提示"余额不足，该模型每次需 N 次额度"。
  - 原因：devA 的 `increment_quota_used_weighted` 内部本就是
    `quota_used + weight <= quota` 才扣。若预检查仍用二元 `>= quota`，边界贵模型
    请求会"聊成功但 RPC 拒扣" → 白嫖。带权重预检查与 RPC 守卫一致，**不漏扣、不超扣**。
  - devA 的 RPC **原样用**，不改、不加新 migration。
- 扣费：`weight > 1` 走 `increment_quota_used_weighted(code, weight)`；
  `weight === 1` 走原 `increment_quota_used`。
- ⚠️ **扣费结果必须检查（小B 复评补充）**：扣费 RPC **不再塞进 `Promise.all`
  静默跑**。`increment_quota_used_weighted` 返回 boolean —— PR-2 要检查
  `{ data, error }`：`error` 或 `data === false`（并发抢额度等导致没扣成）至少
  打 `console.error` / 告警，杜绝"对话成功但扣费被吞"。
- `model_quota_weights` 只用 SQL 维护，本期不做 admin UI（决策 6）。

### W3 · 会话摘要降本（**本期不做** —— W1+W2 上线确认无误后，另开分支制作）

- `lib/summarizer.ts`：**滑动窗口 + 增量摘要**。保最近 N 轮原样；只摘"新溢出
  窗口"的老消息（增量，不重复摘）；摘要器用智谱 `glm-4-flash`（复用 zhipu 适配器）。
- chat route 历史加载改造：加载 `summary_until_at` 之后的消息 + 把
  `conversations.summary_text` 作为一条 system 注入；老历史溢出阈值时触发增量
  摘要、回写 `conversations`。
- 铁律：**当前用户消息永不进摘要**（保真）；短对话（≤ 窗口）完全不调摘要器。
