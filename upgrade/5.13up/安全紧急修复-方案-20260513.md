## 二、修复方案

### 修复 1 · 收紧登录错误文案

**目标**：让外部观察者无法通过错误信息区分"账号是否存在/注册过几次"。

**改 [app/api/auth/login/route.ts](../../app/api/auth/login/route.ts)**：

| 行 | 当前 | 改成 |
|---|---|---|
| :47 | `"账号不存在，请先注册"` | `"账号或密码错误"` |
| :67 | `"密码错误"` | `"账号或密码错误"` |
| :51-55 | `"该手机号存在多个账号，请使用用户名登录"`（status 409） | `"账号或密码错误"`（status 401，**不要** 409）|

> 第三条值得特别说明：v1 没改这条。但它**比前两条还泄漏**——直接确认手机号已注册 2 次以上。代价是真实的多组织用户体验略差（必须改用用户名才能登），但安全收益值。组织用户登录引导已经有"使用用户名登录"的入口，前端不依赖这条 API 的具体提示词。

**保留**的错误（合规价值，且不可被未注册者触发）：
- "该账号已注销/已被禁用/已到期" 等账号状态错误（[:60-94](../../app/api/auth/login/route.ts#L60-L94)）——只对**已通过密码校验**的真用户暴露

**admin/login 入口**：**不动**（已经是统一文案，详见 §一-1c）。

**不做**：v1 提的"假 bcrypt 抗时序侧信道"。理由：
- 公网 RTT 抖动 ~10-50ms 远大于 bcrypt 单次差异
- [app/api/admin/login/route.ts:67-72](../../app/api/admin/login/route.ts#L67-L72) admin 路径多 user 匹配时是 for 循环 N 次 bcrypt，时长随匹配数线性变化，单次"假 bcrypt"根本抹不平
- 加 5 行代码换不到实际安全收益，是过度工程

### 修复 2 · 改密码后强制其他设备重登（按四类语义分别处理）

四条路径**语义不同，处理也必须不同**，不能一把统一。

| # | 文件 | 当前 update 行 | 语义 | 改法 |
|---|---|---|---|---|
| 2.A | [auth/change-password/route.ts:38](../../app/api/auth/change-password/route.ts#L38) | 38-41 | **用户改自己**（user-token） | update 加 `force_relogin_at: new Date().toISOString()`；**保留**当前的 `buildClearCookieHeader()` + `requireRelogin: true`（[:43-47](../../app/api/auth/change-password/route.ts#L43-L47)）—— 现状就是主动让用户重登，**不切到"重发 cookie 保留会话"模式**（避免改语义） |
| 2.B | [admin/change-password/route.ts:40](../../app/api/admin/change-password/route.ts#L40) | 40-43 | **admin 改自己**（users 表路径，被提升的普通用户） | update 加 `force_relogin_at`；**重发 admin cookie**（用 [lib/auth.ts:88](../../lib/auth.ts#L88) `signToken` + [:204](../../lib/auth.ts#L204) `buildAdminSetCookieHeader`）保留当前会话 |
| 2.C | [admin/change-password/route.ts:65](../../app/api/admin/change-password/route.ts#L65) | 65-68 | **admin 改自己**（admins 表路径，内置 admin） | 同 2.B：加 `force_relogin_at` + 重发 admin cookie |
| 2.D | [admin/users/[id]/route.ts:156](../../app/api/admin/users/[id]/route.ts#L156) | 156-159 | **管理员给别人重置**（reset-password action） | update 加 `force_relogin_at`；**绝对不要**碰当前 admin 的 cookie。这正是 force_relogin_at 该干的事——把目标用户 B 的所有会话踢掉，admin A 自己的会话不受影响 |

**为什么 v1 的"四处统一重发 cookie"是错的**：
- 2.A 现状已经主动 `buildClearCookieHeader()` 让用户跳重登，硬切成"重发 cookie 保留会话"等于偷偷改了用户可见行为
- 2.D 当前会话是 admin A，被改密码的是 user B，给 admin A "重发 cookie 保留会话"毫无意义；该被踢的是 user B 的所有会话
- 2.B / 2.C 才是真正"改自己密码、想保留会话不被自己踢"的场景

**用户改自己密码的审计日志**（小补丁）：
- 2.B / 2.C / 2.D 现状都已 `writeAuditLog`（[admin/change-password](../../app/api/admin/change-password/route.ts) 内）
- 2.A 没有 audit。建议补一行 `writeAuditLog({ action: "update", resourceType: "self_password", ... })`（user 侧没有 `writeAuditLog` helper 用过的话需要确认接口；如果改这条会扩大 PR 范围，可以放到下个 patch）

### 修复 3 · 拆分加密 key 和 JWT_SECRET

**目标**：JWT_SECRET 泄漏 ≠ 全平台加密数据沦陷。

**前置步骤（小A 务必先做）**：
```
Grep -r "encrypt\(|decrypt\(" lib app
```
确认所有 `encrypt()` / `decrypt()` 调用点。v1 假设迁移范围只有 `agents.api_key_enc`，但 [lib/crypto.ts](../../lib/crypto.ts) 公开导出 `encrypt`/`decrypt`，可能 workflow / users / 其他表也有加密字段。**漏一张表 = 那张表永久丢数据**。grep 结果列在迁移脚本注释里，逐表迁移。

**改法**：

#### 3.1 加新环境变量

```env
# .env.local（本地）
ENCRYPTION_KEY=<openssl rand -hex 32 生成>
```

`.env.local.example` 同步加注释。**Vercel 生产环境同步加**，但**JWT_SECRET 保持不变**——见 3.3 迁移顺序。

#### 3.2 改 [lib/crypto.ts](../../lib/crypto.ts)

```ts
function getKey(): Buffer {
  const secret = process.env.ENCRYPTION_KEY ?? process.env.JWT_SECRET;
  if (!secret) throw new Error("ENCRYPTION_KEY (or fallback JWT_SECRET) is not set");
  return createHash("sha256").update(secret).digest();
}
```

> 关于 ENCRYPTION_KEY 长度：因为 `getKey()` 走 `sha256(secret).digest()` 派生，ENCRYPTION_KEY **本身没有长度限制**（任意字符串都会派生出 32 字节 key）。推荐 `openssl rand -hex 32` 是为了"够熵 + 不易抄错"，不是强制格式。

**`decrypt()` catch 块改动**（[lib/crypto.ts:35-37](../../lib/crypto.ts#L35-L37)）：

当前：
```ts
} catch {
  return ciphertext; // 解密失败，当作旧明文
}
```

问题：迁移期间一旦 ENCRYPTION_KEY 配错或某条记录漏迁，这里**静默把密文当成 API key 返回**，下游会把"iv:tag:cipher" 这种字符串发给 Coze/Dify，错误信息一片混乱无法定位。

改成：
```ts
} catch (e) {
  throw new Error(`[crypto] decrypt failed: ${(e as Error).message}`);
}
```

> 兼容性：现有"旧明文"分支（[:26](../../lib/crypto.ts#L26) `if (parts.length !== 3) return ciphertext`）保留，那是格式判断、不是失败兜底。这里 catch 的是"格式对但 key/tag 验不过"的真实失败，必须显式报错。

#### 3.3 迁移顺序（**严格按这个顺序**，做错一步可能永久丢数据）

```
[1] 部署前：本地准备
    - .env.local 加 ENCRYPTION_KEY=<新值>
    - JWT_SECRET 保持不变（不要轮换！）
    - 写迁移脚本 scripts/migrate-encryption-key.ts，签名：
        ts-node scripts/migrate-encryption-key.ts \
          --old-secret "$OLD_JWT_SECRET" \
          --new-key "$NEW_ENCRYPTION_KEY" \
          --table agents --column api_key_enc \
          [--dry-run]
      （显式参数，绝不读 process.env，避免 env 已轮换后跑脚本永久丢数据）
    - 脚本内部：sha256(old-secret) 解密 → sha256(new-key) 重加密 → update
    - 加 --dry-run 模式：只解密验证不写回，看有几条解密失败

[2] 备份生产 DB（pg_dump 或 Supabase 控制台 backup）
    - 至少 dump §3.1 grep 出来的所有含加密字段的表
    - 备份留存 ≥ 7 天

[3] 部署生产
    - Vercel 加 ENCRYPTION_KEY 环境变量
    - JWT_SECRET 仍保持不变
    - 部署新代码（getKey 优先读 ENCRYPTION_KEY，缺失 fallback JWT_SECRET）
    - 此时：新数据用 ENCRYPTION_KEY 加密，老数据仍能用 JWT_SECRET（因为 fallback 链）解密
    - 但一旦老数据被 update，会用新 key 重写。所以必须立刻进 [4]

[4] 跑迁移脚本（生产）
    - 先 --dry-run 确认 0 失败
    - 实跑，逐表迁移
    - 验证：随机抽样 10 条 agents 调用，确认能正常调通 Coze/Dify

[5] （可选）轮换 JWT_SECRET
    - 只有 [4] 验证通过后才能做
    - 轮换后 fallback 链断掉，老数据如有遗漏将永久无法解密
    - 建议至少 staging 上观察 1 周再做
```

**回滚预案**：[3] 之后任何一步失败 → restore [2] 的 DB backup → 删 ENCRYPTION_KEY env → 回滚代码到上一版。

**执行人提醒**：第 [1] 步脚本签名是关键。**绝不能让脚本读 `process.env.JWT_SECRET`**——因为脚本如果晚点跑，env 可能已经被轮换或更名，再读就是空值/新值，迁移直接全废。
