## 15. Hotfix · 后台设置页刷新后展示图回退占位

### 现象

用户上传成功 → 登录页主界面正常出图 → 但**后台设置页刷新后**预览框显示"默认深蓝纯色（未配置展示图）"，
点删除按钮也消失（因为 `settings.login_showcase_url` 为空）。

### 根因

v2 方案 §4.2 只列了改 `/api/settings`（公开接口），漏写了 `/api/admin/settings`（后台接口）。
两条接口走的是不同 GET handler：
- `app/api/settings/route.ts` → 登录页等未鉴权页面读，**§14.2 已加** `login_showcase_url` 字段
- `app/api/admin/settings/route.ts` → 后台 `/admin/settings` 页面读，**漏改**，返回 JSON 不带 `login_showcase_url`

后台页 `useEffect` 拿到 `d.login_showcase_url === undefined`，
`setSettings({ ...d })` 把整个对象覆盖（不是合并），导致初始状态里的 `""` 也被冲掉，
渲染时落到"未配置"分支。

### 修复

**文件**：`app/api/admin/settings/route.ts:21-22`

GET 返回对象加一行：
```ts
login_showcase_url: map.login_showcase_url ?? "",
```

下次给"前台公开 + 后台管理"两套接口的设置加字段，
**必须同时改两个 GET**，并且在方案里就列清楚——v2 方案漏掉就是因为我只看了 `/api/settings` 没翻 `/api/admin/settings`。

memory 加一条：品牌设置字段双接口对称检查。

### 16.2 Lint 修法

**问题**：`app/admin/audit-logs/page.tsx:99` 在 `useEffect` body 内同步 `setLoading(true)`，触发 React 19 新规则 `react-hooks/set-state-in-effect`。

**修法**：line 99 上方加一行 ESLint 豁免：

```ts
// eslint-disable-next-line react-hooks/set-state-in-effect -- loading spinner intentionally re-fires when filter/page deps change
setLoading(true);
```

**为什么用 disable 而非重构**：抽 callback 会改 fetch 时序（依赖闭包 + 清理逻辑都得重设），违背收口"行为不变"原则。React 19 这条规则在社区还在争议，一行注释 + 原因说明是合规做法。

### 16.3 Freshness Bug 修复（属于 Fix 2 直接尾巴）

**Bug**：`lib/auth.ts:137` 的 `validateAdminTokenFreshness` **只查 admins 表**，但 `app/api/admin/change-password/route.ts:48` 对 **users 表的管理员**（被提升为管理员的普通用户）改密时写的是 `users.force_relogin_at`，两边对不上号。

**后果**：users 表管理员改密后，他在另一台设备的后台 admin cookie **不会被踢**——属于 Fix 2 改密强制重登机制的漏洞。

**修法**：扩展 `validateAdminTokenFreshness`，admins 查不到时回退查 users 表。代码改动**仅落在该函数内**，行数从 18 行扩到 28 行，不动 change-password / middleware / 其他调用点：

```ts
// admins 没记录 → token.adminId === users.id（被提升为管理员）→ 回退查 users
let forceAt: string | null = adminRow?.force_relogin_at ?? null;
if (!adminRow) {
  const { data: userRow } = await db.from("users").select("force_relogin_at")
    .eq("id", payload.adminId).single();
  forceAt = userRow?.force_relogin_at ?? null;
}
```

修后行为：
- 原生 admin → 跟原来一样 ✅
- users 表管理员 → admins 查不到 → 回退 users 表 ✅
- 都没有 / DB 故障 → fallback 放行（保留原行为） ✅
