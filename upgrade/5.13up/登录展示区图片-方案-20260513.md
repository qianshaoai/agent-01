# 登录页展示区可上传图片 · 方案

版本：5.13up
日期：2026-05-13
状态：v2，已按 devB review 修订

> **v2 修订摘要**（v1 → v2 的关键变化）
> 1. 模板从 logo route 改成 **contact-qr route**（contact-qr 已有 DELETE 实现，少一处出错机会）
> 2. 文件白名单收窄到 `PNG / JPG / JPEG / WEBP`，**排除 SVG / GIF**（XSS 边角风险 + 动图分散注意力）
> 3. **加 5MB 后端 cap + 2MB 前端建议**（展示图比 logo 大数量级，等出问题再加来不及）
> 4. 存储路径前缀沿用 **`logos/`**（和现有 logo / contact-qr 保持一致），不另开 `showcases/`
> 5. 登录页 JSX 的 `<img>` 改成 `absolute inset-0` 显式定位（避免依赖隐式 flex stretch）
> 6. §五 register 同步明确写"留待下个 patch，届时再决定共用字段 vs 独立字段"

## 一、需求

- 登录页左侧大块（v12 是纯深蓝占位）改成展示一张图片
- 图片由 super_admin 在后台"品牌设置"页上传
- 没图片时回退到当前纯深蓝状态

- **DB 表 `system_settings`**：通用 key-value 表，存 logo_url / platform_name / contact_qr_url 等。**新字段直接加 key 就行，不用 migration**
- **公开读取**：[app/api/settings/route.ts](../../app/api/settings/route.ts) GET 返回所有品牌字段（登录页前端已经在用）
- **管理员上传 + 删除模板**：[app/api/admin/settings/contact-qr/route.ts](../../app/api/admin/settings/contact-qr/route.ts) 是最干净的模板——POST 上传 + DELETE 清值，audit log、cookie、权限边界都齐全
- **管理员 UI**：[app/admin/settings/page.tsx](../../app/admin/settings/page.tsx) 已有 logo 和联系二维码两块上传组件

新功能完全可以**抄现有模式**，加一份新文件 + 几行配置即可。

## 四、改动清单

### 4.1 后端 · 新上传路由

新文件 `app/api/admin/settings/login-showcase/route.ts`，**抄 [contact-qr route](../../app/api/admin/settings/contact-qr/route.ts) 模板**（POST + DELETE 都现成），改五处：

| 改动 | 旧 | 新 |
|---|---|---|
| 文件上传路径 | `logos/contact-qr.{ext}` | `logos/login-showcase.{ext}`（**沿用 `logos/` 前缀**，和现有约定一致） |
| 写入 `system_settings` 的 key | `contact_qr_url` | `login_showcase_url` |
| audit_log resourceName | `"联系二维码"` | `"登录页展示图"` |
| 文件白名单 | `["png", "jpg", "jpeg", "webp", "gif"]` | `["png", "jpg", "jpeg", "webp"]`（**排除 GIF**） |
| 错误文案 | `"只支持 PNG / JPG / WEBP 格式"` | `"只支持 PNG / JPG / WEBP 格式"`（保持一致） |

**新增校验**（contact-qr route 没有的）—— 加在 ext 校验之后、buffer 转换之前：

```ts
// 文件大小上限：展示图比 logo 数量级大，必须显式 cap
const MAX_SIZE = 5 * 1024 * 1024; // 5MB
if (file.size > MAX_SIZE) {
  return apiError(`图片不能超过 5MB，当前 ${(file.size / 1024 / 1024).toFixed(1)}MB`, "VALIDATION_ERROR");
}
```

权限边界：跟 logo / contact-qr 一致，**仅 `super_admin`** 可改（contact-qr route 模板里已经有这一段，照抄即可）。

**决策 1 = A**：DELETE 接口直接抄 contact-qr 的 [:53-70](../../app/api/admin/settings/contact-qr/route.ts#L53-L70)——把 `system_settings.login_showcase_url` 设成空字符串（不动 Storage 里的文件，留着无害且节省 IO）。

**决策 4 = A**：URL 后缀加 `?t=${Date.now()}` 破坏浏览器缓存，跟 logo / contact-qr 一致。

**关于 SVG 排除的额外说明**：logo 白名单里有 SVG 是历史遗留（logo 是图标语义、确实有用 SVG 的需求）。展示图是摄影/插画语义，**不需要 SVG**；且 SVG 内嵌 `<script>` 在某些浏览器上下文（非 `<img>` 加载方式）会执行，新增字段不应该把这个边角风险带进来。

### 4.2 后端 · 公开读接口加字段

文件 [app/api/settings/route.ts](../../app/api/settings/route.ts)：

### 4.3 前端 · 登录页消费

文件 [app/login/page.tsx](../../app/login/page.tsx)：

- `siteSettings` 类型（[:16-22](../../app/login/page.tsx#L16-L22)）加 `login_showcase_url: ""`
- 左侧展示区（[:124](../../app/login/page.tsx#L124)，当前是 `<div className="hidden md:block flex-[9] bg-[#001f7a] relative border-r border-white/10" />`）改成：

```tsx
<div className="hidden md:block flex-[9] relative border-r border-white/10 overflow-hidden">
  {siteSettings.login_showcase_url ? (
    <>
      {/* eslint-disable-next-line @next/next/no-img-element */}
      <img
        src={siteSettings.login_showcase_url}
        alt=""
        className="absolute inset-0 w-full h-full object-cover"
      />
      {/* 决策 2 = B：加薄黑色 overlay 平衡视觉重量 */}
      <div className="absolute inset-0 bg-black/15 pointer-events-none" />
    </>
  ) : (
    <div className="absolute inset-0 bg-[#001f7a]" />
  )}
</div>
```

**关键写法说明**：
- `<img>` 用 `absolute inset-0 w-full h-full` 而不是只 `w-full h-full`——避免依赖父容器的隐式 flex stretch 行为。万一未来外层 flex 改了 `items-start` 之类，img 不会塌成 0 高度。
- 兜底分支也用 `absolute inset-0` 保持和有图分支的定位机制一致
- `object-cover`：保证不同比例的图都能填满展示区（多余部分裁掉，不会留白）
- `bg-black/15` overlay：15% 透明度的黑色蒙层，柔化图片亮度，跟右侧深蓝表单视觉重量平衡。如果实测后觉得太重/太轻，可调到 10-20% 之间

### 4.4 前端 · 后台品牌设置页

文件 [app/admin/settings/page.tsx](../../app/admin/settings/page.tsx)：

抄"联系二维码"那块的 JSX 和处理函数（[:278-359](../../app/admin/settings/page.tsx#L278-L359)，含上传 / 删除 / msg flash），新增"登录页展示图"一节：

- 展示当前图（如有）：`<img src={settings.login_showcase_url} className="max-w-full h-auto" />`
- 上传按钮 + 处理 `POST /api/admin/settings/login-showcase`（抄 [handleQrFileChange](../../app/admin/settings/page.tsx#L97)）
- 删除按钮 + 处理 `DELETE /api/admin/settings/login-showcase`（抄 [clearQr](../../app/admin/settings/page.tsx#L117)，决策 1）
- input 的 `accept` 属性写 `.png,.jpg,.jpeg,.webp`（**不带 .svg .gif**，跟后端白名单对齐）
- **决策 3 = A** + 文件大小说明：按钮下方说明文案：
  > "建议尺寸 1200×1000 以上（左侧展示区比例约 6:5），文件大小 ≤ 2MB（最大 5MB），支持 PNG / JPG / WEBP"
- 前端可以加一个软提示：上传时如果 `file.size > 2 * 1024 * 1024` 给个**警告横幅**"图片较大可能影响登录页加载速度"，但**不阻断**——硬限在后端的 5MB

  > **关于"软警告"用什么组件**：admin/settings 页**不用** `useToast`（那是 admin/workflows、users、agents 三个页用的），它用自家 `flash() + setMsg()` 内联横幅（[:36-39](../../app/admin/settings/page.tsx#L36-L39) + [:140-145](../../app/admin/settings/page.tsx#L140-L145)）。但 flash 当前只有 `ok | err` 两种 type，**没有 warning**。最低成本做法是扩展第三种 type，5 行：
  >
  > ```tsx
  > // type 加 "warn"
  > const [msg, setMsg] = useState<{ type: "ok" | "err" | "warn"; text: string } | null>(null);
  > function flash(type: "ok" | "err" | "warn", text: string) { ... }
  >
  > // 渲染处加一行 warn 样式
  > className={`... ${
  >   msg.type === "ok"  ? "bg-green-50 text-green-700"  :
  >   msg.type === "warn" ? "bg-amber-50 text-amber-700" :
  >                         "bg-red-50 text-red-600"
  > }`}
  > // 图标加 warn 分支：msg.type === "warn" ? <AlertTriangle size={15} /> : ...
  > ```
  >
  > **不**用 useToast，避免同页两套提示机制并存（横幅 + 弹窗）。如果未来要把 settings 整页迁 useToast，那是单独 patch 范围。
- 上传成功后页面 state 立即用新 URL 刷新（不需要整页 reload，参考 [:107](../../app/admin/settings/page.tsx#L107)）

## 五、其他要点

| 项 | 方案 |
|---|---|
| 图片填充方式 | `object-cover`（裁满，无留白）。不用 `object-contain` 因为会留黑边显得丑 |
| 文件格式 | **PNG / JPG / JPEG / WEBP**，不含 SVG / GIF（理由见 §4.1） |
| 文件大小 | **后端硬限 5MB，前端软警告 2MB**。展示图比 logo 大数量级，每个用户首屏都要拉这张图，不能"等出问题再加" |
| 响应式 | 移动端登录页已经 `hidden md:block` 把展示区隐藏，新方案不影响 |
| 多租户切图 | 暂时只支持一张图全平台共用。如果未来要按租户切，独立功能再做 |
| **register 页同步** | 本方案不动 register 页（变更记录-20260513.md §9 有"register 同步"待办）。**留待下个 patch**，届时决策：A) register 共用 `login_showcase_url` 一张图两处用；B) register 单独 `register_showcase_url` 字段；C) register 不加展示图保持单栏。三选一不在本方案范围 |
| Storage 残留 | 上传不同 ext 时旧文件不清理（沿袭 logo / contact-qr 现状），未来如要清理 Storage 一并处理 |
