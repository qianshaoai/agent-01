## 10. 审计记录按组织过滤（5.11up 主菜之三）

详细方案见 `upgrade/5.11up/审计记录组织过滤-方案-20260511.md`。

### 10.1 数据模型变更（migration_v32）

`audit_logs` 加两列：
- `admin_tenant_code`：操作发起人所属组织（org_admin 才有值）
- `resource_tenant_code`：被操作资源所属组织（写时反查得到）

加复合索引便于按组织 + 时间查询。

历史数据 backfill（在 migration 里一次性跑完）：
- `admin_tenant_code`：通过 admin_id 反查 admins.tenant_code
- `resource_tenant_code`：按 resource_type 分别反查（notice / user / dept / team / tenant / workflow / workflow_step / agent）
- 全局资源（category / wf_category / user_group / settings / resource_permission）保留 NULL

### 10.2 lib/audit.ts 重写

- 新增 `resolveResourceTenantCode(resourceType, resourceId)`：按 resource_type 分支反查归属组织
- `writeAuditLog` 接受新参数 `adminTenantCode`，并自动反查 resource_tenant_code 落库
- 反查失败统一返回 null（资源已删除等场景不影响主流程）

### 10.3 52 处调用点全部加 `adminTenantCode: admin.tenantCode ?? null`

用 sed 批量处理两种调用格式：
- 多行格式（`adminRole: admin.role ?? "super_admin",`）：在下一行插入 `adminTenantCode` 字段
- 一行 inline 格式（`adminUsername: admin.username, adminRole: admin.role,`）：行内追加

最终 adminRole 总数 = adminTenantCode 总数 = 52，全覆盖。

### 10.4 读取 API 加组织过滤

`app/api/admin/audit-logs/route.ts`：

```ts
if (admin.role === "org_admin" && admin.tenantCode) {
  query = query.or(`admin_tenant_code.eq.${tc},resource_tenant_code.eq.${tc}`);
}
```

super_admin / system_admin 不加条件，看全部。

## 11. DELETE 路由 snapshot tenant_code（修上面遗留 bug）

用户测试：删工作流后审计记录里看不到这条删除。

原因：DELETE 流程先删了 `resource_permissions`，再删工作流本身，最后 `writeAuditLog` 才反查 `resolveResourceTenantCode("workflow", id)` ——但此时 resource_permissions 已经空了，反查返回 null。同时操作人是 super_admin（admin_tenant_code 也是 null），两个都 null 让 org_admin 的 OR 过滤条件匹配不到。

修法：

- `lib/audit.ts`
  - `resolveResourceTenantCode` 从私有函数改为 `export` 供路由调用
  - `writeAuditLog` 增加可选参数 `resourceTenantCode`，传入时优先使用，跳过反查
- 7 个 DELETE 路由改为"先 snapshot 后删除"模式：
  - `workflows/[id]` —— `resolveResourceTenantCode("workflow", id)` 后再删 resource_permissions / workflows
  - `workflow-steps/[id]` —— 同
  - `notices/[id]` —— 同
  - `departments/[id]` —— 同
  - `teams/[id]` —— 同
  - `agents/[id]` —— 同（在 tenant_agents 删除前 snapshot）
  - `tenants/[id]` —— 自己就是被删的组织，直接 `resourceTenantCode: tenant.code`
- `users/[id]` 是软删（status='deleted'，tenant_code 保留），不用 snapshot
