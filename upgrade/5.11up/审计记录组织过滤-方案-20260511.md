# 审计记录按组织过滤 · 方案稿

版本：5.11up
状态：设计稿
日期：2026-05-11

---

## 一、需求

组织管理员能查看审计记录，但只看本组织相关的变动（其他组织的操作不可见）。

超级管理员 / 系统管理员仍看全部。

---

## 二、技术方案：写时打 tenant 标记（高级版）

### 2.1 为什么不在读时做反查

资源类型多（agent / workflow / workflow_step / notice / user / dept / team / tenant 等），每种归属判断方式不一样：

- notice 直接看 `notices.tenant_code`
- workflow 要查 `resource_permissions`（scope=org/dept/team）
- workflow_step 要先查所属 workflow 再查权限
- user / dept / team 看 `tenant_code` 字段
- agent 通过 `tenant_agents` M2M

读时做意味着每次列表请求要做 6+ 次反查，再用 IN 拼接过滤条件，分页/排序也会很别扭。

写时打标记，读时直接 `WHERE admin_tenant_code = $1 OR resource_tenant_code = $1`，性能可控、未来扩展容易。

### 2.2 数据模型变更

`audit_logs` 加两列：

| 列名 | 含义 | 什么时候填 |
|---|---|---|
| `admin_tenant_code` | 操作发起人的所属组织。org_admin 操作时记录其 tenantCode；super/system 留 NULL（他们不属于具体组织） | 写日志时取 admin.tenantCode |
| `resource_tenant_code` | 被操作资源所属的组织 | 写日志时反查得到 |

也加复合索引方便查询：

```sql
CREATE INDEX idx_audit_logs_admin_tenant      ON audit_logs (admin_tenant_code, created_at DESC);
CREATE INDEX idx_audit_logs_resource_tenant   ON audit_logs (resource_tenant_code, created_at DESC);
```

### 2.3 资源到 tenant_code 的映射规则（写时反查）

写 audit_log 前根据 resourceType + resourceId 反查：

| resource_type | 反查逻辑 | 落地结果 |
|---|---|---|
| `notice` | 查 `notices.tenant_code` | notice 的 tenant_code（global 为 NULL）|
| `workflow` | 查 `resource_permissions` 中第一条 scope=org 的 scope_id | 组织 code，多组织取第一条；无则 NULL |
| `workflow_step` | 先查 `workflow_steps.workflow_id` 拿到 workflow，再走 workflow 的逻辑 | 同上 |
| `user` | 查 `users.tenant_code` | tenant_code |
| `department` | 查 `departments.tenant_code` | tenant_code |
| `team` | 查 `teams.dept_id → departments.tenant_code` | tenant_code |
| `tenant` | resourceId 本身就是 tenant code | resourceId |
| `agent` | 查 `tenant_agents.tenant_code` 取第一条（多对多场景） | 第一个绑定的 tenant code |
| `category` / `wf_category` / `user_group` / `settings` / `resource_permission` | 视为全局资源 | NULL |

如果反查失败（资源已被删除等），落 NULL 即可。

### 2.4 读时过滤

`/api/admin/audit-logs` GET：

```ts
if (admin.role === "org_admin") {
  query = query.or(
    `admin_tenant_code.eq.${tc},resource_tenant_code.eq.${tc}`
  );
}
```

含义："本组织管理员发起的" 或 "操作了本组织资源的"。这样既包括 org_admin 自己的操作，也包括 super_admin 影响本组织的操作。

NULL（全局资源的事件、global notice 等）对 org_admin 不可见。这符合"只看本组织相关"的语义。

### 2.5 历史数据 backfill

新增 `supabase/migration_v32.sql`：

```sql
-- 加列
ALTER TABLE audit_logs
  ADD COLUMN IF NOT EXISTS admin_tenant_code    TEXT,
  ADD COLUMN IF NOT EXISTS resource_tenant_code TEXT;

-- 索引
CREATE INDEX IF NOT EXISTS idx_audit_logs_admin_tenant    ON audit_logs (admin_tenant_code, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_audit_logs_resource_tenant ON audit_logs (resource_tenant_code, created_at DESC);

-- backfill admin_tenant_code：通过 admin_id 反查 admins.tenant_code
-- 注：admin_id 可能指向 admins 表也可能指向 users 表（schema 注释里说的），
-- 但 5.7up 之后 admins 表的 admin 才会有 tenant_code，users 不参与审计写入。
UPDATE audit_logs SET admin_tenant_code = a.tenant_code
FROM admins a
WHERE audit_logs.admin_id = a.id
  AND audit_logs.admin_tenant_code IS NULL
  AND a.tenant_code IS NOT NULL;

-- backfill resource_tenant_code（分 resource_type 来）
-- notice
UPDATE audit_logs SET resource_tenant_code = n.tenant_code
FROM notices n
WHERE audit_logs.resource_type = 'notice'
  AND audit_logs.resource_id = n.id::text
  AND audit_logs.resource_tenant_code IS NULL;

-- user
UPDATE audit_logs SET resource_tenant_code = u.tenant_code
FROM users u
WHERE audit_logs.resource_type = 'user'
  AND audit_logs.resource_id = u.id::text
  AND audit_logs.resource_tenant_code IS NULL;

-- department
UPDATE audit_logs SET resource_tenant_code = d.tenant_code
FROM departments d
WHERE audit_logs.resource_type = 'department'
  AND audit_logs.resource_id = d.id::text
  AND audit_logs.resource_tenant_code IS NULL;

-- team（经 dept 到 tenant_code）
UPDATE audit_logs SET resource_tenant_code = d.tenant_code
FROM teams t
JOIN departments d ON t.dept_id = d.id
WHERE audit_logs.resource_type = 'team'
  AND audit_logs.resource_id = t.id::text
  AND audit_logs.resource_tenant_code IS NULL;

-- tenant（resource_id 本身就是 tenant code）
UPDATE audit_logs SET resource_tenant_code = audit_logs.resource_id
WHERE audit_logs.resource_type = 'tenant'
  AND audit_logs.resource_tenant_code IS NULL;

-- workflow：取 resource_permissions 中第一条 scope=org 的 scope_id
UPDATE audit_logs SET resource_tenant_code = rp.scope_id
FROM (
  SELECT DISTINCT ON (resource_id) resource_id, scope_id
  FROM resource_permissions
  WHERE resource_type = 'workflow' AND scope_type = 'org'
  ORDER BY resource_id, created_at
) rp
WHERE audit_logs.resource_type = 'workflow'
  AND audit_logs.resource_id = rp.resource_id::text
  AND audit_logs.resource_tenant_code IS NULL;

-- workflow_step：经 workflow_id → workflow 的 org scope
UPDATE audit_logs SET resource_tenant_code = rp.scope_id
FROM workflow_steps ws
JOIN (
  SELECT DISTINCT ON (resource_id) resource_id, scope_id
  FROM resource_permissions
  WHERE resource_type = 'workflow' AND scope_type = 'org'
  ORDER BY resource_id, created_at
) rp ON ws.workflow_id = rp.resource_id
WHERE audit_logs.resource_type = 'workflow_step'
  AND audit_logs.resource_id = ws.id::text
  AND audit_logs.resource_tenant_code IS NULL;

-- agent（取 tenant_agents 第一条绑定）
UPDATE audit_logs SET resource_tenant_code = ta.tenant_code
FROM (
  SELECT DISTINCT ON (agent_id) agent_id, tenant_code
  FROM tenant_agents
  ORDER BY agent_id, created_at
) ta
WHERE audit_logs.resource_type = 'agent'
  AND audit_logs.resource_id = ta.agent_id::text
  AND audit_logs.resource_tenant_code IS NULL;
```

### 2.6 writeAuditLog 改造

把 tenant 反查逻辑封装到 `lib/audit.ts` 内部，调用方无感：

```ts
async function resolveResourceTenantCode(resourceType, resourceId): string | null {
  switch (resourceType) {
    case 'notice':
      return (await db.from('notices').select('tenant_code').eq('id', resourceId).single()).data?.tenant_code ?? null;
    case 'workflow':
      return (await db.from('resource_permissions').select('scope_id')
        .eq('resource_type', 'workflow').eq('resource_id', resourceId)
        .eq('scope_type', 'org').limit(1).maybeSingle()).data?.scope_id ?? null;
    case 'workflow_step':
      const { data: step } = await db.from('workflow_steps').select('workflow_id').eq('id', resourceId).single();
      if (!step?.workflow_id) return null;
      return resolveResourceTenantCode('workflow', step.workflow_id);
    case 'user':
      return (await db.from('users').select('tenant_code').eq('id', resourceId).single()).data?.tenant_code ?? null;
    case 'department':
      return (await db.from('departments').select('tenant_code').eq('id', resourceId).single()).data?.tenant_code ?? null;
    case 'team':
      const { data: team } = await db.from('teams').select('dept_id').eq('id', resourceId).single();
      if (!team?.dept_id) return null;
      return (await db.from('departments').select('tenant_code').eq('id', team.dept_id).single()).data?.tenant_code ?? null;
    case 'tenant':
      return resourceId; // tenant code 就是 id
    case 'agent':
      return (await db.from('tenant_agents').select('tenant_code').eq('agent_id', resourceId).limit(1).maybeSingle()).data?.tenant_code ?? null;
    default:
      return null;
  }
}
```

writeAuditLog 调整：

```ts
export async function writeAuditLog(params: {
  ...
  adminTenantCode?: string | null; // 由调用方传，简化反查
}) {
  const resourceTenantCode = params.resourceId
    ? await resolveResourceTenantCode(params.resourceType, params.resourceId)
    : null;
  await db.from('audit_logs').insert({
    ...existing fields,
    admin_tenant_code: params.adminTenantCode ?? null,
    resource_tenant_code: resourceTenantCode,
  });
}
```

### 2.7 调用方调整

`requireAdmin()` 返回的 admin 对象本来就有 `tenantCode` 字段（org_admin 才有值，其他角色 null）。调用 writeAuditLog 时多传一个参数。

简化策略：把当前 admin 对象的 tenantCode 自动透传，不让调用方关心。考虑到调用点多，封装一个 helper：

```ts
// 在 lib/audit.ts
export async function writeAuditLogFromAdmin(
  admin: { adminId; username; role; tenantCode? },
  params: Omit<WriteAuditParams, 'adminId' | 'adminUsername' | 'adminRole' | 'adminTenantCode'>
) {
  return writeAuditLog({
    adminId: admin.adminId,
    adminUsername: admin.username,
    adminRole: admin.role,
    adminTenantCode: admin.tenantCode ?? null,
    ...params,
  });
}
```

但这是更彻底的重构，本期就先扩展 `writeAuditLog` 参数 + 让调用方多传 `adminTenantCode: admin.tenantCode ?? null` 一行。

## 五、风险与注意

| 风险 | 处理 |
|---|---|
| 写日志慢一点（多了反查） | 每个 resource_type 最多一次轻量查询，可接受 |
| 反查失败导致 resource_tenant_code = NULL | NULL 对 org_admin 不可见，等于"全局事件"。已被删除的资源历史日志会全部 NULL，但不影响功能 |
| backfill 把已知错误 NULL 锁死 | 不会，backfill 后再写新日志正常反查；老 NULL 数据有遗漏可手工跑 UPDATE 补 |
| 全局资源（category/settings）操作 org_admin 看不到 | 符合"只看本组织"的设计 |
| 性能：列表分页 OR 多 tenant_code 字段 | 加了两个 BTree 索引，分页 + 过滤可接受 |
