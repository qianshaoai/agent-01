# 管理员上下级权限 · 方案稿

版本：5.11up
状态：设计稿，待评审
日期：2026-05-11

---

## 一、需求

下级管理员不能删除或修改上级管理员创建的工作流。

管理员等级（高 → 低）：

```
super_admin   超级管理员（平台拥有者）
    ↓
system_admin  系统管理员（平台运维）
    ↓
org_admin     组织管理员（绑定 tenantCode，限本组织）
```

规则：

- super_admin 可以删/改 任何工作流
- system_admin 可以删/改 system_admin 或 org_admin 创建的，但不能动 super_admin 创建的
- org_admin 只能删/改 自己（org_admin）创建的且在自己组织范围内的工作流

## 二、数据模型变更

### 2.1 新增列

workflows 表新增 2 列：

```sql
ALTER TABLE workflows
  ADD COLUMN IF NOT EXISTS created_by         UUID REFERENCES admins(id) ON DELETE SET NULL,
  ADD COLUMN IF NOT EXISTS created_by_role    TEXT;  -- 'super_admin' | 'system_admin' | 'org_admin'

CREATE INDEX IF NOT EXISTS idx_workflows_created_by_role ON workflows (created_by_role);
```

两个字段都存：

- `created_by` —— 创建者 admin ID。便于查询"谁创建的"和审计追溯
- `created_by_role` —— 创建时的角色快照。即使后续创建者角色变了，原工作流的"上级保护"不变

> 为什么要 snapshot？防止"创建时是 super，后来降级为 org_admin"这种情况下保护级别错位。

### 2.2 历史数据如何处理

历史工作流（current_step_idx = 1.0 时代留下的）created_by 为 NULL，created_by_role 为 NULL。

**策略**：把 NULL 视为 super_admin 创建（最保守）。这意味着历史工作流只有超级管理员可以删改。

如果用户觉得太严，可以加一个 migration：

```sql
-- 备选：把无 creator 的历史工作流降级为 system_admin 创建
UPDATE workflows SET created_by_role = 'system_admin' WHERE created_by_role IS NULL;
```

**建议**：先按"NULL = super_admin"实施。让用户看实际体验后再决定要不要降级。

### 2.3 是否对 workflow_steps 也做检查？

不需要单独检查。原因：

- workflow_steps 通过 workflow_id 外键挂在 workflow 上
- 任何 step 写操作（增/改/删）的前置检查都是"先确认该 workflow 你有权改"
- 所以只要 workflow 层面的写入检查到位，step 自然受保护

具体代码层面，需要给 `/api/admin/workflow-steps/[id]` 和 `/api/admin/workflows/[id]/steps` POST 也加上同一个 guard。

## 三、权限矩阵

| 操作者 → 创建者 | super_admin | system_admin | org_admin |
|---|---|---|---|
| super_admin | ✓ | ✓ | ✓ |
| system_admin | ✗ | ✓ | ✓ |
| org_admin | ✗ | ✗ | ✓（且 tenant 匹配）|

ChatGPT 助记口诀：等级 ≥ 创建者的等级才能动。

枚举 → 数值：super=3, system=2, org=1。
检查公式：`actor_level >= creator_level`。

## 四、后端实施

### 4.1 工具函数

新增 `lib/admin-permissions.ts`：

```ts
type AdminRole = "super_admin" | "system_admin" | "org_admin";

const ROLE_LEVEL: Record<AdminRole, number> = {
  super_admin: 3,
  system_admin: 2,
  org_admin: 1,
};

export function canActOnRole(actorRole: AdminRole, creatorRole: AdminRole | null): boolean {
  // creatorRole 为 null（历史数据）按 super_admin 处理
  const creatorLevel = ROLE_LEVEL[creatorRole ?? "super_admin"];
  return ROLE_LEVEL[actorRole] >= creatorLevel;
}
```

### 4.2 受影响的路由

| 路由 | 方法 | 需要补 |
|---|---|---|
| `/api/admin/workflows` | POST | 写入时填 `created_by` 和 `created_by_role` |
| `/api/admin/workflows/[id]` | PATCH | 调用 `canActOnRole` 检查 |
| `/api/admin/workflows/[id]` | DELETE | 调用 `canActOnRole` 检查 |
| `/api/admin/workflows/[id]/duplicate` | POST | duplicate 也算"修改源" + 新副本 created_by 是当前 admin |
| `/api/admin/workflows/[id]/steps` | POST | 调用 `canActOnRole`（受 workflow 的保护）|
| `/api/admin/workflow-steps/[id]` | PATCH/DELETE | 通过 step 反查 workflow，再调用 `canActOnRole` |

### 4.3 错误信息

403 + 中文 message：

```json
{ "error": "无权操作：该工作流由更高级别的管理员创建" }
```

## 五、前端实施

### 5.1 列表页 `admin/workflows`

每条工作流的"编辑 / 删除"按钮根据 `canActOnRole` 决定：

- 可操作 → 正常显示
- 不可操作 → 按钮变灰 + tooltip "由更高级别管理员创建，无权修改"

API 已经返回 `created_by` 和 `created_by_role` 字段，前端读 `me.role` 和 `wf.created_by_role` 比较即可。

为了让用户能知道"谁建的"，badge 旁边加一行小字：

```
创建者：admin（超级管理员）· 2026-05-11
```

### 5.2 工作流详情步骤

进入步骤编辑页时，如果不可写，整个页面变只读模式（参考 readonly mode 的实现）：

- 隐藏"新增步骤 / 编辑步骤 / 删除步骤"按钮
- 顶部提示横幅"该工作流由 XXX 创建，您无权修改"

### 5.3 dashboard 卡片

"工作流"卡片显示数字时不变。但点进去后受保护的不能改。

## 七、Migration 文件

新增 `supabase/migration_v31.sql`：

```sql
-- migration_v31
-- 版本：v31
-- 来自：5.11up
-- 改了什么表：workflows 新增 created_by + created_by_role
-- 是否需要数据迁移：否（NULL 视为 super_admin 创建）

ALTER TABLE workflows
  ADD COLUMN IF NOT EXISTS created_by         UUID REFERENCES admins(id) ON DELETE SET NULL,
  ADD COLUMN IF NOT EXISTS created_by_role    TEXT;

CREATE INDEX IF NOT EXISTS idx_workflows_created_by_role ON workflows (created_by_role);

-- 添加 CHECK 约束保证 role 字段只能是这三个值之一
ALTER TABLE workflows
  ADD CONSTRAINT chk_workflows_created_by_role
  CHECK (created_by_role IS NULL OR created_by_role IN ('super_admin', 'system_admin', 'org_admin'));
```

## 九、风险

| 风险 | 评估 | 应对 |
|---|---|---|
| 历史工作流被锁死（NULL → super_admin） | 中：org_admin 看到自己原来能改的现在改不了，会困惑 | UI 上明确提示"该工作流由超级管理员管理"，让他们去找超管 |
| Migration 跑了一半失败 | 低：ADD COLUMN IF NOT EXISTS 幂等 | 重跑即可 |
| 后续要扩展更多角色 | 低：枚举 ROLE_LEVEL 即可 | 留好扩展位 |
| created_by_role snapshot 与 admins 表实际 role 不一致 | 设计上就是这样 | 不是 bug，是 feature |
