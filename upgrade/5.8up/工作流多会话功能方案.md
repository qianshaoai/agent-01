# 工作流多会话功能方案

版本：5.9  
状态：设计稿，待评审  
目标：让用户能同时跑多次同一个工作流（处理不同工作），会话之间对话记录完全隔离，并支持命名、保存、随时恢复。

---

## 一、核心概念

| 概念 | 说明 |
|---|---|
| **工作流模板**（workflow） | 管理员定义的流程骨架，不变 |
| **工作流会话**（workflow_session） | 用户某一次执行工作流产生的实例，有自己的名称、进度、对话记录 |

用户可以同时拥有多个会话：同一个"知识库整理"工作流，可以同时跑"项目A资料整理"和"项目B资料整理"，互不干扰。

---

## 二、数据库变更

### 2.1 新增表：`workflow_sessions`

```sql
CREATE TABLE workflow_sessions (
  id              uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         uuid        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  workflow_id     uuid        NOT NULL REFERENCES workflows(id) ON DELETE CASCADE,
  name            text        NOT NULL DEFAULT '未命名会话',
  current_step_idx int        NOT NULL DEFAULT 0,
  status          text        NOT NULL DEFAULT 'in_progress',
    -- in_progress | completed | abandoned
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_wf_sessions_user      ON workflow_sessions (user_id, updated_at DESC);
CREATE INDEX idx_wf_sessions_workflow  ON workflow_sessions (workflow_id);
CREATE INDEX idx_wf_sessions_status    ON workflow_sessions (user_id, status);
```

### 2.2 修改表：`conversations` — 新增 `session_id` 列

```sql
ALTER TABLE conversations
  ADD COLUMN session_id uuid REFERENCES workflow_sessions(id) ON DELETE SET NULL;

CREATE INDEX idx_conversations_session ON conversations (session_id);
```

**设计说明：**
- `session_id` 为 nullable：普通聊天（非工作流）的对话不设 session_id，行为与现在完全一致
- 工作流内创建的对话强制携带 session_id，实现隔离
- 同一个智能体在不同 session 中拥有各自独立的 conversation

### 2.3 迁移文件

- `supabase/migration_v29.sql`：建 `workflow_sessions` 表
- `supabase/migration_v30.sql`：给 `conversations` 加 `session_id` 列

## 三、API 变更

### 3.1 新增：工作流会话 CRUD

```
POST   /api/workflow-sessions            创建会话（开始新工作流）
GET    /api/workflow-sessions            列出当前用户的所有会话
PATCH  /api/workflow-sessions/[id]       更新会话（改名 / 更新进度 / 标记完成）
DELETE /api/workflow-sessions/[id]       删除会话
```

**POST body：**
```json
{ "workflowId": "uuid", "name": "项目A资料整理" }
```

**GET response：**
```json
[
  {
    "id": "uuid",
    "name": "项目A资料整理",
    "workflow": { "id": "uuid", "name": "知识库整理工作流" },
    "currentStepIdx": 2,
    "totalSteps": 5,
    "status": "in_progress",
    "updatedAt": "2026-05-08T10:00:00Z"
  }
]
```

**PATCH body（支持部分更新）：**
```json
{ "name": "新名称" }
{ "currentStepIdx": 3 }
{ "status": "completed" }
```

### 3.2 修改：`/api/conversations`

新增 `sessionId` 查询参数，用于在工作流内查找或创建会话范围内的对话：

```
GET  /api/conversations?agentCode=xxx&sessionId=uuid
POST /api/conversations   body 中新增可选字段 session_id
```

- 带 sessionId 查询时：只返回该 session 内、属于该 agent 的对话
- 不带 sessionId：行为与现在一致（兼容普通聊天）

### 3.3 修改：`/api/agents/[id]/chat`

请求 body 新增可选字段 `sessionId`：

```json
{ "message": "...", "sessionId": "uuid", ... }
```

- 若携带 sessionId：创建/复用该 session 内属于此 agent 的对话，conversation 写入时带上 session_id
- 若不携带：行为与现在完全一致

## 四、前端变更

### 4.1 URL 结构调整

现在：`/agents/CODE?wf=WF_ID&step=N`  
新增：`/agents/CODE?wf=WF_ID&step=N&session=SESSION_ID`

`session` 参数是关键——所有对话操作都通过它定位到正确的隔离上下文。  
`wf` 和 `step` 保留，用于工作流步骤加载和进度条渲染。

localStorage 的 `wf_progress_*` 仍可保留作为短暂缓存，但以服务端 session 为准。

### 4.2 首页：新增"我的工作流"区块

位置：在工作流卡片列表上方（或独立 tab）

```
┌──────────────────────────────────────────────────────┐
│  我的进行中工作流                           [查看全部] │
├──────────────────────────────────────────────────────┤
│  📋 项目A资料整理          知识库整理工作流            │
│     第 3 步 / 共 5 步  · 更新于 05/08 10:00  [继续]  │
├──────────────────────────────────────────────────────┤
│  📋 项目B资料整理          知识库整理工作流            │
│     第 1 步 / 共 5 步  · 更新于 05/07 16:30  [继续]  │
└──────────────────────────────────────────────────────┘
```

点击"继续"直接跳到 `/agents/AGENT_CODE?wf=WF_ID&step=N&session=SESSION_ID`。

### 4.3 工作流卡片：改为"新建会话"入口

现在点击工作流直接进入第一步；新逻辑：

```
点击工作流卡片
    ↓
弹出命名对话框
  ┌─────────────────────────────┐
  │  开始新的工作流              │
  │  工作流名称：知识库整理工作流 │
  │  会话名称：[_______________] │  ← 默认为当前日期+工作流名
  │              [取消]  [开始] │
  └─────────────────────────────┘
    ↓ 点开始
  POST /api/workflow-sessions → 拿到 session id
    ↓
  跳转到第一个步骤
```

### 4.4 智能体页面：适配 session 隔离

**变更点一：读取 session 参数**
```typescript
const sessionId = searchParams.get("session");
```

**变更点二：加载对话列表时带 sessionId**
```
GET /api/conversations?agentCode=xxx&sessionId=SESSION_ID
```
返回的是本次工作流会话专属的对话列表，不会和其他 session 混在一起。

**变更点三：发消息时带 sessionId**
```json
POST /api/agents/xxx/chat  body: { ..., "sessionId": "SESSION_ID" }
```

**变更点四：前进步骤时更新 session 进度**
```
PATCH /api/workflow-sessions/SESSION_ID  { "currentStepIdx": nextIdx }
```
替代现在写 localStorage 的逻辑（localStorage 可保留作乐观更新缓存）。

**变更点五：跨步骤上下文注入**

现在：找"上一个智能体的最新对话"（全局，可能跨 session 污染）  
新逻辑：找"上一个智能体在本 session 内的对话"
```
GET /api/conversations?agentCode=PREV_CODE&sessionId=SESSION_ID
```
完全隔离，互不干扰。

**变更点六：工作流完成时标记 session**
```
PATCH /api/workflow-sessions/SESSION_ID  { "status": "completed" }
```

### 4.5 会话管理页（可选，二期）

路径：`/my-workflows`

- 列出全部 in_progress / completed 会话
- 支持搜索、按工作流筛选
- 每条记录可重命名、删除、查看历史

## 五、改动范围汇总

| 层 | 改动 | 工作量估计 |
|---|---|---|
| DB | 新表 workflow_sessions，conversations 加列 | 小 |
| API | 新增 `/api/workflow-sessions` CRUD | 小 |
| API | `/api/conversations` 支持 sessionId 过滤 | 小 |
| API | `/api/agents/[id]/chat` 接收 sessionId，conversation 写入带 session_id | 中 |
| 前端·首页 | 新增"我的进行中工作流"区块，改工作流卡片点击为命名弹窗 | 中 |
| 前端·智能体页 | 读 `?session=`，所有 conversation/chat 请求带 sessionId，步进时 PATCH session | 中 |
| 前端·进度条 | 不改逻辑，只把 localStorage 写入替换为 PATCH session | 小 |

**总计：约 2~3 天工作量**

---

## 六、兼容性说明

- 现有普通聊天（不通过工作流进入）：`session_id = null`，完全不受影响
- 现有工作流（通过 localStorage 跟踪进度）：老 URL 不带 `session` 参数时，降级为当前行为（localStorage 模式），不强制迁移
- 新工作流：必须通过"新建会话"入口进入，全程带 sessionId

## 七、待决定事项

1. **命名弹窗时机**：点击工作流卡片立即弹，还是进入工作流后再提示保存？  
   建议：立即弹，保持每次都有明确的会话名。

2. **最大会话数限制**：是否限制每个用户每个工作流最多 N 个 in_progress 会话？  
   建议：不限制，前端展示做好分页即可。

3. **会话自动过期**：长时间未更新的 in_progress 会话是否自动归档？  
   建议：不自动，由用户手动删除。

4. **首页展示**：只展示 in_progress 的，还是也展示最近完成的？  
   建议：默认只展示 in_progress，完成的放"查看全部"页。
