---

## 一、需求 & 痛点

**现状**：每个智能体的 API key 各存各的（`agents.api_key_enc`）。key 一旦过期，
每个用到它的智能体都要点进配置弹窗重新复制粘贴，几十个智能体时极其繁琐。

**目标**：API key 集中管理 + 命名 + **一处更新，处处生效**。智能体不再手输 key，
改为从下拉菜单选择已命名的 API。

---

## 二、模块结构

把 PR-A 的「模型接入」一级菜单升级为「API 管理」，下分两类：

```
API 管理（一级菜单，由「模型接入」升级而来）
 ├── 大模型 API   —— OpenAI / 智谱 GLM / DeepSeek 等（= 现在的 model_providers）
 └── 智能体 API   —— 扣子 Coze / Dify / 腾讯元器 / 清言 的平台凭证
```

用户在这里为每个 API 命名、更新 key，智能体侧只引用、不再持有 key。

---

## 三、数据库设计

**核心判断**：大模型 API 和智能体 API 结构几乎一样（平台 + endpoint + key + 名称），
没必要拆两张表。

**推荐方案**：复用 PR-A 已建的 `model_providers` 表，只 ALTER 加 1 列 `category`：

```
model_providers 表（PR-A 已建，本次只 ALTER 加 1 列）
+ category  TEXT  NOT NULL DEFAULT 'model'
+ CHECK (category IN ('model', 'agent'))
```

- `category = 'model'` → 大模型 API
- `category = 'agent'` → 智能体 API（Coze / Dify / 元器 / 清言）
- UI 按 `category` 分两个 tab 展示。
- **category ↔ platform 映射**（建/迁移时据此归类）：
  - `model`：`openai`、`zhipu`
  - `agent`：`coze`、`dify`、`yuanqi`、`qingyan`
  - 新建时所在 tab 决定 category，platform 下拉只列该 category 下的平台。
- 新增索引：
  - `(category, enabled)`：下拉只查启用项时使用。
  - `(category, platform)`：按类型 + 平台筛选时使用。

> 表名建议**保留 `model_providers` 不改** —— 表名是内部标识，改名要动一堆代码引用，
> 收益低风险高。UI 上叫「API 管理」即可，表名与 UI 名称解耦。

迁移文件：`migration_v37_model_providers_category.sql`（编号续 v36，待最终确认编号）。

## 四、智能体如何引用 API

`agents` 表已有 `provider_id`（PR-A/B 加的列，目前只有搭建器发布的新 agent 用到）。
本次让**所有正式 agent（`agents` 表记录）统一走 `provider_id` 引用**：

- 智能体管理的「API 配置」弹窗：主操作改为 **"选择已命名 API"下拉**。
- 选中后 `agent.provider_id` 指向那条命名 API 记录。
- `agents.api_key_enc` 进入**废弃过渡期**，但聊天链路必须调整优先级：
  1. `agent.provider_id` 存在 → 优先使用 `model_providers` 的 key / endpoint / 平台配置；
  2. `agent.provider_id` 为空且 `agent.api_key_enc` 存在 → 使用旧 agent 自带 key；
  3. 仍无配置且是 OpenAI 旧链路 → fallback 到 `tenant.openai_key_enc`；
  4. 都没有 → 返回“智能体未配置模型 API”。

> 关键修正：不能继续保持“agent 自带 key → provider_id”的旧优先级。否则迁移脚本即使回填
> `provider_id`，只要 `api_key_enc` 还保留，线上仍会走旧 key，API 管理里更新 key 不会生效。

provider 异常处理：

- 只要 `agent.provider_id` 非空，就认为该 agent 已迁移到集中 API 管理。
- 绑定的 provider 被删除、禁用、key 为空、endpoint 为空或解密失败时，聊天接口**阻断报错**，提示管理员检查 API 管理配置。
- 不再 fallback 到旧 `agent.api_key_enc`，避免集中禁用/清空 key 后仍被旧 key 绕过。
- 仅当 `agent.provider_id` 为空时，才允许进入旧 `agent.api_key_enc` fallback。

## 六、数据迁移（现有几十个 agent 的 key 怎么归拢）

现有每个 agent 自存 key。要让它们用上命名 API，需一次性迁移：

- **方案 A（推荐）**：脚本扫描所有正式 agent，**相同调用配置的归到同一条命名 API**，
  自动命名（如「扣子-默认-1」「智谱-默认-1」）；名字之后可在 API 管理里手动改。
  迁移后回写 `agent.provider_id`。
- 方案 B：人工逐个梳理（几十个智能体太累，不推荐）。

迁移脚本只新增 `model_providers` 记录 + 回填 `provider_id`，**不删 `api_key_enc`**，
可安全重跑、可回滚。

迁移分组规则：

- 不能按 `api_key_enc` 密文字段直接分组；当前加密使用随机 IV，同一个明文 key 多次加密后的密文也不同。
- 也不能只按明文 key 合并；provider 会承载平台、endpoint、key、默认参数，合并维度至少应为：
  `category + platform + api_endpoint/base_url + key fingerprint`。
- 如 agent 的 `model_params` 中存在会改变调用路由的字段（如 `model`、bot id、workflow id、app id 等），
  也应纳入分组，或保留在 agent 自身参数中并确认不会影响 provider 复用。
- 脚本应逐条解密旧 key，只在内存中计算临时 fingerprint（建议 HMAC-SHA256）。
- HMAC secret 必须使用同一环境的固定值，不能每次运行随机生成；否则 fingerprint / `provider_code` 不稳定，
  重跑会重复创建 provider。建议用单独的 `MIGRATION_FINGERPRINT_SECRET`，仅迁移脚本读取。
- 脚本日志严禁输出明文 key、解密后的 endpoint token、完整 fingerprint；最多输出 agent 数量、平台、生成的 provider_code。
- 重跑幂等策略：生成稳定 `provider_code`，如
  `migrated-{category}-{platform}-{endpointHash前6位}-{fingerprint前8位}`；若已存在则复用并继续回填。
- 若 `provider_code` 冲突但分组内容不一致，脚本应停止并要求人工确认，不自动覆盖。
- 迁移后保留 `api_key_enc`，但由于聊天链路已改为 `provider_id` 优先，API 管理更新才能立即对已迁移 agent 生效。

## 十、风险与回滚

- **旧 agent 兼容**：PR-2 调整优先级后，`provider_id` 为空的未迁移 agent 仍能用自带 key /
  tenant OpenAI key fallback，迁移失败不影响线上对话；`provider_id` 非空的已迁移 agent 则严格走 provider。
- **回滚**：`category` 列有默认值，新增不破坏现有查询；数据迁移脚本只增不删，
  误迁可清掉新增的 `model_providers` 记录 + 把 `provider_id` 置空恢复。
- **集中 API 生效风险**：如果 chat route 没有改成 `provider_id` 优先，保留的旧 `api_key_enc`
  会抢先命中，导致 API 管理更新不生效。本项是 PR-2 的阻断检查。
- **provider 异常风险**：已绑定 `provider_id` 的 agent 遇到 provider 禁用、删除、key 为空等情况时必须阻断报错，
  不 fallback 旧 key，避免绕过集中管理。
- **D5 分阶段删除**：PR-2 先支持下拉引用，PR-3 完成迁移，PR-4 再移除旧手输入口。
  否则配置弹窗可能出现"无可选项"，或旧 agent 还没迁移就无法维护。
