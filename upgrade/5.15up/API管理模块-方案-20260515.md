# API 管理模块 · 方案

日期：2026-05-15
分支：feature/agent-builder-pr-d
状态：**待评审 / D1-D5 待拍板**
关联：承接 5.15up PR-A「模型 API 统一管理」（`model_providers` 表已建）

---

## 一、需求 & 痛点

**现状**：每个智能体的 API key 各存各的（`agents.api_key_enc`）。key 一旦过期，
每个用到它的智能体都要点进配置弹窗重新复制粘贴，几十个智能体时极其繁琐。

**目标**：API key 集中管理 + 命名 + **一处更新，处处生效**。智能体不再手输 key，
改为从下拉菜单选择已命名的 API。

---

## 二、模块结构

把 PR-A 的「模型接入」一级菜单升级为「API 管理」，下分两类：

```
API 管理（一级菜单，由「模型接入」升级而来）
 ├── 大模型 API   —— OpenAI / 智谱 GLM / DeepSeek 等（= 现在的 model_providers）
 └── 智能体 API   —— 扣子 Coze / Dify / 腾讯元器 / 清言 的平台凭证
```

用户在这里为每个 API 命名、更新 key，智能体侧只引用、不再持有 key。

---

## 三、数据库设计

**核心判断**：大模型 API 和智能体 API 结构几乎一样（平台 + endpoint + key + 名称），
没必要拆两张表。

**推荐方案**：复用 PR-A 已建的 `model_providers` 表，只 ALTER 加 1 列 `category`：

```
model_providers 表（PR-A 已建，本次只 ALTER 加 1 列）
+ category  TEXT  NOT NULL DEFAULT 'model'   -- 'model' | 'agent'
```

- `category = 'model'` → 大模型 API
- `category = 'agent'` → 智能体 API（Coze / Dify / 元器 / 清言）
- UI 按 `category` 分两个 tab 展示。

> 表名建议**保留 `model_providers` 不改** —— 表名是内部标识，改名要动一堆代码引用，
> 收益低风险高。UI 上叫「API 管理」即可，表名与 UI 名称解耦。

迁移文件：`migration_v37_model_providers_category.sql`（编号续 v36，待最终确认编号）。

---

## 四、智能体如何引用 API

`agents` 表已有 `provider_id`（PR-A/B 加的列，目前只有搭建器发布的新 agent 用到）。
本次让**所有 agent 统一走 `provider_id` 引用**：

- 智能体管理的「API 配置」弹窗：把"输入 API Key"输入框 → 改成 **"选择已命名 API"下拉**。
- 选中后 `agent.provider_id` 指向那条命名 API 记录。
- `agents.api_key_enc` 进入**废弃过渡期** —— chat route 已有三级 fallback
  （agent 自带 key → provider_id → tenant.openai_key_enc，见 PR-D），
  旧 agent 即使没迁移也不受影响。

## 五、前端改动

| 页面 | 改动 |
|---|---|
| 「模型接入」→「API 管理」 | 顶部加两个 tab：大模型 API / 智能体 API；新增/编辑时带 `category` |
| 智能体管理「API 配置」弹窗 | "输入 API Key" 文本框 → "选择命名 API" 下拉 + "去 API 管理新建"快捷链接 |
| 导航菜单 | 「模型接入」改名「API 管理」 |

下拉项按 `category` 过滤：智能体（Coze 等平台）的配置弹窗只列 `category='agent'` 的
API，大模型类只列 `category='model'`，避免选错。

## 六、数据迁移（现有几十个 agent 的 key 怎么归拢）

现有每个 agent 自存 key。要让它们用上命名 API，需一次性迁移：

- **方案 A（推荐）**：脚本扫描所有 agent，**相同 key 的归到同一条命名 API**，
  自动命名（如「扣子-默认-1」「智谱-默认-1」）；名字之后可在 API 管理里手动改。
  迁移后回写 `agent.provider_id`。
- 方案 B：人工逐个梳理（几十个智能体太累，不推荐）。

迁移脚本只新增 `model_providers` 记录 + 回填 `provider_id`，**不删 `api_key_enc`**，
可安全重跑、可回滚。

## 十、风险与回滚

- **旧 agent 兼容**：chat route 三级 fallback 已就位，未迁移的 agent 仍能用自带 key，
  迁移失败不影响线上对话。
- **回滚**：`category` 列有默认值，新增不破坏现有查询；数据迁移脚本只增不删，
  误迁可清掉新增的 `model_providers` 记录 + 把 `provider_id` 置空恢复。
- **D5 若选"完全删除"手输入口**：需确保迁移脚本先跑完、所有 agent 都有
  可选 API，否则配置弹窗会出现"无可选项"。建议 PR-3 迁移完成后再上 PR-2 的删除。
