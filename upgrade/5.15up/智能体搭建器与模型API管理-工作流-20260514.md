# 智能体搭建器与模型 API 管理 - 具体工作流

**日期**：2026-05-14  
**状态**：规划中，待评审后实施  
**目标版本**：5.15up  
**适用范围**：正式版后台、用户端智能体聊天、工作流绑定体系  
**核心目标**：把现有“后台登记智能体”升级为“后台可配置、可测试、可发布智能体”，并把大模型 API 接入做成统一模型网关。

---

## 0. 当前基础判断

现有平台已经具备以下基础：

- `agents` 表已经保存智能体基础信息、平台类型、API endpoint、加密 API key、`model_params`
- `/api/agents/[id]/chat` 已经负责用户聊天、额度校验、日志记录、历史上下文、流式返回
- `lib/adapters/index.ts` 已有多平台适配器：Coze、Dify、元器、清言、OpenAI-compatible
- 后台 `/admin/agents` 已有智能体管理能力
- 工作流 `workflows` / `workflow_steps` 已经可以绑定智能体
- 权限体系 `resource_permissions` 已经可以控制智能体、工作流可见范围

所以本次不建议重写聊天系统，而是沿着现有链路扩展：

```text
后台模型 API 管理
  -> 智能体搭建器
  -> 测试聊天
  -> 发布到 agents
  -> 用户端复用现有 /agents/[id] 聊天页
  -> 工作流继续绑定 agents.id
```

---

## 1. 产品边界

### 1.1 本次要做

- 后台新增“模型 API 管理”
- 后台新增或改造“智能体搭建器”
- 支持创建草稿、测试、发布
- 支持配置系统提示词、模型参数、开场白、建议问题、上传能力
- 支持模型连通性测试
- 支持智能体测试聊天
- 发布后复用现有用户端智能体卡片、聊天页、工作流绑定

### 1.2 本次不做

- 不做本地部署大模型
- 不做训练或微调
- 不做完整知识库 RAG
- 不做复杂工具调用 DSL
- 不做工作流执行引擎
- 不让普通用户直接填写或看到平台 API Key

---

## 2. 总体工作流

```text
超级管理员配置模型供应商
  -> 测试模型 API 连通性
  -> 创建智能体草稿
  -> 选择模型供应商
  -> 配置提示词、参数、能力、开场白
  -> 在搭建器内测试聊天
  -> 调整配置
  -> 发布智能体
  -> 自动写入/更新 agents
  -> 配置分类与可见范围
  -> 用户端使用
  -> 后台查看调用日志、额度消耗、错误原因
```

## 3. 数据库工作流

### 3.1 新增模型供应商表

新增迁移文件建议命名：

```text
supabase/migration_v35_model_providers.sql
```

（编号基线：master / devB / devB+ 已占用至 v33；v34 留给前置 `migration_v34_logs_status_aborted.sql`；v35 起为 5.15up 新表。详见配套方案文档 §2.2 / §2.2.1）

新增表：

```sql
CREATE TABLE IF NOT EXISTS model_providers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  provider_code TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  platform TEXT NOT NULL,
  api_endpoint TEXT NOT NULL DEFAULT '',
  api_key_enc TEXT NOT NULL DEFAULT '',
  default_model TEXT NOT NULL DEFAULT '',
  default_params JSONB NOT NULL DEFAULT '{}',
  enabled BOOLEAN NOT NULL DEFAULT TRUE,
  created_by UUID,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS model_providers_enabled_idx
  ON model_providers(enabled);
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `provider_code` | 业务编号，如 `openai-main`、`zhipu-backup` |
| `name` | 后台展示名，如“OpenAI 主账号” |
| `platform` | 适配器类型，如 `openai`、`coze`、`dify`、`yuanqi`、`qingyan` |
| `api_endpoint` | 模型接口地址 |
| `api_key_enc` | 加密后的 API Key |
| `default_model` | 默认模型 |
| `default_params` | 默认温度、token、供应商特殊参数 |
| `enabled` | 是否可被智能体选择 |

### 3.2 新增智能体草稿表

新增表：

```sql
CREATE TABLE IF NOT EXISTS agent_drafts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source_agent_id UUID REFERENCES agents(id) ON DELETE SET NULL,
  name TEXT NOT NULL DEFAULT '',
  description TEXT NOT NULL DEFAULT '',
  category_ids JSONB NOT NULL DEFAULT '[]',
  provider_id UUID REFERENCES model_providers(id) ON DELETE SET NULL,
  agent_type TEXT NOT NULL DEFAULT 'chat' CHECK (agent_type IN ('chat', 'external')),
  external_url TEXT NOT NULL DEFAULT '',
  builder_config JSONB NOT NULL DEFAULT '{}',
  model_params JSONB NOT NULL DEFAULT '{}',
  visibility_config JSONB NOT NULL DEFAULT '{}',
  status TEXT NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'testing', 'published', 'archived')),
  published_agent_id UUID REFERENCES agents(id) ON DELETE SET NULL,
  created_by UUID,
  updated_by UUID,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS agent_drafts_status_idx
  ON agent_drafts(status);
```

`builder_config` 建议结构：

```json
{
  "system_prompt": "",
  "opening_message": "",
  "suggested_questions": [],
  "capabilities": {
    "file_upload": true,
    "image_input": true,
    "conversation_summary": true
  },
  "safety_rules": [],
  "output_format": "",
  "tone": "",
  "business_rules": []
}
```

`visibility_config` 建议结构：

```json
{
  "visible_to": "all",
  "scope": [
    { "scope_type": "org", "scope_id": "DEMO" }
  ]
}
```

### 3.3 agents 表是否新增字段

短期可以不新增字段，发布时映射到现有字段：

| draft 字段 | agents 字段 |
| --- | --- |
| `name` | `name` |
| `description` | `description` |
| `provider.platform` | `platform` |
| `provider.api_endpoint` | `api_endpoint` |
| `provider.api_key_enc` | `api_key_enc` 或留空走 provider |
| `model_params + builder_config.system_prompt` | `model_params` |
| `agent_type` | `agent_type` |
| `external_url` | `external_url` |

但中期建议给 `agents` 增加：

```sql
ALTER TABLE agents
  ADD COLUMN IF NOT EXISTS provider_id UUID REFERENCES model_providers(id) ON DELETE SET NULL,
  ADD COLUMN IF NOT EXISTS builder_config JSONB NOT NULL DEFAULT '{}',
  ADD COLUMN IF NOT EXISTS published_from_draft_id UUID REFERENCES agent_drafts(id) ON DELETE SET NULL;
```

推荐本次直接加这三列，后续维护更清楚。

## 4. 后端 API 工作流

### 4.1 模型供应商管理 API

新增目录：

```text
app/api/admin/model-providers/route.ts
app/api/admin/model-providers/[id]/route.ts
app/api/admin/model-providers/[id]/test/route.ts
```

接口清单：

| 方法 | 路径 | 说明 | 权限 |
| --- | --- | --- | --- |
| `GET` | `/api/admin/model-providers` | 列表 | super_admin / system_admin |
| `POST` | `/api/admin/model-providers` | 创建 | super_admin |
| `PATCH` | `/api/admin/model-providers/[id]` | 更新 | super_admin |
| `DELETE` | `/api/admin/model-providers/[id]` | 删除或禁用 | super_admin |
| `POST` | `/api/admin/model-providers/[id]/test` | 测试连通性 | super_admin / system_admin |

连通性测试逻辑：

```text
读取 provider
  -> 解密 api_key_enc
  -> 构造最小 messages
  -> 调用 streamChat 或新增 nonStreamingTestChat
  -> 限制 10 秒超时
  -> 返回 success / latency_ms / sample_text / error
```

测试 prompt：

```text
请只回复“连接成功”四个字。
```

注意：

- 测试接口不写入正式 `messages`
- 可以写入 `logs`，action 用 `model_provider_test`
- 返回错误时要隐藏 API Key、Authorization header

新增目录：

```text
app/api/admin/agent-drafts/route.ts

app/api/admin/agent-drafts/[id]/publish/route.ts
app/api/admin/agent-drafts/[id]/duplicate/route.ts
```

接口清单：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/api/admin/agent-drafts` | 草稿列表 |
| `POST` | `/api/admin/agent-drafts` | 创建草稿 |

| `POST` | `/api/admin/agent-drafts/[id]/publish` | 发布到正式智能体 |
| `POST` | `/api/admin/agent-drafts/[id]/duplicate` | 复制草稿 |

### 4.3 草稿测试聊天

测试聊天不进入正式用户对话，不消耗组织正式额度。建议独立返回 SSE：

```text
读取 draft
  -> 读取 provider
  -> 合并 provider.default_params + draft.model_params + builder_config.system_prompt
  -> 调用 streamChat
  -> SSE 返回
  -> 写 admin 测试日志
```

测试消息结构：

```ts
const messages = [
  { role: "system", content: builderConfig.system_prompt },
  ...testHistory,
  { role: "user", content: message }
];
```

测试限制：

- 每次最多保留前端本地最近 6 轮，不入库
- 单次测试超时 60 秒
- 错误必须返回可读信息，如“模型接口地址无效”“API Key 未配置”“供应商返回 401”

### 4.4 发布逻辑

发布接口：

```text
POST /api/admin/agent-drafts/[id]/publish
```

发布步骤：

```text
校验权限
  -> 读取 draft
  -> 校验必填项
  -> 读取 provider
  -> 生成或沿用 agent_code
  -> upsert agents
  -> 写 agent_categories
  -> 写 resource_permissions
  -> 更新 draft.status = published
  -> 记录 published_agent_id
  -> 写 logs
```

发布校验：

- `name` 必填
- `provider_id` 必填，外链型除外
- `system_prompt` 建议必填，至少 10 个字符
- `agent_type = external` 时 `external_url` 必填
- `agent_type = chat` 时 provider 必须启用
- 分类可以为空，但后台要提示“未分类”
- 可见范围为空时默认仅管理员可见或继承“全部可见”，需产品拍板

推荐默认：

```text
新建发布默认 visible_to = custom
仅创建者所在组织可见
super_admin 可手动改成全平台可见
```

## 6. 现有聊天链路改造

### 6.1 chat route 读取 provider

当前聊天接口从 `agents` 直接读取：

```text
platform
api_endpoint
api_key_enc
model_params
```

改造后读取优先级：

```text
agent 自带 api_key_enc 非空
  -> 使用 agent 自带配置
否则 agent.provider_id 存在
  -> 使用 model_providers 配置
否则 platform = openai 且租户有 openai_key_enc
  -> 使用租户 key
否则
  -> 返回“智能体未配置模型 API”
```

配置合并优先级：

```text
provider.default_params
  < agent.model_params
  < 运行时必要参数
```

### 6.2 system prompt 来源

兼容旧逻辑：

```ts
const systemPrompt =
  agent.model_params?.system_prompt
  ?? agent.builder_config?.system_prompt
  ?? "";
```

推荐发布时同时写：

```json
{
  "system_prompt": "..."
}
```

到 `model_params`，保持旧 route 可读。

### 6.3 开场白来源

现有 `/api/agents/[id]/greeting` 主要拉 Coze 平台开场白。

本次建议调整为：

```text
优先返回 agents.builder_config.opening_message
如果没有，且 platform = coze，再走原 Coze prologue
否则返回 null
```

## 7. 权限工作流

### 7.1 角色权限

| 操作 | super_admin | system_admin | org_admin |
| --- | --- | --- | --- |
| 管理模型 API Key | 可以 | 只读/测试 | 不可以 |
| 创建智能体草稿 | 可以 | 可以 | 可选，建议第二阶段开放 |
| 发布全平台智能体 | 可以 | 不可以 | 不可以 |
| 发布本组织智能体 | 可以 | 可以 | 可选 |
| 删除模型供应商 | 可以 | 不可以 | 不可以 |

建议第一阶段：

```text
只有 super_admin / system_admin 可进入智能体搭建
只有 super_admin 可管理模型 API Key
system_admin 可以选择已启用 provider，但看不到 API Key
org_admin 暂不开放搭建器
```

### 7.2 API Key 展示规则

- 永不返回明文 API Key
- 列表只返回 `has_api_key`
- 编辑时 password input 为空代表不修改 key
- 输入新 key 才覆盖
- 清空 key 需要单独按钮并二次确认

## 11. 风险与处理

| 风险 | 等级 | 处理 |
| --- | --- | --- |
| API Key 泄露 | 高 | 加密存储、脱敏返回、禁止日志记录明文 |
| provider 被删除导致智能体不可用 | 中 | 被引用时禁止硬删除，改为禁用 |
| 旧智能体聊天被破坏 | 高 | chat route 保持旧字段优先兼容，单独做回归 |
| 不同供应商参数差异大 | 中 | 高级参数统一放 JSON，UI 只暴露通用参数 |
| 测试聊天消耗费用 | 中 | 限制测试频率、记录测试日志、后续可加 rate limit |
| system_admin 权限边界不清 | 中 | 第一阶段不允许 system_admin 管理 API Key |

## 13. 决策点

| ID | 问题 | 推荐 |
| --- | --- | --- |
| D-1 | 模型 API Key 是全平台共用还是组织独立 | 第一阶段全平台 provider，后续再做组织级覆盖 |
| D-2 | org_admin 是否能搭建智能体 | 第一阶段不开放，避免权限和费用失控 |
| D-3 | 草稿发布默认可见范围 | 默认本组织或自定义，不默认全平台 |
| D-4 | API Key 存在 provider 还是复制到 agents | 存 provider，agents 只存 `provider_id`，兼容旧智能体自带 key |
| D-5 | 测试聊天是否扣额度 | 第一阶段不扣用户额度，但写审计日志 |
| D-6 | 是否做知识库 | 本次不做，等 builder 跑通后单独立项 |

## 14. 一句话总结

本次 5.15up 的核心不是“再写一个聊天接口”，而是把大模型 API 做成后台可管理资源，再把智能体创建做成“草稿配置 -> 测试 -> 发布”的闭环。这样既能复用现有聊天、权限、工作流体系，也能为后续知识库、工具调用、智能体版本管理打地基。
