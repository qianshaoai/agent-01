## 1. 背景

线上反馈：上传 PDF 提示"文本提取失败，请确认文件未加密"；上传图片智能体识别全部错误。
用户在本地复现：PDF 能正常识别，图片识别全部错误，要求仔细排查。

两条线分别做了**可复现的实证实验**定位真因，结论如下。

---

## 2. 图片识别失败 —— 不是程序 bug，是 Coze 智能体本身无视觉能力

### 排查方法

测试对象 AGTXS-012「知识库入库整理智能体」是 Coze 平台智能体（bot_id `7636677333616918537`）。
用该智能体的 Coze API key 直连 Coze `/v3/chat` API，完全复刻 portal adapter 发出的多模态请求，
两种方式各测一遍：

| 方式 | 说明 | Coze HTTP | 机器人回复 |
|---|---|---|---|
| `object_string` + `file_url` | adapter 当前默认走这条（图片公网链接） | 200，无报错 | "我无法直接访问和读取图片内容" |
| `object_string` + `file_id` | 先上传图片到 Coze 再引用，Coze 原生最优方式 | 200，无报错 | "目前我无法直接读取图片中的文字内容" |

### 结论

- Coze 两次都返回 200、无任何结构性报错 → **portal 发的多模态格式正确，Coze 接收正常**。
- 但这个 bot 自己回复"我的功能是处理您直接提供的文本信息" —— 它用的是纯文本模型，
  未开启图片理解能力。
- 之前那次"识别成《收入证明》模板"是模型在没看到图的情况下**编造**的，所以"全部错误"。
- **portal 端代码无需修改。** 要支持图片识别，需在 Coze 扣子平台把该 bot 的模型
  换成支持视觉的模型 —— 属 Coze 侧 bot 配置，portal 不可控。

## 3. 线上 PDF "文本提取失败" —— standalone 构建漏打原生依赖 @napi-rs/canvas

### 根因

- 本地 `next dev` 用完整 `node_modules`，所以本地测 PDF 正常；线上跑 `output: standalone` 构建产物。
- pdf-parse → pdfjs-dist 依赖原生模块 `@napi-rs/canvas`，pdfjs 在 Node 环境靠它
  polyfill `DOMMatrix` / `ImageData` / `Path2D`。
- 检查 `.next/standalone/.../node_modules/` —— **无 `@napi-rs` 目录**。
  `@napi-rs/canvas` 是原生模块 + optionalDependency，Next.js 文件追踪(nft) 静态抓不到；
  而 `next.config.ts` 的 `outputFileTracingIncludes` 里列了 pdf-parse / pdfjs-dist /
  mammoth / xlsx / jszip，**唯独漏了 `@napi-rs`**。

### 实证实验

把本地 `node_modules/@napi-rs` 临时移走（模拟线上 standalone 缺该模块），再跑 PDF 提取：

```
Warning: Cannot load "@napi-rs/canvas" package
[getText THREW] ReferenceError: DOMMatrix is not defined
```

实验后已还原。这就是线上 PDF 失败的真因。老代码 `catch {}` 把该异常吞掉，
统一误报"文本提取失败，请确认文件未加密" —— 与加密毫无关系。

---

## 4. 修复

### 4.1 next.config.ts —— 把 @napi-rs 纳入 standalone 打包

`outputFileTracingIncludes` 的 `/api/upload` 和 `/api/trial/chat` 两处各加一行：

```
"./node_modules/@napi-rs/**/*",
```

`/**/*` 覆盖 `@napi-rs/canvas`、平台原生二进制子包（`canvas-linux-x64-gnu` 等）、
`wasm-runtime`，全部打进 standalone。

### 4.2 app/api/upload/route.ts —— 异常分类 + 日志

- PDF catch：原 `catch {}` 改为带异常分类。
  - `PasswordException` / 含 password|encrypt → "文件已加密，请上传未加密版本"
  - `InvalidPDFException` / 含 invalid pdf → "文件已损坏或不是有效 PDF"
  - 其它 → "文本提取失败 · {errName}: {errMsg}"（诊断期把真实异常带到前端，定位后可收敛文案）
  - 全部分支加 `console.error` 日志
- PDF 提取成功但内容为空 → 提示"未提取到文字（可能是扫描件 / 图片型 PDF）"
- docx / xlsx / pptx 三处 `catch {}` 补 `console.error` 日志（pdf/docx 之前已有，本次补 xlsx/pptx）

## 5. 部署前提（重要，需确认）

`@napi-rs/canvas` 是原生模块，二进制按平台分包：

- 本地这台是 Windows，`node_modules/@napi-rs/` 只有 `canvas-win32-x64-msvc`
- 线上 Linux 需要 `canvas-linux-x64-gnu`

本修复成立的前提是 **Jenkins 构建在 Linux 服务器上跑 `npm install` / `npm ci`**，
这样会自动装上 Linux 版原生二进制，`@napi-rs/**/*` 这个 glob 才能把它打进 standalone。
若构建流程是把 Windows 上构建好的 `node_modules` 整包上传，则本修复不足以解决问题。
