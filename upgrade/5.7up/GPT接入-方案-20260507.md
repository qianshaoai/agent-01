# GPT 接入 + 历史摘要降本 — 方案（两阶段整合）

**日期**：2026-05-07
**状态**：✅ 探讨阶段，**未动手**
**作者**：用户 + Claude
**前置条件**：5.7up 主菜（管理员权限完善）已上线 master

---

## 0. 背景与目标

为平台引入 OpenAI（GPT）能力，分两阶段：

- **阶段一 · 接入**：让国内组织用户能在平台 UI 里直接对话 GPT，账号是平台的，配额由平台扣
- **阶段二 · 降本**：用国内便宜模型对老对话做摘要，缩短发给 GPT 的上下文，降低 GPT token 成本

两阶段独立可发，但二阶段强依赖一阶段先跑通。

非目标（明确不做）：
- ❌ 不暴露成 OpenAI 兼容 API 网关（不开放外部 SDK 直连）
- ❌ 不让用户填自己的 OpenAI key
- ❌ 不接 GPT Store 的 GPT（OpenAI 没开 API，技术上做不到）
- ❌ 阶段二不对**当前用户消息**做概括（这是用户最初设想里要纠正的点）

## 一阶段 · OpenAI 接入

### 1.1 数据流

```
组织用户 ──► 平台 UI（agents/[id] 聊天页）
              │
              ▼
         /api/agents/[id]/chat（已有）
              │
              ▼
       lib/adapters/index.ts  → openai 适配器（已有）
              │
              │  HTTP 请求带 Authorization: Bearer <租户 OpenAI key>
              ▼
         api.openai.com/v1/chat/completions
              │
              ▼
       SSE 流式响应回平台
              │
              ▼
       记 tokens + 扣 quota + 写 logs（已有 RPC，按新规则改）
              │
              ▼
       返回给用户
```

### 1.5 quota 模型 — 三选一（**这道题最关键**）

GPT 不同模型成本差 100 倍：

| 模型 | input $/1M | output $/1M |
| --- | --- | --- |
| gpt-4o-mini | 0.15 | 0.60 |
| gpt-4o | 2.50 | 10.00 |
| o1 / o1-preview | 15 | 60 |

按现状"调用一次扣 1 次额度"会让用户优先薅最贵的——账扛不住。三条路：

| 路径 | 改动量 | 优点 | 缺点 | 适合 |
| --- | --- | --- | --- | --- |
| **A · 维持按次扣，token 只记录不计费** | 极小 | 不动 quota 模型 | 用户能薅羊毛（永远调 o1） | 短期试水 |
| **B · 加权扣次**：每个模型一个权重，1 次 4o-mini=1，1 次 4o=5，1 次 o1=30 | 中（quota RPC 改造 + `model_weights` 配置表） | 用户认知是"次数"，admin 好理解 | 涨价 / 新模型时要手动调权 | **推荐** |
| **C · 双轨**：`tenants` 加 `token_balance` 字段，调 GPT 走 token 余额，调其他走次数余额 | 大（schema + UI 双显示） | 最准 | 用户 / admin 认知双倍 | 规模上来再做 |

**推荐 B**。配置示例：

```sql
CREATE TABLE model_quota_weights (
  model_id TEXT PRIMARY KEY,
  weight_per_call INT NOT NULL,  -- 一次调用扣多少次额度
  enabled BOOLEAN DEFAULT true
);

INSERT INTO model_quota_weights VALUES
  ('gpt-4o-mini',   1, true),
  ('gpt-4o',        5, true),
  ('o1-preview',   30, false);  -- 默认不开
```

后续涨价 / 新模型只需 `UPDATE` 这张表，不用动代码。

### 1.6 模型白名单

阶段一**只开 2 个**：

- `gpt-4o-mini` — 日常对话主力（便宜、快）
- `gpt-4o` — 复杂任务（长文档、写作、代码）

明确**不开**：
- ❌ `o1 / o1-preview` — 太贵，等有明确推理类需求再开权限位
- ❌ vision / whisper / tts / dall-e — 这些是另开的 SKU，跟"聊天"是两回事

> 决策点 **D-3**：模型选择暴露给谁？
> - 选项 1：admin 在创建智能体时勾选 "GPT 模型 = 4o-mini / 4o"，对用户透明
> - 选项 2：用户聊天时在右上角自己切 — 但用户切 4o 等于多扣 5 次额度，体验和扣费要对齐
>
> 建议选 1，把模型当成智能体配置而非用户运行时选择。

### 1.7 用户端 UI

唯一可能要的小改：智能体卡片右下角加 token 用量小字（"今日已消耗 12,345 token"），但**不是必须**，看产品要不要透明给用户。

## 二阶段 · 历史摘要降本

### 2.1 动机

阶段一上线后会发现一个事实：**多轮对话越聊越贵**。

举例：第 20 轮时，用户每发一条消息，平台都要把前 19 轮历史一起塞给 GPT 当上下文。20 轮 × 平均 200 token/轮 = 4000 input token/请求。按 4o 算就是 4000 × $2.5/1M = $0.01/次，看着不贵但乘以日活、乘以平均对话长度，月底账单很难看。

阶段二的目标：**把老历史压缩，最近消息保真**。

### 2.2 标准方案：滑动窗口 + 增量摘要

```
完整对话历史
  ├─ 第 1～14 轮 (老历史)        ──► 国内 LLM 摘要 ──► [memory_summary]
  └─ 第 15～20 轮 (滑动窗口 6 轮) ──► 原样保留    ──► [recent_messages]

发给 GPT 的 messages：
  [system: agent.system_prompt]
  + (可选) [system: 早期对话摘要：${memory_summary}]
  + recent_messages（包含用户当前这条）
```

### 2.3 跟用户最初设想的关键差异

用户最初想的是"**所有**消息都过国内 LLM 概括"。这个有 3 个硬伤：

1. **当前消息也被概括 → 信息损失**：用户贴一段代码 / 合同原文要 GPT 处理，被概括成 "用户想审查代码 / 合同" 后 GPT 看不到原文，废一半
2. **延迟翻倍**：每条消息都要 2 次串行调用
3. **短对话反而亏钱**：1 轮对话也要调国内模型，但根本没历史可压缩

标准方案把这三点全解决了：
- 当前消息**永远原样**给 GPT
- 短对话（≤ 滑动窗口大小，比如 6 轮）**完全不调**国内模型
- 只有溢出窗口的老消息才被概括

### 2.4 数据模型改动

**`conversations` 表新增**：

| 列 | 类型 | 说明 |
| --- | --- | --- |
| `summary_text` | TEXT | 当前的摘要（增量维护） |
| `summary_until_msg_id` | UUID | 已被摘进 summary 的最后一条消息 ID |
| `summary_updated_at` | TIMESTAMPTZ | 上次更新时间 |

> 决策点 **D-4**：是否新增独立的 `conversation_summaries` 表，支持版本回滚 / 调试？或者就直接挂 `conversations` 三列。建议先挂三列，需要再分表。

### 2.5 聊天路由改造（伪代码）

```ts
// app/api/agents/[id]/chat/route.ts
const SLIDING_WINDOW = 6;            // 滑动窗口轮数（12 条消息）
const SUMMARIZE_THRESHOLD = 10;      // 老历史超过 10 条才开始摘要
const SUMMARIZER_MODEL = "glm-4-flash";  // 复用 zhipu 适配器

async function buildMessagesForGPT(conversationId, agent) {
  const all = await loadAllHistory(conversationId);
  if (all.length <= SLIDING_WINDOW * 2) {
    // 短对话：全原样，不调摘要
    return [{ role: "system", content: agent.system_prompt }, ...all];
  }

  const recent = all.slice(-SLIDING_WINDOW * 2);
  const olderAll = all.slice(0, -SLIDING_WINDOW * 2);

  // 增量摘要：只摘"上次摘到现在"新增的老消息
  const conv = await loadConversation(conversationId);
  const newOlder = olderAll.filter(m => m.id > conv.summary_until_msg_id);

  let summary = conv.summary_text ?? "";
  if (newOlder.length > 0 && (olderAll.length >= SUMMARIZE_THRESHOLD)) {
    summary = await callZhipu({
      model: SUMMARIZER_MODEL,
      messages: [
        { role: "system", content: "你是对话历史压缩助手。把对话提炼成简洁的事实清单，保留关键信息。" },
        { role: "user", content: `已有摘要：\n${summary}\n\n新增对话：\n${formatMessages(newOlder)}\n\n请输出更新后的摘要：` },
      ],
    });
    await saveConversation(conversationId, {
      summary_text: summary,
      summary_until_msg_id: olderAll.at(-1).id,
      summary_updated_at: new Date(),
    });
  }

  return [
    { role: "system", content: agent.system_prompt },
    ...(summary ? [{ role: "system", content: `早期对话摘要：${summary}` }] : []),
    ...recent,
  ];
}
```

### 2.6 国内 LLM 选型

**直接复用现有 `zhipu` 适配器**（[lib/adapters/index.ts](lib/adapters/index.ts) 已接入），跑 `glm-4-flash`：

- 单价：约 ¥0.0001 / 1k token（约 $0.000014），相比 gpt-4o（$0.0025/1k）便宜 **180 倍**
- 速度：< 1s 出摘要
- 接入成本：0（已经在用）

不需要新接入模型。

### 2.7 关键参数

| 参数 | 推荐 | 理由 |
| --- | --- | --- |
| 滑动窗口 | 6 轮（12 条消息） | 主流值，过小会丢上下文，过大白费 token |
| 摘要阈值 | 老历史 ≥ 10 条 | 避免一开始就摘 |
| 摘要模型 | glm-4-flash | 已有 + 便宜 + 够快 |
| 摘要更新策略 | **增量**：每次只摘新冒出来的老消息 | 比"每次全量重摘"省 70%+ 国内模型成本 |

### 2.8 quota 影响

阶段二会**多调一次国内模型**做摘要——这部分要不要扣用户额度？

- **方案 1**：算到 GPT 那次调用的 quota 里（透明给用户）
- **方案 2**：摘要由平台自己买单，不计入用户额度（成本差异 180 倍，影响小）

建议 **方案 2**——成本差异巨大，"为了省你们的钱我们多调一次国内模型反而多扣你一次额度" 是反直觉的产品设计。

### 2.9 阶段二待拍清单

- **D-4** summary 表结构（挂 conversations vs 独立表）
- **D-5** 滑动窗口大小（6 / 8 / 10）
- **D-6** 摘要 prompt 怎么写最稳（事实清单 / 自由摘要 / 角色扮演）
- **D-7** 摘要的 token 成本是否对用户透明（方案 1 vs 2）

## 风险盘点

| 风险 | 等级 | 缓解 |
| --- | --- | --- |
| 平台服务器到 `api.openai.com` 不稳定 | 高 | 上线前压测 + 准备代理回退 |
| OpenAI key 泄露 | 高 | AES 加密落库 + admin 访问审计 + key 用一段时间 rotate |
| 用户薅 o1（如果开放） | 中 | 加权 quota（路径 B） + 默认不开 o1 |
| GPT 新模型上线时权重未配 | 中 | `model_quota_weights` 表 + 默认 weight=10 拒绝调用 |
| 摘要质量差导致 GPT 答非所问 | 中 | 灰度 + 用户反馈通道 + 摘要 prompt 持续调优 |
| OpenAI 涨价 / 政策变动 | 低 | 路径 B 改 weight 即可 |

## 一句话总结

**阶段一是接 OpenAI + 把 quota 模型换成"加权计次"，2.5 天能跑通；阶段二是给老历史加摘要降本，复用现有 zhipu 适配器，又 2.5 天。**

但真正决定项目能不能跑通的是 **D-3、D-5、D-7 三个产品决策**，不是技术。
