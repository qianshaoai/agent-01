## 一、聊天页 UX 微调

### 1. Markdown fenced 代码块加复制按钮

**问题**：bot 输出 ```` ```text ... ``` ```` 代码块时，要复制内容只能整段拖蓝复制，会带上前后描述文字。

**实施**：
- `app/agents/[id]/page.tsx` 新增 `CopyableCodeBlock` 组件
- ReactMarkdown `components.pre` 接管 `<pre>` 渲染：包一层 `<div className="relative group/code">`，右上角放复制按钮
- 按钮点击：`navigator.clipboard.writeText(pre.textContent)`，只取代码块内容
- 默认隐藏，hover 代码块时浮现；已复制 1.5s 内显示反馈状态后还原

**视觉迭代**：
- 第一版：黑底"复制"文字标签 → 用户反馈太丑
- 第二版（当前）：28×28 透明方块 + lucide `Copy` 图标 / 复制后变 `Check` 绿色 + hover 浅灰背景

### 2. 流式生成期间不再强制滚到底

**问题**：bot 流式输出时，每收一段 delta 都把页面滚到底部，用户想边生成边往上看历史不可能。

**实施**：
- `app/agents/[id]/page.tsx` 把自动滚动 useEffect 的依赖从 `[messages]` 改成 `[messages.length]`
- 流式中只是末尾 assistant 的 `content` 变长（同一条消息），messages.length 不变 → 不触发滚动
- 用户发送 / 切换会话 / 重新生成时 length 变化 → 仍会滚到底（保留）
- 想跟最新内容右下角"↓ 回到最新"按钮已存在，无需额外做

## 二、字体大小切换重做（路径 A 已实施）

**问题**：旧方案 `zoom: 1 / 1.15 / 1.30` 是页面密度缩放，不是字体缩放——文字变大的同时 padding / 间距 / 卡片 / 按钮也跟着放大，导致中 / 大档界面变挤、固定定位元素被推出可视区。

**新方案（A 路径，已落地）**：

只放大字号，不放大布局。

- `lib/font-size.ts`：删 `ZOOM_MAP`，`applyFontSize(level)` 改为给 `<html>` 写 `data-font-size="small|medium|large"`，并清理旧 `style.zoom`
- `app/globals.css`：加 root font-size 规则
  - 默认（small） — 不写规则，沿用浏览器默认 16px
  - `:root[data-font-size="medium"] { font-size: 19px; }`
  - `:root[data-font-size="large"]  { font-size: 21px; }`
- `app/layout.tsx`：pre-hydration script 从设置 `style.zoom` 改为设置 `data-font-size` 属性，避免 SSR 与首屏闪烁
- Tailwind v4 的 `text-xs/sm/base/lg` 等是 rem 单位 → 自动跟着 root font-size 缩放
- 项目里大量 `text-[NNpx]` 任意值类**不会**跟着放大，覆盖率 ~50-70%（路径 B 暂未做，等观察效果）

## 五、GPT 接入（两阶段，晚间补加 + 当晚验收完成）

配套方案：[`upgrade/5.7up/GPT接入-方案-20260507.md`](GPT接入-方案-20260507.md)（先于实施落地，对应 commit `555d875`）。

阶段一 + 阶段二在同一个 commit `022d11e` 推上。下面分项目说。

### 5.1 阶段一 · 组织共享 OpenAI key + token 用量捕获 + 加权 quota

**动机**：
1. 国内组织想用 GPT，但每个 agent 单独存 key 不利于多 agent 复用、key rotate 也碎
2. GPT 不同模型成本差 100 倍（4o-mini ↔ o1），按次扣 quota 用户能薅羊毛
3. 没有 token 用量记录就没法做后续聚合分析 / 计费精算

**schema 改动（`migration_v25.sql`）**：
- `tenants` 加 `openai_key_enc TEXT DEFAULT ''` + `openai_key_set_at TIMESTAMPTZ` + `openai_key_set_by UUID`
- `logs` 加 `prompt_tokens INT` + `completion_tokens INT` + `model_used TEXT`
- 新建 `model_quota_weights(model_id PK, weight_per_call INT, enabled BOOL, note TEXT)` 表
- 种子 4 行：`gpt-4o-mini=1` 启用 / `gpt-4o=5` 启用 / `o1-preview=30` 默认禁用 / `o1-mini=15` 默认禁用
- 新增 RPC `increment_quota_used_weighted(p_code, p_weight)`，原子 `quota_used + p_weight ≤ quota` 才扣

**后端实施**：

| 文件 | 改动 |
| --- | --- |
| `app/api/admin/tenants/[id]/openai-key/route.ts` 🆕 | super_admin only。`PUT` 加密落库 + 写审计字段；`DELETE` 清空；`GET` 返回 `{isSet, setAt, setBy}` 元信息（不回传明文 / 不回传密文） |
| `lib/adapters/index.ts` | `AdapterConfig` 加 `onUsage` 回调；`openaiCompatibleStream` 当调用方提供 `onUsage` 时自动加 `stream_options: { include_usage: true }`；usage 帧位置兼容多家厂商 |
| `app/api/agents/[id]/chat/route.ts` | `platform=openai` 且 agent 自带 key 为空兜底到 `tenants.openai_key_enc`；流式 usage 捕获到 `capturedUsage`；按 `model_quota_weights` 折算 weight 走 `increment_quota_used_weighted`；token + model_used 入 `logs` |

**前端实施（`app/admin/tenants/page.tsx`）**：
- `super_admin` 在组织展开行内可见的 OpenAI Key 区块（system_admin / org_admin 都看不到）
- 状态徽章：「未配置」+ 配置按钮 / 「已配置」+ 配置时间 + 更新 / 清空
- 配置表单内联：password 类型输入 + sk- 前缀校验 + Enter 保存 / Esc 取消 + 错误红边
- AES-256-GCM 加密落库（复用 `lib/crypto.ts`），UI 提示「平台不会以任何形式回显或转发明文」

**默认决策（用户用 D-1 / D-2 / D-3 倾向值）**：
- D-1：token 字段挂 `logs`（聚合方便）
- D-2：一组织一把 key（隔离 / 可单独 rotate）
- D-3：模型选择由 admin 在 agent.model_params 里定，用户运行时不可切

### 5.2 阶段二 · 滑动窗口 + 增量摘要降本

**动机**：阶段一上线后多轮对话会越聊越贵——第 N 轮要把前 N-1 轮历史一起发给 GPT，token 线性涨。需要把老历史压缩成摘要。

**schema 改动（`migration_v26.sql`）**：
- `conversations` 加 `summary_text TEXT` + `summary_until_at TIMESTAMPTZ`

**实施**：

| 文件 | 改动 |
| --- | --- |
| `lib/summarizer.ts` 🆕 | 独立模块。env 配置 `SUMMARIZER_ENDPOINT/_API_KEY/_MODEL`；OpenAI 兼容协议；增量更新（已有摘要 + 新冒出来的老消息 → 合并摘要）；任意失败返回 `prevSummary` 不破坏聊天 |
| `app/api/agents/[id]/chat/route.ts` | 加载 history 时按 `summary_until_at` gt 过滤（限 200 条）；history > 12 条时 `older = slice(0, -12)`、`recent = slice(-12)`，older 扔给摘要器 → 落库 `summary_text` + `summary_until_at = older.last.created_at`；构造 messages 时插入 system 摘要消息（在 agent.system_prompt 之后） |
| `.env.local.example` | 加 `SUMMARIZER_ENDPOINT/_API_KEY/_MODEL` 三个变量示例 |

**默认决策（用户用 D-4 / D-5 / D-6 / D-7 倾向值）**：
- D-4：summary 字段挂 `conversations` 两列（不独立表）
- D-5：滑动窗口 6 轮 / 12 条消息
- D-6：摘要 prompt 倾向「事实清单」风格（保留人名 / 地名 / 时间 / 决定 / 用户偏好；删寒暄客套）
- D-7：摘要器调用费用平台自吃，不计入用户额度（成本差 180 倍，反向扣用户额度反直觉）

**摘要器未配置时的退化行为**：
- 不调用国内模型，沿用 5.7up 之前的「截断老历史到 MAX_CONTEXT_TURNS*2 条」
- env 变量缺失也不会破坏聊天，只是省不了 token

### 5.3 验收过程（从失败到打通）

**验收一 · 阶段一 token 捕获（智谱链路代测 OpenAI）**：
- 用户先建了一个 GPT-4o-mini 测试 agent，但生产服务器到 `api.openai.com` 不通（用户那边 key/网络问题），改成跑智谱（OpenAI 兼容协议同样能验证我加的 token 捕获逻辑）
- **第一次失败**：`logs` 表里 `model_used` / `prompt_tokens` / `completion_tokens` 全 NULL
- **根因定位**：我 adapter 里那个判断条件太死板：
  ```ts
  if (wantsUsage && obj.usage && obj.choices.length === 0) { ... }
  ```
  `choices.length === 0` 是 OpenAI 自己的格式（usage 单独出在末帧 choices=[]），但**智谱在带 finish_reason 的最后一帧 delta 里塞 usage**，那一帧 choices.length === 1，永远不进
- **补丁**：去掉 `choices.length === 0` 这个限制，「只要这帧带 usage 就捕获」+ 同时如果还有 delta.content 就照常 yield
- **第二次跑**：`logs` 行新增 `model_used=glm-4-flash, prompt_tokens=255, completion_tokens=48` ✅

**验收二 · 阶段二端到端（关键 — 摘要真的传到主对话模型了吗）**：
- 用户 env 变量 `SUMMARIZER_API_KEY` 第一次填进去时长度仍是 0（编辑器没保存 / 改到了 .example 上），dev 重启后 `isSummarizerConfigured()` 返回 false → 摘要根本没跑
- 用户把真实智谱 key 贴给我，我直接写 `.env.local`，重启 dev
- 用户连续聊 8 轮：「搓手手 / 搓脚脚 / 搓头头 / 搓肘肘 / 搓膝膝 / 扇耳光」等
- 查 `conversations.summary_text` length=116 ✅
- 关键测试问 bot：「你还记得我一开始是搓什么吗」
- bot 答：「一开始你是**搓手手**，然后是搓脚脚，接着是搓头头，最后是搓肘肘」
- 「搓手手」是第 1 条消息，按 16 条对话长度算早被滑出窗口了。bot 答得出 → 摘要确实进了主对话上下文 ✅
- 副作用观察：bot 答的顺序末尾「最后是搓肘肘」其实不对（实际最后是「扇耳光」），是摘要的细节损耗。这是阶段二的已知 tradeoff——压缩省 token 的代价就是细节弱化

### 5.4 阶段一 · 已知未测项

- **真 OpenAI（`api.openai.com`）端到端**：用户那边卡 key/网络。我加的"组织共享 key 兜底"分支按代码逻辑通过智谱（同走 `openaiCompatibleStream`）已验证，但真打 OpenAI 没真测过
- **加权 quota 的边界**：理论上 RPC 的 `WHERE quota_used + p_weight ≤ quota` 是原子的，没压测多用户同时调用同组织刚好卡配额边界的极端场景
- **图片 / 文件附件**：GPT 智能体能不能接图片要看 endpoint 是否指 OpenAI 4o vision，没纳入第一阶段回归
