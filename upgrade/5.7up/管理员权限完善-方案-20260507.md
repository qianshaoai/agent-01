**适用范围**：用户端登录策略 + 全部 admin 接口与页面

## 一、需求原文

来源：桌面 `管理员功能完善.docx`，转 md 整理后：

### 角色定义

| 角色 | 用户端 | 后台 |
| --- | --- | --- |
| **超级管理员** super_admin | **不可登录用户端** | 全部权限（含品牌管理） |
| **系统管理员** system_admin | 可看 + 用**所有**工作流 | 后台**除品牌管理外**全部权限 |
| **组织管理员** org_admin | 可看 + 用**本组织所有**工作流 | 受限：见下表 |

### org_admin 后台权限白名单

可做：
- 查看本组织成员
- 添加 / 修改本组织部门 / 小组
- 调整本组织成员所在部门
- 查看本组织工作流
- 为本组织添加工作流

不可做（隐含）：
- 看其它组织数据
- 改品牌设置
- 改组织码 / 创建组织
- 看智能体管理（属于内容管理，跨组织）
- 操作其它组织成员

### 设置规则

- 管理员只能由**更高级**管理员设置
- 超级 > 系统 > 组织
- 不允许同级互相调整
- 不允许下级调整上级

### 与需求有差距的（必改）

#### 差距 1：**super_admin 不能登录用户端**

- **现状**：`/api/auth/login` 没有按角色拦截。super_admin 用同一手机号若在 users 表也存在，可以正常登录用户端。
- **需求**：super_admin 完全屏蔽用户端登录入口。
- **改法**：`/api/auth/login` 在校验密码通过后，`if (user.role === 'super_admin') return 403 + 提示"超级管理员请从 /admin 入口登录"`。

#### 差距 2：**system_admin 用户端访问所有工作流**

- **现状**：用户端 `/api/workflows` 走 visible_to + resource_permissions 过滤；system_admin 没有特殊豁免，看到的还是按个人/组织规则过滤的子集。
- **需求**：system_admin 在用户端能看到平台**全部**工作流（即使工作流的 visible_to=org_only 且不属于他所在组织）。
- **改法**：`/api/workflows` 加一条豁免——如果当前用户的 role === 'system_admin'，跳过 visible 过滤直接返回所有 enabled=true 的工作流。

#### 差距 3：**org_admin 用户端访问本组织所有工作流**

- **现状**：org_admin 在用户端走的是普通 user 过滤路径——visible_to=all 看得到、visible_to=org_only 且属于自己组织看得到、custom 看权限规则。
- **需求**：org_admin 在用户端能看到所属组织**所有**工作流，**包括**那些 visible_to=custom 但权限规则没有覆盖到他个人 / 部门 / 小组的工作流。
- **改法**：`/api/workflows` 加一条豁免——如果当前用户的 role === 'org_admin'，对 `visible_to IN ('all', 'org_only')` 中属于自己组织的 + custom 但权限规则的 scope_type='org' 且 scope_id=自己组织码的 + ... 走宽松匹配；最简实现：org_admin 时直接返回 `visible_to ∈ ('all', 'org_only')` 中所属本组织的，外加 custom 的本组织工作流。
  - **方案 A（最稳）**：org_admin 时直接 SQL 多取一些：visible_to='all' OR visible_to='org_only'（对个人用户应过滤掉 org_only，但 org_admin 是组织用户，本来就能看 org_only）OR (visible_to='custom' AND id IN (SELECT resource_id FROM resource_permissions WHERE resource_type='workflow' AND scope_type='org' AND scope_id=admin.tenant_code))
  - 这条策略**不绕过** visible_to=personal_only（个人专属）和别的组织 org_only

> 这里要先确认：org_admin 的"本组织所有工作流"是不是包含其它组织设了 visible_to='org_only' 但用 resource_permissions 把"其他组织"列为可见的工作流？文档没明说。**默认收紧解释**：仅本组织自己的工作流。

#### 差距 4：**org_admin 后台能为本组织管部门 / 小组**

- **现状**：`/api/admin/departments/route.ts` GET 没做 admin.role 过滤，POST 直接信任前端传的 tenantCode。**有越权风险**（org_admin 可以传别的组织的 tenantCode）。
- **需求**：org_admin 能管本组织部门 / 小组，但不能跨组织。
- **改法**：departments / teams / [id] 4 个文件全部加：
  - GET：如果 admin.role === 'org_admin'，强制 query.eq('tenant_code', admin.tenantCode)，忽略前端传的 tenantCode
  - POST / PATCH / DELETE：如果 admin.role === 'org_admin'，校验 body.tenantCode（或目标行的 tenant_code）必须等于 admin.tenantCode，否则 403

#### 差距 5：**org_admin 后台能调本组织成员的部门 / 小组**

- **现状**：用户管理 PATCH 已有 admin.role === 'org_admin' 的目标 tenant_code 校验（`/api/admin/users/[id]/route.ts:27`）。
- **需求**：org_admin 改本组织成员的 dept_id / team_id 应该允许（不止改 role / 改组织）。
- **改法**：users/[id] PATCH action=set-dept-team（如未实现）或现有更新接口确认 org_admin 能改 dept_id / team_id 字段，但不能改 tenant_code、不能改 role 到 super_admin / system_admin、不能跨组织操作。当前文件 line 27 的校验已经覆盖大头，只需补一个限制：org_admin 走 set-tenant action 一律 403（5.6up 已处理）。需要复核 set-role 是否限制 org_admin 不能提到 system_admin / super_admin（canAssignRole 应已覆盖）。

#### 差距 6：**org_admin 后台能查 + 添加本组织工作流**

- **现状**：
  - 用户端入口：`/api/workflows` 按可见性过滤，org_admin 已经能看到本组织工作流（差距 3 修完后更全）
  - **后台入口**：`/admin/workflows` 在 admin-layout 当前是 SUPER_SYSTEM only，org_admin 看不到
  - `/api/admin/workflows/*` 接口没有 org_admin 校验
- **需求**：org_admin 能进 /admin/workflows 看本组织工作流 + 创建本组织工作流（visible_to 自动绑到自己组织）
- **改法**：
  - admin-layout：工作流管理改 ALL_ROLES（让 org_admin 也能看到入口）
  - `/api/admin/workflows` GET：org_admin 时只列出**本组织相关的工作流**（visible_to=org_only 且属于本组织 + custom 但权限规则带本组织 + 自己创建的）
  - POST：org_admin 创建工作流时强制 visible_to='org_only' + 自动写入 resource_permissions 绑定本组织
  - PATCH / DELETE：org_admin 只能改 / 删本组织自己创建的工作流；改不了别人创建的或跨组织的
  - 工作流详情里改步骤 / 引用智能体：org_admin 能不能用智能体？**需求没写**——默认收紧解释：org_admin 不进入智能体管理界面，但工作流步骤里**可以选**已存在的智能体（不创建智能体）

#### 差距 7：**org_admin 进不了智能体管理 / 组织码管理 / 品牌设置**

- **现状**：
  - 智能体管理：admin-layout 控制 SUPER_SYSTEM only ✓
  - 组织码管理：SUPER_SYSTEM only ✓
  - 品牌设置：SUPER_ONLY ✓
- 这些已经满足需求，**无需改**。

#### 差距 8：**system_admin 不能改品牌设置**

- **现状**：admin-layout 里品牌设置是 SUPER_ONLY ✓；`/api/admin/settings/*` 路由是否有同样限制需要复核。
- **改法**：复核 `/api/admin/settings/route.ts`、`/api/admin/settings/logo/route.ts`、`/api/admin/settings/contact-qr/route.ts`，确保 PATCH/POST 拦截非 super_admin。

## 四、方案要点 / 约束

### 1. 不动 DB

需求纯权限逻辑层面，没有需要新增列的内容。`super_admin / system_admin / org_admin` 角色和 ROLE_RANK 都已存在。

### 2. 用户端工作流可见性的两条豁免要写在同一处

`/api/workflows/route.ts` 现在有完整的可见性过滤逻辑——加 system_admin / org_admin 豁免要在**这一个文件**里加分支，不要散到多处，避免后续维护时漏改。

### 3. org_admin 创建工作流时自动绑组织

org_admin 创建工作流不应让他选 visible_to——后端**强制**写成 `visible_to='org_only'` 或 `'custom'` + 同时写一条 `resource_permissions(scope_type='org', scope_id=admin.tenant_code)`。前端表单也要把 visible_to 选择器对 org_admin 隐藏 / 禁用。

### 4. org_admin 编辑工作流的步骤里可以选智能体

工作流步骤可以选已有的 agent。org_admin 不能进 /admin/agents 管理智能体本身，但能在工作流步骤里点"选择智能体"下拉选已有的——这个下拉走 `/api/admin/agents` GET。需要确认这个接口对 org_admin 是否允许（目前看 admin-layout 是禁止的，但 API 实际是否拦截？需要复核 + 可能需要放开 GET 给 org_admin）。

### 5. 普通 user 角色不变

需求文档没提 user 角色，按现状保持。`/api/admin/*` 系列接口对 user 角色一律 401。

### 6. 不破坏现有 5.6up 改组织功能

5.6up 已经做了"用户改组织"的全套层级校验。本方案不动 set-role / set-tenant 这两个 action，只复核它们对 org_admin 的限制是否到位。

## 五、决策点（已部分拍板）

### 1. org_admin 用户端"本组织工作流"范围 ✅ 已定

**口径**：在后台 visible_to 设置中**选了该组织 / 其下属部门 / 其下属小组**的工作流，都算本组织工作流。

具体 SQL 条件：
```
visible_to = 'all'
OR (visible_to = 'org_only' AND id IN (SELECT resource_id FROM resource_permissions WHERE resource_type='workflow' AND scope_type='org' AND scope_id = admin.tenant_code))
   /* 兼容旧数据：org_only 但权限规则只指当前组织 */
OR id IN (
  SELECT resource_id FROM resource_permissions
  WHERE resource_type = 'workflow'
    AND (
      (scope_type = 'org' AND scope_id = admin.tenant_code)
      OR (scope_type = 'dept' AND scope_id IN (SELECT id FROM departments WHERE tenant_code = admin.tenant_code))
      OR (scope_type = 'team' AND scope_id IN (SELECT id FROM teams WHERE tenant_code = admin.tenant_code))
    )
)
```

不包含：
- `personal_only` 工作流（个人专属）
- 其它组织的 `org_only` 工作流
- `custom` 但只放权给个别用户（user scope_type）的工作流——除非该用户属于本组织（这条边界保守，不予覆盖）

### 2. org_admin 创建工作流的 visible_to 默认值 ✅ 已定

**强制 `visible_to = 'org_only'`**：让本组织所有成员都能看到 org_admin 新建的工作流。

由后端在 `POST /api/admin/workflows` 收到请求时，如果 `admin.role === 'org_admin'`，强制覆盖前端传的 visible_to 字段为 `'org_only'`，前端表单也对 org_admin 隐藏 visible_to 选择器。

### 3. org_admin 改别人创建的本组织工作流 ✅ 已定

**允许**（决策 B）。"本组织工作流"统一归 org_admin 管，不区分谁创建的。

### 4. `/api/admin/agents` 对 org_admin 放开范围 ✅ 已定

**仅只读** — A 方案。

| 动作 | org_admin 是否允许 |
| --- | --- |
| GET 列表 / 详情 | ✅ 允许（用于工作流步骤选智能体） |
| 改名 / 改描述 / 改分类 | ❌ 拒绝 |
| 改 API key / 改 endpoint / 改 model_params | ❌ 拒绝 |
| 禁用 / 启用切换 | ❌ 拒绝 |
| 删除 | ❌ 拒绝 |
| 创建新智能体 | ❌ 拒绝 |

实施：`/api/admin/agents/route.ts` GET 改成 ALL_ROLES（含 org_admin）；`POST/PATCH/DELETE` 以及 `[id]/route.ts` 全部 PATCH/DELETE 都保持 SUPER_SYSTEM。`/admin/agents` 后台页面对 org_admin 的处理：
- 选项 a：**完全不显示菜单入口**（org_admin 看不到），但若硬访问 URL 仍可看到列表（GET 接口允许）
- 选项 b：菜单入口对 org_admin 显示，但页面里所有"编辑/禁用/删除/新建"按钮全部隐藏 / 禁用

考虑到 org_admin 在工作流步骤里选智能体的下拉**直接从 API 拉**，不经过 /admin/agents 页面，所以 **菜单不必给 org_admin 开**——保持现状（SUPER_SYSTEM only），只改 API GET 的角色限制。

### 5. super_admin 拒登用户端文案 ✅ 已定

文案："请登录后台"
