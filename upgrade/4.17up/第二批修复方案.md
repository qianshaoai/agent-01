#### 问题位置

**选定接入范围（优先级从高到低）**：

1. **第一轮**（核心业务，必须接）：
   - `app/api/agents/[id]/chat/route.ts` — 聊天主链路
   - `app/api/user-agents/[id]/chat/route.ts` — 自定义智能体聊天
   - `app/api/upload/route.ts` — 文件上传

   - `app/api/auth/register/route.ts`
   - `app/api/auth/change-password/route.ts`
   - `app/api/auth/cancel-account/route.ts`

2. **第二轮**（管理后台写操作）：
   - `app/api/admin/users/[id]/route.ts`
   - `app/api/admin/agents/[id]/route.ts`
   - `app/api/admin/tenants/[id]/route.ts`
   - 其他 admin 路由的 PATCH/DELETE

3. **暂不接**：GET 列表接口（噪音大、价值低）

**统一改造模板**：

```ts
// 接入前
export async function POST(req: NextRequest) {
  const admin = await requireAdmin();
  if (admin instanceof Response) return admin;
  // ... 业务逻辑
  return NextResponse.json({ ok: true });
}

// 接入后
import { withRequestLog } from "@/lib/request-logger";

export const POST = withRequestLog(async (req: NextRequest) => {
  const admin = await requireAdmin();
  if (admin instanceof Response) return admin;
  // ... 业务逻辑
  return NextResponse.json({ ok: true });
});
```

**扩展 `withRequestLog` 支持 userId 记录**（可选，`lib/request-logger.ts` 需配合调整）：

```ts
// 在 handler 内手动设置 userId

  const admin = await requireAdmin();
  if (admin instanceof Response) return admin;
  ctx.userId = admin.adminId;  // 注入 userId 到日志
  // ...
});
```

#### 风险
- `withRequestLog` 包裹不能改变原有的返回值类型
- 动态路由的 `params` 参数传递需要确认兼容性（`handler(req, { params })` 这种签名是否被 `withRequestLog` 正确转发）

#### 验证
- 本地启动 dev，调用被接入的接口，观察控制台是否输出结构化 JSON 日志
- 响应头中应有 `X-Request-Id`

### F9：源码文件编码统一为 UTF-8

#### 问题位置
运行检查报告中提到"多个后台页面在终端读取时出现明显乱码"，说明部分源码文件可能是 GBK/ANSI 编码。

#### 修改方案

**第 1 步：扫描所有源码文件的编码**

在项目根目录新建脚本 `scripts/check-encoding.mjs`（临时工具，修完可删）：

```js
import { readdir, readFile } from "node:fs/promises";
import { join } from "node:path";

async function walk(dir, out = []) {
  const entries = await readdir(dir, { withFileTypes: true });
  for (const e of entries) {
    if (e.name === "node_modules" || e.name === ".next" || e.name.startsWith(".")) continue;
    const p = join(dir, e.name);
    if (e.isDirectory()) await walk(p, out);
    else if (/\.(ts|tsx|js|jsx|mjs|cjs|css|md)$/.test(e.name)) out.push(p);
  }
  return out;
}

const files = await walk(".");
for (const f of files) {
  const buf = await readFile(f);
  // UTF-8 BOM
  if (buf[0] === 0xEF && buf[1] === 0xBB && buf[2] === 0xBF) {
    console.log("UTF-8-BOM:", f);
    continue;
  }
  // 尝试 UTF-8 解码
  const text = buf.toString("utf-8");
  if (text.includes("\uFFFD")) {
    console.log("NON-UTF-8:", f);
  }
}
```

运行：`node scripts/check-encoding.mjs`

**第 2 步：转换非 UTF-8 文件**

对扫描出的问题文件，用 VSCode 右下角 "Reopen with encoding" → 选择 GBK 读取 → 再 "Save with encoding" → 选 UTF-8。

或用脚本批量转换（谨慎使用，建议先 git commit 一版）：

```js
import iconv from "iconv-lite";  // 需要 npm install -D iconv-lite
// 或用 Node 内置 TextDecoder("gbk")

const buf = await readFile(path);
const text = new TextDecoder("gbk").decode(buf);
await writeFile(path, text, "utf-8");
```

**第 3 步：加 `.editorconfig` 防止再次污染**

项目根目录创建或更新 `.editorconfig`：

```ini
root = true

[*]
charset = utf-8
end_of_line = lf
indent_style = space
indent_size = 2
trim_trailing_whitespace = true
insert_final_newline = true
```

**第 4 步（可选）：Git hook 检查**

可以在 pre-commit 中加编码检查，但第一阶段不急。

#### 风险
- 批量转换前**务必先 commit 当前状态**，方便出问题时 revert
- 某些中文字符在 GBK → UTF-8 转换后可能需要目视复核

#### 验证
- 终端执行 `cat app/admin/users/page.tsx | head -30` 中文应正常显示
- IDE 打开所有文件，底部编码标识均为 UTF-8

### F11：分类展示查询合并为数据库视图

#### 问题位置
`app/api/agents/route.ts` 约第 113-208 行，当指定 `categoryId` 参数时执行 6 次独立数据库查询：
1. `workflow_categories` — 取分类下的工作流
2. `category_agent_display` — 取手工覆盖配置
3. `agent_categories` — 取分类下的直接绑定智能体
4. `workflows` — 取工作流的可见性
5. `workflow_steps` — 取工作流关联的智能体
6. `agents` — 取外链智能体

虽然用 `Promise.all` 并行化了，但网络往返仍多。

#### 修改方案

**第 1 步：在 Supabase 中创建视图**

新建 `supabase/migration_v20.sql`：

```sql
-- 分类展示视图：给定 category_id 和 tenant_code，返回应展示的 agent_id 列表
-- 合并了自动推导（工作流关联）+ 手工覆盖（category_agent_display）+ 直接绑定（agent_categories）
CREATE OR REPLACE VIEW category_display_agents AS
WITH
  -- 自动推导：通过工作流步骤间接关联到分类的智能体
  auto_via_workflows AS (
    SELECT DISTINCT
      wc.category_id,
      ws.agent_id,
      w.visible_to
    FROM workflow_categories wc
    JOIN workflow_steps ws ON ws.workflow_id = wc.workflow_id
    JOIN workflows w ON w.id = wc.workflow_id
    WHERE ws.enabled = true AND ws.agent_id IS NOT NULL
  ),
  -- 直接绑定：agent_categories 表
  direct_binding AS (
    SELECT ac.category_id, ac.agent_id, NULL::text AS visible_to
    FROM agent_categories ac
  ),
  -- 合并自动+直接
  combined AS (
    SELECT * FROM auto_via_workflows
    UNION
    SELECT * FROM direct_binding
  )
SELECT
  c.category_id,
  c.agent_id,
  c.visible_to,
  cad.is_manual,
  cad.is_hidden
FROM combined c
LEFT JOIN category_agent_display cad
  ON cad.category_id = c.category_id AND cad.agent_id = c.agent_id;
```

**第 2 步：改造路由**

`app/api/agents/route.ts` 原来的 6 次查询改为：

```ts
// 旧：6 次查询 + 在代码里 merge
// 新：一次查询视图
const { data: displayRows } = await db
  .from("category_display_agents")
  .select("agent_id, visible_to, is_manual, is_hidden")
  .eq("category_id", categoryId);

// 后续过滤（is_hidden、visible_to 判断）保留在代码中
```

剩下的可见性判断（visible_to、is_hidden、tenant 过滤）仍需要在应用层做，因为涉及当前用户的 tenant_code。

```sql

RETURNS TABLE (agent_id UUID, visible_to TEXT, is_manual BOOLEAN, is_hidden BOOLEAN) AS $$
```

### F14：`<img>` 替换为 `next/image`

#### 17 处 warning 分布

| 文件 | 位置数 | 图片类型 |
|------|-------|---------|
| `app/page.tsx` | 4 | logo、分类图标、联系二维码、智能体图标 |
| `app/admin/agents/page.tsx` | 3 | 分类图标 |
| `app/admin/settings/page.tsx` | 2 | logo 预览、二维码预览 |
| `app/admin/workflows/page.tsx` | 2 | 分类图标 |
| `app/admin/page.tsx` | 1 | logo |
| `app/login/page.tsx` | 1 | logo |
| `app/register/page.tsx` | 1 | logo |
| `components/agent-card.tsx` | 1 | 分类小图标 |
| `components/layout/admin-layout.tsx` | 1 | logo |

#### 分类策略

**策略 A：替换为 `next/image`（推荐用于尺寸固定的图片）**

适用于：logo、二维码、头像等尺寸确定的图片。

```tsx
// 之前
<img src={siteSettings.logo_url} alt="Logo" className="w-9 h-9 object-contain" />

// 之后
import Image from "next/image";

<Image
  src={siteSettings.logo_url}
  alt="Logo"
  width={36}
  height={36}
  className="object-contain"
/>
```

**前提条件**：需要在 `next.config.ts` 中配置允许的外部图片域名：

```ts
const nextConfig: NextConfig = {
  images: {
    remotePatterns: [
      { protocol: "https", hostname: "*.supabase.co" },
      { protocol: "https", hostname: "your-cdn.com" },
      // ... 其他可能的图片来源
    ],
  },
  // ...
};
```

**策略 B：保留 `<img>` + 忽略 warning（推荐用于动态小图标）**

适用于：分类图标、智能体图标等尺寸小、频繁变动、URL 动态的场景。`next/image` 反而会引入额外的 `<picture>` 包装和布局抖动。

```tsx
{/* eslint-disable-next-line @next/next/no-img-element */}
<img src={cat.icon_url} alt="" className="w-3.5 h-3.5 rounded-[3px] object-contain" />
```

并在该行上方加一行注释说明原因，例如：
```tsx
{/* 小图标不经过 next/image 优化，避免不必要的 request overhead */}
```

#### 推荐方案组合

| 文件 | 策略 | 理由 |
|------|------|------|
| logo（5 处：layout、首页、login、register、admin） | A：next/image | 首屏关键图片，需 LCP 优化 |
| 二维码（2 处：首页联系、settings 预览） | A：next/image | 尺寸固定，值得优化 |
| 分类图标（7 处：admin/agents、admin/workflows、首页、agent-card） | B：eslint-disable | 小于 20px，动态 URL，优化收益低 |
| settings 的 logo 预览 | A：next/image | 同 logo |
| 智能体图标（首页） | B：eslint-disable | 同分类图标 |

#### 风险
- `next/image` 要求 width/height 属性或 `fill`，布局变化需回归测试
- 外部图片域名未配置会 400 错误
- Supabase Storage 的图片 URL 需加入 `remotePatterns`

#### 验证
- 替换为 `next/image` 后观察 Network 面板，图片是否走 `/_next/image` 优化通道
- 登录页、首页、管理后台各处图片显示正常，无布局抖动
