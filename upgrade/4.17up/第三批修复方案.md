## M1：build / EPERM 环境复验

### 问题背景

之前执行 `npm run build`：
- 编译阶段通过
- 末尾出现 `spawn EPERM` 报错

初步判断是 Windows 本地环境的子进程权限问题（Defender / 终端 / 沙箱），**不是业务代码问题**。但没有在干净环境复验过，不能 100% 下结论。

### 目标

判断 EPERM 是"环境特定问题"还是"项目本身的构建问题"，然后：

- 若仅本地特定环境：记录规避方案，关闭此项
- 若任何环境都复现：才需要深入代码层面排查

### 修改方案

#### 第 1 步：明确三种环境复验

| 环境 | 目的 | 操作 |
|------|------|------|
| **本地干净终端** | 排除 VSCode 终端和编辑器干扰 | 关闭 VSCode 所有集成终端，用系统自带 PowerShell 或 cmd 独立窗口跑 |
| **本地 WSL / Linux** | 排除 Windows 平台特定问题 | 在 WSL Ubuntu 中 `npm ci && npm run build` |
| **CI 环境** | 平台无关的最权威验证 | 走 M2 的 CI 配置，触发一次 build |

#### 第 2 步：结果判定矩阵

| 本地终端 | WSL/Linux | CI | 结论 |
|----------|-----------|-----|------|
| ❌ | ✅ | ✅ | Windows 本地特定问题，可能是 Defender / 杀毒软件干扰 |
| ❌ | ❌ | ✅ | Windows 平台问题（Node 版本、权限、路径） |
| ❌ | ❌ | ❌ | 真正的构建问题，需要打开 stack trace 排查 |
| ✅ | ✅ | ✅ | 历史偶发，无需处理 |

#### 第 3 步：按结果处理

**情况 A（最可能）：仅本地 Windows 终端失败**

1. 把项目目录加入 Windows Defender 排除列表：
   - Windows 安全中心 → 病毒和威胁防护 → 管理设置 → 排除项 → 添加文件夹
2. 把 Node 版本统一到 LTS（20.x）
3. 在 `README.md` 或 `SETUP.md` 中记录："Windows 本地构建如遇 EPERM，请将项目目录加入 Defender 排除"
4. 关闭此项

**情况 B：WSL 也失败但 CI 过**

1. 查看完整 stack trace，定位是哪个子进程（常见：pdf-parse、sharp、esbuild 子进程）
2. 尝试删除 `node_modules/.cache`、`.next`，重新 `npm ci`
3. 若仍失败，考虑依赖层面问题（见候选池 L4）

**情况 C：CI 也失败**

走业务代码排查，不在本批方案范围内，单独立项。

### 验收

- 至少在 WSL 或 CI 中**连续跑 3 次** `npm run build` 全部通过
- EPERM 问题在文档中有明确处理说明（无论是修复还是规避）

### 风险

- WSL 首次跑会下依赖，耗时较长
- Defender 排除只在本地生效，不能通用

### M2.1：CI 质量门禁（平台中立方案）

#### 原则

不绑定特定平台（GitHub Actions / Jenkins / GitLab），而是：
1. 先把"要跑什么"固定在 npm scripts 里
2. 任何 CI 平台只要能执行 `npm run ci:check` 就够了
3. 团队后续用哪个平台再挂接

#### 修改方案

**第 1 步：在 `package.json` 中定义统一入口**

```json
{
  "scripts": {
    "ci:lint": "npm run lint",
    "ci:typecheck": "tsc --noEmit",
    "ci:build": "npm run build",
    "ci:test": "echo 'Tests pending (see M2.2 in 4.17up)'",
    "ci:check": "npm run ci:lint && npm run ci:typecheck && npm run ci:build && npm run ci:test"
  }
}
```

**口径约束**：
- `ci:lint` 复用仓库现有 `lint` 脚本（当前 `package.json` 里是 `eslint`），**不引入 `next lint` 这条并行口径**，保证 CI 和本地命令行为一致
- `ci:build` 同理复用现有 `build`，不直接写 `next build`
- `ci:test` 在 M2.2 落地前只是占位，不阻塞 `ci:check` 通过

这样任何 CI 平台都只需要一行：`npm run ci:check`。

**第 2 步：准备两套 CI 模板**

不直接启用，而是作为**文档示例**放在 `doc/ci-examples/` 中，团队选择用哪个再复制到根目录。

`doc/ci-examples/github-actions.yml`：

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run ci:check
        env:
          JWT_SECRET: ${{ secrets.JWT_SECRET }}
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY }}
          SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}
```

`doc/ci-examples/Jenkinsfile.ci`：

```groovy
pipeline {
  agent any
  stages {
    stage('Install') { steps { sh 'npm ci' } }
    stage('Check') {
      steps {
        withCredentials([
          string(credentialsId: 'jwt-secret', variable: 'JWT_SECRET'),
          string(credentialsId: 'supabase-url', variable: 'NEXT_PUBLIC_SUPABASE_URL'),
          // ...
        ]) {
          sh 'npm run ci:check'
        }
      }
    }
  }
}
```

**第 3 步：README 标注启用方式**

在 `README.md` 的"开发流程"小节加一段：

```markdown
## 质量门禁

所有检查通过 `npm run ci:check` 统一入口执行，包含：lint（eslint）、类型检查（tsc）、构建（next build）、测试（vitest，M2.2 落地后接入）。

CI 平台配置模板见 `doc/ci-examples/`。选定平台后，复制对应模板到项目根目录即可启用。
```

#### 验收

- `npm run ci:check` 在本地能一次性跑通
- CI 模板已存入 `doc/ci-examples/`
- README 有使用说明

#### 风险

- 暂不启用 CI，只是具备能力。正式启用需团队决策（选平台、配 secrets、约定"CI 红了不合并"）

### M2.2：测试基础设施（候选增强项）

#### 定位声明（与顶部范围表对齐）

- **M2.2 不进第三批必做面**
- M2.1 落地后，`ci:test` 以 echo 占位，`ci:check` 整体依然能通过，M2.2 延后不会阻塞任何环节
- 本项应作为**独立 sprint** 排期，不要拆散塞进普通迭代

**分支决策**：

| 情况 | 建议 |
|------|------|
| 本批结束时有专门人力和至少 1-2 周周期 | 紧接第三批做，作为本批的后续延伸 |
| 人力紧张或周期不够 | **只做到 M2.1**，M2.2 单独立项，以下内容作为立项预案 |

以下修改方案**是独立 sprint 的实施预案**，不代表本批必做内容。

#### 修改方案

##### 阶段 1：基础设施搭建（1-2 天）

**工具选型**：Vitest（快、TS 友好、与 Next.js 兼容好）

**安装**：

```bash
npm install -D vitest @vitest/ui @testing-library/react @testing-library/jest-dom jsdom
```

**配置 `vitest.config.ts`**：

```ts
import { defineConfig } from 'vitest/config';
import path from 'path';

export default defineConfig({
  test: {
    environment: 'jsdom',
    setupFiles: ['./test/setup.ts'],
    coverage: {
      reporter: ['text', 'html'],
      include: ['lib/**/*.ts'],
      exclude: ['lib/mock-data.ts', 'lib/types.ts'],
    },
  },
  resolve: {
    alias: { '@': path.resolve(__dirname) },
  },
});
```

**更新 `package.json`**：

```json
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run",
    "test:coverage": "vitest run --coverage",
    "ci:test": "vitest run"
  }
}
```

##### 阶段 2：核心 lib 层测试（3-5 天）

**优先级**（按风险从高到低）：

| 模块 | 风险点 | 必测场景 |
|------|--------|---------|
| `lib/crypto.ts` | 数据安全 | 加解密往返、空值、格式错误、密钥缺失 |
| `lib/auth.ts` | 身份安全 | Token 签发、校验、过期、篡改 |
| `lib/permissions.ts` | 权限绕过 | 7 种 scope 的匹配逻辑、个人/组织用户分支 |
| `lib/rate-limit.ts` | 限流正确性 | 窗口滑动、锁定时间、清理逻辑 |
| `lib/validate.ts` | 输入校验 | Schema 通过/失败、超大请求体、非 JSON |
| `lib/api-error.ts` | 错误格式 | 各错误码映射、dbError 错误识别 |

**覆盖率目标**：
- 第一阶段：`lib/` 下核心文件 ≥ 70%
- 不强求页面组件和 API 路由覆盖（成本高、收益低）

**示例**：`lib/crypto.test.ts`

```ts
import { describe, it, expect, beforeAll } from 'vitest';

beforeAll(() => {
  process.env.JWT_SECRET = 'test-secret-at-least-32-characters-long!!';
});

describe('crypto', () => {
  it('encrypt → decrypt roundtrip', async () => {
    const { encrypt, decrypt } = await import('@/lib/crypto');
    const plain = 'sk-test-123';
    const cipher = encrypt(plain);
    expect(cipher).not.toBe(plain);
    expect(decrypt(cipher)).toBe(plain);
  });

  it('encrypt empty returns empty', async () => {
    const { encrypt } = await import('@/lib/crypto');
    expect(encrypt('')).toBe('');
  });

  it('decrypt garbage throws', async () => {
    const { decrypt } = await import('@/lib/crypto');
    expect(() => decrypt('not-a-valid-cipher')).toThrow();
  });
});
```

##### 阶段 3（可选，后续迭代）

- API 路由的集成测试（需要 mock Supabase）
- 关键页面的组件测试
- E2E 测试（Playwright）

**不纳入本批范围**，等阶段 1、2 稳定后再评估。

#### 验收

- `npm run test:run` 可以执行并输出通过/失败
- `lib/` 核心文件覆盖率 ≥ 70%
- 故意改坏一个 `lib` 函数，`ci:test` 会失败
- 新人能跟着 README 跑通测试

#### 风险

- 投入成本大，如果团队人力不足，建议**先只做阶段 1（搭基础）**，阶段 2 分多次迭代补测试
- 测试本身也可能有 Bug，需要评审
- 补历史代码的测试需要深入理解业务，容易写成"为了测试而测试"的低价值用例

## 为什么不做更多

前两批已经解决了"代码正确性"和"代码可读性"。第三批如果还继续铺摊子（数据库改造、监控接入、依赖升级），会出现：

1. **失焦**：本来是"让工程可持续"，变成"顺便把架构也升级了"
2. **风险累积**：多项变更叠加，出问题难回滚
3. **延期**：工作量从 2 天膨胀到 10 天

**第三批必做面只保留两件事**：一个有定论的 build（M1） + 一个统一的质量门禁入口（M2.1）。

测试基础设施（M2.2）是价值最高的候选增强项，但它**独立成 sprint** 才有质量，不能塞进本批末尾赶工。

其余选项放进候选池，等触发信号再拉回来。
