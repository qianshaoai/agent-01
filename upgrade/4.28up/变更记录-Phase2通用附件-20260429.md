# 4.28up · Phase 2 · 体验版通用附件支持

**日期**：2026-04-29
**主分支**：master / devA+
**主题**：把附件上传从「Coze 专属」改造成「平台中立」，让任何 adapter 都能在自己能力范围内使用附件

---

## 1. 背景

Phase 1 完成多平台多轮上下文后，剩余短板是**附件**。

之前 `/api/trial/upload` 把所有文件直接上传到 Coze `/v1/files/upload`，拿 file_id 给 Coze 用。这意味着：

- 元器（Yuanqi）智能体收不到附件 — Coze file_id 是 Coze 私有的，元器看不懂
- 任何未来新接入的平台（智谱、Dify、OpenAI Compatible 等）都要重写一套上传链路

业务方明确：**附件需要通用，即使元器看不懂某些类型，其他智能体也得能用**。

---

## 2. 核心改造

把附件存储**先上 Supabase Storage**（拿公开 URL），让所有平台共享一份资源；Coze 平台时**额外**上传到 Coze 拿 file_id（保留 Coze 原生 RAG 能力）。

| 改前 | 改后 |
|----|----|
| 上传只到 Coze，存 `file_id` | 必到 Supabase 拿 `url`；Coze 平台时**额外**上传 Coze 拿 `cozeFileId` |
| `Attachment.file_id` 是核心字段 | `Attachment.url` 是核心字段；`cozeFileId` / 旧 `file_id` 作为可选 |
| 只 Coze adapter 处理附件 | Coze + Yuanqi adapter 都按自己能力处理；未来平台按相同模式扩展 |

---

## 3. 数据 schema 变更

### 3.1 `Attachment`（前端）

```ts
type Attachment = {
  kind: "image" | "file";
  file_name?: string;
  url?: string;          // ★ 新增 — Supabase 公开 URL，所有平台都能用
  cozeFileId?: string;   // ★ 新增 — Coze 私有 file_id，仅 Coze 平台
  file_id?: string;      // @deprecated 旧字段，向后兼容
  previewUrl?: string;   // 仅本地刚上传的 ObjectURL（session 内有效）
};
```

### 3.2 `ChatAttachment`（adapter）

```ts
type ChatAttachment = {
  kind: "image" | "file";
  fileName?: string;
  url?: string;
  cozeFileId?: string;
  fileId?: string;       // legacy
};
```

### 3.3 `PendingAttachment`（前端 pending 上传状态）

```ts
type PendingAttachment = {
  uploadId: string;      // ★ 新字段名（之前叫 fileId，语义不准确）
  fileName: string;
  bytes: number;
  kind: "image" | "file";
  previewUrl?: string;
  uploading: boolean;
  error?: string;
  url?: string;          // 上传成功后填充
  cozeFileId?: string;   // Coze 平台 + 上传成功时填充
};
```

### 3.4 `trial_messages.attachments`

JSONB 字段，schema 灵活。新结构：

```json
[
  { "kind": "image", "url": "https://...supabase.../trial/uid/123-cat.png", "file_name": "cat.png" }
]
```

旧记录里的 `file_id` 字段保留也能被前端正确处理（fallback 到 chip 渲染）。

## 4. 接口变更

### 4.1 `POST /api/trial/upload`

```text
1. 校验 agent_id + 文件类型 + 大小（保持不变）
2. ★ 必做：上传到 Supabase Storage `uploads/trial/<userId>/<ts>-<safeName>`
   → 拿 publicUrl
3. ★ 可选：仅当 agent.platform === "coze" 时，额外上传到 Coze /v1/files/upload
   → 拿 file_id；失败不阻断（降级到只用 URL）
4. 返回 { kind, file_name, bytes, url, cozeFileId? }
```

旧返回的 `file_id` 字段不再使用。

### 4.2 `POST /api/trial/chat`

请求 body 的 attachments 字段格式扩展：

```ts
attachments: [
  {
    kind: "image" | "file",
    url?: string,
    cozeFileId?: string,
    file_id?: string,        // legacy
    file_name?: string
  }
]
```

后端把字段透传给 adapter（在 `userMessage.attachments` 里），同时保存到 `trial_messages.attachments` JSONB。

## 5. Adapter 行为表

| Platform | 图片 | 文件（PDF/DOCX/etc.）|
|---|---|---|
| **Coze** | 优先 `{type:"image", file_id: cozeFileId}` → 没有则 `{type:"image", file_url: url}` | 必须 `{type:"file", file_id: cozeFileId}`（Coze 不支持外部 URL）；没 file_id 时**降级**为 text 链接 `[文件: name](url)` |
| **元器（Yuanqi）** | `{type:"image", image_url: {url}}` | 不在 content 数组里，**降级**追加到 message text 末尾：`[文件: name](url)` |
| **未来平台** | image_url（默认假设支持） | text 降级 |

---

## 6. 前端渲染逻辑

| 场景 | 行为 |
|----|----|
| 图片附件 + 有 url | 直接 `<img src={url}>` 显缩略图，点击新窗看大图 |
| 图片附件 + 无 url + 有 previewUrl（本地刚上传）| 用 previewUrl 过渡显示 |
| 图片附件 + 都没有 | chip 形式（图标 + 文件名）|
| 文件附件 + 有 url | chip 包成可点击 `<a>` 下载链接 |
| 文件附件 + 无 url（旧 Coze 数据，只有 file_id）| chip 不可点 |

## 9. 已知短板

| 项 | 现状 |
|----|------|
| Coze file_url 用于图片可能依赖 Coze 文档版本 | 如果 Coze 不接受 file_url，回退到 cozeFileId（双轨上传保证有 file_id） |
| 大文件上传双轨耗时 | Coze 平台时上传两次（Supabase + Coze），耗时翻倍。仅在 Coze 用文件类附件时显著 |
| Supabase Storage 公开 URL | 任何拿到 URL 的人都能下载。当前 trial 模块不存敏感数据，可接受 |
| 元器 PDF 等文档 | 只是文本提及，AI 不会真的解析内容。是元器平台限制，不是我们能解 |

## 10. 与 Phase 1 关系

- Phase 1（多轮上下文 + 历史回放）：让所有平台都能多轮聊天 + 回看历史
- Phase 2（通用附件）：让所有平台都能在能力范围内用附件

两 Phase 加起来，体验版**真正具备多平台能力**，未来加任何新平台只要：
1. 写好 adapter（按平台 SDK 文档实现）
2. 在 `lib/trial-agents.ts` 加配置
3. 在 Jenkins 加 credentials
4. push → 自动部署
