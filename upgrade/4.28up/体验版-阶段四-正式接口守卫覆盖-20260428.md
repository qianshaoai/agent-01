**日期**：2026-04-28
**前置**：阶段一已完成（`requireFullUser` 已就绪）。可在阶段二之后立即推进，不必等阶段三。
**目标**：给所有"持 user JWT 就能调"的正式业务接口加 `requireFullUser`，让 trial 账号即使绕过前端直接 curl 也会被 403 拦下

## 1. 阶段目标

完成本阶段后系统状态：

- 体验账号 cookie 调用 `/api/agents/[id]/chat`、`/api/conversations/*`、`/api/files/*`、`/api/users/me` 等正式接口 → 全部 403
- 普通用户 cookie 调用上述接口 → 行为完全不变
- admin 接口（`/api/admin/*`）天然不受影响（走 admin token 校验）

## 3. 实施要点

### 3.1 改动模式（每个文件统一）

入口处插入 3 行：

```ts
import { getPayloadFromRequest, requireFullUser } from "@/lib/auth";

export async function GET(req: NextRequest) {
  const payload = await getPayloadFromRequest(req);
  const guard = requireFullUser(payload);
  if (guard) return guard;

  // ... 原有业务代码
}
```

如果 route 已经在解析 payload（多数 user-jwt 接口都有），就**复用现有 payload 变量**，只插入 `const guard = requireFullUser(payload); if (guard) return guard;` 两行。

### 3.2 改动顺序

按下面顺序处理，每改完一个 file 就 curl 验一下：

1. `app/api/agents/[id]/chat/route.ts`（最高优先级；是 trial 最容易绕来调正式平台的入口）
2. `app/api/agents/route.ts` + `app/api/agents/[id]/route.ts`
3. `app/api/conversations/*`
4. `app/api/upload/route.ts` + `app/api/files/*`
5. `app/api/users/me/route.ts`
6. `app/api/workflows/*`
7. 其他遗漏的 user-jwt 接口（`grep -r "getPayloadFromRequest" app/api/`）

### 3.3 grep 兜底

每个阶段四收尾前必须跑一次（注意：现有用户接口同时使用 `getCurrentUser()` 和 `getPayloadFromRequest()` 两种身份解析方式，必须都覆盖）：

```bash
# 找出所有持 user JWT 的接口（含 getCurrentUser 路径）
grep -rl "getCurrentUser\|getPayloadFromRequest\|verifyToken" app/api/ \
  | grep -v "/api/admin/\|/api/trial/\|/api/settings"
```

注意 `/api/auth/` **不要全量排除**：

- `/api/auth/login`、`/api/auth/logout`、`/api/auth/register` → 排除（trial 必须能登 / 登出）
- `/api/auth/change-password`、`/api/auth/cancel-account` → **必须加 `requireFullUser`**（体验账号不允许改密 / 注销）

清单里每一个文件（除上述明确排除）都必须用上 `requireFullUser`，否则视为漏改。

**改动模式（getCurrentUser 路径）**：

```ts
// 现状
const user = await getCurrentUser();
if (!user) return NextResponse.json({ error: "未登录" }, { status: 401 });

// 改为（在 user 取出后立即拦截 trial）
const user = await getCurrentUser();
if (!user) return NextResponse.json({ error: "未登录" }, { status: 401 });
if (user.userType === "trial") {
  return NextResponse.json({ error: "体验账号不可调用此接口", code: "FORBIDDEN" }, { status: 403 });
}
```

或者：在 `lib/auth.ts` 给 `requireFullUser` 增加一个接受 `getCurrentUser` 返回值的重载，让两种路径调用方式一致。本期为减少改动面，先用直接 inline 判断的方式。

## 6. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 漏掉某个接口未加 guard，trial 账号能调 | §3.3 grep 兜底命令兜清单；每个文件人工对照 |
| 不小心给 `/api/auth/*` 加了 guard，trial 账号无法登录 | §5 严格约束写明，code review 时重点查 |
| guard 误伤普通账号 | `requireFullUser` 实现明确：仅 `userType === "trial"` 才 403，其他用户都通过 |
| `getPayloadFromRequest` 在某些 route 已包装，引入冲突 | 优先用现有 payload 变量；不重复解析 |

## 8. 与阶段三的协作

阶段四交付后再做阶段三时，trial 账号即使在浏览器手输 URL `/`、调用任何正式接口，都会：

- 被 middleware 跳回 `/trial`（前端层）
- 接口直接 403（后端层）

双层兜底，前端跳转失败也不至于污染数据。
