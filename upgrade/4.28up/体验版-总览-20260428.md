# 体验版模块 · 开发方案总览

**日期**：2026-04-28
**依据**：`doc/体验版模块设计方案.md`（已定稿）
**目标**：节前交付一个最小可用的体验版（trial）模块，0 影响现有正式平台
**总工作量预估**：1 ~ 1.5 天
**分支**：`devB`（已基于 `master` 创建）

---

## 1. 阶段拆分原则

- **每个阶段独立可验收**：单独完成不破坏现有功能、不引入半成品坏行为
- **依赖单向、逐层叠加**：后一阶段依赖前一阶段产出，但不反向修改
- **跨阶段不重构**：每个阶段只做自己的事，不顺手改前面的产出
- **每个阶段都有明确验收清单**：通过即可推进；不通过不进入下一阶段

## 2. 阶段总览

| 阶段 | 主题 | 涉及文件 | 行为变化 | 工时 |
|------|------|---------|---------|------|
| 一 | 身份与守卫 | `supabase/migration_trial.sql`、`lib/auth.ts` | trial 身份纳入类型系统；账号已建好；helper 已写完但暂未调用。**对所有现有用户零影响**。 | 1.5h |
| 二 | 体验版后端接口 | `lib/trial-agents.ts`、`app/api/trial/agents`、`app/api/trial/chat` | 体验版接口可用；trial 账号 curl 通；非 trial 账号 403。**前端无入口，普通用户感知不到**。 | 2.5h |

| 四 | 正式接口守卫覆盖 | `app/api/agents/[id]/chat/route.ts` 等正式 user-JWT 接口 | 体验账号即使绕过前端直接 curl 正式接口也会 403。**至此安全边界完整**。 | 1.5h |

## 3. 阶段间的依赖关系

```text
阶段一（身份与守卫）
   │
   ├─→ 阶段二（体验版后端接口）  依赖 requireTrialUser
   │
   ├─→ 阶段四（正式接口守卫覆盖） 依赖 requireFullUser
   │
   └─→ 阶段三（前端页面与路由分流）依赖 trial 类型存在
                                  依赖阶段二的 /api/trial/* 可用
```

阶段二和阶段四在阶段一之后可以并行；阶段三依赖阶段二的接口产出。建议串行：一 → 二 → 四 → 三，理由：

- 一 → 二 → 四：先把所有"身份 + 守卫 + 后端接口"全部就位，且每步都能 curl 验收
- 四 之后再做 三：写前端时所有后端边界都已稳定，不会出现"前端做完发现后端 403 还没生效"

## 4. 共同约束（所有阶段必须遵守）

**禁止**：

- ❌ 不引入任何新 npm 依赖
- ❌ 不改 `users` / `agents` / `tenants` / `logs` 等现有表结构（仅扩展 `users.user_type` CHECK 约束，不加新列、不加新表）
- ❌ 不改注册流程、不改企业用户/个人用户原有逻辑
- ❌ 不在前端任何位置出现 API Token / Bot ID
- ❌ 不向第三方平台发起带空 token 的请求（必须前置校验）
- ❌ 不对正式平台已有 API contract 做破坏性修改
- ❌ 不引入"批量改文件"式重构；插入式扩展为主

**必做**：

- ✅ 所有体验版接口入口必须 `requireTrialUser`
- ✅ 所有 user-JWT 正式接口入口必须 `requireFullUser`
- ✅ 体验账号在 DB 中预置 `first_login = false`（避免首次改密弹窗截断 `/trial` 跳转）
- ✅ Token / Bot ID 走环境变量，不进 Git
- ✅ 体验聊天通过后端代理 + SSE，**不跳第三方网页**

## 8. 范围之外

以下需求**不在本系列范围**，需独立立项评估：

- 多个体验账号（本期只支持 1 个预置账号）
- 体验版调用次数限制
- 体验版聊天历史 / 多轮上下文 / 短期记忆
- 体验版智能体管理后台 / 配置表
- 体验用户批量创建
- 正式用户与体验用户的权限模型完整改造
