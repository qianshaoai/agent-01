# 智能体删除 + 工作流双向引用展示 · 方案

**日期**：2026-04-29
**分支**：devA
**目标**：

1. 智能体管理页支持删除（被工作流引用时阻止删除并提示）
2. 智能体列表/编辑里展示"被哪些工作流引用"，可一键跳转到对应工作流；反过来工作流里绑定的智能体也能一键跳转回智能体管理页

## 2. 设计决策（已与产品对齐）

| 议题 | 决策 |
|------|------|
| 智能体被工作流引用时是否允许删除 | **硬阻止**。后端 409，前端展示引用列表 + 跳转链接，引导用户先去工作流解绑 |
| 引用工作流如何展示 | 智能体行内展示 chip，超过 N 个用「+M 更多」popover 展开全部 |
| chip 显示数量 | 默认 2 个，超出走「+M 更多」按钮 |
| 跳转打开方式 | **同窗口跳转**（保持上下文，浏览器后退即可返回） |
| 跳转方向 | **双向**：智能体行 chip → 工作流页 + 工作流步骤里的"绑定 XXX" → 智能体页 |
| 删除影响策略 | 因为硬阻止，删除时已无引用，无连带影响（`ON DELETE SET NULL` 实际不会触发） |
| **权限边界** | 删除/编辑保持与现有 PATCH 同口径（`requireAdmin`）。本期**不细分 super/system/org_admin**，仍是"任何 admin 都可"。如果产品要求 org_admin 不可删全局智能体，**单独立项**（涉及一批 admin 接口的口径调整，超出本方案） |
| **DELETE 失败语义** | 用 `.delete().eq("id", id).select("id")` 拿到删除行：返回空 → 404；DB error → 500；引用存在 → 409 |
| **统一字段命名** | 后端响应 `used_by` 数组里统一用 `{ id, name, stepCount }`（不再 wfId/wfName 双重命名），前端直接消费 |

## 3. 数据 / 接口变更

### 3.1 在既有 `app/api/admin/agents/[id]/route.ts` 追加 `DELETE` export

注意：**该文件已存在**且包含 PATCH。本方案在同文件 export 一个新的 `DELETE` handler，不新建文件。

```ts
// 伪代码
export async function DELETE(req, { params }) {
  const admin = await requireAdmin();
  if (admin instanceof Response) return admin;

  const { id } = await params;

  // 1) 引用检查（带嵌套 join 拿工作流名）
  const { data: refs, error: refsErr } = await db
    .from("workflow_steps")
    .select("workflow_id, workflows(id, name)")
    .eq("agent_id", id);
  if (refsErr) return dbError(refsErr);

  if (refs && refs.length > 0) {
    // 同一工作流多个 step 引用同一 agent → 聚合到 workflow 维度
    const map = new Map<string, { id: string; name: string; stepCount: number }>();
    for (const r of refs) {
      const wf = r.workflows as { id: string; name: string } | null;
      if (!wf?.id) continue;
      const cur = map.get(wf.id) ?? { id: wf.id, name: wf.name, stepCount: 0 };
      cur.stepCount += 1;
      map.set(wf.id, cur);
    }
    const used_by = [...map.values()];
    // 现有 apiError(message, code) 只收 2 参且 ErrorCode 不含 "AGENT_IN_USE"。
    // 为这一个接口直接 NextResponse.json，避免改全局 apiError 类型签名。
    return NextResponse.json(
      {
        error: `该智能体被 ${used_by.length} 个工作流引用，无法删除`,
        code: "AGENT_IN_USE",
        used_by,
      },
      { status: 409 }
    );
  }

  // 2) 真删；用 .select("id") 区分"被删了 N 行" vs "记录不存在"
  const { data: deleted, error: delErr } = await db
    .from("agents")
    .delete()
    .eq("id", id)
    .select("id");
  if (delErr) return dbError(delErr);
  if (!deleted || deleted.length === 0) {
    return apiError("智能体不存在或已被删除", "NOT_FOUND");
  }
  return NextResponse.json({ ok: true });
}
```

**错误语义清单**：

| 情况 | 响应 |
|------|------|
| 被引用 | 409 `AGENT_IN_USE` + `used_by: [{id, name, stepCount}]` |
| 记录不存在 | 404 `NOT_FOUND` |
| DB 异常 | 500 `dbError` |
| 非 admin | 401（`requireAdmin` 自动返回） |
| 删除成功 | 200 `{ ok: true }` |

注意：连带表 `agent_categories` / `tenant_agents` 通过 schema 已有 `ON DELETE CASCADE` 自动清理，无需手动删。
`workflow_steps.agent_id` 是 `ON DELETE SET NULL`，但因为本方案硬阻止，正常路径不会触发它。

### 3.2 改 `GET /api/admin/agents`：返回引用工作流（**只查当前页**）

每个 agent 多带：

```ts
workflows: [
  { id: "uuid", name: "内容组-创作IP稿件工作流" },
  ...
]
```

**关键：分页范围内查询，不全量扫**。`/api/admin/agents` 是分页接口，只关注当前页内 agent 的引用。否则数据增长后会把"分页接口"的成本变成全局成本。

实现：

1. 现有 SELECT 拉当前页 `agents` 列表 → 拿到 `pageAgentIds: string[]`
2. **仅当 `pageAgentIds.length > 0`** 时：
   - `db.from("workflow_steps").select("agent_id, workflows(id, name)").in("agent_id", pageAgentIds)`
   - 利用 supabase 嵌套 select，一次拿到 step 对应的 workflow 信息（避免再多发一次 query）
3. Node 端聚合 `agent_id → Map<wfId, {id, name}>`，再转 array
4. 合并到响应：每个 agent 行加 `workflows: [...]`

仅 1 次额外 query，且范围限定在当前页 agent，复杂度跟着分页走。

### 3.3 智能体页支持 `?focus=<agentId>`（含跨页处理）

- 客户端读 `useSearchParams().get("focus")` 与 `useSearchParams().get("pageSize")`（**注意：Next.js App Router 在客户端 page 里直接用通常 OK；但若 build 时报 "useSearchParams should be wrapped in a Suspense boundary"，需要把读 query 的逻辑抽成子组件包 `<Suspense>`，或退回到 `window.location.search`**）
- 数据加载完后：找到对应行 → `scrollIntoView({ behavior: "smooth", block: "center" })` + 加 1.5 秒 `ring-2 ring-[#002FA7] ring-offset-2` 高亮动画

**跨页问题（关键，双向都有）**：

`/api/admin/agents` 与 `/api/admin/workflows` 都已分页（默认 50，最大 100），目标记录可能不在第一页。本期对**两个页面**采用同一套两段式策略：

1. **跳转方统一带 `?pageSize=100`**：
   - 工作流页跳智能体页：`router.push("/admin/agents?focus=" + agentId + "&pageSize=100")`
   - 智能体页跳工作流页：`router.push("/admin/workflows?focus=" + wfId + "&pageSize=100")`
   `MAX_PAGE_SIZE=100`（`lib/config.ts:33`），可覆盖大多数场景。
2. **目标页 load 时，把 URL 里的 `pageSize` 传给请求**：
   - agents 页：`fetch("/api/admin/agents?pageSize=" + pageSizeFromUrl + "&page=1")`
   - workflows 页：`fetch("/api/admin/workflows?pageSize=" + pageSizeFromUrl + "&page=1")`
3. **若 agent / workflow 总数超过 100**：本期不实现"按 focus 自动定位页码"。改为兜底 toast：「目标记录不在当前页，请翻页查找」。**未来增强**：后端 GET 增加 `?findId=xxx` 参数返回该记录所在的 page 号，前端再带 `&page=N` 二次请求。
4. 目标找不到（数据已删 / 总数超 100 跨页）：**不报错、不留空白，仅 toast 提示**。

### 3.4 工作流页支持 `?focus=<wfId>`

同 §3.3 全套跨页策略：

- 读 URL 的 `focus` 与 `pageSize`，发请求时把 `pageSize` 透传给后端（默认 100）
- 数据加载完找到目标卡片 → `scrollIntoView` + 高亮 1.5s
- 找不到 → **不报错、不空白**，仅 toast 提示「目标工作流不在当前页，请翻页查找」
- 同样注意 `useSearchParams` 的 Suspense 兼容性

### 3.5 工作流步骤里"绑定：智能体名"改成可点击（**两处都要改**）

工作流页存在两种步骤渲染视图，**都要改**：

| 视图 | 定位 |
|------|------|
| 列表视图 | [app/admin/workflows/page.tsx:736-740](app/admin/workflows/page.tsx#L736) |
| 流程图视图 | [app/admin/workflows/page.tsx:1322-1330](app/admin/workflows/page.tsx#L1322) |

把 `<span>...绑定：{agent.name}</span>` 改成：

```tsx
<button
  onClick={(e) => { e.stopPropagation(); router.push(`/admin/agents?focus=${agent.id}&pageSize=100`); }}
  className="text-xs text-[#002FA7] hover:underline flex items-center gap-1"
  title="跳转到智能体管理"
>
  {iconByType}<span className="truncate">{agent.name}</span>
</button>
```

`stopPropagation()` 防止冒泡触发外层 step 的编辑/选中。

## 4. 前端变更

- 弹窗左下角红色「删除智能体」按钮

- 200 → 关弹窗 + 刷新列表 + toast「删除成功」

- 409 → 弹一个**禁止删除提示弹窗**：
  ```

  该智能体被以下工作流引用，请先在对应工作流中解绑：

  📋 财务-月报模板工作流（1 个步骤）
  
  [前往第一个工作流]  [关闭]
  ```
- 点工作流条目同样可跳转

### 4.3 工作流页 / 智能体页的高亮动效

```css

```

## 9. 风险与缓解

| 风险 | 缓解 |
|------|------|
| GET list 数据增长后变慢 | 已收窄到**额外 1 次** workflow_steps query（嵌套 join workflows），且 `.in("agent_id", pageAgentIds)` 限定当前页 agentIds 范围 |
| **agent / workflow 总数 > 100 时 focus 跨页失败（双向）** | 本期两端跳转 URL 都带 `?pageSize=100` 已覆盖绝大多数场景；超出时 toast 提示翻页。**未来增强**：后端 `?findId=xxx` 返回 page 号或在响应中夹带目标行 |
| 删除请求并发：两个管理员同时删 | 第二个删除返回 deleted.length === 0 → 前端按 404 处理 toast，列表自动刷新 |
| **删除竞态：引用检查通过 → 实际删除前有人新建步骤引用** | 引用检查到删除非事务，存在窗口期。窗口期内若有 step 新建，DB 会触发 `workflow_steps.agent_id` 的 `ON DELETE SET NULL`（**注意是 SET NULL，不是 CASCADE**），结果是该新引用步骤的 agent 字段被置空。这与"硬阻止"语义有短暂偏差但安全。**本期可接受**；如要绝对硬阻止，需要 Postgres 事务 + SELECT FOR UPDATE 或服务端 RPC，超出本期范围 |
| `?focus=` URL 被分享后访问者看不到对应记录 | 加载完后找不到目标 → **不报错、不空白**，仅 toast 提示 |
| `useSearchParams` 在 build 时报需 Suspense boundary | 把读 query 的逻辑抽成子组件包 `<Suspense>`，或退回 `window.location.search` |
| 权限边界：org_admin 删了不该删的全局智能体 | 本期沿用 `requireAdmin` 不细分。**如产品要求 org_admin 不可删全局智能体，单独立项**，影响一批 admin 接口的口径调整 |
