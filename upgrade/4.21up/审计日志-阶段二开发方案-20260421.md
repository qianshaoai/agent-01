## 3. 统一接入模板

所有上述接口按同一模板改造，避免零散风格。

### 3.1 标准接入骨架

```ts
import { writeAuditLog } from "@/lib/audit";

export const POST = async (req: NextRequest) => {
  const admin = await requireAdmin();
  if (admin instanceof Response) return admin;

  // 1. 操作前快照（更新/删除类必做；create 类跳过）
  const { data: before } = await db.from("xxx").select("*").eq("id", id).single();

  // 2. 执行业务
  const { data: after, error } = await db.from("xxx").update(patch).eq("id", id).select().single();

  // 3. 写审计（失败/成功都写）
  if (error) {
    await writeAuditLog({
      actor: { type: "admin", id: admin.adminId, name: admin.username, role: admin.role, tenantCode: admin.tenantCode },
      action: "xxx.update",
      resource: { type: "xxx", id, name: before?.name, tenantCode: before?.tenant_code },
      result: "failed",
      before,
      errorMessage: error.message,
      req,
    });
    return dbError(error);
  }

  await writeAuditLog({
    actor: { type: "admin", id: admin.adminId, name: admin.username, role: admin.role, tenantCode: admin.tenantCode },
    action: "xxx.update",
    resource: { type: "xxx", id, name: after.name, tenantCode: after.tenant_code },
    result: "success",
    before,
    after,
    req,
  });
  return NextResponse.json(after);
};
```

### 3.2 actor 构造约定

所有管理端接口统一用 `actorFromAdmin(admin)`（由 `lib/audit.ts` 导出，见 §3.5）：

```ts
import { actorFromAdmin } from "@/lib/audit";

const actor = actorFromAdmin(admin);
// 等价展开：{ type: "admin", id: admin.adminId, name: admin.username, role: admin.role, tenantCode: admin.tenantCode }
```

禁止在路由内手写等价解构，避免字段名漂移（`adminId` 还是 `id`、`tenantCode` 还是 `tenant_code`）。

登录**失败**时 `admin` 对象不存在，**不允许把用户原始输入直接放进 `actor.name`**（可能是完整手机号）。

actor 必须按下列规则构造：

```ts
const actor = {
  type: "admin" as const,
  id: null,
  name: "unknown_admin",                 // 固定值，不暴露输入
  role: null,
  tenantCode: null,
};

// 原始标识只允许走 extra.identifier_masked，且必须先脱敏
const extra = {
  identifier_masked: maskIdentifier(identifier),
};
```

**脱敏规则**（由 `lib/audit.ts` 导出 `maskIdentifier` 辅助函数，见 §3.5）：
- 手机号（纯数字 7-15 位）：保留前 3 后 4，中间 `****`
- 用户名（含字母/下划线）：保留前 2 字符 + `****`
- 其他：一律替换为 `***`

审计表最终不出现完整手机号或完整账号明文。

### 3.5 `lib/audit.ts` 新增的最小辅助（阶段一基础上的轻量补充）

为避免 2.3 那批十来个文件接入风格漂移，**允许** `lib/audit.ts` 额外导出以下 **最小** 辅助。不新建 helper 文件、不引入分层。

```ts
// 1) 从数据库原始行剔除必删字段（pwd_hash / api_key_enc / 等），返回新对象
//    白名单集中维护，避免各路由各写一份
export function stripSensitiveColumns<T extends object>(row: T | null | undefined): Partial<T> | null;

// 2) 登录失败 / 未登录场景下对原始标识的脱敏（§3.2 引用）
export function maskIdentifier(raw: string | null | undefined): string;

// 3) 便捷 actor 构造器（管理员身份 → AuditEvent.actor）
//    内部统一字段映射，避免各路由写 { id: admin.adminId, name: admin.username, ... } 散落
export function actorFromAdmin(admin: { adminId: string; username: string; role: string; tenantCode?: string | null }): AuditEvent["actor"];
```

**强约束**：
- 以上三个辅助**只允许在 `lib/audit.ts` 内实现**，不拆文件、不分层
- 路由里必须优先用这三个辅助，不得再在路由内手写等价逻辑
- 其他辅助需求本阶段一律记录为备注，不允许自行新增

### 3.3 resource_name 取值约定

- 用户：`real_name || username || phone_masked`
- 组织：`tenants.name`
- 部门/小组/分组：`name`
- 权限：`resource_type:resource_id`（权限本身不是独立资源，用组合值）

### 3.4 before/after 取值约定

| 操作 | before | after |
|------|--------|-------|
| `create` | null | 新建后的完整行 |
| `update` | 改前完整行 | 改后完整行 |
| `delete`（软删） | 删前完整行 | 墓碑化后的行 |
| `delete`（硬删） | 删前完整行 | null |
| `login.success` | null | `{ adminId, role, tenantCode }` |
| `login.failed` | null | null，仅填 `errorMessage` |
| `password.reset` | null（不读密码） | `{ userId, passwordChanged: true }` |

## 4. 失败场景接入要求（对应总方案 4.4）

以下失败场景**必须**写审计（不是主业务成功才记）：

1. 登录失败（密码错 / 用户不存在 / 状态异常 / 限流触发）
2. 修改角色失败（权限不足 / 目标不存在）
3. 删除组织失败（有关联用户阻塞）
4. 删除部门/小组失败（下挂成员阻塞）
5. 权限 assign 重复 / scope 无效
6. 所有 `requireAdmin()` 返回 Response 的早退路径**不写**（非鉴权后业务），避免未登录噪音污染

## 5. 脱敏与数据收缩

### 5.1 必须脱敏的表字段

`lib/audit.ts` 的 `sanitizeAuditData` 已覆盖关键词。但 before/after 来自数据库原始行，以下字段需**显式剔除**再传给 `writeAuditLog`（即使脱敏会替换为 `[REDACTED]`，"存一条 `[REDACTED]` 也是噪音"）：

- `users.pwd_hash`
- `admins.pwd_hash`
- `agents.api_key_enc`

**实现方式**：统一走 `stripSensitiveColumns`（`lib/audit.ts` 导出，见 §3.5），避免各路由手写解构不一致：

```ts
import { stripSensitiveColumns } from "@/lib/audit";

await writeAuditLog({
  ...,
  before: stripSensitiveColumns(before),
  after: stripSensitiveColumns(after),
});
```

`stripSensitiveColumns` 内部维护白名单（`pwd_hash` / `api_key_enc` / 未来新增的敏感列），一改全路由生效。


### 5.2 大对象收缩

`tenants` / `users` 等小行不做额外处理。
权限 `assign` 的 before/after 可能包含完整资源详情，本阶段直接传 `{ resource_type, resource_id, scope_type, scope_id }` 即可，不传全量资源快照。

## 8. 验证方式

### 8.1 单接口验证

每接入一个接口，手动跑一次对应管理后台动作，然后在 Supabase 执行：

```sql
SELECT action, actor_name, resource_type, resource_name, result, created_at
FROM audit_logs
ORDER BY created_at DESC
LIMIT 20;
```

期望能看到新写入的记录。

### 8.2 脱敏验证

对 `user.password.reset` 和 `auth.login.*`：

```sql
SELECT before_data, after_data, diff_data
FROM audit_logs
WHERE action LIKE 'auth.%' OR action LIKE 'user.password.%'
ORDER BY created_at DESC LIMIT 5;
```

期望：无任何 `pwd_hash` / `password` 明文。

### 8.3 失败场景验证

- 用错密码登录 → 应有一条 `auth.login.failed` 记录
- 删除仍有用户的组织 → 应有一条 `tenant.delete` + `result='failed'`

### 8.4 主业务不被阻断验证

临时把 `audit_logs` 表重命名触发 `writeAuditLog` 写入失败：
- 正常业务接口仍然返回 200
- 控制台出现 `[AUDIT-FAIL]` 日志
- 业务数据正确写入

测完恢复表名。

## 9. 风险点

| 风险 | 缓解 |
|------|------|
| 忘记剔除 `pwd_hash` / `api_key_enc` | 代码 review 时对照 §5.1 清单逐项核对；`sanitizeAuditData` 兜底脱敏 |
| before 读两次加重数据库负担 | 更新/删除本就需要判断存在，before 可与原有存在性检查合并 |
| 登录失败高频刷 audit | 已有 rate-limit 15 分钟 5 次上限，单个账号日志量可控 |
| actor_id 在 `admins` 表和 `users` 表间可能有 UUID 冲突 | `actor_type` 区分来源；查询时一律按 `(actor_type, actor_id)` 联合筛 |
| 新接入引入 lint 不一致 | 每个批次完成跑 `npm run ci:check` |
