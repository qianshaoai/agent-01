## 1. 本阶段目标

阶段四完成后，管理端应具备一套可实际使用的审计追查工具，核心目标四条：

1. 任何被接入审计的动作，都能在后台按多维筛选快速定位
2. 任何一条记录都能查看结构化详情（actor / resource / before / after / diff / 环境上下文）
3. 组织管理员只能看到本组织相关日志，平台级动作对其不可见
4. 敏感字段在展示层继续保持脱敏，不因查询页成为二次泄密源

阶段四**不做**：
- 不扩展审计写入范围（智能体 / 工作流 / 前台用户接口等不回头补）
- 不重构 `audit_logs` 表结构
- 不重写 `lib/audit.ts`
- 不引入前端埋点 / 访问轨迹 / 风控评分
- 不接入 Sentry / 短信 / 邮件 / 告警平台
- 不做冷归档 / 链式签名 / 不可篡改审计
- 不做风险等级标注或可疑行为自动识别

## 3. 页面与接口设计

### 3.1 列表接口 `GET /api/admin/audit-logs`

**接口职责**：根据筛选条件返回一页审计记录，统一 `{ data, pagination }` 格式（延续阶段 A1 口径）。

**查询参数**（全部可选，支持组合）：

| 参数 | 类型 | 语义 | 匹配方式 |
|------|------|------|---------|
| `page` | number | 页码，默认 1 | — |
| `pageSize` | number | 每页条数，默认 20、上限 100 | 复用 `PAGINATION` 配置 |
| `startAt` | ISO8601 | 时间范围下限（含） | `created_at >= startAt` |
| `endAt` | ISO8601 | 时间范围上限（含） | `created_at <= endAt` |
| `actorType` | string | `admin` / `user` / `system` | 精确 |
| `actorId` | string | 操作人 ID | 精确 |
| `actorName` | string | 操作人显示名 | 模糊（`ilike %q%`） |
| `actorRole` | string | 操作人角色 | 精确 |
| `tenantCode` | string | 操作人所属组织 | 精确 |
| `action` | string | 动作标识，如 `agent.update` | 精确或前缀匹配 |
| `actionPrefix` | string | 按前缀筛选（如 `agent.`） | `action like 'agent.%'` |
| `resourceType` | string | 资源类型 | 精确 |
| `resourceId` | string | 资源 ID 搜索 | 精确 |
| `resourceName` | string | 资源名称搜索 | 模糊 |
| `result` | string | `success` / `failed` | 精确 |
| `requestId` | string | 链路追踪 ID | 精确 |
| `targetTenantCode` | string | 被操作资源所属组织 | 精确 |

**组合筛选规则**：
- 所有参数之间是 `AND` 关系
- `action` 与 `actionPrefix` 二选一生效，同时传入时以 `action` 精确为准
- `startAt` / `endAt` 可单独传入一侧
- 权限强制条件（见 §4）无法被参数覆盖

**服务端优先原则（不可放宽）**：
- 审计日志的最终可见范围**以服务端过滤结果为唯一事实依据**
- 任何来自客户端的 `tenantCode` / `targetTenantCode` / `actorId` 等参数在违反角色边界时必须被服务端**静默覆盖或忽略**，不得返回提示、不得报错跳过
- 前端无法通过修改下拉选项、URL 参数、请求体伪造来扩大可见范围
- 详情接口（`GET [id]`）在返回前必须再做一次 `tenant_code` / `target_tenant_code` 的归属校验，避免直接猜 ID 越权

**排序**：固定 `created_at DESC`，不允许客户端改排序方向，保证追查逻辑稳定。

**返回结构**：

```json
{
  "data": [
    {
      "id": "uuid",
      "created_at": "2026-04-21T10:00:00.000Z",
      "actor_type": "admin",
      "actor_id": "...",
      "actor_name": "admin",
      "actor_role": "super_admin",
      "tenant_code": "DEMO",
      "action": "agent.config.update",
      "resource_type": "agent",
      "resource_id": "...",
      "resource_name": "案例撰写专家",
      "target_tenant_code": null,
      "result": "success",
      "request_id": "mo8cvp0w-0000",
      "ip_address": "192.168.x.x",
      "has_before": true,
      "has_after": true,
      "has_diff": true,
      "has_error": false
    }
  ],
  "pagination": { "page": 1, "pageSize": 20, "total": 1357 }
}
```

**列表接口刻意不返回** `before_data` / `after_data` / `diff_data` / `error_message` / `user_agent` / `extra` 的完整内容：
- 只用 `has_before` / `has_after` / `has_diff` / `has_error` 布尔标记提示"是否有内容可看"
- 降低列表响应体大小，避免一屏加载几十条时把大 JSON 一起传
- 详情展开再取完整数据

### 3.2 详情接口 `GET /api/admin/audit-logs/[id]`

**建议：本阶段实现独立详情接口**。理由：
- 列表接口已裁掉大字段，前端抽屉展开时必然要二次拉取
- 详情接口可在此处集中做"敏感字段二次脱敏展示兜底"（见 §5）
- 权限校验可复用同一套边界逻辑

**返回结构**：完整 `audit_logs` 行 + 再次经过脱敏兜底的 `before_data` / `after_data` / `diff_data`。

**不做**：不返回任何解密字段、不回查原资源当前值（避免追查时看到"现在的样子"污染历史判断）。

## 4. 权限与可见范围

### 4.1 角色边界（对应总方案 13.1）

| 角色 | 列表可见范围 | 详情可见范围 |
|------|-----------|-----------|
| `super_admin` | 全部 | 全部（仍脱敏） |
| `system_admin` | 全部 | 全部（仍脱敏） |
| `org_admin` | **仅本组织相关** | **仅本组织相关** |
| `user`（普通用户） | **禁止访问**，接口返回 `FORBIDDEN` | 同 |

### 4.2 `org_admin` 可见性硬规则

`org_admin` 只能看到满足下列任一条件的记录：

1. `tenant_code = admin.tenantCode`（本组织管理员做的操作）
2. `target_tenant_code = admin.tenantCode`（本组织资源被操作）

**查询实现方式**（服务端强制，不可被请求参数覆盖）：

```sql
WHERE (audit_logs.tenant_code = :adminTenantCode
       OR audit_logs.target_tenant_code = :adminTenantCode)
```

### 4.3 `org_admin` 看不到的动作

以下动作属于**平台级**，`org_admin` **不可见**：

| 动作族 | 原因 |
|-------|------|
| `settings.*`（系统设置 / logo / contact_qr） | 平台全局配置，与具体组织无关 |
| `notice.*` 中 `target_tenant_code IS NULL` 的全局公告 | 平台级公告 |
| `agent.*` 中没有 `target_tenant_code` 归属的平台级智能体 | 平台资源 |
| `workflow.*` 中平台级工作流 | 同上 |
| 其他组织的 `tenant.*` / `department.*` / `team.*` / `group.*` | 越权 |

实现上：§4.2 的 `tenant_code` + `target_tenant_code` 双字段过滤天然能排除上述动作（平台级动作这两字段均为 null），无需针对 action 再加白名单。

### 4.4 参数注入保护

`tenantCode` / `targetTenantCode` 等查询参数在 `org_admin` 身份下被**静默覆盖**为其自身组织码；客户端无法通过传参"尝试看别的组织"。这是 §4.2 硬规则的补充实现约定。

### 4.5 服务端优先原则（权限硬控制）

审计日志的可见范围控制**只在服务端生效**，前端**不承担任何权限控制职责**：

- 前端的筛选条、下拉框、默认值、URL 参数**只负责"展示当前可选条件"**，**不负责**"决定用户能看到什么"
- 前端在 URL / 本地状态 / 请求体中保留或传入越权参数时，服务端必须一视同仁地静默覆盖或忽略，**不得因为"前端没显示"就认为不会传**
- 任何"前端隐藏 = 不可见"的依赖都违反本原则
- 验收必须包含"绕过前端直接打接口"的越权场景（见 §10.2）

**实现提醒**：
- 列表接口在解析完查询参数后、拼 SQL 前，必须根据当前 `admin.role` 重置或追加权限条件
- 详情接口在拿到单条记录后、返回前端前，必须再做一次归属校验
- 两处校验逻辑独立落地，不互相依赖

### 4.5 权限实现位置

- 使用阶段一/二已有的 `requireAdmin()`
- `org_admin` 的双字段过滤直接写在 `/api/admin/audit-logs/route.ts` 中，不抽新 helper
- 详情接口取到单条后再做一次 `tenant_code` / `target_tenant_code` 校验，防止猜 ID 越权

## 5. 展示与交互要求

### 5.1 列表布局

沿用现有管理后台风格（表格 + 筛选条，参考 `app/admin/logs/page.tsx` 的现有运行日志页），不重做设计系统。

**列表列**（固定 9 列，避免信息过载）：

| 列 | 来源字段 | 说明 |
|----|---------|------|
| 时间 | `created_at` | 本地化到秒 |
| 操作人 | `actor_name` + `actor_role` | 两行展示，角色用小徽章 |
| 组织 | `tenant_code` | `-` 表示平台级 |
| 动作 | `action` | 整条如 `agent.config.update`，用 monospace 字体 |
| 资源类型 | `resource_type` | 带颜色徽章 |
| 资源名称 | `resource_name` | 超长省略，hover 显示完整 |
| 结果 | `result` | `success` 绿 / `failed` 红 |
| 请求 ID | `request_id` | monospace，可点击复制 |
| 操作 | — | "查看详情" 按钮打开抽屉 |

**不做**：不在列表列里展示 `before_data` / `after_data` / `diff_data` 任何片段，避免一屏加载过重。

### 5.2 筛选条布局

顶部筛选条分两行：

- **第一行（主筛选）**：时间范围、动作（支持前缀下拉，如 `agent.*`）、资源类型、结果
- **第二行（细筛选）**：操作人（模糊）、组织（下拉）、资源名称（模糊）、请求 ID、搜索按钮 + 重置按钮

筛选实时生效（`onChange` 后 400ms 防抖），避免每次改筛选都发请求。

### 5.3 详情抽屉

点击"查看详情" → 右侧打开抽屉（宽度约 `720px`，不全屏），避免切走列表上下文。

**抽屉分五段**：

1. **基础信息段**：时间、操作人、角色、组织、动作、资源类型、资源名称 / ID、结果、请求 ID、IP、User-Agent
2. **差异段 `diff_data`**：**首屏默认展开**，按键名分行渲染 `{ 字段: { before: 旧, after: 新 } }`
   - 优先展示，因为 diff 是追查最高价值信息
   - 若 `diff_data` 为空但 `before_data` / `after_data` 存在，提示"本次变更未生成字段级差异，请查看完整快照"
3. **原始快照段 `before_data` / `after_data`**：**折叠**，点击展开
   - 使用轻量 JSON 展示方式（见 §5.4），不要求虚拟化 / 不要求分页
   - 继续脱敏（见 §5.5）
4. **异常段**：仅 `result = failed` 时出现，展示 `error_message`
5. **扩展段 `extra`**：折叠展示（例如 `config_changed` / `identifier_masked` 等）

### 5.4 JSON 可读性（第一版克制实现）

本阶段只做最小可读实现，**不追求高级 JSON 浏览器行为**。

**第一版要求**：
- `diff_data` 优先展示（详情抽屉首屏默认展开）
- `before_data` / `after_data` / `extra` 折叠展示，点击展开
- 允许直接使用 `<pre><code>` + `JSON.stringify(value, null, 2)` 作为兜底渲染方式
- 若实现成本低，可以加上轻量高亮（key 蓝、string 绿、`[REDACTED]` 红徽章），但**不作为验收硬性要求**

**本阶段明确不做**（作为后续增强项，不在本阶段范围）：
- 不做大对象分页虚拟化
- 不做多层按需展开的交互式 JSON 树
- 不引入 `react-json-view` 等第三方 JSON 浏览器库
- 不做复杂性能优化组件

**原则**：阶段四的首要目标是"能用"，不是"好看"。如果后续真遇到性能瓶颈（例如某条记录 `before_data` 超过几十 KB 导致页面卡顿），单独作为增强项评估，不拉高阶段四实现门槛。

### 5.5 展示层脱敏兜底

虽然写入时已经通过 `sanitizeAuditData` + `stripSensitiveColumns` 双道防线，展示层仍做第三道：

- 前端在渲染 `before_data` / `after_data` / `diff_data` 时，若遇到字符串值包含 `[REDACTED]`，显示为红色徽章"已脱敏"
- 前端不做主动替换（避免错误判断引起误伤），只做"强调展示"

### 5.6 交互要求

- 请求 ID 列一键复制到剪贴板（便于跨系统排查）
- 列表空态："没有符合条件的审计记录"，提示可调宽时间范围
- 筛选无结果时不隐藏筛选条，用户能立即改筛选
- 切换页码保留筛选条件
- URL 参数与筛选条件双向绑定（`?action=agent.delete&page=2`），便于分享/跳转追查链接

## 6. 查询性能考虑

### 6.1 依赖阶段一既有索引

阶段一已建索引：
- `created_at DESC`
- `(action, created_at DESC)`
- `(resource_type, resource_id, created_at DESC)`
- `(actor_type, actor_id, created_at DESC)`
- `(tenant_code, created_at DESC)`
- `(result, created_at DESC)`
- `request_id`

本阶段**不新增索引**。如果后续实测存在慢查询，再单独立 migration。

### 6.2 查询策略

- 默认只查最近 30 天（`startAt` 未传时，接口隐式限定 `created_at >= now() - interval '30 days'`），避免全表扫描
- 若用户显式传入更早的 `startAt`，放开限制
- `pageSize` 上限 100（复用 `PAGINATION.MAX_PAGE_SIZE`）
- 所有模糊搜索字段（`actorName` / `resourceName`）只在已有精确筛选（`tenantCode` / `actionPrefix` / `resourceType` 等至少一项）命中后才启用，避免全表 ilike

### 6.3 org_admin 查询路径

`org_admin` 的 `(tenant_code OR target_tenant_code) = X` 走 `tenant_code` 索引；`target_tenant_code` 当前无独立索引。若 `org_admin` 场景查询偏慢，后续评估是否加 `(target_tenant_code, created_at DESC)` 索引——本阶段先观察。

## 7. 导出策略设计

### 7.1 本阶段结论：**暂不实施导出**

原因：

1. **高敏感性**：审计数据含 `before_data` / `after_data`，即便已脱敏，汇集成 CSV/JSON 下载到本地仍是放大泄露面的动作
2. **无明确业务触发**：目前尚无外部合规要求强制离线归档
3. **一旦实施需要同时实施"导出行为的审计"**（`audit.export`），增加阶段四工作量

### 7.2 若未来启用，设计预留

当未来需要导出时，须满足：

| 维度 | 要求 |
|------|------|
| 权限 | 仅 `super_admin`；`system_admin` 需二次审批；`org_admin` 不允许 |
| 范围 | 单次导出上限 10,000 条，且必须带时间范围约束（最多 90 天） |
| 格式 | CSV（不含 `before_data` / `after_data` 完整 JSON，只展开列表同款 9 列） |
| 审计 | 导出动作本身必须写入一条 `audit.export` 记录，`extra` 含导出参数摘要、命中条数、导出者 |
| 触发方式 | 独立接口 `POST /api/admin/audit-logs/export`，**不在**列表接口上加 `format=csv` 旁路 |

上述内容在本阶段**只落文档、不落代码**。

## 8. 不改动的部分（严格边界）

**禁止**：
- 不改 `audit_logs` 表结构（含索引）
- 不改 `lib/audit.ts`（除非查询/展示层实测需要补 `desensitizeForDisplay` 辅助；但优先让前端处理）
- 不改阶段一 / 二 / 三已接入的写入路径
- 不动现有 `logs` 表和 `/api/admin/logs` 页面
- 不加新 npm 依赖（含 JSON viewer 组件库）
- 不做导出（见 §7）
- 不做告警 / Sentry / 通知
- 不做跨 `audit_logs` 的关联查询（例如把 `actor_id` 联到 `admins` / `users` 表取最新名字）—— 审计数据是时间点快照，不做后续状态回填

## 9. 实施批次拆分

阶段四内部拆 3 个子批次，每个子批次单独汇报：

| 批次 | 内容 | 文件 | 预计改动量 |

## 10. 验证方式

### 10.1 后端验证（批次 4.1）

```bash
# super_admin 登录后
curl -b "ai_portal_admin_token=<JWT>" \
  "http://localhost:3000/api/admin/audit-logs?action=agent.config.update&pageSize=5"
# 期望：返回 { data, pagination }，data 每条含 has_before / has_after / has_diff

# 详情
curl -b "..." "http://localhost:3000/api/admin/audit-logs/<id>"
# 期望：返回完整行，敏感字段值为 [REDACTED]
```

### 10.2 权限边界验证（批次 4.1，全部绕过前端直接打接口）

| 角色 | 操作 | 期望 |
|------|------|------|
| `super_admin` | 无 `tenantCode` 参数查列表 | 返回全部数据 |
| `system_admin` | 同上 | 返回全部数据 |
| `org_admin`（DEMO） | 无 `tenantCode` 参数查列表 | 自动只返回 `tenant_code=DEMO OR target_tenant_code=DEMO` 的记录 |
| `org_admin`（DEMO） | 传 `tenantCode=OTHER` 尝试越权 | 参数被静默忽略，仍只返回 DEMO 相关；接口不报错、不提示 |
| `org_admin`（DEMO） | 传 `targetTenantCode=OTHER` 尝试越权 | 同上，静默覆盖 |
| `org_admin`（DEMO） | 直接查一条 `settings.logo.update` 详情（无 tenant 归属） | 403 |
| `org_admin`（DEMO） | 直接猜一个其他组织日志的 UUID 打详情接口 | 403，不返回任何记录内容 |
| `user`（普通用户） | 查任意接口 | 401 / 403 |

**关键验收点**：上述所有用例都**不经过前端页面**，直接用 curl 或 HTTP 客户端打接口。这是验证"权限硬控制在服务端"的唯一正确方式。

### 10.3 前端验证（批次 4.2）

- 各筛选项可单独生效、可组合
- 列表列显示与 §5.1 一致，无大 JSON 溢出
- 点击某条 → 抽屉打开 → diff 默认展开 → 折叠段能展开 / 收起
- 含 `[REDACTED]` 的字段显示红色"已脱敏"徽章
- 请求 ID 一键复制成功
- URL `?action=xxx&page=2` 能还原筛选状态

### 10.4 追查场景模拟

模拟三种真实追查：

1. **"某个智能体今天被谁改过 API Key"** → 筛 `action=agent.config.update` + `resourceName=xxx` + 时间范围 → 应能在列表看到至少一条并在 `extra.config_changed.api_key=true`
2. **"昨晚一次登录失败高峰"** → 筛 `actionPrefix=auth.login.` + `result=failed` + 时间范围 → 应能看到对应记录
3. **"某个请求 ID 整个链路做了什么"** → 筛 `requestId=xxx` → 应返回该请求触发的所有审计记录

三种场景都能在 30 秒内定位到结果视为合格。

## 11. 风险点

| 风险 | 缓解 |
|------|------|
| 列表查询响应体过大 | §3.1 裁掉大字段；列表只返回布尔 has_* 标记 |
| `org_admin` 查询走不上索引 | §6.3 先观察；必要时单独加索引，不在本阶段做 |
| 前端 JSON 渲染大对象卡顿 | 本阶段不做虚拟化；可用 `<pre>` 兜底；若后续实测瓶颈再独立评估 |
| `org_admin` 越权路径遗漏 | §10.2 强制验证 5 种角色/场景矩阵 |
| 敏感信息展示层泄漏 | §5.5 展示层第三道防线；§3.2 详情接口再做一次脱敏兜底 |
| 筛选参数被 URL 篡改 | §4.4 服务端静默覆盖越权参数；§4.2 硬规则不可绕过 |
| 审计页自身被误审计造成无限循环 | `GET /api/admin/audit-logs*` 本身**不写审计**（只读 + 对当前资源无变更） |
| 时间筛选默认 30 天导致误以为"没有历史数据" | §5.2 空态文案明确提示"当前仅查询最近 30 天"，并给一键放宽按钮 |

## 14. 后续增强项建议（非本阶段）

审计体系四阶段完成后，以下条目作为独立立项候选，**不属于本阶段实施**：

1. **导出与合规**：`audit.export` 动作 + CSV 导出 + 审批流
2. **风险标注**：对 `action` 做静态分级（high / medium / low），列表以颜色区分
3. **可疑行为检测**：同一 `actor_id` 短时间内多次 `result=failed`、同一 IP 多 actor 等规则
4. **告警联动**：`[AUDIT-FAIL]` 前缀对接 Sentry；特定 action 写入时触发邮件/短信通知
5. **归档策略**：超过 N 天的审计记录迁移到冷归档表 / 对象存储
6. **不可篡改保证**：审计记录链式签名或 WORM 存储
7. **外部 API**：开放只读审计查询 API 给外部合规系统
8. **前台用户侧审计**：接入前台账号改密 / 注销 / user_agents 等动作
9. **任务执行链路审计**（对应总方案 7.1 `task.*` 预留）

每条后续增强项启动前都需要独立评估触发信号和边界，不在本阶段范围内展开。
