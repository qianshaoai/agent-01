在 `logs` 表之后追加 `audit_logs` 表与索引的 `CREATE TABLE IF NOT EXISTS`，保持新建库也能直接拿到该表。

**表级约定**（与现有表风格保持一致）：
- `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- 建表后追加 `ALTER TABLE audit_logs DISABLE ROW LEVEL SECURITY;`（现有所有表同一写法）
- 索引命名沿用 `audit_logs_xxx_idx` 格式，与 `logs_xxx_idx` 风格对齐
- `before_data` / `after_data` / `diff_data` / `extra` 四个 JSONB 字段允许 NULL（`null` 表示"本动作无此数据"而非 `{}`，便于后续筛选）

### 2.2 脱敏规则（对应总方案 9）

**键名匹配策略：大小写不敏感 + 同时匹配下划线/驼峰/空格变体 + 关键词包含匹配**

匹配算法：
1. 把 key 小写并去掉 `_` / `-` / 空格（例如 `api_key` / `apiKey` / `API-KEY` → `apikey`）
2. 再用下列关键词做 `includes` 判断

**关键词清单**（只要 key 归一化后包含其一即脱敏）：
- `password`
- `pwdhash`（对应 `pwd_hash`）
- `secret`（覆盖 `api_secret` / `apiSecret` / `clientSecret` 等）
- `token`（覆盖 `token` / `accessToken` / `refreshToken` / `access_token` / `refresh_token` / `idToken` 等）
- `cookie`
- `authorization`
- `apikey`（覆盖 `api_key` / `apiKey` / `api_key_enc` / `apiKeyEnc` 等）
- `session`

命中后统一替换为字符串 `"[REDACTED]"`。

**手机号字段**：归一化 key 包含 `phone` / `mobile` / `cellphone` 之一时，保留前 3 后 4，中间用 `****` 替代（长度 < 7 的值原样保留）。

**大文本**：字符串长度 > 1024 → 截断为前 1024 字符 + `"…[truncated]"`。

**嵌套处理**：
- 对象递归处理每个 key
- 数组按元素递归（每个元素仍走上述规则，不做额外结构变换）
- 保持原结构类型和深度

### 2.3 diff 规则（对应总方案 8）

**定位**：轻量 diff，不承诺精细差异。目标只是"看出改了哪些字段"，不是"还原完整变更轨迹"。

**规则**：
- 平铺字段用 `===`（字符串/数字/布尔）直接对比
- 一层嵌套对象用 `JSON.stringify` 对比后再逐键展开（`modelParams` 等 JSON 列就是一层）
- **数组统一按"整体字段变化"处理**：不做引用比较，不做长度启发式，不做元素级深比较。两个数组一律 `JSON.stringify` 对比：字符串相等视为未变；不等则整体作为 `{ before: 旧数组, after: 新数组 }` 记录
- 只记录真正变化的键
- 输出形如：
  ```json
  {
    "enabled": { "before": true, "after": false },
    "permissions": { "before": [...], "after": [...] }
  }
  ```

**明确不做**（留到后续阶段评估）：
- 数组元素级 diff
- 深度嵌套对象的路径展开
- 变更语义推断（如"新增了某个权限"而非"数组变了"）

### 2.4 writeAuditLog 行为

1. 深拷贝 `before` / `after`，分别 `sanitizeAuditData`
2. `diff` 未传且 before/after 皆为对象 → 自动 `buildAuditDiff`；diff 本身也脱敏
3. 从 `req.headers` 抽取：
   - `ip_address`：优先 `x-forwarded-for` 第一段（按 `,` split 后 `[0].trim()`），回退 `x-real-ip`
   - `user_agent`：`user-agent` 原样
   - `request_id`：`x-request-id` 原样（由 middleware 注入）
4. 写入 `audit_logs` 表
5. **写入失败只 `console.error`，不抛错**，返回 `Promise<boolean>`（`true` = 写入成功，`false` = 失败）
6. **错误日志必须带上下文**，统一前缀 `[AUDIT-FAIL]`，格式：
   ```
   console.error("[AUDIT-FAIL]", JSON.stringify({
     action, resource_type, resource_id, actor_id, request_id, error: err.message
   }));
   ```
   便于后续接入告警（如 Sentry）时按 `[AUDIT-FAIL]` 前缀筛选，也便于阶段二灰度期团队观察写入健康度。

## 5. 风险点

| 风险 | 缓解 |
|------|------|
| actor_id 不加外键，删除用户后指向失效 | 按总方案预期：审计要保留历史轨迹，墓碑化后仍能通过 actor_name 追溯 |
| 大 JSON 写入导致表膨胀 | 大文本已做 1KB 截断；第一期只记关键动作不记列表查询 |
| 写入失败吞错 | 只吞表级错误，同时 `console.error` 带 `[AUDIT-FAIL]` 前缀日志可观察 |
