# 4.21 总设计方案：审计日志体系建设与实施规划

**日期**：2026-04-21  
**优先级**：P0  
**适用范围**：`agent-01` 当前项目  
**目标执行者**：Claude / Codex / 任意后续接手实施的工程师  

---

## 1. 背景与设计目标

当前项目已经具备以下能力：

1. 用户端与管理端双入口
2. 用户、组织、部门、团队、用户组等管理能力
3. 智能体、工作流、分类、权限分配等后台配置能力
4. 聊天、上传、语音、公告、品牌设置等业务能力
5. 现有运行日志 `logs`，可记录部分行为结果与耗时

但目前缺少一套真正面向线上治理的**业务审计日志体系**。  
这会直接带来以下问题：

1. 无法准确追溯“谁在什么时候修改了什么”
2. 无法还原关键资源变更前后的状态
3. 无法区分普通运行日志与高价值敏感操作日志
4. 管理员越权、误操作、错误配置后难以排查责任链
5. 上线后很难支撑安全、运维、合规、客户追责场景

因此，本方案目标不是做“更多日志”，而是建立一套**独立、统一、可追溯、可检索、可脱敏的审计日志体系**。

---

## 2. 本次建设目标

本次方案要解决的核心问题只有四个：

1. 记录谁做了什么
2. 记录操作影响了什么资源
3. 记录资源变更前、变更后、差异内容
4. 支持按时间、人员、资源、组织、结果快速追查

本次方案完成后，项目至少应具备：

1. 独立的 `audit_logs` 审计日志表
2. 统一的服务端审计写入能力
3. 关键后台管理动作的审计接入
4. 审计日志后台查询页
5. 敏感字段脱敏机制
6. 审计日志接入规范与后续扩展规划

---

## 3. 定位与边界

### 3.1 审计日志不是运行日志

必须明确区分两种日志：

1. **运行日志**
   用于记录接口调用结果、耗时、错误、链路排查  
   当前已有 `logs`，属于这一类

2. **审计日志**
   用于记录关键业务操作、敏感配置变更、权限变更、删除禁用等治理类动作  
   本方案新增 `audit_logs`

一句话区分：

- `logs` 解决“系统怎么跑的”
- `audit_logs` 解决“谁改了什么”

### 3.2 本次纳入范围

本次方案纳入的审计范围：

1. 认证与账号安全相关动作
2. 用户、组织、部门、团队、用户组管理
3. 角色、权限、资源可见范围变更
4. 智能体创建、修改、删除、启停、分配
5. 工作流创建、修改、删除、步骤调整、可见范围变更
6. 系统设置、品牌、公告、二维码等后台配置变更

### 3.3 本次暂不纳入范围

本次不做或不强制做的内容：

1. 页面访问轨迹级别的全量审计
2. 所有普通列表查询的访问审计
3. 前端纯 UI 交互埋点
4. 不可篡改链式签名审计
5. 审计日志独立归档服务
6. 风险评分模型与自动风控策略

本次目标是先完成**业务审计闭环**，不是一次性做完整安全平台。

---

## 4. 总体设计原则

### 4.1 统一入口原则

所有审计日志必须通过统一服务端工具写入，禁止在各接口里手写零散结构。

### 4.2 业务优先原则

优先记录真正有治理价值的动作，不做大量低价值噪音日志。

### 4.3 脱敏优先原则

审计日志必须可追责，但不能变成敏感数据的二次泄露源。

### 4.4 成功失败都要记录

高风险操作不仅成功要记，失败也要记，以便发现攻击、误操作或越权尝试。

### 4.5 结构化优先原则

审计日志不应只记录一句话描述，必须包含结构化字段，支持查询与筛选。

### 4.6 渐进式落地原则

先覆盖高风险后台动作，再逐步扩展到更多场景，避免一次性改动过大。

---

## 5. 总体架构

建议整体架构拆为四层：

1. **采集层**
   各管理类 API、认证类 API、后续任务执行类 API 触发审计事件

2. **标准化层**
   统一的 `lib/audit.ts` 负责：
   - 标准字段填充
   - actor / resource 归一化
   - before / after / diff 计算
   - 敏感字段脱敏

3. **存储层**
   统一写入 `audit_logs` 表

4. **展示层**
   管理后台增加审计日志页，支持筛选、查看详情、追查上下文

## 6. 数据模型设计

### 6.1 新增表：`audit_logs`

建议字段如下：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `created_at` | TIMESTAMPTZ | 创建时间 |
| `actor_type` | TEXT | `admin` / `user` / `system` |
| `actor_id` | UUID / TEXT | 操作者 ID |
| `actor_name` | TEXT | 操作者显示名 |
| `actor_role` | TEXT | 操作者角色 |
| `tenant_code` | TEXT | 操作者所属组织 |
| `action` | TEXT | 动作标识，如 `agent.update` |
| `resource_type` | TEXT | 资源类型，如 `agent` |
| `resource_id` | UUID / TEXT | 资源 ID |
| `resource_name` | TEXT | 资源名称 |
| `target_tenant_code` | TEXT | 被操作资源所属组织 |
| `result` | TEXT | `success` / `failed` |
| `ip_address` | TEXT | 来源 IP |
| `user_agent` | TEXT | 请求 UA |
| `request_id` | TEXT | 请求链路 ID |
| `reason` | TEXT | 操作原因，可选 |
| `before_data` | JSONB | 操作前快照 |
| `after_data` | JSONB | 操作后快照 |
| `diff_data` | JSONB | 差异内容 |
| `error_message` | TEXT | 失败原因 |
| `extra` | JSONB | 额外扩展字段 |

### 6.2 索引建议

建议建立以下索引：

1. `created_at DESC`
2. `(action, created_at DESC)`
3. `(resource_type, resource_id, created_at DESC)`
4. `(actor_type, actor_id, created_at DESC)`
5. `(tenant_code, created_at DESC)`
6. `(result, created_at DESC)`
7. `request_id`

### 6.3 保留现有 `logs`

现有 `logs` 表保留，不合并、不替代、不做兼容式混写。

## 8. 数据快照与差异设计

### 8.1 `before_data`

记录操作前的资源快照。

适用场景：

1. 更新前读取旧值
2. 删除前读取资源当前状态
3. 禁用前读取当前状态

### 8.2 `after_data`

记录操作完成后的资源快照。

适用场景：

1. 创建成功后的新资源
2. 更新成功后的新状态
3. 禁用后的最终状态

### 8.3 `diff_data`

只记录真正变化的字段。

建议格式：

```json
{
  "enabled": { "before": true, "after": false },
  "name": { "before": "旧名称", "after": "新名称" }
}
```

### 8.4 差异计算策略

第一期建议采用**轻量级对象差异策略**：

1. 比较平铺字段
2. 比较常见一层嵌套对象
3. 不做复杂数组深比较
4. 对大对象仅记录关键字段差异

原因：

1. 足够满足上线治理需求
2. 复杂 diff 成本高、稳定性差
3. 当前项目核心资源结构相对可控

## 9. 敏感字段脱敏设计

### 9.1 必须脱敏字段

以下字段不得原样写入审计日志：

1. 密码
2. `pwd_hash`
3. `api_key`
4. `api_secret`
5. `api_key_enc`
6. token
7. session
8. cookie
9. 完整手机号  
   视场景可部分脱敏

### 9.2 脱敏策略

建议统一规则：

1. 密钥类字段：只记录“已修改”，不记录明文
2. 密码类字段：统一替换为 `[REDACTED]`
3. 手机号：保留前 3 后 4，中间脱敏
4. 大文本：可截断存储，防止日志膨胀

### 9.3 关键要求

必须做到：

1. 审计日志可追责
2. 审计日志不可泄密

不能出现：

1. 智能体 API Key 在审计页明文展示
2. 用户密码相关字段进入日志
3. 完整 token 被写入日志

## 10. 接入方式设计

### 10.1 统一工具文件

建议新增：

`lib/audit.ts`

负责：

1. 标准化 actor 信息
2. 标准化 resource 信息
3. 自动脱敏
4. 自动生成 diff
5. 写入 `audit_logs`

### 10.2 核心调用方式

建议提供统一函数：

1. `writeAuditLog()`
2. `buildAuditDiff()`
3. `sanitizeAuditData()`

### 10.3 接入规范

关键资源操作按以下模式接入：

1. 操作前读取旧数据
2. 执行业务写操作
3. 操作成功后写入成功审计
4. 发生异常时写入失败审计

### 10.4 请求上下文采集

审计写入时应尽量补齐：

1. `request_id`
2. `ip_address`
3. `user_agent`
4. 当前登录身份
5. 当前组织信息

## 13. 权限与访问控制

### 13.1 谁可以看审计日志

建议第一期权限如下：

1. `super_admin`：查看全部
2. `system_admin`：查看全部或平台级范围
3. `org_admin`：只查看本组织相关日志
4. 普通用户：无审计日志访问权限

### 13.2 访问边界

必须遵守：

1. 组织管理员不能看到其他组织的审计日志
2. 组织管理员不能看到平台级敏感配置日志
3. 审计详情中的敏感字段仍需脱敏

### 13.3 导出策略

第一期建议：

1. 可以暂不开放导出
2. 如开放导出，必须单独审计“导出行为”

### 14.1 存储增长

审计日志属于持续增长型数据，必须预留增长预期。

第一期建议：

### 14.3 保留策略

## 15. 失败与异常处理策略

### 15.1 审计写入失败处理原则

原则建议如下：

1. 对核心业务操作，不应因审计日志失败直接阻断主业务
2. 但应记录系统错误，便于后续排查

### 15.2 推荐策略

第一期建议：

1. 主业务成功
2. 审计写入失败
3. 返回业务成功，但记录错误日志并标记告警

原因：

1. 避免后台管理因为审计表偶发问题完全不可用
2. 同时保证可观察性

### 15.3 特殊场景

对于极高风险动作，后续可考虑升级为：

1. 审计写入失败则阻断操作

但这不作为第一期强要求。

## 17. 验收标准

完成第一期后，至少应满足以下验收点：

1. 已存在独立的 `audit_logs` 表
2. 审计日志与现有 `logs` 明确分层
3. 关键后台动作可记录 `actor`、`action`、`resource`
4. 更新类操作可查看 `before_data`、`after_data`、`diff_data`
5. 敏感字段不会以明文进入审计日志
6. 后台可按时间、人员、资源、结果筛选审计记录
7. 组织管理员无法越权查看其他组织日志
8. 操作失败时也能保留失败审计

---

## 18. 风险与注意事项

### 18.1 不要把审计日志做成另一套敏感数据仓库

如果不脱敏，审计日志本身会变成新的泄密源。

### 18.2 不要把所有行为都审计

日志噪音过大，会导致真正关键动作被淹没。

### 18.3 不要在每个接口里手写不同结构

必须统一格式，否则后续查询和维护成本会很高。

### 18.4 不要与运行日志混用

混用会导致定位模糊，后期难以治理。

---

## 19. 本方案结论

本项目上线前，审计日志是高优先级建设项。  
最合适的落地方式不是继续扩充现有 `logs`，而是新增独立的 `audit_logs` 体系，并通过统一服务端写入工具覆盖高风险后台动作。

第一期目标应当聚焦：

1. 建表
2. 统一写入
3. 脱敏
4. diff
5. 核心后台接口接入
6. 后台查询页

只有先把这套基础设计定下来，后续无论是做权限治理、任务执行智能体、人工审核链路，还是做上线合规，都能有一个稳定的可追溯底座。
