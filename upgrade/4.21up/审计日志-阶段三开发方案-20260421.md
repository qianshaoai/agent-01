## 1. 本阶段目标

完成"智能体 + 工作流 + 工作流步骤 + 系统设置 + 公告"五大模块的审计接入，覆盖总方案 7.3 高风险动作清单中与**配置型资源**相关的动作（6/7/8/9/10）。

阶段三**不做**：

- 不接入前台用户接口（`user-agents` 等）

- 不重构已完成的阶段一、阶段二产出
- 不引入新 helper 文件，仍复用 `lib/audit.ts`

## 2. 接入范围（按总方案 11.2 顺序）

### 2.1 智能体管理（`agent.*`）

| 路由 | 方法 | 动作 |
|------|------|------|
| `app/api/admin/agents/route.ts` | POST | `agent.create` |
| `app/api/admin/agents/[id]/route.ts` | PATCH（含启停切换） | `agent.update` / `agent.enable` / `agent.disable` |
| `app/api/admin/agents/[id]/route.ts` | PATCH（含 API Key / Endpoint / model params 类字段） | `agent.config.update`（含 `agent.apikey.rotate` 语义） |
| `app/api/admin/agents/[id]/route.ts` | DELETE（若存在） | `agent.delete` |

**动作分流规则**（**必须按请求意图判定，不依赖 `after`**）：

- action 在**鉴权通过后、业务执行前**就要确定，保证成功和失败都能稳定归类
- 分流依据是**请求体本身出现了哪些字段**（请求意图），不是"改完比对 before/after"
- 具体判定优先级（自上而下）：
  1. 请求体中出现 API Key / API Endpoint / model params **类字段**（即当前接口入参中对应"API 密钥"或"调用配置"或"模型参数"的字段）→ `agent.config.update`
  2. 请求体中只出现 `enabled`（或等价布尔启停字段）且值为 `true` → `agent.enable`；值为 `false` → `agent.disable`
  3. 其余 PATCH → `agent.update`
- **字段名示例 `apiKey` / `apiEndpoint` / `modelParams` 仅用于说明"哪一类字段"**，不做强绑定。实际实现时以真实接口入参与当前代码结构为准；只要语义上对应 API Key / Endpoint / model params，就归入 `agent.config.update`
- 即使业务执行失败（DB 写入失败、参数冲突），action 仍必须是已确定的那一个，不得因拿不到 `after` 而退化成通用 `agent.update`

**不单列**：
- `agent.permission.assign`：权限变更走阶段二的 `permission.*`，不在智能体模块重复审计

### 2.2 工作流管理（`workflow.*`）

| 路由 | 方法 | 动作 |
|------|------|------|
| `app/api/admin/workflows/route.ts` | POST | `workflow.create` |
| `app/api/admin/workflows/[id]/route.ts` | PATCH | `workflow.update` / `workflow.enable` / `workflow.disable` |
| `app/api/admin/workflows/[id]/route.ts` | DELETE | `workflow.delete` |
| `app/api/admin/workflows/[id]/duplicate/route.ts` | POST | `workflow.duplicate` |

**动作分流规则**（**必须按请求意图判定，不依赖 `after`**）：

- action 在**鉴权通过后、业务执行前**就要确定，失败场景下也能稳定归类
- 判定依据：请求体中出现的字段集合
  1. 请求体中**只**出现 `enabled`（或等价布尔启停字段）且值为 `true` → `workflow.enable`；值为 `false` → `workflow.disable`
  2. 其他字段（`name` / `description` / `visible_to` / `sort_order` / `categoryIds` 等）任一出现 → `workflow.update`
- **工作流本体接口的 PATCH 不包含 steps 变更**；步骤走 §2.3 的独立 action，避免把步骤变更塞进 `workflow.update` 的 diff
- 失败场景（如 name 冲突）仍按上述请求意图写入已确定的 action，不得因失败退化为通用 action

### 2.3 工作流步骤管理（`workflow_step.*`）

| 路由 | 方法 | 动作 |
|------|------|------|
| `app/api/admin/workflows/[id]/steps/route.ts` | POST | `workflow_step.create` |
| `app/api/admin/workflow-steps/[id]/route.ts` | PATCH | `workflow_step.update` |
| `app/api/admin/workflow-steps/[id]/route.ts` | DELETE | `workflow_step.delete` |

**顺序调整**：
- 不单列 `workflow_step.reorder` action
- 若 PATCH 只修改 `step_order`，一律归为 `workflow_step.update`，diff 自然会体现为 `{ step_order: { before, after } }`
- 如有批量排序接口出现，本阶段**不纳入**，留到阶段四评估

- action 在**鉴权通过后、业务执行前**确定，失败场景（例如 `org_admin` 越权尝试禁用他人公告）也能稳定归类

- 失败场景不得因拿不到 `after` 而退化为通用 `notice.update`

## 3. 统一接入模板

延续阶段二 §3 模板。所有动作统一走：

```ts
import { writeAuditLog, actorFromAdmin, stripSensitiveColumns } from "@/lib/audit";

export const PATCH = async (req: NextRequest, { params }) => {
  const admin = await requireAdmin();
  if (admin instanceof Response) return admin;
  const actor = actorFromAdmin(admin);

  const body = await req.json();

  // 1. action 必须在业务执行前确定，按"请求意图"判定（§2 分流规则）
  //    只能依赖：路由语义 + 请求体字段集合；禁止依赖 after
  const action = decideActionFromRequest(body);

  // 2. 读旧值（更新/删除类必做；仅用于 before_data、resource_name 兜底、脱敏等）
  const { data: before } = await db.from("xxx").select("*").eq("id", id).single();

  // 3. 执行业务
  const { data: after, error } = await db.from("xxx").update(patch).eq("id", id).select().single();

  // 4. 审计（失败/成功都写，action 已提前确定）
  if (error) {
    await writeAuditLog({
      actor,
      action,                                             // 失败场景 action 仍然明确
      resource: { type: "xxx", id, name: before?.name, tenantCode: before?.tenant_code },
      result: "failed",
      before: stripSensitiveColumns(before),              // after 缺失不影响归类
      errorMessage: error.message,
      req,
    });
    return dbError(error);
  }

  await writeAuditLog({
    actor,
    action,                                               // 成功场景使用同一 action
    resource: { type: "xxx", id, name: after.name, tenantCode: after.tenant_code },
    result: "success",
    before: stripSensitiveColumns(before),
    after: stripSensitiveColumns(after),                  // after 仅用于补充成功场景快照
    req,
  });
  return NextResponse.json(after);
};
```

**约束总结**：
- `action` 的唯一判定依据是**请求意图**（路由 + 请求体字段集合），不得依赖 `after`
- `after` 的职责仅限于：成功场景下的资源状态补充与快照
- 失败场景下 `after` 可能缺失，但 `action` 必须仍然稳定、可归类

### 3.1 resource_name 取值

| 资源类型 | resource_name 来源 |
|----------|------------------|
| `agent` | `before/after.name` → fallback `agent_code` |
| `workflow` | `before/after.name` |
| `workflow_step` | `before/after.title`（或 `step_order + title`） |
| `settings` | 固定字符串：`"platform_settings"` / `"brand_logo"` / `"contact_qr"` |
| `notice` | 截断的公告内容前 20 字 + `...`，避免把长文本塞 name |

### 3.2 target_tenant_code 取值

- `agent` / `workflow` / `settings` / 公告（全局）：`null`
- 公告（组织级）：`notice.tenant_code`
- `workflow_step`：归属于所属 workflow，取 workflow 的 `target_tenant_code`（即 `null`，因 workflow 本身无 tenant 归属）

## 4. 脱敏与数据收敛要求（本阶段核心）

### 4.1 必须显式剔除的列（调用 `writeAuditLog` 前必剔）

统一走 `stripSensitiveColumns`（阶段一底座已提供，白名单集中维护）：

| 表 | 必剔字段 |
|----|---------|
| `agents` | `api_key_enc` |
| `workflows` | — |
| `workflow_steps` | — |
| `settings`（若有 secret 类） | 暂无；若未来新增需同步扩 `stripSensitiveColumns` 白名单 |
| `notices` | — |

**强约束**：
- 阶段一的 `sanitizeAuditData` 是"兜底"，`stripSensitiveColumns` 是"显式"。两道防线，不可省
- 若在实施中发现 `stripSensitiveColumns` 白名单遗漏，必须扩白名单，**禁止**在路由内手写解构

### 4.2 `model_params` 内嵌敏感字段（智能体特有）

**背景**：`agents.model_params` 是 JSONB 自由字段，可能包含嵌套 `api_secret` / `client_secret` / `refresh_token` 等（尤其清言适配器会在 `model_params.api_secret` 存二级密钥）。

**策略**：
- 依赖阶段一 `sanitizeAuditData` 的**递归脱敏**（§2.2 已规定：关键词 `includes` 匹配 + 归一化键名大小写不敏感）
- `writeAuditLog` 内部会对 `before_data` / `after_data` / `diff_data` 递归过 `sanitizeAuditData`，嵌套的 `api_secret` 会自动替换为 `[REDACTED]`
- **验证动作**：阶段三实施后必须人工验证一次 —— 保存一个清言智能体，查 `audit_logs`，确认 `after_data->>model_params->>api_secret` 为 `[REDACTED]` 而非明文

### 4.3 敏感修改的"已修改"语义

对 `agent.config.update` 这类动作，`stripSensitiveColumns` 剔除 `api_key_enc` 后，`before_data` / `after_data` / `diff_data` 都不会再显示该字段。但审计页需要能看到"API Key 被改过"这个事实。

**解决方式**：在 `writeAuditLog` 调用时显式传入 `extra`。

> **说明**：以下示例中 `apiKeyChanged` / `apiEndpointChanged` / `modelParamsChanged` 和请求体字段 `body.apiKey` / `body.apiEndpoint` / `body.modelParams` 均为**语义占位符**，与 §2.1 的口径保持一致：
> - 实际实现时以真实接口入参与当前代码结构为准
> - 只要语义上属于 API Key / Endpoint / model params 类字段发生变化，即应在此处标记为 `true`
> - `extra.config_changed` 记录的是"哪一类配置是否发生变更"，不是在绑定请求体字段命名

```ts
// 以下变量名与字段路径仅为语义示例
const apiKeyChanged = body.apiKey !== undefined && body.apiKey !== "";
const apiEndpointChanged = body.apiEndpoint !== undefined && body.apiEndpoint !== before?.api_endpoint;
const modelParamsChanged = body.modelParams !== undefined
  && JSON.stringify(body.modelParams) !== JSON.stringify(before?.model_params);

await writeAuditLog({
  ...,
  extra: {
    config_changed: {
      api_key: apiKeyChanged,          // true = 已修改（不记录值）
      api_endpoint: apiEndpointChanged,
      model_params: modelParamsChanged,
    },
  },
});
```

`extra` 只记布尔值"是否修改"，不记任何明文。`config_changed` 内的键名（`api_key` / `api_endpoint` / `model_params`）作为**审计侧的稳定语义契约**保留，独立于后端字段命名。

### 4.4 文件上传类动作（logo / contact_qr）

**规则**：
- **不记录** 文件二进制、dataURL、base64 原文
- **记录** 以下结果信息到 `after_data`：
  - 上传后的 CDN URL 或 Supabase Storage 路径（如 `settings.logo_url`）
  - 文件名（`file.name`）
  - 大小（`file.size`，字节）
  - MIME（`file.type`）
  - 成功/失败
- `before_data` 记录 `settings` 表对应字段的旧 URL
- `diff_data` 自然退化为 `{ logo_url: { before: 旧URL, after: 新URL } }`

### 4.5 只记摘要不记完整快照的场景

以下场景的 `before_data` / `after_data` **不记完整资源快照**，只记关键字段摘要：

| 动作 | 摘要内容 |
|------|---------|
| `workflow.duplicate` | `{ source_workflow_id, source_name, new_workflow_id, new_name, steps_copied: N }` |
| `workflow_step.delete` | `{ id, title, step_order, workflow_id }`（不记 `description` / `button_text` 长文本） |
| `settings.update` | 只记 diff 命中的字段对；`platform_name` / `help_doc_url` 等原样记录 |
| `notice.update`（内容 > 500 字） | 内容字段截断到前 500 字，后接 `...[truncated by audit]`；其他字段正常 |

### 4.6 轻量 diff 在工作流/步骤下的边界

复述阶段一 §2.3 的"整体字段变化"原则，在本阶段的具体落地：

| 字段 | diff 表现 |
|------|----------|
| `workflow.visible_to = "custom"` + `permissions[]` 数组 | 数组按"整体变化"处理：`{ permissions: { before: [...], after: [...] } }`；不做元素级 diff |
| `workflow.categoryIds[]` | 同上，整体变化 |
| `workflow_step.agents`（关联对象） | 若只比 `agent_id` 字段对就够，不展开嵌套 agent 对象 |
| `workflow_steps` 数组整体排序变更 | **不在** `workflow.update` 的 diff 里体现；每个 step 自己走 `workflow_step.update` |

**明确不做**：
- 不做 `permissions` / `categoryIds` 数组的元素级差异（哪个被添加、哪个被移除）
- 不做 workflow → steps 的跨表 diff
- 不做 diff 语义推断

## 5. 失败场景接入要求

以下失败场景**必须**写审计（延续阶段二 §4 原则）：

1. 创建智能体失败（agent_code 冲突 / 参数非法）
2. 删除智能体失败（被工作流引用阻塞，若有此约束）
3. 更新工作流失败（name 重复 / visible_to 非法）
4. 删除工作流失败（被其他引用阻塞）
5. 删除工作流步骤失败（最后一个步骤不能删，若有此约束）
6. 系统设置保存失败（磁盘满 / Storage 上传失败）
7. 公告创建失败（org_admin 越权尝试创建全局公告）
8. 公告 PATCH / DELETE 失败（org_admin 操作非本组织公告触发 403）

**不写**的情况：
- `requireAdmin()` 直接返回 401（未登录），不写审计
- 参数基础校验失败（`!name.trim()` 之类）可选：若能读到 `actor` 就写，否则不写

## 8. 验证方式

### 8.1 单接口功能验证

每个批次完成后，手动触发对应动作，查 `audit_logs`：

```sql
SELECT action, actor_name, resource_type, resource_name, result, created_at
FROM audit_logs
WHERE action LIKE 'agent.%' -- 或 workflow.% / settings.% / notice.%
ORDER BY created_at DESC
LIMIT 20;
```

### 8.2 敏感字段脱敏验证（**本阶段强制**）

对 3.1 批次完成后：

```sql
-- 查任意智能体 config.update / create 记录，确认 API Key / model_params 无明文
SELECT id, action, after_data, diff_data
FROM audit_logs
WHERE action IN ('agent.create', 'agent.update', 'agent.config.update')
ORDER BY created_at DESC
LIMIT 5;
```

验收点：
- `after_data->>'api_key_enc'` 不存在或为 `null`（已被 `stripSensitiveColumns` 剔除）
- `after_data->'model_params'` 任意层级中不存在原始 `api_secret` / `client_secret` 等明文，出现时必为 `[REDACTED]`
- `extra->'config_changed'` 字段体现"是否修改"的布尔值

### 8.3 文件类动作验证

对 3.4 logo 上传：

```sql
SELECT after_data, extra
FROM audit_logs
WHERE action = 'settings.logo.update'
ORDER BY created_at DESC LIMIT 3;
```

验收点：
- `after_data` 只含 URL / 文件名 / 大小 / MIME，**不含** base64 / 二进制
- 字段总字节数在几百字节量级，不是几 MB

### 8.4 步骤 diff 边界验证

对 3.3 批次：修改一个步骤的 `step_order`，查 diff：

```sql
SELECT diff_data FROM audit_logs
WHERE action = 'workflow_step.update'
ORDER BY created_at DESC LIMIT 1;
```

验收点：
- `diff_data` 只有 `{ step_order: { before: N, after: M } }`
- 不出现 `workflow_id` 跨表引用（步骤的 `workflow_id` 字段本身不变就不进 diff）

### 8.5 主业务不被阻断验证

延续阶段二 §8.4：临时把 `audit_logs` 改名，触发写入失败，确认：
- 智能体创建、工作流保存、logo 上传全部正常 200
- 控制台出现 `[AUDIT-FAIL]` 前缀日志，上下文含 `action` / `resource_type` / `resource_id`

## 9. 风险点

| 风险 | 缓解 |
|------|------|
| `model_params` 嵌套密钥脱敏遗漏 | 阶段一 `sanitizeAuditData` 已按"归一化 + 关键词包含"匹配；§8.2 强制人工验证 |
| 文件上传把 base64 塞进审计 | §4.4 明确只记 URL / 元数据；路由层不传 `file` 对象给 `writeAuditLog` |
| 数组字段 diff 噪音大（permissions / categoryIds） | §4.6 明确数组整体变化；不做元素级，避免误导 |
| `agent.config.update` 与 `agent.update` 分流不一致 | §2.1 给出明确判定规则：按请求体中敏感字段是否出现 |
| 步骤 `step_order` 频繁调整刷日志 | 短期可接受；若量大，阶段四评估批量 reorder action |
| 公告长内容塞满 audit 表 | §4.5 公告内容 > 500 字截断 |
| `workflow.duplicate` 把复制出来的新 workflow 全量 steps 塞审计 | §4.5 只记摘要 `{ steps_copied: N }` |
