## 2. 后端：新建 bulk API

`app/api/admin/users/bulk-set-tenant/route.ts`（新文件）：

- `POST /api/admin/users/bulk-set-tenant`
- Body: `{ userIds: string[], tenantCode: string | null }`
  - `tenantCode = null` 或 `'PERSONAL'` → 调到个人空间
  - 其它字符串 → 目标组织码
- 限制：
  - org_admin → 403
  - userIds 必填，非空，每项是字符串
  - 单次最多 200 人（`BULK_LIMIT`）
- 处理：
  - 一次拉所有目标用户的基础信息
  - 循环每人：跳过已删除 / super_admin / 已在目标组织内 → 收集到 failed[]
  - 其余调 RPC `change_user_tenant`，单条失败原因纳入 failed[]
  - 成功的写一条 audit_log（含 `detail.bulk=true` + `fromTenantCode`），`resourceTenantCode` 用目标组织（PERSONAL 则 null），便于 org_admin 在审计页看见迁入自己组织的事件
- 返回：`{ succeeded: [{id, phone}], failed: [{id, phone?, reason}] }`，整体永远 200

## 7. 用户列表角色优先级排序

用户反馈："让系统管理员排序优先级 > 组织管理员"。原 `.order("role")` 按字母序得到
`org_admin < super_admin < system_admin < user`，跟权限层级不符。

### migration_v33.sql

在 `users` 表加一个生成列 `role_priority`：

```sql
role_priority INT GENERATED ALWAYS AS (
  CASE role
    WHEN 'super_admin'  THEN 4
    WHEN 'system_admin' THEN 3
    WHEN 'org_admin'    THEN 2
    ELSE 1
  END
) STORED;
```

加索引 `idx_users_role_priority (role_priority DESC, created_at DESC)`。

### app/api/admin/users/route.ts

`.order("role")` 替换为 `.order("role_priority", { ascending: false })`。

效果：列表按 super → system → org → 普通用户 排，同优先级再按 created_at desc。

## 10. 工作流进度条点击套娃修复

用户反馈：在工作流页面点已完成的步骤会发送一大段文本到当前对话，多次点击形成"套娃"。

### 根因

`app/agents/[id]/page.tsx` 的 `handleStepBarClick(idx)` 把那一步整条对话拼成
`"用户: xxx\n助手: yyy..."` 长字符串（截 2000 字）后 `handleSend({ text: ctx })` —— 当成用户消息直接入库 + 发给 AI。下次进入 N+1 步注入上下文时又把这一坨包进去，雪球。

### 修法

复用现有 `workflowContext` 字段（chat route 早就只拼到 AI prompt 不写 displayContent）：

- `handleStepBarClick` 不再 `handleSend`，改成把 ctx 挂到 `workflowContextRef.current` + 设 `stepReferenceLabel`
- 输入框上方加 chip "📎 参考：{步骤标题}  ✕"
- 用户主动 send 时 chip 一次性消费 + UI 摘掉
- 切会话/切 agent 时同步清掉 chip

### 引用标记入库

为了让历史回看时知道哪条消息带过参考：

- 前端 send body 多带一个 `workflowReferenceLabel`
- 后端 chat route 在 displayContent 前 prepend `[参考：xx]\n` 标记
- 前端 `parseUserContent` 解析这个标记 → 渲染成消息气泡里的小 chip
- 乐观气泡和 DB 重读路径都识别这个 stepReference 字段

修改：app/agents/[id]/page.tsx（handleStepBarClick / parseUserContent / Message type / 渲染 chip / 输入框上方 chip）

### 关键决策

- 超分母 clamp 到 100% + 文字"已含历史 N 次"（tenant 续费/重置后会出现）
- 入口：顶栏额度胶囊点击弹出 popover

### 实施前 SQL 核对

跑了两条 SQL 确认口径与生产实际值集一致：

```sql
SELECT status, count(*) FROM logs GROUP BY status;
-- 结果：success 528 / error 29 / 无 aborted
SELECT action, count(*) FROM logs GROUP BY action;
-- 结果：chat 381 / trial_chat 143 / upload 22 / admin_change_user_tenant 11
```

结论：

- `eq("status", "success")` 正确，无任何"该计未计"
- `eq("action", "chat")` 正确，`trial_chat` / `upload` / `admin_*` 都不算用量
- 发现遗留 bug：`schema.sql` 的 CHECK `(status IN ('success','error'))` 在生产生效，chat route 写 `'aborted'` 那条 insert 全部被 DB 静默拒绝，aborted 调用 0 条入档。**遗留 TODO**：补一个 migration_v34 放宽 status CHECK 加 `'aborted'`，本期不做

## 12. tenants 初始密码字段清理

用户发现"组织初始密码"在编辑组织弹窗里还在要求填，但似乎没用了。

### 根因

`tenants.pwd_hash` 是早期"用户首次登录用组织码作为初始密码自动开户"流程的字段。自助注册上线后：

- `app/api/auth/login/route.ts` 只校验 `users.pwd_hash`，根本不读 `tenants.pwd_hash`
- `app/api/auth/register/route.ts` 也只 hash 用户自己输入的密码

所以这列写进去就再也没人读，是僵尸字段。

### 最小动清理（避免老数据迁移风险）

- `app/admin/tenants/page.tsx`：隐藏"组织初始密码"输入框，校验不再要求 `initialPwd`
- `app/api/admin/tenants/route.ts` POST：不再要求 `initialPwd`；`pwd_hash` 写占位墓碑值 `"deprecated:tenant-pwd-no-longer-used"`（满足 NOT NULL 约束）；移除 bcrypt 导入
- `app/api/admin/tenants/[id]/route.ts` PATCH：删 initialPwd 分支；移除 bcrypt 导入
- DB 列保留（NOT NULL 约束还在），等下次大清理一起 migration 删
