# 会话全局搜索 · 方案

版本：5.12up
状态：设计稿，待评审
来源：用户列出的"功能欠缺"清单，列在"先做（1 周内）"优先级

---

## 一、目标

把当前"只能在单个智能体 sidebar 里搜聊天"扩成**全平台搜**：

- 用户在任何页面都能从顶栏入口搜自己的所有对话
- 按关键词检索 conversation.title + messages.content
- 命中后能跳到对应 agent 页面 + 滚动到命中消息

---

## 二、用户场景（为什么要做）

| 场景 | 现状痛点 |
|---|---|
| "上个月跟某客户讨论的 XX 方案在哪？" 记不清是哪个 agent | sidebar 搜索只在单 agent 内 |
| "我以前问过这个问题，bot 回答过的，再用一次" | 翻历史会话费时 |
| "上次工作流 A 类似项目的产出" | 多 session 之后会话越来越多，靠记忆找不回 |
| 跨会话复用经验 | 工作流多会话功能上线后，本来就鼓励"同 workflow 多次跑"，必然产生搜索需求 |

---

## 三、决策点（请你拍板）

### 决策 1 · 检索范围

- **A**：只搜自己的对话
- **B**：org_admin 可搜本组织所有人的对话（合规风险大，需要事先和客户约定）
- **C**：先 A，后期再考虑 B（不阻塞主功能）

**倾向 C** → 主版本只做 A，B 留到客户明确要的时候

### 决策 2 · 检索算法

- **A**：PostgreSQL 全文索引（`tsvector` + `tsquery`，原生支持中文需配 zhparser 或 jieba 插件）
- **B**：简单 `ILIKE %关键词%`（无须建索引，能跑就行；超过几万条消息会慢）
- **C**：向量检索（OpenAI / 本地 embedding 模型 + pgvector），支持语义搜索

| | A | B | C |
|---|---|---|---|
| 实施难度 | 中（Supabase 不一定支持 zhparser） | 极低 | 高 |
| 中文体验 | 好 | 一般（必须连续字符匹配） | 最好 |
| 性能（10w 消息） | 毫秒级 | 秒级 | 毫秒级 |
| 月成本增加 | 0 | 0 | embedding API 费用 |
| 维护成本 | 低 | 极低 | 中（要管 embedding 时效） |

**倾向 B**（极简版） → 用 `ILIKE` 起步，先看数据量。这个项目当前用户量 + 配额限制下，单用户消息数应该几千以内，ILIKE 完全够用。如果未来确实慢了，迁移到 A（PG 全文索引）改 1 个 query 就行，迁移成本低。C 留给"语义检索"作为高级功能。

### 决策 3 · 搜索什么内容

- **A**：只搜消息正文
- **B**：消息正文 + 会话标题
- **C**：消息正文 + 会话标题 + 智能体名 + 工作流名 + session 名

**倾向 B**。C 信息维度多了但容易召回噪音（搜"客户"返回一堆名字含"客户"的 agent 列表）。B 已经能覆盖 95% 用户需求。

### 决策 4 · 搜索入口位置

- **A**：顶栏右侧加搜索图标，点击展开搜索框（类似 GitHub 的 `/`）
- **B**：首页顶部固定一个大搜索条
- **C**：单独一个 `/search` 路由
- **D**：A + 快捷键 `Ctrl+K` 弹出对话框（Linear / Notion 风格）

**倾向 D**：A 做底线（永远可达），加 Ctrl+K 提升熟练用户效率。B 太占首页空间。C 太重不直接。

### 决策 5 · 结果展示

每一条命中显示：

```
┌──────────────────────────────────────────────┐
│ [Bot/User]  内容萃取师 · 工作流：企业培训 · 2026-05-10 14:23   │
│ "...对了，跟小米发布会那次客户的对话我记..." 高亮"小米发布会" │
└──────────────────────────────────────────────┘
```

点击 → 跳到对应 agent 页面 + URL 带 `?conv=<convId>&highlight=<msgId>` 滚到该条消息高亮 2 秒。

**附加问题**：
- 一条会话有多条命中怎么显示？
  - **A**：每条命中独立一行
  - **B**：按会话聚合（"内容萃取师 · 3 条命中"，展开看每条）
  - **倾向 B**（信息密度高，避免一搜搜出 50 条相同会话刷屏）

- 默认排序？
  - **A**：按相关度（ILIKE 没有相关度，会退化成时间序）
  - **B**：按时间倒序（最新优先）
  - **倾向 B**（用户多半是找"最近聊过的"）

### 决策 6 · 性能 / 上限

- 每页返回多少条命中：**默认 20，可分页加载更多**
- 单关键词最短长度：**至少 2 个字符**（防止单字符疯狂返回所有消息）
- 是否限速：**短期不需要，看后期再加**

## 四、技术方案（基于上面倾向选项）

### 4.1 后端 API

新建：`app/api/search/conversations/route.ts`

```
GET /api/search/conversations?q=关键词&page=1&pageSize=20

权限：需登录（user JWT），只返回 user_id = current user 的数据

实现：
  1) 同时跑两个查询（限当前用户）
     - conversations 表：title ILIKE %q%
     - messages 表：content ILIKE %q%
  2) messages 表查询 JOIN conversations 拿到 conv.title / agent_code / session_id
     再 JOIN agents 拿 agent.name
  3) 按 conversation_id 聚合（按会话归集多条命中）
  4) 排序：每个会话取最新命中消息的 created_at 倒序

Response:
{
  data: [
    {
      conversationId: "uuid",
      conversationTitle: "小米发布会准备",
      agentCode: "content-extractor",
      agentName: "内容萃取师",
      workflowName: "企业培训课程开发" | null,
      sessionId: "uuid" | null,
      latestHitAt: "2026-05-10T14:23:00Z",
      hits: [
        {
          messageId: "uuid",
          role: "user" | "assistant",
          snippet: "...小米发布会那次客户的对话...",
          createdAt: "2026-05-10T14:23:00Z",
        },
        // 最多 3 条 / 会话；点击会话再展开看全部
      ],
      totalHitsInConversation: 7,
    },
  ],
  pagination: { page, pageSize, total },
}
```

**snippet 生成规则**：截取命中关键词前后各 30 字符，关键词位置高亮（前端做高亮 marking）。

### 4.2 前端

#### 4.2.1 顶栏入口（决策 4=D）

`components/layout/admin-layout.tsx` 和首页 `app/page.tsx` 顶栏右侧加：

```tsx
<button onClick={openSearch} className="...">
  <Search size={16} />
  <span className="hidden md:inline">搜索  Ctrl+K</span>
</button>
```

全局快捷键 `Ctrl+K`（Mac 上 `Cmd+K`）由 root layout 监听。

#### 4.2.2 搜索弹窗组件

新建 `components/search/global-search-modal.tsx`：

```
┌──────────────────────────────────────────┐
│ 🔍 [搜索手机号 / 智能体 / 对话内容…    ]  │
├──────────────────────────────────────────┤
│ 内容萃取师 · 企业培训 · 5/10 14:23       │
│ ├ "...对了，跟小米发布会那次客户的对话..." │
│ ├ "...小米发布会上的演讲节奏..."          │
│ └ "...小米发布会要求精简..."              │
│   还有 4 条命中 →                        │
│                                          │
│ 课程架构师 · 5/8 09:12                   │
│ └ "...参考小米发布会的流程..."            │
│                                          │
│ ... (共 23 条会话)                       │
│                                          │
│ [↑↓ 选择] [Enter 打开] [Esc 关闭]        │
└──────────────────────────────────────────┘
```

- 输入框打字 → debounce 250ms → 调 GET /api/search/conversations
- 按会话聚合分块
- 键盘 ↑↓ 切换选中行，Enter 打开
- 打开方式：`router.push(/agents/{agentCode}?conv={convId}&highlight={msgId})`
- 工作流会话内的命中 URL 加 `&session={sessionId}`

#### 4.2.3 agent 页面接住 `?highlight` 参数

`app/agents/[id]/page.tsx` 接收 `?highlight=msgId`：

- 加载消息后 scrollIntoView 到该消息
- 给该消息加 2 秒淡黄色高亮动画然后淡出
- 清掉 URL 上的 highlight 参数避免分享时干扰

### 4.3 数据库索引（可选优化）

如果 ILIKE 性能不达标，再加：

```sql
-- 给 messages.content 加 GIN trigram 索引（PG 标准做法）
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_messages_content_trgm ON messages USING gin (content gin_trgm_ops);
CREATE INDEX idx_conversations_title_trgm ON conversations USING gin (title gin_trgm_ops);
```

**起步不上**，先看真实查询时间。如果用户搜索经常 > 1 秒再补这两个索引。

### 4.4 audit / 隐私

- 搜索行为不写 audit_log（搜索本身不是写操作，量太大也没意义）
- 但要确保：API 严格按 `user_id = current_user.id` 过滤，绝对不能漏给别人

## 六、风险

| 风险 | 缓解 |
|---|---|
| ILIKE 在 10w+ 消息时慢 | 上文 4.3 的 GIN 索引一键解决，提前在 schema 注释里 TODO 标好 |
| 搜索结果泄漏给别人 | API 一定带 `user_id = current_user.id`；写完手动 review 一遍权限边界 |
| 关键词高亮在 markdown 里渲染错位 | 高亮只在 snippet 文本里做（非 markdown），打开后在 agent 页只滚到位置不再额外高亮 |
| `?highlight` 参数被分享出去 | 第一次 useEffect 执行后立刻 `router.replace` 删掉它 |
| 同一会话命中 100 条把弹窗刷屏 | 单会话最多展示 3 条 + "还有 N 条" 链接 |

---

## 七、和现有功能的协同

- **工作流多会话（5.9up）**：搜索结果若命中工作流 session 内的对话，URL 带 `session_id`，跳过去仍然保持 session 隔离
- **审计日志（5.11up）**：搜索行为不入 audit，但跳转访问历史会话本身有访问记录
- **历史工作流（5.9up）**：搜索能命中已归档的会话；点击进入会自动用 `?readonly=1`（如果该会话已 abandoned/completed）
