# 进度条点击套娃修复 · 方案

版本：5.12up
状态：设计稿，待评审
来源：用户反馈 — 工作流页点已完成步骤会发送一大坨文本到当前对话，多次点击后形成"套娃"

---

## 一、问题现象

用户进入工作流第 N 步（智能体页面），顶部进度条上可以点击之前已完成的步骤。

**当前行为**：点击 step K（K < N）→ 拉取 step K 的整条对话 → 拼成 `"用户: ...\n助手: ..."` 长字符串（截到 2000 字）→ **作为用户消息**发送给当前 step N 的智能体。

**结果**：
- 用户气泡里出现一大坨他从没说过的话（实际是别的步骤的历史对话）
- 这条"用户消息"入库
- 下次进入 N+1 步时，跨步骤上下文注入会拉 N 步的最新对话 → 里面就包含了刚才那一大坨 → 又被原样塞进 N+1 步的第一条 user message → 套娃

每多点一次进度条 + 多走一步，体积膨胀。

## 二、根因

文件 `app/agents/[id]/page.tsx`，函数 `handleStepBarClick(idx)`：

```ts
async function handleStepBarClick(idx: number) {
  ...
  const lines: string[] = [];
  for (const m of msgs) {
    if (m.role !== "user" && m.role !== "assistant") continue;
    const cleaned = m.content.replace(/\n\n\[附件内容\][\s\S]*$/, "").trim();
    if (cleaned) lines.push(`${m.role === "user" ? "用户" : "助手"}：${cleaned}`);
  }
  let ctx = lines.join("\n\n");
  if (ctx.length > 2000) ctx = ctx.slice(0, 2000) + "…（内容过长已截断）";

  handleSend({ text: ctx });   // ← 这里直接把 ctx 当成用户输入发出去
}
```

走 `handleSend` 之后，`ctx` 同时成为：
1. 入库的 `messages.content`（用户气泡显示）
2. 发给 AI 的 user 消息内容

## 三、解决方案

**核心思路**：复用已有的 `workflowContext` 字段（chat route 早就支持），让 ctx 只拼到给 AI 的 prompt 里，**不进** displayContent（不入库不显示）。

### 3.1 后端路径（已有，不动）

`app/api/admin/agents/[id]/chat/route.ts` 当前逻辑：

```ts
const wfCtx = workflowContext?.trim() || null;
const aiUserContent = wfCtx
  ? `【上一步工作记录】\n${wfCtx}\n\n【当前问题】\n${userContent}`
  : userContent;

// 入库的是 displayContent，不含 wfCtx
await db.from("messages").insert({
  conversation_id: convId,
  role: "user",
  content: displayContent,  // 不含 wfCtx
});
```

满足条件：传 workflowContext 字段 → AI 能看到 → DB 不存。我们只要把 ctx 从"用户消息"挪到这里就行。

### 3.2 前端改动

`handleStepBarClick(idx)` 重写：

```ts
async function handleStepBarClick(idx: number) {
  if (idx === resolvedStepIdx || idx >= resolvedStepIdx) return;
  if (loading || streaming) return;
  const clickedStep = wfSteps[idx];
  if (!clickedStep?.agents?.agent_code) return;

  // 5.12up · 不再把 ctx 作为用户消息发出，改成挂到 workflowContextRef 上，
  // 等用户下一次主动 send 时静默拼入 AI prompt（不入库不显示）
  try {
    const convRes = await fetch(`/api/conversations?agentCode=${...}&pageSize=1`);
    ...
    const ctx = buildCtxString(msgs);
    workflowContextRef.current = ctx;
    workflowContextSentRef.current = false;  // 重置，让下次 handleSend 一次性消费
    setStepReferenceLabel(clickedStep.title);  // 显示 chip
  } catch {}
}
```

### 3.3 UI · 输入框上方加 chip

输入框上方加一行（仅在 `stepReferenceLabel` 非空时显示）：

```
┌────────────────────────────────────────────────┐
│ 📎 已携带「明确课程开发需求」对话作为参考  [✕]   │
└────────────────────────────────────────────────┘
[输入框........................................][发送]
```

点 `[✕]` 移除：清空 `workflowContextRef.current` 和 `stepReferenceLabel`。

### 3.4 一次性消费

`handleSend` 现有逻辑已经实现"一次性"：

```ts
const wfCtxToSend = workflowContextSentRef.current ? undefined : (workflowContextRef.current ?? undefined);
if (wfCtxToSend) workflowContextSentRef.current = true;
```

发送成功后清掉 chip（`setStepReferenceLabel(null)`），引用消失。

### 3.5 子决策（已拍）

- **多次点击进度条**：后一次**覆盖**前一次（chip 显示最新一步的标题）
- 理由：累加可能又把"长"问题搬过来；覆盖更可控

## 六、影响范围

仅前端，**不动后端**、**不动数据库**。

- 老数据无影响（已经入库的"长用户消息"还在那里，不主动迁移）
- 工作流跨步骤上下文的"首次进入注入"路径（5.8up 的 outline 机制）不受影响，仍按原方式工作
- session 隔离逻辑、历史回看（readonly 模式）等不变

## 七、风险 & 缓解

| 风险 | 缓解 |
|---|---|
| 用户点了进度条以为发送了消息，但其实没发 | chip 文案明确：「已携带 ... **作为参考**」+ 提示「请继续输入您的问题」 |
| chip 在用户切换会话后未清除 | 在 `setActiveConvId` 时同步清掉 `workflowContextRef` + `stepReferenceLabel` |
| workflowContext 2000 字限制仍然存在 | 保持现状，截断后追加 `…（内容过长已截断）`；如果用户反馈需要更长再放宽 |
| 多次点击进度条覆盖 ctx，用户可能没注意之前的 ctx 没了 | chip 显示"已携带「最近一次点击的步骤」"，覆盖时显式更新文案 |

## 八、回归测试用例

1. 工作流走到 step 3 → 点进度条 step 2 → 显示 chip → 输入"分析一下" → 发送
   - 用户气泡只有"分析一下"
   - bot 回复基于 step 2 上下文（说明 ctx 进了 AI prompt）
2. 同上场景，发完之后再发"再总结一下"
   - chip 消失
   - bot 不再有 step 2 ctx（一次性消费已结束）
3. 点 step 2 → 又点 step 1 → chip 文案更新为"已携带「step 1 标题」"
4. 点 chip 的 ✕ → chip 消失，ctx 清空
5. 切换会话（点击 sidebar 别的对话）→ chip 自动消失
6. 之前已有"套娃用户消息"的旧会话打开 → 历史显示不变（不做老数据清洗）
