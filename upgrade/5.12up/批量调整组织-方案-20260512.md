# 批量调整组织 · 方案

版本：5.12up
状态：设计稿，待评审
来源：用户反馈"管理员页面已经有批量启用/禁用，希望同样支持批量调整组织"

---

## 一、现状

后台 `/admin/users` 页面已有批量操作栏（已选 N 人 + 批量启用 + 批量禁用 + 取消选择）。
单人调整组织的能力已经存在：

- API：`PATCH /api/admin/users/[id]` 带 `{ action: "set-tenant", tenantCode }`
- 后端调用 `change_user_tenant` RPC（5.6up）原子地完成 tenant_code 更新、清空 dept_id/team_id、写 force_relogin_at（强制重登）
- 前端入口：用户详情抽屉里的"修改所属组织"按钮

但没有批量入口，要把同一批人都调到目标组织得一个一个点，效率低。

---

## 二、目标

在批量操作栏增加「批量调整组织」按钮，点击弹出目标组织选择器，确认后把选中的全部用户一次性移到目标组织。

---

## 三、决策点（请拍板）

### 决策 1 — 是否允许批量把人调到"个人用户"（清空 tenant_code）

- **A**：允许，目标组织选项里加一项「个人空间（无组织归属）」
- **B**：只允许调到具体组织（不允许变回个人）

考量：实际运维里"把人移成个人"很少用，但 5.6up 的单人功能已支持。批量功能保持一致更可预测。倾向 A。

### 决策 2 — 选中的用户里如果有不同来源（已分布在多个组织 / 个人混组织）

- **A**：直接全调走，不管原属
- **B**：阻止操作，弹错"必须选同组织的人"
- **C**：允许，但弹窗里清晰列出"将从 X 个不同组织迁出 N 人到 Y 组织"

倾向 C — 透明给管理员看，但允许做。

### 决策 3 — org_admin 能用这功能吗

5.6up 的单人 set-tenant 已 403 拒绝 org_admin。批量保持一致 → org_admin 看不到批量"调整组织"按钮。

倾向 **不允许 org_admin**。

### 决策 4 — 后端实现路径

- **A**：前端循环调单人 API（N 次 HTTP）→ 实现最快，进度可视化
- **B**：新建一个 bulk RPC `bulk_change_user_tenant(p_user_ids uuid[], p_new_tenant_code text)`，一次事务做完 → 性能最好，全失败 / 全成功
- **C**：新建一个 bulk API `/api/admin/users/bulk-set-tenant`，后端循环调单人 RPC → 折中

考量：用户量上百以后 A 慢；B 强一致性但需要写 migration_v33；C 性能凑合，事务粒度按单人，部分失败时容易处理。

倾向 **C**：先做 bulk API 兼顾性能和实现速度，bulk RPC 留作未来优化。

### 决策 5 — 部分失败如何展示

如果选了 8 人，2 人失败（比如已删除 / 是 super_admin 不可改 / 网络抖动）：

- **A**：弹窗显示"成功 N、失败 M"+ 失败的具体原因列表
- **B**：弹窗只显示"成功 N、失败 M"，详情看审计记录
- **C**：失败的把 ID 高亮在表格里，让管理员一眼能看到哪几人没改成

倾向 **A** — 弹窗里列具体 phone / nickname + 失败原因（简短），管理员能立刻分辨。

## 四、方案细节（基于上面倾向选项）

### 4.1 UI · 批量操作栏

`app/admin/users/page.tsx`：

```

                                  ↑ 蓝色按钮

                                  含已删除用户时禁用 + tooltip
```

### 4.2 后端 · 新建 bulk API

`app/api/admin/users/bulk-set-tenant/route.ts`（新）：

```ts
POST /api/admin/users/bulk-set-tenant
Body: {
  userIds: string[],          // 选中的用户 UUID 数组
  tenantCode: string | null,  // 目标组织码；null = 调到个人空间
}

Response: {
  succeeded: Array<{ id: string; phone: string }>,
  failed: Array<{ id: string; phone?: string; reason: string }>,
}
```

后端流程：

1. requireAdmin + 权限检查：org_admin → 403
2. 校验 userIds 非空、长度 ≤ 200（防误操作）
3. 一条 SQL 拉全部目标用户的 phone / nickname / status / role / tenant_code
4. 循环每个用户：
   - 已删除（status='deleted'）→ 失败，原因"用户已删除"
   - super_admin（role='super_admin'）→ 失败，原因"不能移动超级管理员"
   - 目标 tenant 与当前相同 → 失败，原因"已在该组织内"（避免无意义 force_relogin）
   - 否则调 `change_user_tenant` RPC；失败时把 RPC 错误信息纳入 failed
5. 每个成功的用户单独写一条 audit log（resourceType=user, action=update, detail.bulk=true）
6. 返回 succeeded / failed 列表

### 4.3 个人空间分支

`change_user_tenant` RPC 当前签名 `(p_user_id uuid, p_new_tenant_code text)`，传 `''` 或 `null` 时行为待确认。

- 如果 RPC 已支持 null，bulk API 直接透传
- 如果不支持，要 migration_v33 修 RPC 加一段"如果 p_new_tenant_code IS NULL THEN tenant_code = NULL, ..."

实施时先看 5.6up 的 RPC 怎么写，再决定要不要补 migration。

### 4.4 前端 · 调用 + 结果展示

```ts
async function bulkSetTenant(userIds: string[], tenantCode: string | null) {
  const res = await fetch("/api/admin/users/bulk-set-tenant", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ userIds, tenantCode }),
  });
  const data = await res.json();
  // 弹窗展示：
  //  - 成功 N 人（绿色）
  //  - 失败 M 人（红色，每条带 phone + reason）
  //  - 按钮：[再来一次]（清掉失败的，重试） + [关闭]
  showResultModal(data);
  // 刷新列表
  await load();
  // 清空选中（成功的已经被移走了）
  setSelectedIds(new Set());
}
```

### 4.5 边界 / 风险

| 风险 | 缓解 |
|---|---|
| 一次选太多人（比如 200+）卡服务 | 后端硬限制 200，UI 也提示「单次最多移动 200 人」 |
| 强制重登影响在线用户 | 弹窗里明确写"会强制重登"，并提示"建议在低峰期操作" |
| 部分失败时管理员不知道哪些没成 | A 方案弹窗列具体失败用户 + 原因 |
| 重复点确认按钮发了两次 | 弹窗 confirm button `disabled={loading}`；后端幂等（同 tenant 直接 fail） |
| audit log 一次操作生成几百条 | 单条写入累积，对 audit_logs 表压力可控；若担忧后续可以加 detail.bulk_group_id 把同一次批量挂同一 ID 便于查询 |
