## 一、在整体里的位置

知识库功能拆成两条并行线：

- **方案 A · 入库侧** —— 把知识库做出来、能装资料（建库、传文档、切块向量化、管理后台）。
- **方案 B（本文件）· 使用侧** —— 让智能体用上知识库：搭建器绑定、对话检索注入、
  chat 权限校验。

B 做完的交付物：**搭建器里能给智能体勾选知识库；用户与该智能体对话时，系统自动检索
知识库相关片段注入上下文**。B 还顺带补掉一个既有安全缺口（chat 不校验可见性）。

---

## 二、本方案范围

**做**（= 母方案 PR-4 + PR-5）：

- 搭建器集成：搭建器加「知识库」分区（绑定知识库多选）；发布 / 重发布时按
  `builder_config.knowledge_base_ids` **全量同步**到 `agent_knowledge_bases` 关联表；
  草稿里指向已删知识库的脏 id 在 `load()` 时过滤。
- **chat 前置可见性校验**（既有安全缺口）：chat route 现在不校验当前用户对该 agent
  是否可见 —— 挂知识库后会变成"猜到 code 就能问出库里资料"。先补校验、再接检索。
- 对话检索接入：`lib/kb/retrieve.ts` 调 `match_kb_chunks` RPC 取 top-K 片段，注入
  上下文；检索失败降级（当作无知识库正常回答，不阻断对话）。

**不做**（属方案 A）：向量基建迁移、embedding 适配器、文档摄取、知识库管理后台。

B **依赖方案 A 的基础**，但代码层面可与 A 同时开工 —— 关键约束如下。

### 3.3 文件与合并

- ⚠️ **PR-B2 动 `chat` route，是全功能最高风险点** —— 建议 B 的 PR 里 PR-B2 最后合，

- 两边都合完后做一次**联合联调**（建库→传文档→绑智能体→对话检索），约 1 天，双方一起。

## 四、设计要点

完整设计见母方案 §4.3 / §4.5，这里只列 B 半要点：

- **搭建器集成**：5 分区 → 6 分区，新增「知识库」分区（多选绑定）；绑定写
  `builder_config.knowledge_base_ids`；发布时全量同步 `agent_knowledge_bases`
  （与「可见范围发布翻译」同模式）。
- **chat 新流程**：① 取 agent、判 `enabled` 后**先校验用户对该 agent 有访问权**
  （按 `resource_permissions`），无权 → 403；② 查 `agent_knowledge_bases` 是否绑库；
  ③ 有 → 用户问题向量化 → 调 `match_kb_chunks` RPC 取 top-K；④ 片段拼成 context
  注入；⑤ 其余流程不变。检索失败 → 降级，不阻断对话。
- ⚠️ **个人用户口径**：chat 权限校验"与前台列表同一套"，但前台列表对个人用户绕过
  权限表。个人用户对知识库型 agent 的可见性**随「可见范围方案」D4 最终结论**；若库含
  租户敏感资料，建议把个人用户也纳入权限约束（详见母方案 §4.3）。
