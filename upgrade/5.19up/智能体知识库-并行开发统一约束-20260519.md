## 一、结论

知识库 A / B **可以并行开发**，但不是无前置的自由并行。

- **A 是基础设施线**：建表、RPC、embedding、摄取、知识库后台。
- **B 是使用接入线**：搭建器绑定、发布同步、chat 权限校验、对话检索注入。
- **PR-A1 是全局第一前置**：迁移、表结构、RPC 签名、共享类型必须先定、先合、先在 dev 库执行。
- **B 可以提前写代码**：按冻结契约写，缺 A 实现时用桩；但本地端到端联调必须等 A1/A2 完成。

并行收益成立：A 约 10 人日，B 约 5.5 人日，若串行约 15.5 人日；按本约束并行后，关键路径约为 A 的 10 人日 + 联合联调 1 人日。

## 三、冻结契约

以下内容一旦 PR-A1 定稿，A / B 均不得擅自修改。确需变更时，必须先更新母方案和本文档，再同步另一条线。

### 3.1 数据表

PR-A1 负责创建，B 只使用：

```text
knowledge_bases
kb_documents
kb_chunks
agent_knowledge_bases
```

关键约束：
- 本期全站统一一个 embedding 模型和一个向量维度。
- `kb_chunks.embedding` 使用固定 `vector(N)`，N 由 D1 决定。
- `agent_knowledge_bases` 由 A 的迁移创建，但业务写入由 B 的发布流程负责。

### 3.2 检索 RPC

PR-A1 固定签名：

```sql
match_kb_chunks(
  p_kb_ids uuid[],
  p_query vector,
  p_top_k int,
  p_threshold float
)
```

返回字段至少包含：

```text
content
similarity 或 distance
document_id
```

B 通过 `lib/kb/retrieve.ts` 调用该 RPC，不在业务代码里直接拼 `ORDER BY embedding <=> ...`。

### 3.3 共享类型

PR-A1 创建 `lib/kb/types.ts`，至少包含：

```ts
export type KbSearchResult = {
  content: string;
  document_id: string;
  similarity?: number;
  distance?: number;
};
```

B 可以 import 类型，但不得修改该文件。若 B 需要新增字段，先提给 A 修改契约。

### 3.4 Embedding 适配器

PR-A2 创建 `lib/kb/embed.ts`，冻结导出函数：

```ts
export async function embedTexts(texts: string[]): Promise<number[][]>;
export async function embedQuery(text: string): Promise<number[]>;
```

B 在 A2 合并前可以用本地桩开发，但最终代码只能依赖上述导出。

## 五、分支与合并顺序

推荐分支：

```text
feature/kb-a
feature/kb-b
```

合并顺序：

1. **PR-A1 必须最先合并**：迁移、RPC、共享类型、切块工具。
2. PR-A1 合并后，必须在 dev 库执行迁移并确认 `match_kb_chunks` 可调用。
3. PR-A2 / PR-A3 / PR-B1 可并行合并，前提是没有文件冲突。
4. **PR-B2 最后合并**：它改 `chat` route，是最高风险点。
5. A/B 都合并后做联合联调，不允许跳过。

## 七、关键风险约束

### 7.1 Chat 路由

`app/api/agents/[id]/chat/route.ts` 是最高风险文件。

PR-B2 必须做到：

- 先补 agent 可见性校验，再做知识库检索。
- 无权用户访问任意 agent code 返回 403。
- 检索失败时降级为无知识库回答，不阻断对话。
- 非知识库智能体路径保持原行为。
- 扣子 / Dify / 元器 / 清言等外部平台智能体不接知识库检索。

### 7.2 个人用户权限

个人用户访问知识库型 agent 的口径必须与「搭建器发布可见范围」方案 D4 保持一致。

若 D4 本期维持现状：
- B 不改 `/api/agents` 个人用户列表逻辑。
- 文档和回归说明必须注明：个人用户仍可能看到全部 enabled agent。

若 D4 改为纳入权限表：
- 必须另开独立任务做存量数据补权限和 `/api/agents` 逻辑调整。
- 不得混入知识库 A/B 任一 PR。

### 7.3 摄取上限

A 必须实现 D9：

- 单文档最大提取文本约 20 万字符。
- 单文档最大 500 chunk。
- embedding 批量约 64 块 / 请求。
- 超限后文档 `status=failed`，提示拆分上传。
- 不允许静默截断后标记 `done`。

## 八、自测与联调

### 8.1 A 自测

- 建知识库成功。
- 上传支持格式文档成功。
- 超 D9 上限文档失败且提示拆分。
- 文档状态 `pending → indexing → done / failed` 正确。
- 删除文档清理 chunks。
- 重建索引可重复执行。
- 引用反查可查 `agent_knowledge_bases`。

### 8.2 B 自测

- 搭建器可选择多个知识库。
- 草稿中已删除知识库 id 会被过滤。
- 发布 / 重发布全量同步 `agent_knowledge_bases`。
- 无权用户 chat 返回 403。
- 无知识库 agent 不走检索。
- 检索失败降级，不影响正常回答。
- 非知识库智能体、扣子 / Dify 智能体路径不受影响。

### 8.3 联合联调

必须覆盖完整链路：

1. A 建知识库。
2. A 上传文档并索引完成。
3. B 在搭建器绑定该知识库。
4. B 发布智能体并同步 `agent_knowledge_bases`。
5. 用户对话命中检索，回答包含注入资料影响。
6. 无权限用户不能通过 agent code 问出知识库内容。

## 九、禁止事项

- A 不得修改搭建器、发布接口、chat route。
- B 不得修改迁移、embedding、摄取、知识库后台。
- 不得在未更新契约的情况下改表结构 / RPC 签名 / `embed.ts` 导出。
- 不得提交真实 `.env.local` 或 embedding key。
- 不得把个人用户权限改造混入知识库 A/B。
- 不得把异步摄取、引用来源展示、命中测试 UI 塞入本期。

---

## 十、结论

允许并行，但必须按以下节奏：

1. 先拍 D1 / D1-2 / D1-3 / D2 / D9。
2. A1 先做、先合、先跑迁移。
3. B 按冻结契约和桩并行开发。
4. B2 最后合。
5. 最后做 A+B 联合联调。

只要上述约束执行，A/B 并行风险可控，且能明显缩短整体交付周期。
