## 一、背景与问题

智能体搭建器「发布设置 → 可见范围」下拉现有 4 档（仅本人 / 本组织 / 全平台 /
自定义范围）。发布接口 `app/api/admin/agent-drafts/[id]/publish/route.ts`
**完全不读 `visibility_config`、也不写 `resource_permissions`** —— 这个下拉是摆设。

### ⚠ v2 修正：可见性当前到底怎么判定（小B P1）

原方案写「不写权限行 = 前台谁都看不见」**是错的**。当前 `app/api/agents/route.ts`：

- **组织用户**：走 `resource_permissions` 过滤 —— 没有命中本人组织/部门/小组/用户的
  权限行就看不到。
- **个人用户**（`user.isPersonal`）：**直接取全部 `enabled` 的 agent，不查
  `resource_permissions`**。

所以「不写权限行」只对**组织用户**=不可见；**个人用户照样能看到所有已启用 agent**。
这直接影响下面所有选项的语义 —— 见决策点 **D4**。

---

## 二、目标

搭建器「可见范围」的选择 → 发布时翻译成该 agent 的 `resource_permissions` 行，
做到发布即对选定范围（**组织用户侧**）可见。模型对齐智能体管理「权限设置」那一套
（`resource_permissions` 表，与工作流同一套，`scope_type` 共 7 种）。

个人用户侧是否同步纳入权限约束 —— 见 D4，建议拆为独立项。

## 三、设计

### 3.1 可见范围选项（精简为 3 档）

| 选项（UI 文案） | 内部 `visible_to` | 发布时写入 `resource_permissions` |
| --- | --- | --- |
| **全平台可见** | `all` | 1 条 `scope_type=all` |
| **本组织可见** / **指定组织可见** | `org` | 按发布者角色，见下 |
| **暂不公开**（仅搭建器内自测） | `owner_only` | 不写任何行 |

**第二档按发布者角色分（用户拍板 D1）**：

- **组织管理员（org_admin）** → 该档显示「**本组织可见**」，**无多选**；固定写 1 条
  `scope_type=org, scope_id=<该 org_admin 所属组织码>`。org_admin 不能把 agent 发到
  别的组织（与工作流 `validateOrgAdminPermissions` 同口径，后端再校一道）。
- **超级 / 系统管理员** → 该档显示「**指定组织可见**」，下方**组织多选**；每个选中
  组织写 1 条 `scope_type=org`。

其它：

- **「仅本人可见」改名「暂不公开」**（小B P2）：实际语义是"不写权限行"，不是真"仅
  创建人可见"（草稿创建人是 admin、未必对应前台 user）—— 用诚实文案「暂不公开」。
- **去掉「自定义范围」**（占位无功能）。
- 内部枚举值 `owner_only` 保留不变（仅改 UI 文案），避免动存量草稿数据。

### 3.2 `visibility_config` 结构

```ts
type VisibilityConfig = {
  visible_to: "all" | "org" | "owner_only";   // 去掉 "custom"
  scope: string[];                            // visible_to="org" 时为选中的组织码数组
};
```

旧草稿兼容：`load()` 读到 `visible_to="custom"`（旧值）→ 降级为 `"owner_only"`
（安全侧，不误开放）。`agent_drafts.visibility_config` 是 jsonb 列，**无需 DB 迁移**。

### 3.3 发布翻译（`publish/route.ts`）+ 事务与校验（小B P3 / P4）

**新增 RPC `set_agent_permissions(p_agent_id uuid, p_perms jsonb)`**（`supabase/rpc.sql`）：
plpgsql 函数体即单事务 —— `DELETE` 该 agent 的全部 `resource_permissions` 行，再按
`p_perms` `INSERT`，要么全成、要么全不动。**权限替换原子化。**

publish 流程（**顺序关键** —— 小B 复评：校验先于一切 DB 写）：

1. **校验 `visibility_config` / `scope`** —— 放在 agent upsert **之前**（与现有
   name / provider 校验同段）：
   - `visible_to="org"`：`scope` 去重、非空，每个组织码必须在 `tenants` 表存在。
   - **发布者是 org_admin**：`scope` 必须 = 其所属组织（前端本就固定本组织，后端
     再校一道防手改 —— 与工作流 `validateOrgAdminPermissions` 同口径）。
   - 任一不过 → 返回 422，**此时尚无任何 DB 写**，不留中间态。
   > 实现注意（小B 复评）：provider / tenant 的**查询是读操作**，提前到校验段没问题；
   > 真正必须放在校验**之后**的是写操作 —— agent upsert、draft status 更新、权限写入。
2. （现有）upsert agent。
3. 按 `visible_to` 组装 `p_perms`（`all`→`[{scope_type:all}]`；`org`→每组织码一条；
   `owner_only`→`[]`），调 `set_agent_permissions` RPC。**检查 error**：失败 → 返回
   500、**且不把 draft 标 `published`**（可重试），不静默成功。
4. 全部成功后再更新 draft 状态、写审计。

> **残余态登记**：agent upsert 与 perm RPC 不在同一事务。若 perm RPC 失败，会留下
> 「agent 已 upsert + enabled，但权限未设」的中间态。因 publish **幂等**（按
> `published_from_draft_id` 重发布即重做），重试即修复；且权限未设的 agent 对组织
> 用户本就不可见。可接受性请小B复评确认（或上一步把 agent 先 `enabled=false`、
> 权限成功后再置 true —— 增一次写，彻底消除中间态，备选）。

### 3.4 搭建器 UI（`agent-builder/[id]/page.tsx`）

- 「可见范围」下拉改 3 档（文案见 3.1）。
- 搭建器需知当前管理员角色（`GET /api/admin/me`，与工作流页同款）：
  - **org_admin** → 第二档显示「本组织可见」，无多选；`scope` 由后端按其组织填，
    前端不收（也不拉 tenants 列表）。
  - **超管 / 系统管理员** → 第二档显示「指定组织可见」，下方**组织多选**（复选列表），
    选中组织码存入 `visibility_config.scope`。
- 组织列表（仅超管/系统管理员需要）：`load()` 并行加载 tenants。**注意**（小B P4）：`/api/admin/tenants`
  默认分页 100，组织数可能超 100 → 用**分页循环拉全量**（参照 R4 `fetchAllAgents`
  写法），不能只取第一页。
- **发布弹窗提示**（小B P5 / D3）：重发布时（`draft.status==="published"`）弹窗里
  明确提示：「发布将按搭建器「可见范围」**覆盖**该智能体在「权限设置」里的现有权限」。
- 校验：选「指定组织」但未勾任何组织 → 发布按钮拦下提示。

## 六、决策点（请产品 / 小B 定）

| 编号 | 问题 | 方案默认 |
| --- | --- | --- |
| D1 | 「本组织 / 指定组织」怎么呈现 | ✅ **按角色**：org_admin → 本组织可见（固定）；超管/系统 → 指定组织（多选） |
| D2 | 要不要加「个人/组织用户」（user_type）档 | ✅ **不加**（保持 3 档简洁，细粒度走权限设置页） |
| D3 | 重发布覆盖手工加的细粒度权限 | ✅ **可接受** + 发布弹窗加覆盖提示 |
| **D4** | **个人用户是否纳入 `resource_permissions` 约束** | ✅ 本功能**不碰** `/api/agents`；个人用户纳入与否拆为产品另议的独立项，不阻塞本功能 |

**D4 详述**（小B P1 引出）：当前个人用户绕过权限表、看所有 enabled agent。

- 选项 a：**纳入** —— 改 `app/api/agents/route.ts` 个人用户分支也按 `resource_permissions`
  过滤。但存量 agent 没有面向个人用户的权限行 → 会从个人用户视图**集体消失** →
  必须给存量 agent 补权限（数据迁移）。补哪种要分清（两者语义不同、别混）：
  - `scope_type=all` —— 个人用户**和**组织用户都可见，影响面最大；
  - `scope_type=user_type, scope_id=personal` —— **只**补个人用户可见，不动组织侧。
  **这是独立的、有回归风险的较大改动。**
- 选项 b：**不纳入**（维持现状）—— 个人用户看所有 enabled agent。则本功能的「可见
  范围」**只对组织用户生效**；「指定组织」「暂不公开」对个人用户不起作用，UI 文案
  需诚实标注。

**建议**：本 5.19up 功能只做"写 `resource_permissions`"（组织用户侧立即正确）；D4
选 a 的话，把「个人用户纳入权限约束 + 存量补权限」**拆成独立项**单独评估，不塞进
本功能。请产品拍 D4。
