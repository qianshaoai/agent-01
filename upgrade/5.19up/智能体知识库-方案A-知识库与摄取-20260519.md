## 一、在整体里的位置

知识库功能拆成两条并行线：

- **方案 A（本文件）· 入库侧** —— 把知识库做出来、能装资料：建库、传文档、文档切块
  向量化入库、知识库管理后台。
- **方案 B · 使用侧** —— 让智能体用上知识库：搭建器绑定、对话检索注入、chat 权限校验。

A 做完的交付物：**管理员能在后台建知识库、把文档喂进去并完成索引**。此时知识库"装好了
资料"，但还没有智能体在用它 —— 那是 B 的事。

---

## 二、本方案范围

**做**（= 母方案 PR-1 + PR-2 + PR-3）：

- 向量基建：迁移（启用 `pgvector` + 4 张表 + `match_kb_chunks` 检索 RPC + HNSW 索引）。
- `lib/kb/` 基础：共享类型、文本切块、embedding 适配器、KB 专用文本提取（不截断）。
- 文档摄取流程：上传 → 提取 → 切块 → 向量化 → 写 `kb_chunks`；文档状态机；重建索引。
- 知识库管理后台 `/admin/knowledge-bases`：建库、传文档、文档列表 / 状态、删除、重建、
  "被哪些智能体引用"反查。

**不做**（属方案 B）：搭建器「知识库」分区、发布同步关联表、chat 权限校验、检索注入。

## 三、并行边界与前置依赖（关键）

A 是 B 的前置基础，但**代码层面两边可同时开工** —— 关键约束如下。

### 3.1 A 给 B 提供的（B 依赖 A）

1. **迁移（PR-A1）** —— 4 张表 + `match_kb_chunks` RPC。**A1 是整个知识库功能的第一块，
   必须最先做、最先 merge、最先在 dev 库跑**；B 的本地联调要等这个迁移跑过。
2. **`lib/kb/embed.ts`（PR-A2）** —— embedding 适配器。B 的 chat 检索要拿它把用户问题
   向量化。
3. **`lib/kb/types.ts`** —— 知识库 / 文档 / 片段 / 检索结果的共享类型。

### 3.2 冻结契约（两边照此并行写代码，不得擅改）

- **4 张表结构** + **`match_kb_chunks(p_kb_ids uuid[], p_query vector, p_top_k int,
  p_threshold float)` 签名** —— 见母方案 §4.1，定稿即冻结。
- **`embed.ts` 导出函数签名** —— A 在 PR-A2 定稿后冻结；B 在此之前照签名写代码、用桩
  集成，A 交付后换真实现。
- `agent_knowledge_bases` 表由 A 的迁移创建，但**只有 B 读写它**（B 发布同步 / chat
  检索）；A 不碰该表。

### 3.3 文件与合并

- **A 与 B 改的代码文件零重叠**（见 §八 与方案B §八）。唯一共享文件是当日变更记录 ——
  各自只追加自己段落。
- **合并顺序**：PR-A1（迁移）最先合、最先在 dev 库跑。其余 A 各 PR 与 B 各 PR 代码
  文件互不重叠，可独立合、无顺序要求。
- 两边都合完后，做一次**联合联调**（建库→传文档→绑智能体→对话检索全链路），约 1 天，
  双方一起。

## 四、设计要点

完整设计见母方案 §4.1 / §4.2 / §4.4，这里只列 A 半要点：

- **数据模型**：`knowledge_bases` / `kb_documents` / `kb_chunks` / `agent_knowledge_bases`
  4 表 + pgvector + HNSW 向量索引 + `match_kb_chunks` RPC。**全站统一一个 embedding
  模型 / 一个向量维度**（`vector(N)` 的 N 由 D1 定、迁移写死）。
- **摄取**：上传 → KB 专用提取（**不按 3 万字符截断**）→ 切块（~500 token / 块 +
  ~80 重叠）→ 批量 embedding → 写 `kb_chunks`；文档状态机 `pending/indexing/done/failed`；
  超 **D9 硬上限**（最大 ~20 万字符 / 最大 500 chunk / 批量 ~64）→ 文档置 `failed` +
  提示拆分，不静默截断。
- **管理后台**：知识库 CRUD、文档上传 / 列表 / 状态 / 删除 / 重建、"被哪些智能体引用"
  反查（查 `agent_knowledge_bases` 关联表）。
