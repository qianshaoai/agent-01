## 一、背景与目标

**背景**：管理员用「智能体搭建器」搭出来的智能体（`agents`，走 openai / 智谱大模型
API），目前只有系统提示词，没有"挂资料"的能力。用户希望给它加知识库 —— 上传文档、
对话时自动检索相关内容、让智能体基于资料回答。

**目标**：在搭建器里给智能体配「知识库」。一套标准 RAG：文档入库 → 切块 → 向量化 →
对话时按问题检索 top-K 片段 → 注入上下文。

## 三、现状盘点

**已有、可直接复用：**

## 四、整体设计

### 4.1 数据模型（新增 4 张表 + 1 个检索 RPC + 启用 pgvector）

⚠️ **本期全站统一一个 embedding 模型、一个向量维度**（小B finding 5）。`vector(N)`
的 N 在迁移里写死，**不支持不同知识库混用不同维度的模型**。`knowledge_bases.embedding_model`
只作记录（标明这批向量由哪个模型生成），不构成"每库可选不同模型"的能力。

```
knowledge_bases   知识库
  id / name / description / created_by_admin
  embedding_model            （记录用：这批向量由哪个模型生成；全站统一，见 D1）
  status / created_at / updated_at

kb_documents      知识库里的文档
  id / kb_id(FK→knowledge_bases, ON DELETE CASCADE)
  filename / file_type / storage_path
  status         pending | indexing | done | failed
  chunk_count / error_msg / created_at

kb_chunks         文档切出来的片段（带向量）
  id / document_id(FK→kb_documents, ON DELETE CASCADE) / kb_id
  chunk_index / content(TEXT) / token_count
  embedding  vector(N)        ← N = 全站统一维度（D1 定，迁移写死）
  created_at
  索引：HNSW 向量索引 ON embedding；普通索引 ON kb_id

agent_knowledge_bases   智能体 ↔ 知识库 绑定（关联表，小B finding 6 新增）
  agent_id(FK→agents, ON DELETE CASCADE)
  kb_id(FK→knowledge_bases, ON DELETE CASCADE)
  PRIMARY KEY (agent_id, kb_id)
```

**检索 RPC**（小B finding 3 新增，PR-1 一并设计）：
```sql
match_kb_chunks(p_kb_ids uuid[], p_query vector, p_top_k int, p_threshold float)
  → 返回 top-K 片段（content / 相似度 / document_id）
```
原因：Supabase / PostgREST 无法直接表达 `ORDER BY embedding <=> $query`，向量距离
排序必须封成 RPC，后端用 `db.rpc("match_kb_chunks", {...})` 调。

⚠️ **D1（embedding 模型）必须在 PR-1 写迁移之前拍板** —— 维度写死在 `vector(N)`，
模型没定迁移就动不了。

**智能体 ↔ 知识库绑定（小B finding 6 修订，不再只靠 jsonb）**：
- **草稿侧**：`agent_drafts.builder_config.knowledge_base_ids: string[]`（jsonb，
  搭建器里勾选的意图）。
- **发布侧**：发布 / 重发布时按 `builder_config.knowledge_base_ids` **全量同步**到
  `agent_knowledge_bases`（与可见范围 / 工作流的"全量替换"同口径）。
- 对话检索、知识库详情页"被哪些智能体引用"、删库时的关联清理 —— **都走
  `agent_knowledge_bases` 关联表**，不靠扫 jsonb。
- 删知识库：关联表 `ON DELETE CASCADE` 自动清绑定行；草稿 `builder_config` 里的
  脏 id 由搭建器 `load()` 按"知识库已不存在"过滤掉。

### 4.2 文档摄取流程（入库）

管理员往某个知识库上传文档时：

1. 文件传 Supabase Storage（复用现有上传，存 `kb/` 目录）。
2. 用 **KB 专用提取入口**取全文（**不按 3 万字符截断**，见第三节 finding 2）。
3. **切块**：按 ~500 token / 块、块间重叠 ~80 token 切分，尽量在段落 / 标题边界断开
   （`lib/kb/chunk.ts`，新建）。
4. **向量化**：对每个块批量调 embedding API（`lib/kb/embed.ts`，新建），拿到向量。
5. 写 `kb_chunks`；文档 `status` 走 `pending → indexing → done`，失败置 `failed`
   + `error_msg`。

⚠️ 摄取是耗时操作（大文档切几百块 + 几百次 embedding 调用）。本期方案：
**同步摄取 + 单文档硬上限（D9）+ 状态字段如实反映进度**；超 D9 上限的文档直接置
`failed` 并提示拆分、不静默截断；`status` 字段已为将来改后台异步任务留好位置（D6）。

### 4.3 对话权限前置校验 + 检索接入（`chat` 路由）

⚠️ **必须先补一个既有安全缺口（小B finding 1）**：`app/api/agents/[id]/chat/route.ts`
现在只按 agent_code 取 agent、判 `enabled`，**不校验当前用户对该 agent 是否可见**
（`resource_permissions`）。挂知识库前，这是"猜到 code 就能聊"；挂知识库后会升级成
"猜到 code 就能问出知识库里的资料"—— info leak。**PR-5 必须先补 chat 前置可见性
校验，再接 RAG。**

`chat` route 新流程：

0. 取到 agent、判 `enabled` 后，**校验当前用户对该 agent 有访问权**（按
   `resource_permissions`，与前台智能体列表 / 工作流同一套可见性判断）。无权 → 403，
   不进入检索 / 对话。
1. 查该 agent 在 `agent_knowledge_bases` 里是否绑定了知识库。
2. 有 → 对**本轮用户问题**调 embedding 得到查询向量。
3. 调 `match_kb_chunks(绑定的 kb_ids, 查询向量, topK, 阈值)` RPC 取 top-K 片段（D4）。
4. 把检索到的片段拼成一段 system / context 消息，注入到发给 `streamChat()` 的
   messages 里（历史消息之前、当前问题之前）。
5. 其余流程（配额、扣费、SSE）不变。

- 无绑定知识库 / 非搭建器智能体 → 不走检索，零影响（但**第 0 步权限校验对所有
  agent 生效**）。
- 检索失败（embedding 调用挂 / RPC 报错）→ **降级**：当作无知识库正常回答，不阻断对话。
- `greeting` 路由开场白一般不需检索，本期不接（D7）。

⚠️ **个人用户口径要与「可见范围方案」D4 联动（小B finding 7）**：第 0 步说"与前台
智能体列表同一套"，但前台列表对**个人用户**目前是绕过权限表、看所有 `enabled` agent
（见可见范围方案 D4 现状）。即不动这条，个人用户仍能访问任意"已启用 + 已绑知识库"的
智能体 —— **知识库若含租户资料，会外溢给个人用户**。
本方案口径：个人用户对知识库型 agent 的可见性**与「可见范围方案」D4 的最终结论保持
一致**；**若知识库可能含租户敏感资料，强烈建议把个人用户也纳入 `resource_permissions`
约束**（对应可见范围 D4 选项 a，属独立项）。本期方案先把缺口点明、随 D4 一起拍。

### 4.4 知识库管理后台（新增 `/admin/knowledge-bases`）

- 知识库列表：增删改、看文档数 / 片段数。
- 进某个知识库：上传文档、文档列表（带 `status`）、删除文档、重建索引。
- 侧边菜单「内容」组加入口（与「智能体搭建」并列）。

### 4.5 搭建器集成（`agent-builder/[id]/page.tsx` + publish 路由）

- 5 分区 → **6 分区**，新增「知识库」分区。
- 该分区：一个多选，列出所有知识库，勾选的写进 `builder_config.knowledge_base_ids`。
- **发布逻辑要改**（小B finding 6）：发布 / 重发布时按 `builder_config.knowledge_base_ids`
  全量同步到 `agent_knowledge_bases` 关联表 —— 与「可见范围发布翻译」同模式。

## 七、决策点（请产品 / 小B 拍板）

| 编号 | 问题 | 方案推荐 |
|---|---|---|
| **D1** | embedding 模型选型（**决定向量维度，必须最先定**） | OpenAI `text-embedding-3-small`（1536 维，便宜、效果好）；若顾虑跨境用智谱 embedding。维度以所选模型文档为准 |
| **D1-2** | embedding 配置形态（小B finding 4 新增） | embedding 的 endpoint / 模型 / 返回结构与 chat completions 不同，**不能简单"复用 model_providers"**。三选一：① 复用某个 openai 大模型 API 的 key、但单独配 embedding endpoint；② API 管理加 `category=embedding`；③ 知识库模块级单独配一处 embedding 配置。**推荐 ③** —— 全站统一一处，最简、与"全站统一维度"一致 |
| **D1-3** | embedding 配置落在哪（小B finding 9 新增 · PR-2 取配置之处） | ③ 选定后须明确落点：① **首期只走环境变量**（`KB_EMBEDDING_ENDPOINT/MODEL/KEY/DIM`，0 UI、最快、key 不入库更安全、与 `ENCRYPTION_KEY` 等密钥同口径，改配置需重部署）；② 存 `settings` 表 + 后台配置页；③ 新表 `kb_embedding_settings` + 配置页。**推荐 ①（首期 env）**，后台可视化配置入口列为后续；选 ②/③ 则 §九 涉及文件须补配置 API + 页面 |
| D2 | 知识库归属 / 共享 + 谁能管理 | **全局共享** —— 任何搭建器智能体能绑任意知识库。但须明确**哪些管理员能创建 / 管理 / 绑定知识库**（建议限超管 / 系统管理员，org_admin 不可），否则全局共享叠加多租户会变成"租户资料串库"。本期不做创建者私有 / 租户隔离 |
| D3 | 一个智能体绑几个知识库 | **支持多个**（`knowledge_base_ids` 数组），检索时跨库取 top-K |
| D4 | 检索参数 | top-K = 5；可选相似度阈值（低于阈值的片段丢弃，防噪声）。先给默认值，后续可调 |
| D5 | 切块大小 | ~500 token / 块 + ~80 token 重叠，段落 / 标题边界优先 |
| D6 | 文档摄取同步还是异步 | **本期同步**（+ 单文档大小限制）；`status` 字段已留好，异步后台任务作为后续可选 |
| D7 | 检索结果给用户看「引用来源」 / greeting 是否接检索 | **本期都不做**，列入打磨版 |
| D8 | embedding 调用 + 检索增大的 prompt 的 **token / 配额口径** | 需明确：① embedding 调用本身的 token 现在不计入 `logs`；② 检索注入会让 prompt token 变大、被 W1 统计到。建议本期 embedding 不计费、只在文档维度记调用量，对话侧照 W1/W2 现状走 |
| **D9** | 单文档摄取硬上限（小B finding 8 新增） | "不截断"≠"无上限" —— 20MB 的 PDF/Excel/PPT 可能切出几千块，撞超时 + embedding 成本失控。须定死：单文档**最大提取文本 ~20 万字符**、**最大 500 chunk / 文档**、embedding **批量 ~64 块 / 请求**；超上限 → 文档 `status=failed` + 提示"文档过大，请拆分后上传"，**不静默截断**。具体数值可调，但必须有 |

> ⚠️ **D1 是开工前置**：`vector(N)` 列维度在 PR-1 迁移里写死，模型没定就动不了 PR-1。

## 八、风险

- **越权访问知识库资料（小B finding 1，安全硬门槛）**：chat route 原本不校验 agent
  可见性 —— 挂知识库后等于"猜到 agent_code 就能问出库里资料"。PR-5 的前置可见性校验
  必过：无权用户 / 越权 code 一律 403，回归必测越权场景。
- **租户资料串库**：全局共享知识库下，若管理权限不收口，A 租户的智能体可能绑上含
  B 租户敏感资料的库。D2 须明确"谁能建 / 绑知识库"的边界。
- **个人用户资料外溢（小B finding 7）**：前台列表对个人用户绕过权限表 —— 知识库型
  agent 若不一并约束个人用户，含租户资料的知识库会外溢给个人用户。口径随「可见范围
  方案」D4，详见 §4.3。
- **跨境 Supabase**：摄取要写大量 `kb_chunks`、检索要做向量查询，跨境抖动会放大延迟 /
  失败。摄取须可重试、可重建索引；检索失败要降级（检索不到就当无知识库正常回答，
  绝不阻断对话）。
- **摄取耗时 / 成本失控**：大文档同步摄取可能撞部署环境请求超时，切出几千块还会让
  embedding 成本失控。靠 **D9 单文档硬上限**（最大字符数 / 最大 chunk 数 / 批量大小）
  兜底；PR-2 还要确认部署环境（Jenkins / pm2 自托管 or Vercel）的超时上限。
- **检索质量**：切块大小、top-K、阈值直接影响"答得准不准"。PR-5 留可调参数，上线后按
  实际效果调，别一次写死。
- **`pgvector` 扩展**：需在 Supabase 后台启用 `vector` 扩展（用户操作）；HNSW 索引
  需 pgvector 0.5+，确认 Supabase 实例版本。
- **token / 配额（D8）**：检索注入让每次对话的 prompt 变大 —— 加权扣额度不受影响
  （按模型不按 token），但 W1 的 token 统计数字会上升，属预期，需在变更记录里说明。
- **对话链路改动**：`chat` route 是最高风险模块，PR-5 必须回归"非知识库智能体、
  扣子 / Dify 智能体"完全不受影响。
