## P0 · standalone 发布静态资源补拷（落地）

### 现象（方案 §二）

`next.config.ts` 用了 `output: "standalone"`；Next.js 16 不会把 `.next/static` 和 `public`
拷进 standalone 产物，直接跑 `node .next/standalone/agent-01/server.js`：

- `/api/health` → 200（误判"服务没问题"）
- `/_next/static/...` → 404
- `/favicon.ico` 等 public 资源 → 404
- 前端 JS / CSS 加载失败、白屏

### 修复

**新建 [`scripts/copy-standalone-assets.mjs`](../../scripts/copy-standalone-assets.mjs)**：

- 在 `.next/standalone/` 下递归查找含 `server.js` 的目录作为"运行时根"
  （单包项目 = standalone 根；monorepo / 有父级 lockfile 时 = 子目录如 `agent-01/`）—— 不
  硬编码 `agent-01`，将来父级 lockfile 删除后无需改脚本。
- 用 Node `fs.cpSync({recursive:true, force:true})` 跨平台拷贝；`public` 不存在时跳过；
  完成后打印运行根 + 启动命令。
- 二次收口：拷贝前先清理旧目标目录，避免旧 chunk 残留；`.next/static` 作为必需资源，
  若缺失则脚本直接失败，阻止坏 standalone 产物进入部署流程。

**`package.json` 加 `build:standalone` 脚本**：

```json
"build:standalone": "next build && node scripts/copy-standalone-assets.mjs"
```

部署链路只需把 `next build` 替换为 `npm run build:standalone`，其余不动。

### 验收（方案 §二.4 标准）

| 检查项 | 结果 |
|---|---|
| `.next/standalone/agent-01/.next/static` 存在 | ✅ 含 `chunks/` `media/` 等 |
| `.next/standalone/agent-01/public` 存在 | ✅ 含 `favicon.ico` 等 |
| `ci:typecheck` / `ci:lint` / `ci:build` | ✅ 通过 |
| `node .next/standalone/agent-01/server.js` + `/_next/static/...` 200 | ✅ 本地 standalone 验证通过 |
| `node .next/standalone/agent-01/server.js` + `/favicon.ico` 200 | ✅ 本地 standalone 验证通过 |

补验命令：

```txt
npm run build:standalone
node .next\standalone\agent-01\server.js
GET /_next/static/chunks/02ad55c0df2a1468.js -> 200
GET /favicon.ico -> 200
```

部署机仍需上线前复验同一组检查，避免 CI 产物与部署打包流程不一致。

## P1 · 知识库 prompt 单点化 + 失败 / 空命中口径收紧

### 起因

5.19up 三轮收口把硬规则 prompt 内联到 `chat/route.ts` 与 `test-chat/route.ts`，两处长串重复、
易漂移；同时存在两个隐患：① 检索"空命中"时不注入任何规则、② 检索"失败"时只 `console.warn`
后降级 —— 两种情况下，绑了知识库的智能体仍会用常识答题，**用户无法区分这是"知识库里的答案"
还是"模型自己的常识"**，与 5.19up 验收锁的「以知识库为唯一事实依据」口径相违。

### 改动

**新建 [`lib/kb/prompt.ts`](../../lib/kb/prompt.ts)** —— 知识库注入提示词的单点出口，导出 2 个函数：

- `buildKbStrictAnswerPrompt(chunks)` —— 正常检索路径
  - **chunks 非空**：原 5.19up 三轮的硬规则文案（不可用常识反驳资料、禁词列表等），并在
    ③ ④ 处加严：「资料未覆盖时，必须直接答『知识库中没有找到相关资料，无法回答该问题。』，
    不要另起段补充常识」。
  - **chunks 为空（新增分支）**：返回新的硬规则要求模型直接答固定句「知识库中没有找到相关
    资料，无法回答该问题。」，禁止任何常识补答。
- `buildKbUnavailablePrompt()` —— 检索异常降级路径（新增），要求模型直接答固定句「知识库
  暂时不可用，无法基于知识库回答该问题。」，禁止常识补答。

**`app/api/agents/[id]/chat/route.ts`** ⚠ 最高风险文件：
- 内联硬规则字符串 → `kbInjectText = buildKbStrictAnswerPrompt(chunks)`。
- catch 分支由「只 `console.warn`，不注入」→ 仍 `console.warn` 留可观测性 + 注入
  `buildKbUnavailablePrompt()`，让模型显式告知"知识库不可用"，**不再用常识冒充知识库答案**。
- 其余流程不变；非知识库智能体（无 `kbIds`）零影响。

**`app/api/admin/agent-drafts/[id]/test-chat/route.ts`**：同 chat route 同步同一套（保持
chat / test-chat 同口径，5.19up 已确立的原则）。

### 语义变化（重要 · 写给小B 复审）

| 场景 | 5.19up 行为 | 5.20up 行为 |
|---|---|---|
| 知识库 chunks 命中 ≥ 1 | 注入硬规则 + 资料块 | 同（文案在 ③ ④ 处略严，禁止补充常识段） |
| 知识库 chunks 命中 = 0 | **不注入**，模型按"无知识库"用常识答 | **注入空命中规则**，模型答"没找到相关资料" |
| 知识库检索抛错（embedding / RPC） | `console.warn` + **不注入**，模型按"无知识库"用常识答 | `console.warn` + **注入不可用规则**，模型答"知识库暂时不可用" |

**带来的副作用**：
- 知识库智能体绑了库以后，命中差或检索失败时不再"伪装成普通智能体"用常识答，会明确说"没找到 / 不可用"。
  这是有意为之 —— 否则用户分不清答案来自哪里，违反验收期口径。
- 不影响 **未绑知识库** 的智能体（`kbIds.length === 0` 时 `kbInjectText` 仍为空，行为完全不变）。
- 不影响 **非 openai / 智谱平台**（条件检索本就跳过这些平台）。

### 验证

- `ci:typecheck` / `ci:lint` / `ci:build` 通过。
- 端到端复测项（待人工跑）：
  - 绑库 + 命中：按 KB 严格答（保持 5.19up 验收行为）。
  - 绑库 + 空命中：答"知识库中没有找到相关资料，无法回答该问题。"（新增行为，须人工验证）。
  - 绑库 + 检索失败（断 embedding provider 或 v38/v39 未跑）：答"知识库暂时不可用，无法基于
    知识库回答该问题。"（新增行为，须人工验证）。
  - 未绑库 / 扣子 / Dify / 工作流跑批：行为不变。

## P1 · AdminLayout 加 fullBleed / hideFooter prop（配套知识库接回布局）

知识库列表 / 详情页本次接回 `AdminLayout` 统一导航，但页面本身是沉浸式宽屏 + 自带页脚，
直接套默认 `AdminLayout` 会出现「右侧内容被 `max-w-[1600px]` 居中 + 底部多一条白色 footer」
的体感倒退。

**`components/layout/admin-layout.tsx`**：`AdminLayout` 新增两个可选 prop：

- `fullBleed?: boolean`（默认 `false`）—— 跳过默认的 `p-5 sm:p-7 max-w-[1600px] mx-auto`，
  让页面自己铺满右侧内容区。
- `hideFooter?: boolean`（默认 `false`）—— 隐藏默认 footer，避免沉浸式页面底部露出白框。

知识库列表 / 详情页传 `fullBleed hideFooter` 后维持原全宽体验；其它管理页未传 prop，默认值
等同于原行为，**零影响**。
