## 三、P0：standalone 发布静态资源缺失

### 现象

项目 `next.config.ts` 配置了：

```ts
output: "standalone"
```

回归时使用 `.next/standalone/agent-01/server.js` 启动服务后：

- `/api/health` 返回 200；
- `/_next/static/...` 返回 404；
- `.next/standalone/agent-01/.next/static` 不存在；
- `.next/standalone/agent-01/public` 不存在。

这说明 API 正常，但前端静态资源没有进入 standalone 运行目录。

### 风险

如果线上直接运行 standalone 目录，但部署流程没有额外拷贝静态资源：

- 页面 JS / CSS 加载失败；
- 用户看到白屏或页面样式丢失；
- `/api/health` 仍可能是 200，容易误判为“服务正常”。

### 已落地修复

小A已补：

- 新增 `scripts/copy-standalone-assets.mjs`。
- `package.json` 新增：

```json
"build:standalone": "next build && node scripts/copy-standalone-assets.mjs"
```

脚本职责：

- 在 `.next/standalone/` 下递归查找包含 `server.js` 的目录，作为 standalone 运行时根目录。
- 先清理旧目标目录，再将 `.next/static` 拷贝到 `<serverRoot>/.next/static`，避免旧 chunk 残留。
- 将 `.next/static` 视为必需资源；若源目录不存在，脚本直接失败，阻止坏产物上线。
- 先清理旧目标目录，再将 `public` 拷贝到 `<serverRoot>/public`；若项目没有 `public`，允许跳过。
- 兼容当前 `agent-01` 子目录形态，也兼容未来 root 推断变化后 standalone 根目录变化。

### 本地验证结果

已执行：

```powershell
npm run build:standalone
```

结果：

```txt
.next/static -> .next\standalone\agent-01\.next\static
public       -> .next\standalone\agent-01\public
```

随后用 standalone server 实测：

```txt
GET /_next/static/chunks/02ad55c0df2a1468.js -> 200
GET /favicon.ico                             -> 200
```

说明 P0 的本地发布态风险已闭环。

### 部署机仍需复验

执行：

```powershell
npm run build:standalone
```

检查路径：

```txt
.next/standalone/agent-01/.next/static
.next/standalone/agent-01/public
```

启动：

```powershell
$env:PORT="3108"
node .next\standalone\agent-01\server.js
```

验证：

```txt
GET /api/health               -> 200
GET /_next/static/.../*.js    -> 200
GET /favicon.ico              -> 200 或 304
GET /admin/agent-builder      -> 页面资源正常加载
```

### 验收标准

- 部署流程从 `npm run build` 改为 `npm run build:standalone`。
- standalone 运行目录包含 `.next/static` 和 `public`。
- 本地或部署机运行 standalone server 后，前端页面不白屏。
- 部署机复验 `/_next/static/...` 和 `/favicon.ico` 均返回 200 或 304。

## 四、P1：真实 chat 端到端回答需人工确认

### 当前已验证

本轮已验证 DB/RAG 读链路：

- `knowledge_bases` 有 active KB。
- `kb_documents` 有 done 文档。
- `agent_knowledge_bases` 有 agent 与 KB 的绑定。
- `kb_chunks` 有向量切片。
- `match_kb_chunks` RPC 可命中样例 chunk，similarity=1。

### 未自动验证原因

没有直接自动调用 `/api/agents/[id]/chat`，因为该接口会：

- 写入 conversation / messages / logs；
- 调真实模型；
- 消耗模型额度。

### 风险

RAG 读链路正常，不等于最终模型回答一定严格按知识库输出。

尤其 5.19up 验收已确认：

- `glm-4-flash` 会用世界常识反驳知识库，不能作为知识库智能体基准模型。
- `glm-4-air` 能严格按知识库回答，是当前最低可用模型。

### 人工验收步骤

使用已绑定测试知识库的智能体，模型选择 `glm-4-air`。

测试 1：

```txt
wsl 的代号是什么？
```

期望回答包含：

```txt
plamguns
```

测试 2：

```txt
西红柿会怎么样？
```

期望回答遵循知识库内容，不使用常识反驳。

测试 3：

```txt
葡萄是什么味道？
```

期望回答遵循知识库内容，不擅自纠正为常识。

### 验收标准

- `glm-4-air` 下，已绑定 KB 的智能体能稳定引用 KB 内事实。
- 不把 `glm-4-flash` 作为知识库智能体默认/推荐模型。
- 若 UI 仍允许选择 `glm-4-flash`，需要在上线清单里明确“不适合知识库强事实问答”。

### 5.20up 代码收口

- 搭建器智谱预设模型将 `glm-4-air` 标为“知识库推荐”，不再把 `glm-4-flash` 标为推荐。
- 草稿绑定了知识库且当前有效模型为 `glm-4-flash*` 时，前端发布按钮会提示并阻止发布弹窗。
- 发布接口增加后端兜底：绑定知识库的智谱智能体不能用 `glm-4-flash*` 发布，避免绕过前端或供应商默认模型仍是 flash。
- 组织管理员无知识库可选时，搭建器提示“知识库由平台管理员统一维护”，不再引导其去没有权限的知识库管理页。

## 五、P1：知识库禁用与发布绑定边界

### 当前已验证

代码与 SQL 侧已有双保险：

- `migration_v39_kb_chunks_active_filter.sql` 已将 `match_kb_chunks` 增加 `knowledge_bases.status='active'` 过滤。
- 发布时如果草稿绑定 disabled KB，会返回校验错误，避免“发布成功但检索时静默不命中”的体验不一致。

### 风险

如果目标 Supabase 没跑 v39：

- disabled KB 仍可能被 RPC 命中；
- 后台“停用知识库”按钮会形同虚设。

### 验收 SQL

在 Supabase SQL Editor 执行：

```sql
SELECT pg_get_functiondef(oid) AS def
FROM pg_proc
WHERE proname = 'match_kb_chunks';
```

验收标准：

```txt
返回定义中包含：kb.status = 'active'
```

端到端验收：

```txt
停用一个已绑定 KB -> 该 agent 对话不再命中该库片段
重新启用该 KB -> 该 agent 对话恢复可命中
```

## 六、P1：网络代理误伤外部服务

### 现象

回归期间出现过：

- Supabase 偶发 `fetch failed` / TLS handshake 失败；
- `open.bigmodel.cn` PowerShell 请求失败；
- 本机代理 fake-ip 解析到 `198.18.0.x`。

### 风险

以下链路可能被误判为代码问题：

- 知识库上传；
- 文档向量化；
- 测试聊天；
- 模型供应商连通性测试；
- git 推送。

### 处理口径

5.20up 不强制改代码处理代理问题，但上线前必须确认部署机网络。

建议代理直连规则：

```txt
DOMAIN-SUFFIX,supabase.co,DIRECT
DOMAIN-SUFFIX,bigmodel.cn,DIRECT
DOMAIN-SUFFIX,github.com,DIRECT
```

### 验收标准

- 部署机能稳定访问 Supabase REST / Storage。
- 部署机能稳定访问 `open.bigmodel.cn`。
- 上传知识库文档后，状态能从 `pending` 进入 `done`。
- 测试聊天不会出现稳定复现的 `fetch failed`。

## 八、P2：Next 构建警告收口

### 警告 1：多 lockfile 导致 root 推断

现象：

```txt
Next.js inferred your workspace root...
Detected additional lockfiles:
G:\zhinengticang\package-lock.json

```

建议：

- 若仓库根目录的 `package-lock.json` 无实际用途，删除或统一依赖入口。

```txt
The "middleware" file convention is deprecated. Please use "proxy" instead.
```

建议：

- 后续单独做一次 `middleware.ts` 到 `proxy.ts` 的迁移。
- 迁移后重点回归登录、鉴权、后台路由保护。
