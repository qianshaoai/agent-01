### ✅ 3. 系统角色越权 bug 修复（已完成 2026-04-10）
- **现状**
  - [app/api/admin/users/[id]/route.ts:27-36](app/api/admin/users/[id]/route.ts#L27) 的 `set-role` 处理器**只验证了登录态**，不校验当前管理员自己的等级
  - 任何通过 admin 登录的人都能把任意用户改成任意角色（包括把普通用户提升为超管）
- **问题**
  - 严重越权漏洞，组织管理员理论上可以把自己或别人改成超管
- **改造**
  - JWT `AdminPayload` 需增加 `role` 字段（目前只有 `adminId` + `username`）
  - `set-role` 接口加角色层级校验：
    - 超级管理员：可改任何人为任何角色
    - 系统管理员：可改普通用户/组织管理员，**不可**改超管/系管
    - 组织管理员：只能在自己组织内操作，**不可**改系统级角色
    - 普通用户：完全无权进入后台
  - 角色层级定义为常量表（`ROLE_RANK`）方便复用
- **涉及**：`lib/auth.ts`、users API `set-role`、admin 登录接口

### ✅ 6. 用户可见范围查询接口（已完成 2026-04-10）
- **现状**
  - 代码中**完全没有**"这个用户能看到什么"的计算逻辑
  - `resource_permissions` 表只支持正向查询（资源 → 允许谁），没有反向（用户 → 能看到哪些资源）
- **问题**
  - 详情抽屉无法展示"可见工作流/可见智能体"
- **改造**
  - 新建 `lib/permissions.ts`，封装 `getVisibleResourcesForUser(userId)` 工具函数
  - 计算逻辑：遍历 `resource_permissions`，按以下规则聚合该用户可见资源：
    - `scope_type='all'` → 全部可见
    - `scope_type='user_type'` → 匹配 user_type
    - `scope_type='org'` → 匹配 tenant_code
    - `scope_type='dept'` / `team'` → 匹配 dept_id / team_id
    - `scope_type='group'` → 检查 user_group_members
    - `scope_type='user'` → 精确匹配 user_id
  - 新建 `GET /api/admin/users/[id]/visibility` 接口返回 `{ workflows, agents }`
- **涉及**：`lib/permissions.ts`（新建）、新 API 路由

## 关键风险点

1. **需求 4（RBAC）是本轮工作量最大的一项**，牵涉所有 admin 接口的数据范围控制，改动面广。建议先把 `super_admin / system_admin` 两级做出来验证通路，再铺 `org_admin`。
2. **需求 10（智能体分类多选）需要数据迁移**，线上已有数据要通过迁移脚本平滑过渡，不能丢失。
3. **需求 3 的越权 bug 应当作为紧急修复立即处理**（即使只是在现有 `set-role` 接口加一行层级检查），不要等到阶段 2。
4. 需求 5 的"详情抽屉"是独立共享组件（drawer），做出来后可以复用到组织、智能体等其它管理页，投资回报率高。
5. 分类图标上传要复用现有 logo 上传的 Supabase Storage 模式（`/api/admin/settings/logo` 那套），不要新造一套。
